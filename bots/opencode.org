Tips and conventions learned from past sessions.  Loaded each session
to avoid re-discovering the same things.

** General til knowledge
- ~mode test~ is required for test files (not ~mode gui~).
- ~mode test~ provides ~assert(__LOC__, cond)~ and ~assert_eq(__LOC__, a, b)~.
- Float equality goes through ~a.eq(b)~ on F32; there is no ~assert_eq_f32~.
- Forward references work: functions can reference globals declared later.
- ~func()~ with no explicit return type works; the compiler infers it.
- ~proc~ is for side-effect-only procedures; ~func~ is for value-returning functions.
  Default to ~func~ unless the compiler tells you otherwise.
- ~format(...)~ takes ~..Str~ variadic but the til compiler may not expand
  variadics inside ~proc~ bodies correctly.  Use ~print~/~println~ chains
  instead of ~format~ when in a ~proc~, or test it first.

** til operators and desugaring
- ~x / y~ desugars to ~x.div(y)~ via UFCS.  F32 division works:
  ~1.to_f32() / 2~ == 0.5.
- ~1 / 2~ with both I64 stays I64 (integer division -> 0).
- Unary ~-~ desugars to ~.neg()~.  F32 has no ~.neg~ method, so ~-floor_half~
  does not work.  Use ~0 - x~ instead.
- ~<~, ~>~, ~<=~, ~>=~ desugar to ~.lt()~, ~.gt()~, etc.
- ~x | y~ is bitwise OR (for U32 flags like ~FLAG_WINDOW_RESIZABLE | FLAG_WINDOW_MAXIMIZED~).

** til float literals
- There are no float literals like ~0.5~.  Use ~1.to_f32() / 2.to_f32()~ instead.
- ~0.5.to_f32()~ panics at build time (~Str.to_i64: invalid char in '0.5'~).

** Struct fields and mutation
- Struct fields CAN be mutated: ~camera.position.x = new_x~ works.
- ~Array.new("Type", Type.size(), count)~ creates an Array.
- ~Array.set(idx, own val)~ — note the ~own~ keyword (required for owned params).
- Arrays do NOT have a ~.push()~ method.
- Trailing commas are allowed and encouraged on multi-line constructors.

** GUI / Raylib
- ~UpdateCameraPro~ hard-codes ~moveInWorldPlane=true~.  For 3D flight,
  use ~CameraMoveForward~ with ~moveInWorldPlane=false~.
- ~GetMouseDelta()~ accumulates movement from before window creation.
  Call it once before the game loop (after ~DisableCursor~) to drain the
  initial spike, otherwise frame 1 sees a huge delta and the camera
  jumps.
- ~CheckCollisionBoxSphere~ is the standard AABB-vs-sphere test.
- ~xvfb-run --auto-servernum timeout N bin/prog~ runs GUI programs
  headlessly for testing.  Use ~stdbuf -oL~ to force line-buffered
  stdout if output is missing.

** ASAN leak baseline and delta tracking
- Every merge-commit procedure includes ASAN leak checks. The point is
  to report the *delta* from the known baseline, not whether leaks exist.
- Current baseline (commit 9fef36d78, 2026-05-04):
  - interpret: 254,233 bytes / 48,601 allocs
  - translate: 145,706 bytes / 25,250 allocs
- Report delta as: "interpret +N% / -N%, translate +N% / -N%". A 0%
  delta means no change from baseline. A positive delta means the commit
  increased leaks; the user decides whether to keep or reject.

** Build & debug workflow
- Use ~tmp/~ for ad-hoc test files.
- Run with ~bin/til run tmp/test.til > tmp/test.log 2>&1~.
- Look at generated C in ~gen/til/<name>.c~ to understand desugaring.
- ~make > /dev/null 2>&1~ to build ~bin/til~ (needed after translation).
- ~bin/til build~ does translate + C compile; ~bin/til translate~ does
  just the til->C step (faster iteration).
- ~systemd-run --user --scope -p MemoryMax=64G make clean && make test~
  prevents OOM from memory leaks during full test suite.

** Conventions
- UPPERCASE for module-level constants (~INIT_POS~, ~KEY_W~, ~MENU_*~).
- snake_case for functions/variables.
- Trailing commas on multi-line constructors/arrays.
- Module-level ~mut~ globals for shared mutable state (e.g. ~mut camera := ...~).

** REPL usage and help()
- Pipe REPL input via a tmp file: ~printf 'cmd1\ncmd2\nexit(0)\n' > tmp/repl_input && cat tmp/repl_input | bin/til~
  Always include ~exit(0)~ as the last line to terminate cleanly.
  The REPL exits on EOF; ~exit(0)~ runs in the subprocess and just makes
  the turn complete before the pipe closes.
- ~help("name")~ prints /// doc comment + kind/signature/info. Works
  for top-level decls, struct/enum members (qualified: ~"Vec.push"~),
  and user-defined variables (shows inferred type like ~x is a I64~).
  Undefined symbols print ~undefined symbol: <name>~.
- REPL turns are slow on the first turn (~80s on this host, core blob
  cache-fill) but subsequent turns are fast because the core packed
  info/docs are cached after the first successful typecheck.
- ~printf '...' > tmp/repl_input && cat tmp/repl_input | bin/til~
  is the correct pattern, NOT ~printf '...' | bin/til~ (EOF timing).
- Non-REPL pipelines (~til interpret/build/run/test/translate~) do NOT
  populate the doc registry; ~help()~ only works in the REPL.

** Doc vs docgen changes
- *Doc change*: modifications to documentation-only files (e.g.
   ~issues/open/189.org~, ~bots/*.org~, ~README~). These do NOT affect the
  binary or boot files. Commit with ~Doc: ~ prefix, skip ~make test~.
- *Docgen change*: adding or modifying ~///~ doc comments in core
  library files (~src/core/vec.til~, etc.). These ARE embedded into
  ~boot/til.c~ during bootstrap because the lexer copies them into
  the AST and the boot snapshot includes the full AST text. So even
  a one-line ~///~ comment forces a full ~make clean && make test~
  cycle (~13s wall, ~300s CPU). This is tracked as issue #183.
- *Source change*: any code change to ~.til~ files. Always requires
  ~make clean && make test~ + ASAN before committing.
- File change to ~test/repl_help.til~ or ~src/tests.til~ that only
  adds/removes a test registration line: test-only, still requires
  make test but may skip ASAN if no core code changed.

** git
- ~git reset --soft HEAD~1~ keeps changes in worktree; followed by
  ~git checkout HEAD -- doc/totals.csv img/totals.svg~ to discard
  auto-generated totals.
- Never commit without being asked.
- Doc-only commits: prefix with ~Doc: ~.
- ~git reflog~ recovers accidentally dropped commits.
