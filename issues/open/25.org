** WIP #25 Support for the 4 main operative systems

:DEPENDENCIES: [[../fixed/12.org][#12]] (self-hosting, FIXED)
:RELATED: [[./151.org][#151]] (TempleOS), [[./152.org][#152]] (SDK-less macOS cross), [[./232.org][#232]] (hosted 32-bit)

  til should build and run on the 4 main operative systems, and not
  just natively: the goal is full cross-compilation -- build a til
  binary for any target from any host. Hosts are GNU/Linux, macOS and
  Windows; targets add TempleOS. The matrix is 3x4 + 1x1 = 13
  combinations (TempleOS as a host only targets itself; replacing
  gcc/clang on TempleOS is out of scope).

  The original priority order was Linux, TempleOS, macOS, Windows.
  Reality inverted it: mingw-w64 made Windows cheap, so it landed
  second. macOS is now the active front and the greatest remaining
  challenge; TempleOS goes last ([[./151.org][#151]]).

  Status (2026-07):

  - GNU/Linux: DONE. Native host plus 6 Linux targets (x64, arm64,
    riscv64, x86, arm32, riscv32). CI cross-compiles all six and runs
    x86/arm32/riscv32 under qemu. The --cc override supports odd
    toolchains (the riscv32 job uses a Bootlin gcc).
  - Windows as a target: DONE. mingw-w64 cross from Linux; vendored
    libraylib-win64.a / libtinyfd-win64.a; mode gui links
    gdi32/opengl32/winmm/...; [[file:../../src/c/ext.c][src/c/ext.c]] has full _WIN32 coverage
    (CreateProcess, LoadLibrary/GetProcAddress FFI loading, psapi
    RSS). nng cross-builds for win64 too (cmake + mingw, Makefile
    NNG_WIN_LIB), so the nng mode-server examples (sender/receiver/
    publisher/subscriber) build for windows. nng requires the UCRT, so
    the win64 build forces -D_UCRT and the builder appends -lucrtbase +
    the winsock libs (ws2_32/mswsock/advapi32) to the program link. CI
    cross-builds hello_script.exe and runs it on a real windows-latest
    runner. make build_win sweeps every example (37 .exe): the portable
    ones via the bin/%.exe pattern rule, the nng mode-server ones via
    win_server (--extra-modes + the win64 nng); only the direct-FFI
    raylib.til demo and the standalone-unbuildable custom_modes.til /
    bench_hashmap.til are excluded. 32-bit windows-x86 is tracked in
    [[./232.org][#232]].
  - wasm32 (bonus target beyond the 4): DONE via emscripten
    (make build_wasm, single-file .html output; was #228).
  - Windows as a host (bin/til.exe running ON Windows): DONE. The
    compiler cross-builds for windows and runs there -- til.exe
    interprets and translates natively. The CI windows-x64-host-run job
    downloads the cross-built til.exe to a real windows-latest runner
    and runs both (also verified locally under wine). What it took (all
    issue-#25 FFI-portability work): (1) libffi 3.7.1 cross-built for
    win64 -- Makefile LIBFFI_WIN_LIB target + target_ffi_lib returning
    its stable mingw-triple path, statically linked into til.exe so
    ext_func dispatch works; (2) -Wl,--export-all-symbols on the windows
    link (mingw's -rdynamic): without it the PE has no export table, so
    the interpreter's ffi_global_symbol (GetProcAddress) resolves
    nothing and ffi_prep_cif faults on a NULL ffi_type;
    (3) detect_current_target reads host_os()/host_arch() (compile-time)
    instead of spawning uname, which does not exist on windows --
    til.exe was defaulting to a linux target and emitting linux link
    flags; (4) toolchain_command picks plain gcc on a windows host vs
    the x86_64-w64-mingw32- cross prefix elsewhere. `make build_win_host`
    builds bin/til.exe. Interpret and translate need no C toolchain; the
    CI windows-x64-host-run job now also installs mingw gcc and runs
    `til.exe build` (compile + link) natively, producing and running a
    real .exe -- the full-compiler-on-windows proof (get_bin_dir walks
    up from the download dir to the checkout root, so src/c/ext.c and
    boot/ resolve; hello_script uses no runtime ext_func so no libffi is
    needed). Separately, the same CI job runs the
    cross-built nng sender.exe on the windows runner -- proving nng
    works on real winsock, which wine cannot (its winsock is
    incomplete: nng loads and the socket opens under wine, but listen
    returns "Not supported"). The libffi-win64 also unlocks runtime
    ext_func in cross-compiled user programs.
  - macOS: DONE on both arches (arm64 + x64). CLI (cli/script/pure/lib)
    cross-compiles from Linux via the SDK-less vendored toolchain
    (vendor/macos: zig darwin headers + libSystem.tbd) AND builds
    natively; the mac native ASAN suites run green on the CI runners
    (macos-15-intel + macos-latest). GUI (the 15 mode-gui examples) and
    the nng custom modes (sender/receiver/publisher/subscriber) build
    NATIVELY on a mac -- gui links the real Apple frameworks, the nng
    modes link the natively-built libnng; make build_mac covers all of
    them and both mac CI jobs run it. GUI CROSS from Linux is NOT yet
    wired but IS feasible (green-lit phase 3, see the plan below) -- the
    Cocoa/OpenGL frameworks are the same kind of always-present system
    lib as libSystem, so the vendored-.tbd-stub trick that makes the CLI
    cross work extends to them; only a mac-built libraylib.a + trimmed
    framework stubs need producing, both vendorable. (An earlier note
    here wrongly called gui cross impossible; corrected.) Until phase 3
    lands, the linux make build_mac skips the 15 gui examples and native
    mac gui is the supported path.
  - macOS as a HOST targeting the others: the macos-host-cross-build CI
    job builds bin/til natively on a mac, then cross-compiles
    hello_script for windows (brew mingw-w64 -> the x86_64-w64-mingw32
    cross-gcc toolchain_command already selects) and for linux (a
    prebuilt cross-gcc from the messense tap, via --cc); the artifacts
    run on real windows / linux runners. No cross libffi is needed --
    compiled programs bind ext_func to direct C symbols and never
    reference it. This closes the last two matrix cells (the CLI path;
    gui cross from a mac is the same Apple-SDK block as elsewhere).
  - TempleOS: deferred, split out to [[./151.org][#151]]. #12 is fixed so the hard
    dependency is satisfied; what remains is the HolyC backend itself
    (Lang.HolyC and the holyc toolchain entry are placeholders today).

     mirror (vendor/bindings/libffi.til) was 32 bytes, but the C
     ffi_cif ends with FFI_EXTRA_CIF_FIELDS -- `unsigned
     aarch64_nfixedargs` on darwin-arm64 (also `unsigned is_variadic`

     WRITES it (vendor/libffi/src/aarch64/ffi.c:617). So every

*** macOS plan

   Why it is the hard one: unlike Windows there is no mingw equivalent
   for Mach-O. Apple's SDK (libc headers, libSystem .tbd stubs, Cocoa/
   OpenGL frameworks) is not redistributable, so the windows-x64
   recipe -- vendor a cross compiler plus static libs, link on Linux --
   does not transfer. Split the work so the SDK problem only blocks
   the parts that genuinely need the SDK:

**** Phase 1: native macOS host (arm64 first, x64 second)

     Build and test til ON a Mac using GitHub's macOS runners
     (macos-latest is arm64 today; macos-15-intel is x64). No SDK
     problem there: the runner ships Xcode clang. Work items:

     - Makefile portability: -lrt (in RAYLIB_FLAGS, linked into
       til_boot too) and -latomic (NNG_FLAGS) do not exist on macOS;
       gate on uname -s. xvfb-run is Linux-only: mac runners are
       headless, so run the suite without the wrapper and without gui
       tests at first (bin/tests --no-gui -- same reason Linux CI
       needs xvfb at all). -rdynamic/-ldl reportedly work with Apple
       clang; verify.
     - The til_boot cc line uses only -Wall -Wextra -Werror; Apple
       clang needs the -Wno-* extras toolchain_extra_args already adds
       for clang builds (sometimes-uninitialized, self-assign,
       unused-function) -- or the warnings fixed for real.

**** Phase 2: SDK-less cross-compilation Linux -> macOS (CLI/server)

     No zig dependency -- replicate the mechanism zig uses internally.
     What zig actually does (zig cc is clang underneath):

     - vendors the darwin libc headers in its own tree
       (lib/libc/include/any-darwin-any/ in ziglang/zig, harvested
       from Apple SDKs by the now-archived
       ziglang/fetch-them-macos-headers tool);
     - vendors lib/libc/darwin/libSystem.tbd, a YAML text stub listing
       libSystem's exported symbols: the linker resolves -lSystem
       against the stub and the real libSystem.dylib binds at runtime
       (stable ABI -- same idea as mingw import libs);
     - links with its own Mach-O linker and ad-hoc code-signs arm64
       output (mandatory on Apple Silicon);
     - compiles compiler-rt builtins from source for the target;
     - bundles no frameworks: gui cross is SDK-bound even for zig.

     Every piece has a freely available non-zig equivalent: clang is
     already a Mach-O cross-compiler on any host, ld64.lld
     (clang -fuse-ld=lld) links Mach-O and ad-hoc-signs arm64, and
     libSystem.tbd is one vendorable text file. The vendored header
     tree is the heavy part -- and precompiled per-target ext objects
     ([[./152.org][#152]], macOS-only; linux/windows targets keep the current
     compile-ext.c-with-the-cross-gcc flow that already works) make it
     mostly unnecessary:

     - Precompile ext-macos-arm64.o / ext-macos-x64.o (plus
       libffi-macos-*.a) on the Phase 1 mac CI job, where the real SDK
       lives, and commit them like libraylib-win64.a. That absorbs
       every system-header need in ext.c (dlfcn, spawn, sysconf, ...).
     - The generated C is nearly freestanding already: gen/til/*.c
       includes only stdio/stdlib/string/stdarg/limits plus ext.h, and
       ext.h includes only stdbool/stdint (clang provides those
       itself, stdarg too). Finish the job -- #152 step 1: route the
       remaining direct libc uses (printf family, fprintf(stderr,...),
       malloc/free/realloc, mem*/str*) through ext.h prototypes or
       til_* wrappers so the generated C compiles with clang's builtin
       headers alone, no Apple headers on the host. One darwin quirk
       forces part of this anyway: Apple's stdio.h does
       #define stderr __stderrp, so raw fprintf(stderr, ...) in
       generated C cannot be hand-prototyped portably -- stderr users
       must go through ext.o.
     - Vendor a libSystem.tbd (checked against zig's copy) and link:
       clang -target arm64-apple-macos11 -fuse-ld=lld -lSystem plus
       the precompiled objects. No crt1.o exists on modern macOS
       (LC_MAIN entry point), so no CRT objects to obtain. If the
       generated C ends up referencing compiler-rt builtins, build the
       few needed ones from llvm compiler-rt (permissive license) --
       Linux clang packages do not ship libclang_rt.osx.a.
     - Host toolchain deps: distro clang + lld, nothing else. Plan B
       if shrinking the generated C's libc surface drags on: vendor
       the darwin libc headers the way zig does (do the license
       diligence at vendoring time; the tree already vendors X11
       headers, so there is precedent).
     - osxcross (user-extracted SDK) stays a documented --cc
       alternative for whoever wants Apple's real toolchain; never a
       dependency.
     - CI: a linux job cross-builds examples/hello_script.til for
       macos-arm64 and macos-x64 (file output grepped for Mach-O),
       uploads artifacts; a macos job downloads and runs them -- the
       exact windows-x64-build / windows-x64-run pattern.

**** Phase 3: the rest of the mac matrix

     - mode gui cross from Linux (GREEN-LIT -- correcting the earlier
       "SDK-bound, impossible" note, which was wrong): DOABLE with the
       same tbd-stub trick already used for libSystem. A .tbd is just a
       text symbol-list stub (Apple's own SDK ships frameworks as .tbd
       stubs; `tapi` makes them), and the real Cocoa/OpenGL/... dylibs
       live on EVERY mac -- they bind at runtime exactly like libSystem
       (vendor/macos/libSystem.tbd). So no user SDK is needed to LINK;
       only to produce two build-once artifacts, both of which fit the
       vendor model (committed like libraylib-win64.a + libSystem.tbd):
       1. libraylib-macos-arm64.a / -x64.a -- raylib built for macOS.
          The ONLY step that genuinely needs a real SDK: raylib's mac
          backend is Objective-C that #includes Cocoa/OpenGL headers.
          So build it on the mac CI runner (Apple SDK present) and
          commit the archives (raylib is zlib-licensed, freely
          redistributable, exactly like the committed libraylib-win64.a).
          Regenerate on a raylib upgrade.
       2. Minimal framework .tbd stubs for exactly the frameworks raylib
          pulls (Cocoa/OpenGL/IOKit/CoreVideo/CoreAudio). Derive them
          from the mac raylib.a's own undefined symbols (llvm-nm -u,
          which runs on linux against the committed Mach-O archive),
          mapped to their owning framework; vendor under
          vendor/macos/frameworks/ next to libSystem.tbd. Same category
          of artifact as the vendored libSystem.tbd -- do the per-file
          license diligence the vendor/macos README already flags.
       Then extend the builder: for a mac gui target add the mac
       raylib.a + `-framework OpenGL -framework Cocoa -framework IOKit
       -framework CoreAudio -framework CoreVideo` + the stub search path
       (-F vendor/macos/frameworks), and drop mode gui from the linux
       MAC_CROSS_EXAMPLES_SRC exclusion so `make build_mac` sweeps the
       15 gui examples on linux too -- the mac analogue of build_win
       sweeping windows gui via the redistributable mingw import libs.
       CI: a linux job cross-builds the gui examples (file -> Mach-O),
       the mac runner launches them (headless caveat, same as native
       gui). osxcross with a user SDK stays a documented fallback, never
       a dependency. Mechanism web-verified: a .tbd stub plus the
       runtime-present framework links + runs with no full SDK
       (developer.apple.com stub-library docs; wailsapp/wails #4267).
     - macOS as a host targeting the others: windows-x64 via
       brew mingw-w64; linux-* needs a Linux cross toolchain on the
       mac (Apple clang cannot link glibc ELF without a sysroot --
       homebrew musl cross-gcc for static musl binaries is the simple
       first step, and the Phase 2 trick works in reverse too:
       precompiled ext-linux-*.o plus vendored stubs). Verify on the
       macos-latest CI job once Phase 1 lands.
