** WIP #158 One pass per phase across the whole pipeline

   Progress / current state (read the source, the original list below
   is stale):

   - lexer (~tokenize~), parser (~parse~): single pass. Done.
   - initer (~init_declarations~ / ~init_declarations_unit~): now a
     single AST walk that buckets each Decl into index lists; the
     post-walk loops iterate those small filtered lists (not the AST)
     in dependency order (type names in scope -> aliases -> layouts ->
     size/derived methods). The 12 sub-procs the old list named are
     gone. Done.
   - typer (~infer_body~ -> ~body_pre_passes~): the 7 desugar/insert
     sub-passes are one prefilter + one move-out/rebuild/move-back.
     That is [[./146.org][#146]]'s scope. Done.
   - constfolder (~constfolder_imported~): the three setup sub-passes
     (scan_names, register_defs, register_globals) re-walked the AST
     even though every statement already carries all the data each one
     needed and none reads the scope the others build. Collapsed into a
     single walk (~constfolder_collect_and_register~). Done. (constfolder's
     state -- macro/func name sets and the fold-results map -- now also
     lives on ~Context~ as ~constfolder_macros~/~constfolder_funcs~/
     ~constfolder_known~, per the [[./178.org][#178]] direction.)

   Remaining phases keep multiple walks, but these are NOT redundant
   re-walks -- they are an irreducible dependency chain. Each later
   walk consumes the *complete* output of an earlier walk over the whole
   program, so it cannot start until the earlier one finishes (top-level
   forward references / graph reachability / C output ordering):

   - interpreter (~interpret~ / its imported variant): register_defs ->
     {register_aliases, init_ns} -> copy_alias_ns -> eval. register_defs
     must precede the rest (an alias ~Foo := Bar~ or a method body can
     reference a name defined later). Merging register_aliases with
     init_ns is NOT behaviour-preserving: ~interpreter_init_ns~ calls
     ~eval_expr~ on every namespace-member default, which reads the
     ~global~ scope that ~register_aliases~ writes -- a ns default that
     names a type by its *alias* would resolve under the current
     "all aliases first" ordering but not under an interleaved walk.
     copy_alias_ns reads the namespaces init_ns fills, so it stays last.
   - scavenger: build top maps -> seed worklist -> BFS -> filter. BFS
     follows edges, not source order; the sweep needs the full mark.
   - builder: header/forward decls -> globals -> function bodies. C
     requires a declaration before its use. [[./178.org][#178]] already moved the
     builder's redundant lookup *tables* onto Context.

   The one redundancy left is cross-phase, not within-phase: constfolder's
   name classification (and the interpreter's def/alias/ns registration)
   re-derive facts the typer already computed. The storage half of that
   is done -- [[./178.org][#178]] / [[./147.org][#147]] moved the per-phase symbol/cache state
   onto Context. What remains is to *populate* the foldable-name set
   during the typer's existing walk so constfolder can drop its scan step
   entirely. The typer is the correct capture point because initer runs
   before the typer lifts lambdas and generates derived methods that
   constfolder must also see. That is the natural next step and is tracked
   here.

** Original report (kept for reference)

Each phase listed in doc/pipeline.org (lex, parse,
   init_declarations, type_check, constfolder, scavenge, interpret /
   build) should walk the AST exactly once. At each statement, all
   the work that phase needs to do should happen before moving on.

   Today this is not true. Concrete evidence read directly from the
   source:

   - ~tokenize~ (src/self/lexer.til:259) -- single pass over the
     source string. OK.

   - ~parse~ (src/self/parser.til:1319) -- single pass over the
     token stream. OK. After the main loop it appends pre-collected
     ~fn_sig_decls~ and ~lambda_decls~ -- no extra AST walks.

   - ~init_declarations~ (src/self/initer.til:1870-1887) calls 12
     separate sub-procs, each taking ~program~ and walking it:
       register_struct_definitions, register_funcsig_aliases,
       generate_missing_struct_clones, generate_missing_struct_deletes,
       register_enums_and_generate_nonclone_methods,
       generate_missing_enum_clones, register_type_aliases,
       compute_all_struct_layouts, generate_size_methods,
       generate_derived_methods, register_function_definitions,
       register_top_level_values.

   - ~type_check~ -> ~infer_body~ runs 7 sub-passes per body
     (already covered in [[./146.org][#146]]): desugar_set_literals,
     desugar_map_literals, desugar_variadic_calls,
     desugar_kwargs_calls, hoist_fcall_args, insert_field_deletes,
     insert_free_calls.

   - ~constfolder~ (src/self/constfolder.til:506) has 3 numbered sub-passes
     (collect names, set up scope, fold calls).

   - ~scavenge~ (src/self/scavenger.til:212) has 5 numbered
     sub-passes (build top maps, seed worklist, BFS, filter
     top-level decls, filter ns methods).

   - ~interpret~ (src/self/interpreter.til:1622) walks core and user
     programs through 5 sub-procs each before evaluation:
     ~ffi_init_scan_program~, ~interpret_register_defs~,
     ~interpret_register_aliases~, ~interpreter_init_ns~,
     ~interpret_copy_alias_ns~, then ~eval_body~. So core/user are
     each walked ~6 times before execution.

   - ~build~ (src/self/builder.til:3617-3699) calls roughly a dozen
     emit_* helpers, each walking core and user programs:
     build_register_lookups, build_register_funcsig_names,
     emit_monolithic_header, emit_ext_func_declarations,
     emit_all_forward_declarations, emit_global_declarations,
     emit_function_bodies, emit_dyn_call_bodies,
     emit_dyn_fn_wrappers, emit_dyn_has_bodies,
     emit_collection_helpers, plus emit_test_main /
     emit_script_main / emit_lib_init at the end.

   What I want: one pass per phase. Not one pass for some sub-passes
   inside the typer ([[./146.org][#146]]'s scope). Every phase: lexer, parser,
   initer, typer, constfolder, scavenger, builder, interpreter -- one
   walk over the AST (or token stream / source for lexer/parser).
