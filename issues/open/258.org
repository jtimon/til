** OPEN #258 codegen: clone of a borrowed view is elided, moving the borrow (the other half of #250)

[[../fixed/250.org][#250]] fixed the *delete-ordering* half of the
substr-view-clone miscompile (~extend_hoist_view_lifetimes~ in
~src/self/garbager.til~: an owned local's scope-end delete is now
scheduled after the clone that reads its view, not before). This is the
*other* half: in some shapes the ~.clone()~ of a borrowed view is
*elided entirely* and the borrow itself is moved into the assign target,
which then dangles once the source buffer is reused.

It bit master directly: a refactor changed the arg parser from
~flag.substr(N, ...).clone()~ to ~flag.strip_prefix("...").clone()~.
~strip_prefix~ returns a view (it is ~self.substr(...)~), and
~cli.cc = flag.strip_prefix("--cc=").clone()~ in ~parse_args~ compiled to
a *move of the view*, corrupting ~--cc~ and breaking the linux-riscv32
CI ("sh: 1: U: not found"). #250's fixed.org wrongly assumed the
~format(...)~ ~--cc~ workaround was still in place; the refactor had
replaced it. Re-fixed at the call sites for now (~--cc~ and ~--prefix=~
use ~format(flag.strip_prefix(...))~, which materializes an owned copy).

*** Generated C (the smoking gun)

~cli.cc = flag.strip_prefix("--cc=").clone()~ (equivalently
~flag.substr(5, ...).clone()~) emits:

#+begin_src c
Str *hoisted__Str_X = Str_substr(flag, 5, len-5);   // view INTO flag's buffer
cli->cc = DEREF(hoisted__Str_X);                     // clone ELIDED: the view is moved in
free(hoisted__Str_X);                                // frees only the view wrapper
#+end_src

There is no ~Str_clone~ at all. ~cli->cc.c_str~ now points into ~flag~'s
buffer (an argv element); when that storage is reused during
~load_program~, ~cli.cc~ reads garbage.

*** Where it is NOT (already ruled out)

The elision is *upstream of the asap pass* (~src/self/garbager.til~).
Forcing the two garbager clone-to-move elision branches in
~insert_post_stmt_deletes~ to stand down (the #238 ~var_aliases_target~
guard plus a view guard) does NOT stop it -- the clone is already gone by
the time the garbager runs. So the drop happens in the typer hoist pass,
the scavenger DCE, or precomp.

*** Hard to reduce

The clone is *kept* (and the code works) for every minimal shape tried:
  - owned-local source: ~mut a := Str.with_capacity(..); h.s = a.substr(...).clone()~
  - value-param source: ~mk := func(flag: Str) ... h.s = flag.substr(...).clone()~
  - ref-into-Vec source: ~ref flag : Str = v[0]; h.s = flag.substr(...).clone()~
Only the full ~parse_args~ context triggers the elision: a field of the
*returned* struct, assigned inside a ~while~ loop over a ~ref flag : Str
= args[i]~ binding, read long after the loop. Reproducing it minimally is
the first task.

*** Next steps

1. Find a minimal repro (vary: returned-struct field vs local; inside a
   loop vs not; ~ref~-into-collection source vs owned).
2. Bisect which pass drops the clone: dump the post-typer, post-scavenger,
   and post-precomp AST for ~parse_args~ and see where
   ~<view>.clone()~ becomes bare ~<view>~.
3. Likely fix: the pass that turns ~x = view.clone()~ into a move at the
   view's last use must treat a *borrowed view* as non-owning and keep the
   clone (the move is only valid when the clone-arg owns its buffer). This
   is the dual of [[../open/256.org][#256]] (which wants *more* moving of
   genuinely-owned locals); here we want *less* moving of borrows.
4. Once fixed, drop the ~format(...)~ workarounds in ~parse_args~
   (~--cc~, ~--prefix=~) and confirm ~flag.strip_prefix(...).clone()~
   round-trips.

*** Why medium

Silent data corruption (serious), but the triggering shape is narrow and
the call sites that hit it are worked around. Same severity class as
#238 / #250.
