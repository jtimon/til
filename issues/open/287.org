** OPEN #287 Key-value for-in over maps: `for k, v in map`

Related: builds on [[./284.org][#284]] (collection for-in via unchecked
  unsafe_get); sibling of the value/key iteration cleanups below.

  Maps expose internal unchecked indexed accessors -- ~key_ptr(i)~ (key)
  and ~unsafe_get(i)~ (value, formerly ~val_ptr~) -- and consumer code
  reaches into them with hand-rolled counting loops to walk entries::

    for gi in 0..ctx.generic_funcs.len() {
        ref gname : Str = ctx.generic_funcs.key_ptr(gi)
        ref gdef  : Expr = ctx.generic_funcs.unsafe_get(gi)
        ...
    }

  This leaks the map's storage internals into callers. The for-in idiom
  fixes the value-only and key-only cases already (see below), but there
  is no clean idiom for walking key+value pairs. Add one:

    for k, v in map {
        <body using k and v>
    }

*** Desugar

  ~keys~ and ~values~ are parallel, index-aligned Vecs in every map
  (Map, HashMap, DynMap, DynHashMap -- sorted order for Map/DynMap,
  dense insertion order for the hash maps). So `for k, v in map` lowers
  to a single lockstep index walk over the two backing Vecs::

    _i := 0
    while _i < map.keys.len() {
        ref k := map.keys.unsafe_get(_i)     // unchecked: _i < len
        ref v := map.values.unsafe_get(_i)
        <body>
        _i++
    }

  This is O(n) (no per-element lookup), uses the unchecked Vec.unsafe_get
  from #284, and -- crucially -- needs NO key_ptr / value accessor on the
  map itself: it goes straight through the public ~keys~ / ~values~ Vec
  fields. Two binding vars, both refs; support ~for mut k, v~ /
  ~for k, mut v~ as the element-mutation needs dictate.

  Protocol detection (in src/self/typer.til, alongside
  ~desugar_for_in_collection_stmt~): a two-variable for-in requires the
  iterated type to expose index-aligned ~keys~ and ~values~ collections
  (fields, or keys()/values() accessors -- decide one). Single-variable
  for-in is unchanged.

*** Payoff: stop leaking the maps' indexed accessors

  With this plus #284 already in, every consumer counting-loop over a map
  becomes a for-in, and the indexed accessors retreat to the map
  internals (binary search) where they can be made ~priv~:

  - value-only (8 loops) -> ~for v in map.values~ (needs only #284):
    typer.til check_unused_locals / check_unused_mut_locals /
    check_unused_priv_top_level (~3340/3372/3401), the two
    set_struct_def loops (~4779/4789), normalize_scope_bindings (~4868,
    `for mut v`), interpreter.til scope_free (~2669), garbager.til
    collect_scope_locals (~124).
  - key-only (2 loops) -> ~for k in map.keys~ (needs only #284):
    loader.til wildcard-method expansion (~550 and ~554).
  - key+value (3 loops) -> ~for k, v in map~ (needs THIS issue):
    interpreter.til ns_fields walk (~1661), garbager.til imported walk
    (~1028), initer.til generic_funcs walk (~3447).

  After the conversions, the only remaining users of ~key_ptr~ and the
  indexed ~unsafe_get~ are the maps' own binary-search ~has~/~get~/~set~
  (which fundamentally need O(1) indexed key/value access and cannot be
  for-in loops). At that point mark ~key_ptr~ ~priv~ in src/core/map.til
  and src/std/dyn_map.til so it stops leaking; ~unsafe_get~ stays public
  only insofar as single-var for-in over ~.values~ wants it on Vec (it is
  already a Vec method).

*** Notes

  - key_ptr / unsafe_get are NOT being deleted -- the binary search needs
    them. This issue removes their use from *consumer* code and lets the
    map-internal ones go private.
  - The eight value-only and two key-only conversions do not depend on
    this issue (only on #284) and could land first; the three pair loops
    are what require `for k, v`.

*** Regression test

  test/for_in*.til: assert `for k, v in m` visits every entry with the
  right key/value pairing, for Map and HashMap, including a `for mut`
  variant that mutates values in place. Optionally assert (via
  ~til translate~) the lowered loop body calls no throwing get and no
  map-level key accessor -- just lockstep Vec.unsafe_get.
