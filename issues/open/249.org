# priority: low
** OPEN #249 parser.til cleanup: DRY argument loops, binop clones, flag switches

  Review of ~src/self/parser.til~ (2050 lines) found several
  simplification / DRY / optimization opportunities. None change
  behaviour; tests pass as-is. Listed roughly by value. Sibling of
  #248 (the same kind of review for ~lexer.til~).

*** Triplicated call-argument loop (highest value)
  The argument-parsing loop -- named-arg lookahead, splat / rest-pattern,
  ~own~, the #56/#57 anon struct/enum lift, trailing comma -- is
  copy-pasted across three sites, plus a fourth reduced copy:
    - ~parse_call~ (~754-844)
    - postfix method call in ~parse_primary~ (~1163-1209)
    - statement method call in ~parse_statement_ident~ (~1572-1618)
    - the chain-call loop (~1636-1659), a stripped-down 4th copy
  The code itself flags the hazard: "keep all three sites in sync"
  (~1184, ~1593). ~150 lines kept consistent by hand. Extract a single
  ~parse_call_args(mut p, mut call)~ that appends children to an
  already-built FCall; collapses all four and removes the sync trap.

*** ~make_binop_call~ deep-clones both operands (optimization)
  ~make_binop_call~ (~871) does ~own lhs.clone()~ and ~own rhs.clone()~.
  All 8 callers do ~e = make_binop_call(e, ..., rhs)~ then discard the
  originals. Take ~own lhs~ / ~own rhs~ to remove two full AST clones
  *per binary operator* in every expression parsed -- a real hot-path
  saving with no behaviour change.

*** Seven FCall flag-set switches, 6 lines each to flip one bool
  The ~does_throw~ / ~is_bang~ / ~is_splat~ setters all repeat
    switch X.node_type {
    case NodeType.FCall(fcd): fcd.FLAG = true
    case: UNREACHABLE(__LOC__)
    }
  Three helpers (~set_throw~ / ~set_bang~ / ~set_splat~, or one taking
  the flag) replace ~42 lines and roughly half the file's 14
  ~UNREACHABLE~s. The StructDef/EnumDef mutation switches
  (~sdd.fields.push~, ~edd.variants.push~, ~sdd.ns_decls.push~, ...)
  share the same shape; a ~with_structdef~ / ~with_enumdef~ helper
  absorbs the rest.

*** Postfix ~?~ / ~!~ handling duplicated
  The "if Question advance + set does_throw; if Bang advance + set
  is_bang" block appears in ~parse_call~ (~846-866) and the method-call
  path (~1211-1230). Fold into one ~parse_throw_bang_suffix(mut p, mut
  call)~ (which itself uses the flag-set helpers above).

*** Namespace-decl parsing duplicated between struct and enum
  ~parse_struct_def~ (~541-573) and ~parse_enum_def~ (~654-693) parse
  the same ~[priv] [mut] name [: Type =] | [:= ] value~ shape into a
  ~Declaration~. Only the push target differs (~sdd.ns_decls~ vs
  ~edd.ns_decls~). Factor into ~parse_ns_decl(mut p, member_priv)
  returns Declaration~.

*** ~match~ arm loop overlaps the new ~parse_switch~
  The inline ~match~ arm loop in ~parse_primary~ (~1074-1101) is nearly
  identical to ~parse_switch~ (added in #244): ~KwCase~, optional
  ~if~ guard, ~:~, body. They differ only in body shape (match arm =
  single value-expr wrapped in Body; switch arm = statement list). A
  shared case-arm helper parameterised by the body parser would DRY
  both.

*** ~e.node_type = NodeType.Body; e.children = Vec.new(Expr)~ move-out dance
  This hand-rolled "take ownership of ~e~" idiom appears 5x (postfix
  ~.~ / ~[]~ / ~++~ / ~--~, and range in ~parse_expression~). A
  ~take_expr(mut e) returns Expr~ helper names the intent and DRYs it.

*** ~explicit_type = X; til_type = til_type_from_explicit_type(X)~ pairs (13x)
  This two-line pair appears at 13 sites. Either a ~set_type(mut decl,
  name)~ helper, or have the ~Declaration~ constructor derive
  ~til_type~ from ~explicit_type~ so callers stop passing both.

*** Larger refactors (lower ROI, do only if touching the area)
  - The 7 precedence procs (~parse_multiplicative~ .. ~parse_logical_or~)
    are structurally identical -- only the token set and method name
    differ. A Pratt-style precedence table would replace ~120 lines
    with one driver.
  - ~parse_primary~'s ~e_set~ ladder (~881-1112) is a long
    ~if e_set.not().and(t.type.is(...))~ chain where several branches
    ~return~ directly anyway. A ~switch t.type~ with early returns
    would read far better but is a sizable rewrite.
