** WIP #232 Hosted 32-bit targets: Linux and Windows

:DEPENDENCIES: [[./81.org][#81]]

  Add hosted 32-bit targets as cross-compilation targets. This covers
  normal OS targets with C toolchains and emulators available from a
  Linux host: ~linux-x86~, ~linux-arm32~, ~linux-riscv32~, and
  ~windows-x86~.

  The shared compiler work is auditing generated C for 64-bit pointer
  assumptions and replacing them with ~sizeof(void*)~, ~UPtr~, or the
  concrete runtime type size. linux-x86 is the canary because it can run
  under qemu-i386; windows-x86 should run under Wine if available.

  Out of scope:
  - Browser WASM is [[./228.org][#228]].
  - TempleOS is [[./151.org][#151]].

*** Scope

  1. Add ~Target.LinuxX86~, ~Target.LinuxArm32~, ~Target.LinuxRiscv32~
     to the enum in [[file:../../src/self/targets.til][src/self/targets.til]] and wire all switch cases.
  2. Add ~Target.WindowsX86~ to [[file:../../src/self/targets.til][src/self/targets.til]] and wire all switch cases.
  3. Fix hardcoded 64-bit values in [[file:../../src/self/builder.til][src/self/builder.til]] and
     [[file:../../src/c/ext.c][src/c/ext.c]]:
     - ~dyn_size_of~ must use ~sizeof(Dynamic)~ instead of hardcoded 8.
     - ~get_payload~ must use ~sizeof(void*)~ instead of ~sizeof(I64)~.
  4. Cross-build vendor libs for 32-bit:
     - x86: ~gcc -m32~ raylib/tinyfd
     - arm32: ~arm-linux-gnueabihf-gcc~ raylib/tinyfd
     - riscv32: ~riscv32-linux-gnu-gcc~ raylib/tinyfd
     - windows-x86: mingw i686 raylib/tinyfd
  5. Add CI jobs for linux-x86 (with qemu-i386 test) and linux-arm32
     (file-format verification; no emulation runner available).
  6. Test cross-compiled binaries.

*** Linux progress

  - Linux 32-bit targets added to enum, all switch cases wired.
  - ~dyn_size_of~ and ~get_payload~ fixed for 32-bit pointer widths.
  - linux-x86 cross-compiles and runs under qemu-i386.
  - CI jobs added for linux-x86 (file + qemu) and linux-arm32 (file).
  - riscv32: target defined, but no standard sysroot package exists
    for linking 32-bit RISC-V binaries on CI yet.
*** Windows progress

  - ~Target.WindowsX86~ added to the enum and wired through every
    targets.til switch: target_from_str/to_str, target_is_32bit (so
    UPtr/USize resolve to U32), toolchain_command ->
    i686-w64-mingw32-gcc, executable/shared extensions, system libs
    (-lpsapi), and the ffi_lib path. A new ~is_windows_target~ helper
    folds the builder's mingw link handling (--export-all-symbols, the
    linux->win link-flag rewrite, the nng swap) onto both windows
    targets at once.
  - til cross-compiles a CLI program to a 32-bit windows .exe:
    ~til build --target=windows-x86 examples/hello_script.til~ produces
    a PE32 Intel 80386 binary with clean KERNEL32+msvcrt imports. The
    32-bit codegen itself was already correct (linux-x86 shares it). CI's
    windows-x64/x86 build job cross-compiles it and the run job executes
    it on windows-latest via WOW64.
  - windows-x86 GUI + runtime-FFI DONE. The 32-bit vendor libs build via
    the mingw i686 toolchain (Makefile RAYLIB_WIN32_LIB / TINYFD_WIN32_LIB
    with an -win32 suffix, LIBFFI_WIN32_LIB in its own i686-w64-mingw32/
    subdir); mingw ships the win32 GL/GDI backend, so raylib/tinyfd
    cross-build from source with NO 32-bit X11 sysroot. target_gui_libs
    and target_ffi_lib return the -win32 paths + the win GL/GDI libs
    (gdi32/opengl32/winmm/...). ~til build --target=windows-x86
    examples/hello_gui.til~ produces a PE32 mode-gui binary. One 32-bit
    codegen fix was needed in [[file:../../src/modes/gui.til][src/modes/gui.til]]: make_tone hardcoded
    ~.cast(U64)~ for malloc's count and ptr_add's offset, but both params
    are ~UPtr~ (U32 on a 32-bit target); switched to ~.to_usize()~ which
    tracks the target width (identical U64 on 64-bit hosts). ~make
    build_win32~ sweeps the portable + gui examples for windows-x86
    (mirrors build_win; nng server examples stay win64-only). CI's
    windows-x64/x86 build job now also cross-links hello_gui for both
    arches and asserts PE32+/PE32 (build-only, like the mac gui-cross job:
    a raylib window cannot open headless).

*** Linux GUI progress

  - linux-x86 GUI DONE, and cheaper than feared: raylib compiles -m32
    against the VENDORED X11 headers (vendor/x11/include, same as the
    native recipe -- headers are arch-independent) and GLFW dlopens
    libX11 at runtime, so the final link needs NO 32-bit X11 sysroot;
    gcc-multilib (32-bit libc) is the only dep, which the CI linux-x86
    job already installs. Makefile RAYLIB_X86_LIB / TINYFD_X86_LIB build
    the -x86 archives; builder.til's link-token rewrite now also covers
    the 32-bit linux targets (it previously only fired for windows/wasm/
    mac, so the inherited 64-bit libraylib.a stayed in the link and
    clashed with i386 output). Verified end to end: ~til build
    --target=linux-x86 examples/hello_gui.til~ produces an ELF 32-bit
    Intel 80386 gui binary that opens a window, initializes GL
    (llvmpipe) and loads textures/fonts under qemu-i386 + Xvfb on a
    64-bit host, alive at the 3s timeout (exit 124). One asterisk on
    "no 32-bit X11 needed": that is true for BUILDING (vendored
    headers + dlopen), but RUNNING needs the i386 X11/GL runtime libs
    -- without them GLFW's dlopen of libX11 fails and raylib segfaults
    windowless (first master run 29622850794 caught exactly that). The
    CI linux-x86 job installs libx11-6:i386 + friends and requires
    exit 124 from the smoke run.

*** Remaining

  - Verify linux-arm32 and linux-riscv32 sysroot/toolchain assumptions
    on the actual CI image.
  - GUI vendor libs for linux-arm32/linux-riscv32: unlike linux-x86
    (multilib on the same machine), these need full cross X11/GL
    sysroots for the target arch -- the expensive part. target_gui_libs

    FFI for the linux 32-bit targets is unaffected (no GUI libs needed).
