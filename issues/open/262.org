** WIP #262 lazy_func / lazy_proc: lazily-evaluated call arguments

v1 status (implemented):
  - ~lazy_func~ / ~lazy_proc~ keywords parse to ~FuncType.LazyFunc~ /
    ~FuncType.LazyProc~ (lexer, parser, ~FuncType~ enum + name mappers).
  - A lazy callable's body is a single ~return <expr>~. Calls are inlined
    by ~expand_lazy_calls_in_program~ (src/self/loader.til), a pre-typer
    pass: it clones ~<expr>~ and substitutes each parameter Ident with the
    matching argument expression (call-by-name), then the typer types and
    lowers the result normally. When ~<expr>~ is a ~match~, only the taken
    arm runs, so unused args never evaluate -> short-circuit.
  - The lazy defs themselves are left in place and type/emit as ordinary
    (uncalled) functions, like macros, so the builder needs no changes.
  - Regression test: test/lazy.til (functional and/or, nested lazy calls,
    and decisive short-circuit proofs where an unused side-effecting
    argument must not run).

Remaining (deferred):
  - Variadic, Lisp-style ~and~/~or~ folding (and N-ary lazy callables).
  - First-class lazy callables (stored in a var / funcptr) and recursive
    lazy callables, which cannot be inlined and need real thunks/closures.
  - Multi-statement lazy bodies (v1 requires a single ~return <expr>~).
  - Migrating the core ~and~/~or~ to lazy callables (closes #117).
  - call-by-need (memoize a reused argument once) vs the current
    call-by-name (re-evaluate on each use).

Introduce ~lazy_func~ and ~lazy_proc~ as new callable kinds (alongside
  ~func~/~proc~ and ~ext_func~/~ext_proc~). When you call a lazy callable,
  its arguments are NOT evaluated eagerly at the call site; each argument
  expression is treated as a thunk and is only evaluated if and when the
  body uses it. Most of the work should live in the desugarer.

Motivation:
  - It unblocks short-circuiting ~and~/~or~ (#117). With lazy args, ~and~
    is just:
        and := lazy_func(a: Bool, b: Bool) returns Bool {
            if a { return b } else { return false }
        }
    and ~b~ is never evaluated when ~a~ is false. ~or~ is the dual.
  - It enables a variadic, Lisp-style ~and~/~or~ that short-circuits across
    N operands, e.g. ~and(p, q, r, ...)~ folding to nested ~if~s, instead of
    the chained ~.and(...).and(...)~ method form.

Sketch of the desugaring approach:
  - At a lazy call site, the desugarer inlines the callee's body and splices
    each argument expression in at the points where the corresponding
    parameter is used (call-by-name / macro-style expansion). For ~and~/~or~
    this collapses to the ~if~ forms above with no thunk allocation.
  - Variadic lazy and/or desugar by right-folding the operand list into
    nested conditionals.

Open questions / honest caveats (these are why this is its own issue, not a
  trivial rewrite):
  - "All in the desugarer" only holds when the lazy callee can be inlined at
    the call site. A first-class lazy func (stored in a var, passed as a
    funcptr) or a recursive one cannot be fully desugared by inlining; those
    cases need real thunks/closures or must be disallowed initially.
  - Side-effect / evaluation-order semantics change by design: an unused lazy
    argument's side effects never run. This needs to be documented clearly.
  - Each lazy parameter that is used more than once in the body would
    re-evaluate its argument unless we bind it once; decide whether lazy means
    call-by-name (re-eval) or call-by-need (memoized once).

Note: the binary short-circuit ~and~/~or~ from #117 could also be done as a
  narrow, special-cased desugar rewrite (~a.and(b)~ -> ~if a { b } else
  { false }~) without any of this general machinery. This issue is the
  general path that additionally buys user-defined lazy callables and the
  variadic form; #117 can be satisfied by either.
