# priority: high
** OPEN #184 Finish the bare ~get_payload~ -> switch destructure refactor

Goal: remove the remaining bare ~get_payload(...)~ calls in the
   self-host and replace them with idiomatic switch destructuring,
   so payload extraction is type-safe and the payload reads are
   colocated with the variant tag check.

   Pattern to apply:

   #+begin_src til
   // before
   if e.node_type.is(NodeType.Decl) {
       ref d : Declaration = get_payload(e.node_type)
       ... use d ...
   }
   // after
   switch e.node_type {
   case NodeType.Decl(d):
       ... use d ...
   case:
   }
   #+end_src

   And the bare-binding form:

   #+begin_src til
   // before
   ref d : Declaration = get_payload(stmt.node_type)
   // after
   switch stmt.node_type {
   case NodeType.Decl(d):
       ... rest of function body ...
   case:
       UNREACHABLE(__LOC__)
   }
   #+end_src

*** Plan to close

    Add a typer.til rule that rejects user-code calls to
    ~get_payload~ outside the four sites above.
    Typer-emitted ~Ident("get_payload")~ from switch case
    desugaring stays allowed because it is constructed inside
    the compiler, not written by the user. The check fires on
    AST shape (NodeType.Ident("get_payload") inside an
    NodeType.FCall) and consults a known-callers allow-list keyed
    on enclosing function name + file path.

*** ~.is~ vs ~.eq~ on enum tags

    Style rule that fell out of this work:

    - ~x.is(Variant)~ is the right call for tag-only checks. It
      compiles to a bare tag compare.
    - ~x.eq(Variant)~ is payload-aware deep equality. Using it for
      tag-only checks is misleading and can be more expensive.

    All ~x.eq(EnumName.Variant)~ literal forms in src/self/ have
    been flipped to ~x.is(EnumName.Variant)~.

*** Lessons learned (for future similar work)

    1. *Verify "X is broken" claims empirically before believing
       them.* The session-summary note that said "switch on owned
       ~Value~ types or ~self~ inside ~Value.clone~ causes issues
       (recursive codegen, ownership)" was wrong: ~Value.to_str~
       was already using ~switch self~ in the same file, and
       converting ~Value.eq~ and ~Value.clone~ to switch built and
       passed all tests. Don't carry forward folklore from a
       compacted summary -- run the experiment.

    2. *Read the actual switch desugar before reasoning about it.*
       In [[file:../../src/self/typer.til:make_switch_case_condition][src/self/typer.til:make_switch_case_condition]], a case
       with a payload binding desugars to:

       #+begin_src til
       if X.is(Variant) {
           ref binding : payload_type = get_payload(X)   // OwnType.Ref
           ... case body ...
       }
       #+end_src

       The case-binding is a borrow (~OwnType.Ref~), zero
       allocation. If something around the switch leaks, the leak
       is on the surrounding code's side, not the desugar (this is
       what [[./175.org][#175]] captures: ~mut X := ""; X = case_bound_str~ leaks
       both the ~""~ and the new ref-to-mut clone).

    3. *Don't hoist case-bound names into ~mut Str~ locals.* When
       a single helper handles two variants that share the same
       payload type (e.g. ~FieldAccess~ + ~FieldAssign~ both carry
       ~Str~), the obvious ~mut X := ""; switch ... case A(s): X = s
       case B(s): X = s~ pattern leaks both ~""~ and the
       ref-to-mut clone (see [[./175.org][#175]]). Extract a small
       helper that does the work inside each case arm instead.

    4. *Trust but verify session-summary file lists.* When a
       summary says "all sites in file X converted", grep before
       you commit. Some sites get missed; the typer.til squash
       still left one bare ~get_payload~ at line 1042 that nobody
       caught for two rebases.

    5. *Use ~UNREACHABLE(__LOC__)~ in default arms.* When the
       switch is genuinely exhaustive over the variants the code
       expects, the catch-all ~case:~ should be ~UNREACHABLE(__LOC__)~.
       Skipping it makes the next maintainer wonder if a real
       fallthrough is intended.

    6. *Prefer rebase over cherry-pick when integrating with
       master.* Cherry-pick rewrites authorship and ordering, which
       complicates "is my work merged or not" questions later. The
       answer is almost always "rebase".
