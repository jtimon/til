# category: features
** OPEN #212 Add a `cast(e, T)` builtin (UFCS-callable as `e.cast(T)`) to replace the integer `.to_*` conversion matrix

*** Summary

til used to convert between primitive types with an N x M matrix of
hand-written `.to_<Type>()` methods, each backed by a trivial extern C
function. This issue introduced a single `cast` builtin for the
*numeric width/sign* conversions, while KEEPING the method form for
conversions that are really parsing or formatting (notably `to_str`
and `Str.to_*`).

The chosen interface (see "Interface" below) is a two-argument builtin
`cast(e, T)` callable UFCS-style as `e.cast(T)`, NOT Jai's prefix
`cast(T) expr`. The Jai form is kept below as background/contrast.

*** Status (shipped)

The core arc is in master:

    src/core/cast.til, typer special-case in infer_cast_fcall, builder

*** How Jai does it

  - `cast(T) expr` -- value-directed explicit cast. ONE builtin handles
    every type pair: int<->int, int<->float, pointer, enum<->int. On
    lossy numeric narrowing it inserts a runtime range check in debug
    builds; `cast,no_check(T) expr` and `cast,trunc(T) expr` opt out.
  - `xx expr` -- context-directed autocast. The target type is inferred
    from context (assignment LHS, parameter type, return type) and the
    cast is synthesized: `float x = xx my_int;`, or `foo(xx i)` where
    `foo`'s parameter type drives the conversion.
  - No silent implicit numeric conversions otherwise (a deliberate
    anti-C choice).

The structural point: `cast` is one builtin keyed on the TARGET type,
and `xx` reads the target from context. Neither needs a per-type method
to exist.

*** Interface (decided): `cast(e, T)`, UFCS-callable

Rather than copy Jai's prefix operator, til models `cast` as an ordinary
two-argument builtin so it composes with existing machinery (UFCS,
Dynamic, and type-name-as-argument) and needs NO new keyword or grammar:

    cast := ext_func(e: Dynamic, T: Type) returns Dynamic {}

Usage, plain and UFCS:

    a : U32 = 0
    b : I32 = cast(a, I32)      // plain call
    b : I32 = a.cast(I32)       // UFCS, the intended spelling

Design notes:

  - `returns Dynamic` is a STOPGAP. The true signature is
    `cast(e: Dynamic, T: Type) returns T` -- the result type is the
    *value* of the T argument -- but til has no dependent/type-parametric
    return syntax yet. Until it does, `cast` declares `returns Dynamic`
    and the typer special-cases it (see below). When a `returns T` form
    lands, drop the special-case and the stopgap.
  - Typer special-case: when resolving a `cast` call, read the second
    argument's `Type` value and set the call expression's result type to
    exactly that type -- NOT generic `Dynamic`. That makes
    `b : I32 = a.cast(I32)` type-check directly instead of leaning on
    Dynamic->I32 narrowing at the assignment.
  - `T: Type` as an argument reuses the same type-name-as-value mechanism
    that `Vec.new(Str)` / `Set.new(Str)` already rely on; the parser/typer
    turns the type name `I32` in argument position into a `Type` value.
  - UFCS: `a.cast(I32)` desugars to `cast(a, I32)` through the existing
    `try_ufcs_rewrite` path, so `cast` is reachable as a method on any
    value without per-type definitions.
  - Codegen: the builder intercepts `cast` and emits `(<CType>)e`; there
    is no real C body despite the `ext_func` shape (like other builtins).
  - Ergonomics: `b : I32 = a.cast(I32)` restates `I32` on both sides.
    That is the accepted cost of NOT adopting `xx`-style autocast; the
    explicit target at the call site doubles as the safety contract.

*** How til did it before (historical)

Each numeric type carried a fan of `.to_<Type>()` methods (I64 had 9,
U8 had 7, etc.), each declared in src/c/ext.h and defined in
src/c/ext.c as a trivial `(T)a` C cast. Pairs nobody wrote routed
through the I64 hub (e.g. `Str.to_i32 = self.to_i64().to_i32()`), and
the typer carried an `is_self_conversion_method` guard
(src/self/typer.til) purely to reject `x.to_u32()` when `x` was
already U32. The four same-size reinterpret methods (I64.to_u64,
U64.to_i64, I32.to_u32, U32.to_i32) were the most clearly redundant:
each emitted a bare C cast that `cast` now spells uniformly.

*** Why a cast-style construct looks better

  0. The motivating case: same-width sign reinterpret. Turning a U64
     into an I64 (or U32<->I32) is a free, no-op reinterpret at the
     machine level -- the bits do not change, and til already emits it
     as a bare C cast. `u.cast(I64)` keeps that zero cost and C
     semantics under ONE spelling, replacing the current pile of
     per-type methods (I64.to_u64, U64.to_i64, plus the dead, redundant
     U64.from_i64) that all compile to the same `(T)x`.
  1. No N x M method matrix. `x.cast(I32)` works for any value x.
     Adding a new numeric type today means writing conversions to/from
     every existing type in three places (.til, ext.h, ext.c); with
     `cast` it is zero.
  2. Deletes the C backing surface. The `U8_to_i64`-style functions are
     just `(I64)a`; codegen can emit the C cast inline, dropping the
     conversion block in ext.h/ext.c and the .til wrappers.
  3. Collapses many method names into one. `is_self_conversion_method`
     (the typer guard rejecting `x.to_u32()` on a U32) becomes
     unnecessary -- there is no per-type conversion method name left to
     misuse; the target is just the T argument to one `cast`. (til does
     NOT adopt Jai's `xx` autocast; the target is always written
     explicitly as T.)
  4. No hub-routing: `cast` goes direct, losing the `.to_i64().to_i32()`
     two-step that exists only because the direct pair was not authored.
  5. Uniform across kinds: the one builtin could later cover pointer and
     enum casts, which today would each need bespoke methods.

*** What to KEEP as methods (the dividing line)

The clean rule: `cast` is for representation conversion the CPU does
directly; methods stay for operations that allocate, validate, or run an
algorithm. Those are not casts and should not masquerade as one.

  - `to_str` on every type -- it is FORMATTING, not reinterpretation: it
    allocates a heap Str, and for floats/uints runs real logic
    (F32_to_str uses sprintf, U64_to_str_ext is a divide-by-10 loop --
    src/c/ext.c:399+,464+). `42.cast(Str)` would be a lie. KEEP to_str.
  - `Str.to_i64` / `to_f32` / `to_u8` ... -- PARSING with validation that
    panics on bad input (src/core/str.til:265,314). It is a fallible
    function, not a bit-level cast; `"123".cast(I64)` should not silently
    parse. KEEP these.
  - F32 <-> integer -- the one case where `cast` is NOT free. Same-width
    int sign-reinterpret and int narrowing are pure bit operations, but
    float<->int genuinely recomputes a value (rounding/truncation in
    hardware). It is still a primitive machine conversion, not parsing,
    so it can live under `cast` (C spells it the same way). But it breaks
    the "cast is a free reinterpret" intuition, so it is worth calling
    out explicitly or keeping as a method (to_f32/to_i64) if we want the
    cost to stay visible in the name. Lowest-priority piece either way.

*** Remaining open questions

  - F32 <-> integer: deliberately kept as `.to_f32()` / `.to_i64()`
    methods for now. It is the one machine conversion that genuinely
    recomputes a value (rounding/truncation), not a bit reinterpret,
    so keeping the cost visible in the name is the conservative
    default. Folding it into `cast` (Jai-style: any primitive machine
    conversion) is a coherent alternative -- file a follow-up if/when
    that uniformity is wanted.
  - Pointer / enum casts: scope whether `cast` should subsume any
    existing pointer / enum-int conversions or stay integer-only.
  - Dependent return type: the honest signature is
    `cast(e: Dynamic, T: Type) returns T`. Once til has a way to spell
    a return type that depends on a `Type`-valued argument, drop the
    `returns shallow Dynamic` stopgap in src/core/cast.til AND the
    typer's result-type override in infer_cast_fcall AND the two
    typer-override short-circuits in src/self/builder.til
    (fcall_returns_dynamic, fcall_return_ctype). The same-size-pair
    legality check in infer_cast_fcall still stays.
  - Followup cleanup: `is_self_conversion_method` in src/self/typer.til
    is now unreachable for the four same-size pairs (their methods are
    gone, so any `x.to_u32()` on a U32 fails earlier with "no method").
    Resolved: KEEP. The check is still reached for user struct types
    via ASCII case-insensitive matching (`val.to_terrselfaaa()` on
    `TerrSelfAaa`) -- without it the typer would surface a generic
    "no method" instead of the clearer "redundant self-conversion"
    diagnostic. test/typer_errors.til:519-541 exercises the path. The
    function already carries an explanatory comment.
