# priority: medium
** OPEN #255 lexer: richer number literals (hex/binary/octal, digit separators, exponents)

The lexer only recognises base-10 integers and a single optional decimal
point (~tokenize~ Number case: digits, then ~.~ + digits). Production
lexers for similar languages (Zig, Odin) also accept:

- hex ~0x1F~, binary ~0b1010~, octal ~0o17~ (and/or ~0~-prefixed),
- digit separators ~1_000_000~, ~0xff_ff~,
- exponents ~1e9~, ~1.5e-3~ (and hex-float forms).

*** Scope
This is a *feature*, not an optimisation -- only worth doing if/when the
language wants these literal forms.

Touch points:
- ~src/self/lexer.til~: extend the Number scanning (and add the relevant
  digit-class helpers; today ~is_digit~ is decimal-only).
- The number *value* is parsed later from the token text (~literal_num_node~
  -> ~Str.to_i64~ / ~Str.to_f32~), so those parsers must learn the new
  bases/separators too, or the lexer must normalise the text.
- ~F32~/integer ~to_str~/~to_repr~ round-tripping should stay consistent.
- Add lexer + parser tests (~test/parsing.til~, a lexer test) covering each
  new form and rejecting malformed ones (e.g. ~0x~ with no digits,
  trailing ~_~).

*** Why medium
Pure additive feature with a clear spec; no urgency, but a natural gap
versus comparable languages and self-contained once the value parsers are
extended.
