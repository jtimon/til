  *** 2026-05 [[./175.org][#175]] series (DONE)

  All four landed under doc/self.org's two-pass workflow.

  1. ~Typer: insert exit-deletes through nested Body blocks~ (4f10e0d).
     ~insert_exit_deletes~ recurses into nested ~NodeType.Body~ stmts
     so the desugared switch sentinels (~_sw_NodeType_*~ etc.) get
     freed before deeply-nested early returns. -44% bytes / -43%
     allocs.

  2. ~[[./175.org][#175]] follow-up: extend save_old_delete to user-written .clone()
     RHS~ (7a2d233). The original [[./175.org][#175]] fix only fired for the
     implicit clone path; ~mut X := ""; X = Y.clone()~ in
     ~extend_ref_local_lifetimes~ and similar sites was missed
     because ~rhs~ is an FCall, not the auto-clone-wrapped
     ~Ident~/~FieldAccess~ shape. Also dropped the redundant
     ~stmt.til_type = existing.clone()~ from the original fix
     (already set from rhs.til_type). -28% bytes / -8% allocs.

  3. ~[[./175.org][#175]] follow-up 2: widen save_old_delete to any FCall returning
     OwnType.Own~ (064ef0d). ~mut X := ""; X = expect_text(...)~
     in ~parse_struct_def~ (and many ~parse_*~ siblings) is the
     same [[./175.org][#175]] outer-scope shape but rhs is an arbitrary
     heap-returning FCall. ~fcall_returns_own~ already exists in the
     typer to discriminate heap-returning callees from shallow-return
     ones, so the trigger now uses it directly. -7.8% bytes / -6.9%
     allocs.

  4. ~free the heap wrapper after value-copy ownership transfer to
     a Shallow struct field~ (ff6ce78). Struct-typed locals are
     heap-promoted by ~promote_own_transferred_locals~. When
     ownership is "transferred" via ~Decl(field = name, ...)~ where
     the receiving field is Shallow, the constructor only
     value-copies the inner struct; the field's own destructor frees
     the inner data, but nothing freed the heap wrapper.
     ~insert_post_stmt_deletes~ now emits ~free(own name)~ after
     such transfer stmts, gated by a new
     ~transfer_is_shallow_field_value_copy~ predicate so other
     transfer shapes (which already consume the wrapper) aren't
     double-freed. -13.9% bytes / -4.4% allocs.

  5. ~widen save_old_delete to LiteralStr RHS~ (a886407). Same
     outer-scope [[./175.org][#175]] shape but ~rhs~ is a literal string, e.g.
     ~mut ftype := fd.explicit_type; ftype = "I64"~ inside a nested
     ~if~ in ~compute_struct_layout~. ~Str_lit~ returns a heap Str
     pointer, so the existing ~save_old_delete~ codegen already
     handles it; just add a third ~else if~ branch in
     ~infer_assign_stmt~ alongside the FCall-returning-Own and
     auto-clone-wrap branches. -480 bytes / -44 allocations.

  6. ~free heap-promoted Struct/Enum local before shallow-return~
     (this commit). When a function declared ~returns shallow X~
     returns a heap-promoted local of Struct/Enum type (e.g.
     ~mut result := ""; ...; return result~ in ~str_view~), the
     existing codegen emits ~return DEREF(rv)~ which value-copies
     the struct out of the heap wrapper but never frees the
     wrapper. Builder.til now emits the box-then-free pattern
     ~{ T _r = DEREF(rv); free(rv); return _r; }~ for this case.
     -384 bytes / -43 allocations vs the post-Vec.new-regression
     baseline. Small per call, but the leaking site (~str_view~)
     was hit on every ~dyn_fn~ / ~dyn_size_of~ dispatch.

  Cumulative for the series:
  ~-212,268 bytes (-63%) / -22,348 allocations (-51%)~ vs pre-fix
  baseline. (Net reduction smaller than peak because the 8e30822
  ~Vec.new~ migration reintroduced ~17K bytes of leaks via more
  ~dyn_fn~/~dyn_size_of~ dispatch sites.)

  *** Open buckets (rough size order)

  1. **FieldAssign auto-delete on owned-typed fields**.
     ~obj.field = X.clone()~ overwrites the old value of the field
     without running the field type's destructor. ~162 sites in
     ~src/self/~ assigned to ~til_type~/~node_type~/etc. Tried the
     naive "always free old field on FieldAssign" approach this
     session and it regressed because ~stmt.til_type = existing.clone()~
     in the same [[./175.org][#175]] path was itself leaking each Type clone:
     so every save_old_delete trigger added a Type leak. Need an
     opt-in flag set by the typer (like ~AssignData.save_old_delete~)
     plus a discriminator for sites that move ownership instead
     of value-copying.

  2. **Module-global Set save/restore idiom in
     [[file:../../src/self/builder.til][src/self/builder.til]]**.
     ~saved := X; X = new(); ...; own X.delete(); X = saved~ on
     ~stack_locals~, ~ref_locals~, ~unsafe_to_hoist~, etc. The
     [[./175.org][#175]] fix excludes module-global targets to avoid double-free
     with this idiom. Refactor the idiom (e.g. wrap in a Saver
     helper, or move the state into a function-local that gets
     auto-cleanup) so the global exclusion in
     ~assign_target_is_func_local~ can go away.

  3. **Program-scope locals at process exit**.
     ~collect_scope_locals~ sets ~skip_scope_delete = true~ for
     locals at the program scope (root TypeScope). For CLI scripts
     that run to completion this is mostly cosmetic, but it trips
     ASAN every time. Decide whether to free, or document the
     filter rule for ASAN runs.

  4. **AST clone heritage** (top remaining cluster on the
     post-[[./175.org][#175]] baseline). Top three current leak sites on
     ~cli_args interpret~:
     - ~795 bytes / 159 objects~ Str_clone via Vec_clone via
       Expr_clone (whole-Vec deep clone of Expr children).
     - ~544 bytes / 44 objects~  Str_clone via NodeType_Ident via
       NodeType_clone via Expr_clone (Ident payload clones).
     - ~448 bytes / 88 objects~  same shape, different sites.
     These are AST nodes cloned during desugar passes that compound
     across passes. ~[[./146.org][#146]]~ (single-walk typer) is the umbrella
     fix. Spot-fixes that helped in the past: drop unnecessary
     ~.clone()~ calls (Bucket A), ownership transfer instead of
     deep clone (Bucket D, [[./144.org][#144]]).

  5. **resolve_imports loop-shadowed sub_ast** (Bucket E).
     ~sub_ast := parse(tokens, ...)~ at [[file:../../src/self/loader.til][src/self/loader.til]] around
     line 126 is shadowed each iteration; its children are cloned
     into ~merged~ but ~sub_ast~ itself is not deleted before the
     next iteration. Either fix the typer's ASAP-delete logic for
     loop-shadowed locals, or add an explicit ~free(own sub_ast)~
     after the merge.

  6. **TokenType_clone / lookup_keyword in the lexer** (Bucket F).
     Str clones for keyword/identifier payloads. The token Vec
     should be freed at the end of parse; audit token lifetime
     and check for missing ~Vec.delete~ on the tokens Vec.

  *** Lessons learned

  - **Leak attribution = allocation site, not ownership-loss site.**
    Bucket C looked like it was caused by a clone inside
    ~insert_exit_deletes~ but the leak was in ~lp.ast~ retention
    (Bucket D). When a fix at the attributed site doesn't move the
    baseline, look one layer up.

  - **Two-pass workflow** (~make test_two_pass~, "Two-pass:" commit
    prefix; see ~doc/self.org~). Use whenever a fix changes which
    delete calls the typer emits. The new emit lands in boot/til.c
    atomically with the source change.

  - **Verify the per-test ASAN delta after every commit.** Test
    pass count alone doesn't tell you whether a leak fix moved the
    needle. ~bin/til_asan interpret examples/cli_args.til hello
    world~ is the cheapest signal.

  - **Heap-promoted Struct locals carry two cleanup decisions:**
    (a) the inner data and (b) the wrapper. The "ownership
    transferred" tracking handles (a) correctly but used to leave
    (b) leaking when the transfer was value-copy to a Shallow
    field. Recently fixed; the same shape may recur in other
    transfer patterns.

  - **Import order can move the ASAN baseline by tens of percent**
    (alphabetizing core.til's imports in 9d0b82d cut strings.til
    interpret leaks by 47%). Likely a teardown-order effect that
    causes more owned values to be freed before exit. Never
    attributed to a specific code path; worth a follow-up if a
    similar import reorder lands again.
