  ~make test_asan~ split into two targets:

  - ~test_asan~ (NEW): runs the suite with the regular ~bin/til~ but
    passes ~--asan~ through to ~tests.til~, which splices it into
    every ~til build~/~run~/~test~ invocation. The compiler isn't
    sanitized so its leaks don't show up; only the *compiled*

    correctness bugs in codegen output without the noise of til's

    via ~bin/til_asan~ AND passes ~--asan~ down. Both compiler and

    superset of ~test_asan~: ~cmd_run~ / ~cmd_test_build_run~
    explicitly re-set ~ASAN_OPTIONS=detect_leaks=1~ when spawning the
    compiled binary so the child's leak check doesn't inherit the
    suppression from its asan-built parent compiler.

  ~add_cli_pair~ takes the flag for its ~run~-pair, ~mode_flags~ for

  Fixes that landed (ext-variadic / ext-kwargs ownership):

  - **~ffi_run~ variadic ext_func leak** (FIXED). ~test/kwargs.til~'s
    sibling shape in ~examples/ffi/ext_module.til~ -- 3 ~sum_all(...)~
    calls leaked the synthesized ~_va_Array_NNN~ (48-byte Array
    struct + calloc'd slot buffer + Str_clone of the elem type name)
    once per call. 228B / 9 allocs total. Root cause: the parser and
    typer both unconditionally marked variadic / kwargs params as
    ~OwnType.Own~ (parser.til ~line 238/226, typer.til line 2356/2359
    in ~infer_func_def_expr~). The convention works for regular til
    funcs because the codegen auto-inserts ~Array.delete(parts)~ /
    ~DynMap.delete(opts)~ at the callee's scope exit. ~ext_func~ /
    ~ext_proc~ bodies are user-written C with no auto-cleanup, so the
    caller's ~_va_Array_NNN~ / ~_kw_NNN~ local "transferred ownership"
    to a callee that never freed it -- ~insert_post_stmt_deletes~
    skipped the local's last_use cleanup (own_transfer.gte(0)),
    expecting the callee to free. Fix: at both sites
    (parser parse_func_def variadic/kwargs branches + typer
    infer_func_def_expr's param-walk override), keep variadic/kwargs
    params at ~OwnType.Ref~ when the surrounding func is ext_func or
    ext_proc, leaving the caller as owner. The normal ASAP delete at
    ~_va_Array_NNN~'s last_use then fires. test_asan: 109/117 ->
    110/117. Same one-line guard plugs the bug class for any future
    ext-variadic/ext-kwargs callsites without per-signature
    bookkeeping.

  - **~ext_libc_run~ ~LiteralStr.field~ wrapper leak** (FIXED). The
    binding pattern ~setenv("TIL_ISSUE_176".c_str, ...)~ leaked one
    ~sizeof(Str)~ (16 bytes) per ~"...".c_str~ site -- three
    setenv/getenv/unsetenv args plus the setenv value arg = 64B / 4
    allocs. Root cause: ~emit_str_lit_expr~ in
    [[file:../../src/self/builder.til][src/self/builder.til]] always emits a heap-allocated wrapper
    (~({ Str *_lit = malloc(sizeof(Str)); *_lit = (Str){...}; _lit; })~)
    so callsites that consume the literal as ~own val~ (e.g.
    ~Array.set~, ~Vec.push~) can ~free(own val)~ it. For a
    ~LiteralStr.field~ read (no ownership transfer), the wrapper is
    gratuitous and leaks. Fix: special-case ~NodeType.FieldAccess~
    over a ~LiteralStr~ child in ~emit_expr~ to emit a stack
    compound literal with dot-access:
    ~(Str){.c_str = (void *)"...", .count = N, .cap = TIL_CAP_LIT}.field~.
    No malloc, no leak. test_asan: 111/117 -> 112/117. Compiler
    baseline unchanged (82,960B / 16,571 allocs on
    ~bin/til_asan interpret test/strings.til~).

  - **~ref~ FieldAssign-to-Ref-field leak** (FIXED). ~test/ref.til~'s
    ~test_ref_struct_field~ leaked b's 16B RefNode wrapper. Source:
    ~a := RefNode(value=10); b := RefNode(value=20); a.next = b~,
    where ~next~ is declared ~ref mut next : RefNode = NULL~. b is
    borrowed into a -- a keeps b alive but doesn't own it. Two bugs:
    (1) ~expr_transfers_own~ in [[file:../../src/self/typer.til][src/self/typer.til]] treated any
    non-Shallow FieldAssign as a transfer, so b's scope-end delete
    was skipped (own_transfer.gte(0) blocks it in
    insert_post_stmt_deletes). Master leaked b silently. Fixed by
    narrowing to ~is(OwnType.Own)~ -- Ref fields take a borrow, not
    an ownership transfer. (2) Once (1) was fixed, b's last_use was
    the FieldAssign stmt itself, so the typer emitted
    ~RefNode_delete(b)~ right after ~a.next = b~ -- dangling
    ~a.next~ for the later asserts (and for ~c := a.clone()~'s
    ~c.next~ alias). Fixed by extending b's last_use through the
    end of the function body: any FieldAssign-to-Ref-field where
    the RHS is an Ident extends that local's last_use to
    ~body.children.len() - 1~. Conservative (b lives through
    every shallow-clone alias of the holder), but safe. Pinning a
    tighter bound would require alias analysis of clones.
    test_asan: 112/117 -> 113/117. Compiler baseline: 82,960B/16,571
    -> 83,227B/16,638 (+267B/+67 allocs from the extension's Str
    clones; acceptable).

  - **~dynamic~ test_dynamic_local malloc leak** (FIXED). 4B/1 alloc.
    ~test/dynamic.til~'s ~test_dynamic_local~ allocated a 4-byte
    buffer via ~buf : Dynamic = malloc(4)~ and never freed it.
    ~Dynamic~ is the typer's escape hatch for raw ~void*~ -- the
    type system intentionally doesn't track its lifetime
    (~type_to_name(Type.Dynamic)~ returns ~""~, so
    ~make_delete_call~ produces no cleanup; the binding's
    ~own_type~ stays Shallow because the malloc-RHS Dynamic decl
    isn't recognized as heap-owned by ~promote_own_transferred_locals~,
    which only promotes Struct/Enum locals). Auto-freeing arbitrary
    Dynamic locals is unsafe -- the same ~void*~ slot could hold a
    non-heap pointer (e.g. ~ref x : Dynamic = vec.get(i)~), and the
    typer can't tell the difference. Same rule as Str views from
    ~Str.substr~: cleanup of opaque-pointer-typed locals is the
    user's responsibility. Fix: add ~free(own buf)~ to the test.
    test_asan: 113/117 -> 114/117.

  - **~enums_lifecycle~ Vec.set inner Str leak** (FIXED). 2B/1 alloc.
    ~test/enums_lifecycle.til~'s ~test_vec_payload_clone~ replaced an
    existing element via ~bag1.items.set(0, own "x")~ where the
    previous slot held ~"a"~ (a Str with a 2-byte c_str). ~Vec.set~
    in [[file:../../src/core/vec.til][src/core/vec.til]] is plain
    ~memcpy(slot, val) + free(val)~ -- the prior slot's inner heap
    data (here Str.c_str) is silently leaked because the bytes get
    overwritten without running elem_delete on them first. Root cause
    is a Vec.set API design choice: drop-old vs bit-overwrite. The
    compiler's typer relies on bit-overwrite semantics at ~46+
    ~children.set(i, own X)~ sites where the prior slot was logically
    a placeholder (NoDefaultArg, Unfilled CtorArg) or had been moved
    out -- making Vec.set drop-old globally caused a 6x leak baseline
    regression and a Map.get panic in the interpreter (some sites
    transitively held shared pointers that elem_delete then double-
    consumed). Fix: add a new ~Vec.replace(i, own val)~ method that
    explicitly drops the old slot via elem_delete, leave ~Vec.set~
    as-is, and update the user-visible call site to use ~replace~.
    Same model as Rust's ~vec.swap_remove~ vs ~vec[i] = val~ split:
    bit-overwrite stays the low-level primitive, drop-old is opt-in.
    test_asan: 114/117 -> 115/117. Compiler baseline unchanged
    (88,498B/17,782 -> 88,702B/17,718 interp; 85,751B/17,444 ->
    86,267B/17,459 translate; +204B/-64 and +516B/+15 are noise).

  Non-leak ASAN findings (these are real correctness bugs the
  pre-split ~test_asan~ couldn't see -- the compiler runs ASAN-clean
  on these programs, only the compiled output trips ASAN):

  - ~test/vecs.til~ panic (FIXED): ~test_vec_dynamic~ pushed two
    Dynamic pointers to uninitialized 8-byte malloc'd buffers and
    relied on the resulting bytes differing across the two slots --
    Vec.push for an elem_type of Dynamic does ~memcpy(slot, val, 8)~,
    reading 8 bytes from the *pointee* (the malloc'd buffer's
    contents), not storing the pointer value itself. Under glibc
    malloc those buffers happened to contain different garbage so
    ~is(a, b)~ (4-byte compare) returned false; ASAN's allocator
    fills new allocations with a consistent magic pattern, making
    the two slots compare equal. Fix: ~memset~ each buffer with a
    distinct non-zero sentinel (170, 187) before pushing -- now the
    test verifies that distinct stored bytes round-trip through
    Vec.push/get regardless of allocator.
    Also fixed ~test_vec_pop~'s ~popped : I64 = v.pop()~ leak (8B/1
    alloc): Vec.pop returns a malloc'd Dynamic but the typer doesn't
    auto-free heap-owned primitives (only Struct/Enum locals are
    heap-promoted via ~promote_own_transferred_locals~). Same
    user-responsibility model as ~free(own buf)~ in test_dynamic_local
    -- added an explicit ~free(own popped)~. test_asan: 115/117 ->
    116/117.
  - ~svg2ttf_ok~ global-buffer-overflow (FIXED). ~build_post~ in
    ~lib/fonts.til~ constructs a 32-byte big-endian TTF "post"
    table header; precomp folds it to a Value.Struct(Str) which
    ~value_to_expr~ rewrites as a LiteralStr after escaping the
    runtime bytes back to source form (via ~str_to_source_form~).
    Every ~\xNN~ hex escape gets a trailing ~""~ (close-quote +
    open-quote) appended so the C lexer doesn't greedily consume
    the next char into the escape (e.g. ~\x03"" 0~ vs ~\x030~).
    Codegen's ~c_lit_byte_count~ in
    [[file:../../src/self/builder.til][src/self/builder.til]] is
    supposed to compute the actual byte count of the resulting C
    string (used for ~Str.count~), but treated the two ~"~ chars
    as regular bytes. The C compiler concatenates the two adjacent
    string literals and ~""~ contributes zero bytes to the global
    array -- the emitted ~Str.count~ was thus over-counting by 2
    for every hex escape, leading to ~Str.clone~ memcpy'ing past
    the end of the global. ASAN catches it as
    global-buffer-overflow (33-byte global, 34-byte read). Fix:
    in ~c_lit_byte_count~, skip two-char ~""~ sequences (advance
    by 2 without incrementing count). Same shape as the existing
    ~\xNN~ skip. **test_asan now 117/117 -- the compiled output
    of every test runs ASAN-clean.**
  - **Kwargs ~elem_type~ leak** (FIXED). ~test/kwargs.til~ compiled
    leaked 384 bytes / 24 allocations (16 bytes per kwarg-fcall
    invocation -- one ~sizeof(Str)~ per call). Symptom-level cause:
    ~build_kwargs_dynmap_set~ in [[file:../../src/self/typer.til][src/self/typer.til]] was calling
    ~set_own_arg~ on the ~elem_type~ arg of the generated ~DynMap.set~
    call, but ~DynMap.set~'s signature in [[file:../../src/core/dyn_map.til][src/core/dyn_map.til]]
    declares ~elem_type~ as a borrowed ~Str~ (no ~own~ qualifier).
    Builder honored the bogus own bit and emitted
    ~({ Str *_oa = malloc(sizeof(Str)); *_oa = literal; _oa; })~ for
    the arg; ~DynMap.set~ stored only the type-name *contents* inside
    the per-key Tuple (not the wrapper) and never freed the
    malloc'd wrapper.

    Root cause: every ~build_*_*~ desugar site (~build_kwargs_dynmap_set~,
    ~build_variadic_array_set~, ~build_builtin_vec_push~, Set-literal
    ~Set.add~ in ~desugar_set_literal_decl~, Map-literal ~Map.set~ in
    ~desugar_map_literal_decl~) was *hardcoding* per-arg ~set_own_arg~
    calls based on its own knowledge of each callee's signature.
    Synthesized FCalls are created during ~body_pre_passes~, *after*
    the per-stmt type-check that runs
    ~validate_fcall_own_args~/~infer_field_access_fcall~, so they
    never go through the typer's own-args consistency check; the
    desugar was the only enforcement and it could drift silently from
    the callee's actual signature (which is what happened with
    ~elem_type~).

    Fix: new ~sync_own_args_from_callee~ helper next to ~make_ns_call~
    in [[file:../../src/self/typer.til][src/self/typer.til]] that resolves the synthesized FCall's
    callee via ~scope.get_struct~ + ~find_namespace_func~ and ORs
    ~set_own_arg~ bits for each param whose ~own_type~ is
    ~OwnType.Own~. All 5 build sites drop their manual ~set_own_arg~
    litanies and end with one call to the helper, so "which positional
    args are own" is now derived from the callee's actual signature in
    [[file:../../src/core/dyn_map.til][dyn_map.til]] / [[file:../../src/core/map.til][map.til]] / [[file:../../src/core/vec.til][vec.til]] / [[file:../../src/core/set.til][set.til]] / [[file:../../src/core/array.til][array.til]] instead of
    redeclared at each desugar site. test_asan: 101/117 -> 102/117.
    Same fix plugs the entire bug class (any future signature change
    propagates automatically). The remaining ~set_own_arg~ callers
    (~make_delete_call~ -- conditional ~arg_is_own~ for the
    borrowed-wrapper-only delete pattern; ~make_free_call~ -- ~free~
    is a builtin Ident, not a namespace call; struct-ctor own-field
    handling at line 1054; UFCS bit-shift at 1789; pure-splat
    variadic at 7712) are intentionally outside the namespace-call
    shape the helper resolves.

  - ~issues_summary~: ~AddressSanitizer: heap-use-after-free~ at
    ~gen/c/issues_summary.c:380~ in ~I8_neq~ -- FIXED (workaround).
    Root cause: ~Str.substr~ returns a view (~.cap=CAP_VIEW~,
    ~c_str~ borrows the source's buffer) and the typer's ASAP-delete
    pass doesn't track view-borrow chains. In ~parse_issue~, the
    only direct use of ~content~ after the ~after_first~ substr was
    the substr itself, so the typer scheduled
    ~Str_delete(content)~ between the substr and its first use --
    dangling every downstream view (~after_first~,
    ~first_line~, ~title~, ~num_str~, ~after_num~). Even chaining
    ~content.substr(...).clone()~ doesn't help: the typer lowers the
    chain into separate C statements and inserts the delete *between*
    ~Str_substr~ and ~Str_clone~. Workaround: extract the parse body
    into ~parse_issue_body(fname, content: Str)~ -- borrowed params
    aren't subject to ASAP-delete, so ~content~ stays live for the
    whole body. Real fix is typer-level: teach
    ~extend_ref_local_lifetimes~ (or its equivalent) to recognize
    view-returning callees and extend the source's lifetime through
    the view's last use. Same fix would let the widespread
    ~substr(...).clone()~ idiom in ~src/til.til~ /
    ~src/self/builder.til~ / ~src/self/lexer.til~ be safe for owned
    locals too (currently only safe for borrowed sources like
    function params / struct fields).

  *** 2026-05 [[./175.org][#175]] series (DONE)

  All four landed under doc/self.org's two-pass workflow.

  1. ~Typer: insert exit-deletes through nested Body blocks~ (4f10e0d).
     ~insert_exit_deletes~ recurses into nested ~NodeType.Body~ stmts
     so the desugared switch sentinels (~_sw_NodeType_*~ etc.) get
     freed before deeply-nested early returns. -44% bytes / -43%
     allocs.

  2. ~[[./175.org][#175]] follow-up: extend save_old_delete to user-written .clone()
     RHS~ (7a2d233). The original [[./175.org][#175]] fix only fired for the
     implicit clone path; ~mut X := ""; X = Y.clone()~ in
     ~extend_ref_local_lifetimes~ and similar sites was missed
     because ~rhs~ is an FCall, not the auto-clone-wrapped
     ~Ident~/~FieldAccess~ shape. Also dropped the redundant
     ~stmt.til_type = existing.clone()~ from the original fix
     (already set from rhs.til_type). -28% bytes / -8% allocs.

  3. ~[[./175.org][#175]] follow-up 2: widen save_old_delete to any FCall returning
     OwnType.Own~ (064ef0d). ~mut X := ""; X = expect_text(...)~
     in ~parse_struct_def~ (and many ~parse_*~ siblings) is the
     same [[./175.org][#175]] outer-scope shape but rhs is an arbitrary
     heap-returning FCall. ~fcall_returns_own~ already exists in the
     typer to discriminate heap-returning callees from shallow-return
     ones, so the trigger now uses it directly. -7.8% bytes / -6.9%
     allocs.

  4. ~free the heap wrapper after value-copy ownership transfer to
     a Shallow struct field~ (ff6ce78). Struct-typed locals are
     heap-promoted by ~promote_own_transferred_locals~. When
     ownership is "transferred" via ~Decl(field = name, ...)~ where
     the receiving field is Shallow, the constructor only
     value-copies the inner struct; the field's own destructor frees
     the inner data, but nothing freed the heap wrapper.
     ~insert_post_stmt_deletes~ now emits ~free(own name)~ after
     such transfer stmts, gated by a new
     ~transfer_is_shallow_field_value_copy~ predicate so other
     transfer shapes (which already consume the wrapper) aren't
     double-freed. -13.9% bytes / -4.4% allocs.

  5. ~widen save_old_delete to LiteralStr RHS~ (a886407). Same
     outer-scope [[./175.org][#175]] shape but ~rhs~ is a literal string, e.g.
     ~mut ftype := fd.explicit_type; ftype = "I64"~ inside a nested
     ~if~ in ~compute_struct_layout~. ~Str_lit~ returns a heap Str
     pointer, so the existing ~save_old_delete~ codegen already
     handles it; just add a third ~else if~ branch in
     ~infer_assign_stmt~ alongside the FCall-returning-Own and
     auto-clone-wrap branches. -480 bytes / -44 allocations.

  6. ~skip heap-promotion for Struct local whose only use is a
     shallow return~ (this commit). When a function declared
     ~returns shallow X~ has a Struct local whose ~last_use~ is
     the matching ~Return~ stmt and ~own_transfer < 0~, the
     existing ~promote_own_transferred_locals~ pass would still
     heap-promote it (because ~Struct~ locals always promote).
     The codegen then emits ~Str *result = Str_lit("", 0ULL)~ +
     mutations + ~return DEREF(result)~ -- value-copies the
     struct out and orphans the wrapper. Concrete site:
     ~str_view~ in [[file:../../src/self/interpreter.til][src/self/interpreter.til]], hit on every
     ~dyn_fn~ / ~dyn_size_of~ dispatch (and there are a lot more
     of those after [[file:../../issues/fixed/190.org][#190]]'s ~Vec.new~ migration).
     ~promote_own_transferred_locals~ now skips promotion when the
     last_use stmt is a ~Return~ that returns the local by name and
     a new ~typer_in_shallow_return~ flag (set/restored in
     ~infer_func_def_expr~) confirms the surrounding func is
     shallow. The result is a stack-typed local; the codegen
     emits ~Str result = ...; ...; return result;~ which
     value-copies cleanly with no wrapper to leak.
     -145 bytes / -5 allocations vs the post-Vec.new-regression
     baseline. Small per call but eliminates the ~str_view~ wrapper
     leak that compounded across every dispatch site, and applies

    that explains which are safe** (2026-05, after the bare-FCall /
    enum-payload fixes landed). With those two fixes merged
    (interpret -3.0% / translate -2.3% per asan.org), re-attributed on
    the new master (interpret 123548 / translate 120261, default ctx).
    The remaining top interpreter clusters are ~make_str_value~ Str
    values reached via ~eval_body~ / ~eval_call~ / ~try_eval_call~.
    Tried two more frees, both REVERTED:

    1. ~scope_free~ deep-free. ~scope_free~ (interpreter.til) only frees
       ~Value.Enum~ data per local binding; ~Value.Struct~ (incl. Str
       with its c_str) is skipped, so every owned struct/Str at any
       freed scope -- most visibly the top-level decls in the ~global~
       scope freed at ~interpret_units~' end (~3.3KB) -- leaks.
       (~cleanup_value_payload~'s own doc comment even claims it is
       "Used by scope_free", but scope_free uses an inline Enum-only
       switch.) Swapping in ~cleanup_value_payload(b.cell.val)~ ->
       SEGFAULT during self-compile; the milder ~free_value(b.cell.val)~
       (frees the struct wrapper but not c_str) -> ~malloc_consolidate():
       unaligned fastbin chunk~ heap corruption. Cause below.

    2. Free the ~str_view~ source temporaries in ~h_dyn_fn~ /
       ~h_dyn_size_of~ (dispatch.til). These do
       ~tnv := eval_expr(...); name := str_view(tnv)~ and never free
       ~tnv~ (~256B+ / 16 objs). Adding ~cleanup_value_payload(tnv)~
       after the last use -> PANIC "dyn_fn: unknown .clone" during
       self-compile (a *later* call read an empty/freed type_name).

    Root cause (the #127 core): ~shallow_copy_value~ -- used by
    ~eval_expr~ for *every* Ident (~return shallow_copy_value(cell.val)~)
    and by scope binding -- is a raw ~memcpy(r, v, Value.size())~. It
    copies the ~Value~ struct, i.e. the ~si.data~ POINTER, so copies
    SHARE the heap payload with the source cell. The ~borrowed~ flag is
    not set on these shares. Freeing any ~eval_expr~ result (or any
    scope-bound struct) therefore frees data still owned by the
    originating variable -> UAF / double-free / corruption.

    Why the two LANDED fixes are nonetheless safe, and these are not:
    they free only where the TYPER GUARANTEES an ownership transfer to
    that position. A bare call-statement's discarded return is owned
    (or borrowed-flagged, which cleanup_value_payload skips). An enum
    payload built from a variable is a typer-enforced MOVE -- verified:
    ~b := Box.Tagged(s); print(s)~ is rejected with "use of 's' after
    ownership transfer", so the source is provably dead and the enum is
    the sole owner. ~h_dyn_fn~'s ~str_view~ args and ~scope_free~'s
    bindings have NO such transfer -- they are borrowed/shared -- so
    freeing them corrupts the owner. RULE: only free an interpreter
    Value at a site where the typer marked the feeding expression
    ~own~ / transferred; never free a bare ~eval_expr~ result or a
    scope binding's struct payload until #127 gives Values real
    ownership (deep-copy in ~shallow_copy_value~, or a reliable
    per-Value ~owns_data~ flag distinct from ~borrowed~). That #127
    change is the unblock for the entire interpreter-side cluster
    (scope structs/Strs, FFI args, dispatch temps).

  *** Lessons learned

  - **Leak attribution = allocation site, not ownership-loss site.**
    Bucket C looked like it was caused by a clone inside
    ~insert_exit_deletes~ but the leak was in ~lp.ast~ retention
    (Bucket D). When a fix at the attributed site doesn't move the
    baseline, look one layer up.

  - **Two-pass workflow** (~make test_two_pass~, "Two-pass:" commit
    prefix; see ~doc/self.org~). Use whenever a fix changes which
    delete calls the typer emits. The new emit lands in boot/til.c
    atomically with the source change.

  - **Verify the per-test ASAN delta after every commit.** Test
    pass count alone doesn't tell you whether a leak fix moved the
    needle. ~bin/til_asan interpret examples/cli_args.til hello
    world~ is the cheapest signal.

  - **Heap-promoted Struct locals carry two cleanup decisions:**
    (a) the inner data and (b) the wrapper. The "ownership
    transferred" tracking handles (a) correctly but used to leave
    (b) leaking when the transfer was value-copy to a Shallow
    field. Recently fixed; the same shape may recur in other
    transfer patterns.

  - **Import order can move the ASAN baseline by tens of percent**
    (alphabetizing core.til's imports in 9d0b82d cut strings.til
    interpret leaks by 47%). Likely a teardown-order effect that
    causes more owned values to be freed before exit. Never
    attributed to a specific code path; worth a follow-up if a
    similar import reorder lands again.

  - **~Vec.set~ blindly overwrites the slot without ~elem_delete~,
    but adding a naive deep-delete before the set regresses
    sibling paths.** Bucket 1 records this for ~FieldAssign~ on
    owned-typed fields; the same shape applies to
    ~Vec.set(stmt_idx, own block.clone())~ in
    ~replace_body_stmt_with_block~ (called from
    ~desugar_for_in_range_stmt~ /
    ~desugar_for_in_collection_stmt~). The pre-fix ASAN leak from
    that site on ~strings.til translate~ was ~347 bytes / 27
    objects; adding ~Expr.delete(own body.child(stmt_idx),
    false)~ before the set regressed the total from 84193 ->
    105730 bytes, with ~+22KB of new leaks reported under
    ~desugar_user_func_fcall_args~, ~infer_field_access_fcall~,
    and ~infer_literal_expr~. Confirms bucket 1's "naive auto-
    delete chains through ~stmt.til_type = existing.clone()~
    -shaped sibling leak sources"; same opt-in flag + move-vs-
    value-copy discriminator is needed here too. Tests stayed
    115/115 so the regression is leak-only -- not a correctness
    break.

  - **A "mystery" leak that should already be freed by an
    existing cleanup loop is often a leak the next master commit
    plugs.** Spent a session staring at a
    ~make_str_value_view -> expr_to_value -> is_known ->
    track_literal_known_only~ leak where ASAN said ~1.4KB / 88
    objects leaked from the 16-byte ~Str~-struct allocation, but
    every code path I could trace said
    ~cleanup_value_payload~'s end-loop in ~precomp_imported~ /
    ~process_program~ should free them. Considered adding
    counters / a gdb watchpoint to find the gap.  Next rebase on
