# priority: high
** OPEN #204 Asap destruction is its own phase, not part of the typer

   [[./158.org][#158]] sets the direction: one AST walk per phase, with
   later phases reading caches the earlier phases left on the
   ~Context~. The typer phase's job is to resolve names, infer types,
   and reject invalid programs (see [[../../doc/pipeline.org][doc/pipeline.org]]).

   Today ~infer_body~ / ~infer_body_unit~ in
   ~src/self/typer.til~ also runs ~insert_free_calls~ at the end of
   every body. That call drives the whole asap-destruction machinery:

   - ~collect_scope_locals~ - per-binding live-range bookkeeping
   - ~extend_ref_local_lifetimes~ - propagate Ref aliasing
   - ~check_use_after_own_transfer~ - diagnose use-after-move
   - ~promote_own_transferred_locals~ - flip Decl ownership to Own
     where a stack value would leak
   - ~insert_exit_deletes~ /
     ~insert_nested_exit_deletes~ /
     ~insert_exit_deletes_into_stmt~ - emit deletes on
     return/break/continue paths
   - ~insert_post_stmt_deletes~ - emit the per-stmt last-use cleanup
   - ~insert_assign_delete~ - emit the pre-reassign delete
   - ~insert_free_calls~ - the orchestrator

   That's a sizable second pass over every body, with its own data
   model (~LocalInfo~) and several helpers (~field_assign_needs_delete~,
   ~alias_used_in_stmts~) -- all wedged into the typer for historical
   reasons. It muddies the pipeline diagram, it doubles the typer's
   surface area, and it makes it hard to talk about "the typer walk"
   in [[./158.org][#158]] terms.

   Goal: make asap destruction its own phase that runs after
   ~type_check~ and before ~precomp~ in
   [[../../doc/pipeline.org][doc/pipeline.org]].

   Plan, smallest safe steps first:

   1. DONE. Extracted the asap code into ~src/self/garbager.til~ as its
      own ~mode lib~ module. Call sites stayed in ~infer_body~.

   2. DONE. Documented the new phase in
      [[../../doc/pipeline.org][doc/pipeline.org]] and updated
      [[../../doc/facts.org][doc/facts.org]] to say "garbager phase".

   3a. DONE. Decoupled ~infer_body~ from asap. The proc no longer
       calls ~insert_free_calls~; the scope-management sites in
       ~typer.til~ (infer_func_def_expr, infer_if_stmt,
       infer_while_stmt, the Body/Catch arms of infer_body_stmt, and
       type_check_unit / type_check for the program body) now invoke
       ~garbager_destroy_body(body, scope)~ from garbager.til just before
       tearing the per-body scope down. ~infer_body~ is now purely
       in-place type inference + ~body_pre_passes~ + ~reregister~.

   3b. PARTIAL. Lifted the program-body asap call into a new
       ~garbager_destroy(ctx)~ entry point in ~src/self/garbager.til~ and
       wired it into the tail of ~init_and_type_program~
       (loader.til) so both ~prepare_program~ and ~cmd_doc~ see the
       post-asap program AST. ~type_check_unit~ / ~type_check~ no
       longer call ~garbager_destroy_body~ on the program body; that
       work happens once at the end of typing instead of once per
       file. The walk uses ~ctx.scope~ as the body's scope (it
       already lives on ~Context~), which is why the program body
       could move without scope-cache infrastructure.

       Sub-body asap (function bodies + nested if/while/block/catch
       scopes) still runs inline from the typer's scope-management
       sites in ~typer.til~. Per-body scopes are stack locals freed
       by ~TypeScope.delete~ once ~infer_body~ returns, so they need
       to either: (a) become first-class entries on ~Context~
       ([[./158.org][#158]] cache work) so ~garbager_destroy~ can re-walk
       the AST and look scopes up by body identity; or (b) be
       rebuilt on the fly inside ~garbager_destroy~ from each body's
       Decls + the enclosing FuncDef's params + ~ctx.scope~ as the
       parent. (a) composes better with [[./158.org][#158]].

   3a removes the asap coupling from the typer's per-body machinery;
   3b finishes [[./158.org][#158]]'s "one walk per phase" by turning
   asap into a single top-level walk after type_check.
