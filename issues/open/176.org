# priority: high
** WIP #176 FFI: support const char* params (avoid libc prototype conflicts)

Symptom:

   - Declaring libc functions like ~setenv~ as ~ext_func~ in Til can fail C
     compilation with "conflicting types" because generated code emits a
     prototype that does not match system headers.

   Root cause:

   - Per ~doc/qualifiers.org~, til qualifiers currently do not generate C
     ~const~ qualifiers at all. Non-~mut~ args still emit ~Type *arg~, not
     ~const Type *arg~.
   - libc headers declare many APIs with ~const char *~ (e.g. ~setenv~). If til
     codegen also emits a prototype for the same symbol but without ~const~ (or
     with a different pointer base type like ~U8 *~), the C compiler rejects it
     as a conflicting declaration.

   Current workaround:

   - Provide a thin wrapper in [[file:../../src/c/ext.c][src/c/ext.c]] (e.g. ~til_setenv~) and bind that
     from Til instead of binding libc ~setenv~ directly.

   Desired fix:

   - Add const-correct C emission:
     - When an argument is non-~mut~, emit ~const Type *~ where safe.
     - Provide an escape hatch for APIs that legitimately mutate through a
       borrowed pointer.
   - Extend FFI/type mapping to cover common C string pointer shapes:
     - ~const char *~ and ~char *~ (distinct from ~U8 *~ / ~I8 *~).
   - Reduce collisions with system headers:
     - Either avoid emitting prototypes for known libc symbols when the correct
       header is present, or ensure emitted prototypes exactly match the libc
       signature.

** Progress

   Done (commits on this branch):

   - ~is_skip_ext_decl~ in ~src/self/builder.til~ now suppresses emitted
     prototypes for ~setenv~, ~getenv~, ~unsetenv~, ~putenv~, ~system~,
     ~atoi~, ~atol~, ~atof~ so the libc declarations in ~stdlib.h~ are not
     overridden by a conflicting til-generated prototype.
   - At each call site of those libc functions, the generated C emits
     explicit ~(char *)~ casts around ~U8~ / ~I8~ ref arguments and wraps
     the call expression in a ~(U8 *)~ / ~(I8 *)~ cast when the libc
     prototype returns a ~char *~ (e.g. ~getenv~). The casts make the
     U8/char and unsigned/const conversions explicit, so the generated
     code compiles under ~-Wall -Wextra -Werror~ without any new
     diagnostics being silenced. Helpers ~is_libc_str_fn~,
     ~callee_param_needs_char_cast~ and ~libc_call_return_cast~ in
     ~src/self/builder.til~ drive this.
   - Const-correct C emission for ~ext_func~ / ~ext_proc~ prototypes:
     ~emit_param_list~ takes a ~for_ext_decl~ flag and, when true,
     prepends ~const~ to non-~mut~ Ref pointer params. Variadic /
     kwargs / shallow / own params are unchanged. ~mut~ remains the
     escape hatch. ~emit_ext_func_declarations~ also detects when an
     ~ext_func~ name is shadowed by a regular ~func~ / ~proc~
     definition (the forward-decl-via-ext_func trick used in
     ~src/self/builder.til~ for ~callee_return_ctype~ /
     ~resolve_callee_name~) and skips emitting a redundant prototype
     in that case -- the func body's own forward-decl emission is
     authoritative. For example, ~greet := ext_func(name: Str) returns
     Str {}~ in ~examples/ffi/ext_module.til~ now emits
     ~Str * greet(const Str *)~, and ~InitWindow(... title: U8)~ in
     ~src/lib/raylib.til~ emits ~void InitWindow(I32, I32, const U8 *)~,
     while ~callee_return_ctype~ in ~src/self/builder.til~ keeps just
     the func-body forward decl with no const (matching the body).
   - Bullet 2 ("FFI types ~CStr~ / ~char *~ distinct from ~U8 *~ /
     ~I8 *~") is covered by til's existing ~I8~ primitive: per
     ~src/c/ext.h~, ~I8~ is a typedef for C ~char~, so ~ref I8~ params
     emit ~I8 *~ (= ~char *~) and ~const I8 *~ (= ~const char *~) with
     the const-correct emission. No new primitive type is needed.
     ~U8~ stays as ~unsigned char~ for raw byte buffers and the
     callee-side cast helpers in ~src/self/builder.til~ bridge the two
     where users pass ~Str.c_str~ (~U8 *~) into libc functions.

   Regression test ~test/ext_libc.til~ binds ~setenv~, ~getenv~,
   and ~unsetenv~ directly via ~ext_func~ and asserts the round-trip
   works. 117/117 tests pass with const emission turned on.

   Not done yet:

   - The const-correct ~ext_func~ emission only fires for the ~units~
     side (user file + mode auto-import + transitive imports through
     them, which covers ~src/lib/*~ and ~src/modes/*~). Bindings
     declared in ~src/core/*.til~ (loaded via core auto-import) get
     their C prototypes from the hand-written ~src/c/ext.h~, which
     still declares them without ~const~. Making that fully
     const-correct means updating ~src/c/ext.h~ and the matching
     ~src/c/ext.c~ definitions by hand. The mechanical sweep is
     entirely additive (no behavioural change at the ABI level) but
     spread across many functions; left for a separate change.
