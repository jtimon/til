** OPEN #116 Imported files are parsed in isolation

Imported files are currently lexed and parsed as standalone units, then
  merged later. They do not see declarations from the importing file during
  parse/bootstrap time.

  Current behavior in ~src/til.til~:
  - parse one file
  - extract ~import("...")~ calls from its AST
  - recursively read, lex, and parse each imported file separately
  - only after that merge:
    - core
    - core imports
    - mode imports
    - user imports
    - user file

  This means import order in the source tree does not imply visibility while
  parsing imported files. Imported files are self-contained until the later
  AST merge.

  Concrete blocker example:
  - we tried to unify ~StructDefData~ with the existing ~StructDef~ type
  - the natural cleanup would be to define one canonical ~StructDef~ in
    ~core.til~ and use it as the ~ExprData.StructDef~ payload type in
    ~ast.til~
  - that fails during bootstrap, because ~ast.til~ is parsed in isolation and
    cannot see ~StructDef~ from ~core.til~ at the time ~ExprData~ is declared
  - moving the ~import("ast.til")~ line within ~core.til~ does not help,
    because imported files are not parsed in the importer’s declaration
    context

  What this blocks:
  - unifying local AST payload helper types with canonical types defined in
    importer files
  - some circular/meta-type cleanups that look valid after merge but fail
    during isolated parse/bootstrap
  - relying on importer-local predeclarations to shape imported AST types

  Relation to [[./122.org][#122]] (cancelled):
  Both [[./116.org][#116]] and [[./122.org][#122]] stem from the same root cause -- imported files
  not sharing context with the importer. [[./122.org][#122]] was the typing-phase
  symptom (global ~current_mode~ applied to all files), [[./116.org][#116]] is the
  parsing-phase symptom (files can't see each other's declarations).
  [[./122.org][#122]] was solved by per-path mode tracking (~context_set_mode_from_path~
  in 55ffae8, file-declared mode priority in f0a7aaa). [[./116.org][#116]] remains
  open because parse-time isolation is a deeper architectural constraint.
  Any future mode-sensitive typing rule (func/proc exemptions, pure/pura
  restrictions, etc.) is already handled by the mode infrastructure.

  Likely directions:
  - keep such canonical payload types local to the imported file for now
  - or introduce a true predeclared/builtin meta-type path
  - or change import/bootstrap semantics so imported files can see a shared
    prelude / prior declarations before parsing

  Immediate outcome:
  - keep ~StructDefData~ for now
  - treat the failed ~StructDef~ unification as a symptom of the import model,
    not just a naming conflict
