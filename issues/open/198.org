   Port rstil's exception/error system to til (~throw~, ~catch~,
   ~throws~, postfix ~?~ on calls) and gate it on a single shared
   lowering so the same implementation serves both the interpreter
   ([[file:../../src/self/interpreter.til][src/self/interpreter.til]]) and the builder/ccodegen
   ([[file:../../src/self/builder.til][src/self/builder.til]]) paths.

   Reference: rstil at https://github.com/jtimon/rstil. Canonical
   surface tests live in [[https://github.com/jtimon/rstil/blob/master/src/test/errors.til][rstil's src/test/errors.til]] -- copy that
   file (or its core subset) verbatim into ~src/test/errors.til~ and
   make it pass on both pipelines.

   *** Surface syntax (mirror rstil)

   Identical syntax to rstil:

   #+begin_src til
   MyError := struct { mut msg: Str = "" }

   maybe_fail := func(x: Bool) throws MyError {
       if x { throw MyError(msg="boom") }
   }

   caller := proc() {
       maybe_fail(true)?
       catch (err: MyError) { println(err.msg) }
   }
   #+end_src

   - ~throws T1, T2, ...~ on a func/proc signature declares the set
     of error types it can raise.
   - ~throw E~ raises an error of type E (a Str literal, an I64,
     or a struct instance). The raised type must be in the enclosing
     function's ~throws~ set or be caught locally by a downstream
     catch block.
   - Postfix ~?~ on a call (~f(x)?~) is the mandatory annotation on
     calls to throwing functions. It is *not* propagation by itself;
     whether the error is caught locally or propagates up is decided
     by the presence of a matching catch block after the call.
   - ~catch (err: T) { body }~ matches the most recent pending throw
     of type T in the enclosing body and binds ~err~ as a const local
     of type T inside the catch body.

   Defer the bang ~!~ operator (rstil's "panic on throw" sugar) to a
   follow-up. Not required for the initial port.

   *** Lexer / parser additions

   Mirror rstil's [[file:../tmp/rstil/src/rs/parser.rs][parser.rs]] / [[file:../tmp/rstil/src/rs/lexer.rs][lexer.rs]]:

   - New token types: ~Throw~, ~Catch~, ~Throws~, ~QuestionMark~.
   - Func/proc signature parser accepts an optional
     ~throws T1, T2, ...~ clause after the returns clause and stores
     it on the FuncSig as ~throw_types: Vec<ValueType>~ (til:
     ~throw_types: Vec[TilType]~ on FuncSig in [[file:../../src/core/ast.til][src/core/ast.til]]).
   - FCall nodes gain a ~does_throw: Bool~ flag set when the call is
     followed by ~?~. The parser sets it; the typer validates it.
   - New AST node types ~Throw~ (1 child = thrown expression) and
     ~Catch~ (3 children: Ident var name, Ident type name, Body).
     See ~parse_throw_statement~ / ~parse_catch_statement~ in rstil's
     parser.rs for the exact shapes.

   *** Typer responsibilities (validation + lowering)

   This is where til diverges from rstil. In rstil the throw/catch
   *lowering* lives in [[file:../tmp/rstil/src/rs/ccodegen.rs][ccodegen.rs]] (~440 lines of error_handling
   helpers around hoisted call sites, error label tables, goto
   jumps, struct local hoisting for catch-safe destructors, etc.).
   The interpreter has its own runtime mechanism (an
   ~EvalResult { is_throw, thrown_type, value }~ variant + a
   ~pending_throw~ state machine in ~eval_body~). Two separate
   implementations of the same semantics, kept in sync by hand.

   In til we instead lower throw/catch *in the typer*, before
   precomp / initer / interpreter / builder run. After the typer,
   the AST contains no Throw or Catch nodes -- only ordinary til
   constructs that every downstream stage already knows how to
   evaluate and emit.

   Validation pass (port from [[file:../tmp/rstil/src/rs/typer.rs][rstil typer.rs]] ~check_body_returns_throws~,
   ~check_fcall~, the ~thrown_types~ accumulator):

   - For each function body, walk in order tracking a list of
     ~thrown_types~ active at the current point.
   - ~Throw E~: append ~typeof(E)~ to ~thrown_types~. Reject throwing
     a struct *definition* (require an instance).
   - FCall ~f(...)?~: append each type in ~f.sig.throw_types~ to
     ~thrown_types~. Require ~?~ iff ~f.sig.throw_types~ is non-empty;
     reject ~?~ on non-throwing calls and on constructors.
   - ~Catch (err: T) { body }~: require T be in the current
     ~thrown_types~; remove all entries equal to T; type-check the
     body with ~err: T~ in scope (and field symbols ~err.field~ for
     struct types). Code after a catch is reachable again
     (Bug #96 in rstil -- carry the same fix forward).
   - At end of function: any leftover ~thrown_types~ that is not in
     this function's declared ~throws~ set is a type error. Any
     declared ~throws~ entry that was never raised is also a type
     error (rstil enforces both directions; do the same here).

   Lowering pass (replaces rstil's ccodegen error-handling block):

   For each function body that contains throw/catch or throwing
   FCalls, rewrite to plain til as follows.

   1. At function entry, declare local error state:

      #+begin_src til
      mut _err_kind : I64 = 0           // 0 = no error
      mut _err_T1 : T1 = T1()           // one slot per declared throw type
      mut _err_T2 : T2 = T2()
      ...
      #+end_src

      Each declared error slot gets a stable numeric tag. For
      function-local types raised but caught entirely before they
      leave the function, the slots are local-only. For declared
      ~throws~ types, the slot is part of the function's outbound
      error channel.

   2. ~throw E~ (where E has type T) becomes:

      #+begin_src til
      _err_T = E
      _err_kind = <tag for T>
      // followed by an early exit appropriate to the surrounding
      // construct: see step 4
      #+end_src

   3. ~f(args)?~ becomes a plain call ~f(args, &_err_kind, &_err_T1, ...)~
      style invocation against the lowered signature: throwing
      callees gain extra out-parameters (or, equivalently, an
      ~_err_*~ struct argument) for each declared throw slot, and
      callers reuse their own ~_err_*~ slots. Concretely, mirror
      the rstil ccodegen ABI but expressed in til-level AST nodes
      that the existing builder.til already emits unchanged.

   4. ~catch (err: T) { body }~ at the end of a body becomes:

      #+begin_src til
      if I64.eq(_err_kind, <tag for T>) {
          err := _err_T
          _err_kind = 0          // consumed
          { body }
      }
      #+end_src

      For multiple catches in sequence, generate a chain of ~if~
      branches with the same shape.

   5. Code *between* a throwing call and its catch (Bug #34 in
      rstil's tests) must not run on the error path. Lowering wraps
      the inter-call statements in ~if I64.eq(_err_kind, 0) { ... }~
      so they are guarded by the no-error condition. Equivalently,
      the typer can synthesize a ~Body~ wrapper and fold subsequent
      stmts into it -- whichever shape composes most cleanly with
      the existing typer passes (~insert_exit_deletes~,
      ~promote_own_transferred_locals~, etc.).

   6. Function exit / propagation: at every return point of a
      throwing function, if ~_err_kind != 0~ the lowered code
      copies ~_err_*~ to the caller-provided error out-slots and
      returns; otherwise the normal return value flows out. This
      is identical in spirit to rstil ccodegen's "propagate or
      goto local catch label" decision, just expressed at the
      til-AST level.

   The post-lowering AST uses only: declarations, mut locals,
   assignments, ~if~, calls, returns, and field access. No new node
   kinds reach the interpreter or builder.

   *** Interpreter changes

   With the lowering done in the typer, [[file:../../src/self/interpreter.til][src/self/interpreter.til]]
   ideally needs *zero* throw-specific code. The interpreter just
   evaluates the rewritten AST.

   Two small interpreter-side concerns to verify:

   - The lowered ABI uses extra out-parameters on throwing
     functions. Make sure the interpreter's call dispatch handles
     the extra params naturally (it should -- they are normal
     ~mut ref~ params after lowering).
   - Make sure ~f(...)?~ removed from the AST by lowering doesn't
     leave a stale ~does_throw~ flag that any later stage trips on;
     either clear it during lowering or keep it as a no-op hint.

   Contrast with rstil's interpreter.rs: that file gained an
   ~EvalResult.is_throw~ + ~pending_throw~ control loop in
   ~eval_body~ (~150 lines, ~lines 2003-2152). With desugaring in
   the typer, we drop this entirely.

   *** Builder / ccodegen changes

   [[file:../../src/self/builder.til][src/self/builder.til]] should also need *no* new error-specific
   logic. The lowered AST uses constructs (mut locals, if, mut ref
   params, return) the builder already emits. This is the whole
   point of doing the desugaring in the typer: rstil's
   ccodegen.rs ~440-line throw block (hoisting, label tables,
   goto, struct local pre-declaration for catch-safe destructors,
   ~_ret~ out-pointer plumbing) collapses to "the typer made it
   plain til, the builder emits plain til."

   One thing to watch: rstil's ccodegen pre-declares struct locals
   at the top of any function with a catch (Issue #117 there) so
   that goto-into-catch doesn't skip past their constructor. In our
   lowered form there is no goto -- catches are ~if~-guarded
   sequential blocks -- so this concern goes away. But verify on
   the ASAN run that locals declared between a throwing call and
   its catch get destroyed correctly on the error path. The
   existing ~insert_exit_deletes~ / ~collect_scope_locals~
   machinery should handle it; if not, the fix lives in the typer
   (extend insert_exit_deletes to follow the lowered if-branches),
   not in builder.til.

   *** Why typer-side lowering vs rstil's ccodegen-side

   - One implementation, two pipelines. Today rstil maintains the
     same semantics in two places (interpreter.rs ~pending_throw~
     loop + ccodegen.rs goto/label generator) and they have
     drifted in the past (Bug #34 was an asymmetry between them).
     A single AST rewrite in the typer can't drift from itself.
   - The downstream stages stay simple. builder.til already emits
     mut locals, if, and out-params; interpreter.til already
     evaluates them. No new node kinds = no new code in either.
   - The typer already has the infrastructure for AST rewrites
     (~insert_exit_deletes~, ~promote_own_transferred_locals~,
     desugaring of switch and forin, etc.). Adding a throw-lowering
     pass fits the existing pattern.
   - It composes with the existing leak/destructor passes for free:
     the lowered locals are ordinary locals and ~insert_exit_deletes~
     handles them on every exit edge.
