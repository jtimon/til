# priority: low
** OPEN #257 Centralize primitive size/align: stop hardcoding I64=8, U8=1, ... in layout code

  *Symptom / smell.* The primitive byte size and alignment of every
  builtin scalar is hardcoded as a literal in multiple places instead
  of being looked up from a single source of truth. The clearest
  example is the inline cascade inside ~compute_struct_layout~
  (~src/self/initer.til:1671-1696~):

  #+begin_src til
                if ftype.len().eq(0) { fsz = 8; falign = 8 }
                else if ftype.eq("I64") { fsz = 8; falign = 8 }
                else if ftype.eq("U8") { fsz = 1; falign = 1 }
                else if ftype.eq("I8") { fsz = 1; falign = 1 }
                else if ftype.eq("I16") { fsz = 2; falign = 2 }
                else if ftype.eq("U16") { fsz = 2; falign = 2 }
                else if ftype.eq("I32") { fsz = 4; falign = 4 }
                else if ftype.eq("U32") { fsz = 4; falign = 4 }
                else if ftype.eq("F32") { fsz = 4; falign = 4 }
                else if ftype.eq("Bool") { fsz = 1; falign = 1 }
  #+end_src

  We already expose ~I64.size()~, ~U8.size()~, ... at the language
  level (the interpreter helper ~elem_size_for_type~,
  ~src/self/interpreter.til:475-485~, *already* calls them), so the
  layout code is re-deriving knowledge it could ask for.

  *Where the duplication lives.* The same primitive size/align table
  is spelled out independently in at least three spots:

  1. ~src/self/initer.til:1671-1696~ -- inline cascade in
     ~compute_struct_layout~ (sets ~fsz~/~falign~ per field).
  2. ~src/self/initer.til:1715-1742~ -- ~type_size_align(ftype, scope,
     ctx, sz, al)~, the proc used by ~compute_enum_layout~ for payload
     sizing. Same literals, slightly different type coverage (this one
     also handles ~U64~/~F32~/~Dynamic~; the cascade above omits ~U64~
     entirely and instead leans on the ~ftype.len().eq(0)~ default of
     8, which is fragile).
  3. ~src/self/interpreter.til:475-485~ -- ~elem_size_for_type~, which
     already does it the right way via ~I64.size()~ / ~U8.size()~ /
     ... but stops short of ~F32~/~U64~ and falls back to a bare ~8~.

  (The ~sizeof(...)~ emissions in ~src/self/builder.til~ are a separate
  case: that is C codegen handing the width decision to the C compiler,
  which is fine and not part of this issue.)

  *Why it matters.* Three hand-maintained copies of the scalar width
  table is a correctness hazard: they already disagree on which types
  they cover (e.g. the ~compute_struct_layout~ cascade has no ~U64~
  arm, so a ~U64~ field only gets 8/8 by accident of the empty-string
  default). Any future width change (or a new primitive) has to be
  mirrored by hand across all of them, and a missed copy is a silent
  layout bug.

  *How rstil does it.* Per ~doc/rstil/context.org~ (lines 138-144),
  the Rust prototype computes layout through a *single* pure helper,
  ~get_type_size~ (plus ~calculate_field_offset~ / ~get_field_offset~ /
  ~get_field_type~), over the cached ~scope_stack~. There is no
  scattered literal table -- every phase that needs a size calls the
  one function. Our ~Context~ already mirrors the cache side of this:
  it carries ~struct_layouts := Map.new(Str, StructLayout)~ for
  struct/enum layouts (~src/self/context.til:293~) -- but primitives
  are the one kind of type with no entry anywhere, so they leaked back
  out as literals.

  *Proposed refactor.*

  - Add one canonical helper -- e.g. ~primitive_size_align(name: Str,
    mut sz: USize, mut al: USize) returns Bool~ (returns false for
    non-primitives) -- as the single source of truth for builtin
    scalar widths. One table, covering *all* primitives
    (~I8/U8/I16/U16/I32/U32/I64/U64/F32/Bool/Dynamic~ and ~USize~ ->
    map to its concrete width, see ~type_from_name_init~'s
    ~USize -> U32~ note).
  - Route all three sites through it:
      - ~compute_struct_layout~'s inline cascade -> call the helper,
        keep only the struct/enum/FuncPtr fall-through arms.
      - ~type_size_align~ -> delegate its primitive arms to the helper.
      - ~elem_size_for_type~ -> either delegate to the helper or keep
        the ~T.size()~ form but back it by the same table so it can't
        drift.
    This collapses ~type_size_align~ and the cascade toward rstil's
    single ~get_type_size~ shape.
  - Decide the source of the numbers. Two options, pick one in the
    implementing commit:
      (a) Keep the literals but in *one* place (simplest; the helper
          is the only file that knows "I64 is 8").
      (b) Derive from the autogenerated ~<T>.size()~ methods /
          ~ctx.struct_layouts~ so even primitives have a stored size,
          fully matching rstil's "ask the Context, never hardcode"
          model. Heavier, but removes the last literals.
    Recommendation: (a) first as a pure dedup with no behavior change,
    then consider (b) as a follow-up if we want primitives represented
    in ~struct_layouts~ too.

  *Acceptance.* No primitive width literal (~fsz = 1~, ~sz = 8; al =
  8~, etc.) appears in more than one function; ~compute_struct_layout~,
  ~type_size_align~, and ~elem_size_for_type~ all agree on every
  primitive (including ~U64~/~F32~/~Dynamic~) because they share one
  table; ~make clean && make test~ stays green (layout-sensitive tests
  -- struct/enum size, array element stride -- are the regression
  guard).
