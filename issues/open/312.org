** OPEN #312 Top-level numeric for-loop in a `mode script` panics OutOfBounds in the typer

A `for i in 0..N { ... }` written at the TOP LEVEL of a `mode script` crashes
the typer with an OutOfBounds panic, instead of type-checking. It fails
identically whether interpreted or compiled, and independently of `--usize`
(reproduces at both U32 and U64 -- it predates and is unrelated to the #309
default flip; the flip work merely surfaced it).

*** Reproduction

  #+begin_src til
  mode script
  for i in 0..3 { println(i.to_str()) }
  #+end_src

    $ bin/til repro.til
    ./src/self/typer.til:5029:48: panic: OutOfBounds
    $ bin/til build -o /tmp/repro repro.til
    ./src/self/typer.til:5029:48: panic: OutOfBounds

A statement after the loop does not help (still panics). Moving the loop into
a function body types fine -- function bodies go through `infer_body`, not the
top-level `infer_body_unit`.

*** Workarounds

- Put the loop inside a `func`/`proc` and call it.
- Use a `while` loop at the top level instead of `for-in`.

*** Likely root cause (starting point, not verified as the fix)

`infer_body_unit` (src/self/typer.til:5021):

  #+begin_src til
  mut i : USize = 0
  while i < body.children.len() {
      infer_body_stmt(scope, body, i, ...)          // may advance `i`
      constfolder_classify_stmt(body.children[i], ctx)   // 5029: children[i]
      i++
  }
  #+end_src

`infer_body_stmt` advances `i` when it lowers/splices statements (e.g.
`i = i + inserted` at ~typer.til:4867 for match lowering, and the for-in
desugaring inserts a loop scaffold). After it returns with `i` advanced,
`constfolder_classify_stmt(body.children[i], ctx)` at line 5029 re-reads
`body.children[i]` with the advanced index, which can point at or past the end
of `body.children` -> OutOfBounds. `infer_body` (5006), used for function
bodies, does not have this trailing `children[i]` read, which is why only
top-level units crash.

Candidate fixes to evaluate: re-read `body.children.len()` / bounds-check
before the classify call, capture the pre-advance index for the classify, or
move `constfolder_classify_stmt` to run on the statement `infer_body_stmt`
actually processed rather than `children[i]` after the advance.

*** Regression test

Add a `mode script` fixture with a top-level `for i in 0..N` loop (both
interpret and compile) to the suite once fixed.
