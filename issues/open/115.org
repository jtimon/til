** WIP #115 Reduce primitive special-casing; keep ext_struct, document its contract

*Progress.*

  - Layout contract documented: doc/ffi.org now has an "ext_struct vs
    struct" section spelling out that ~struct~ reserves the right to
    reorder fields while ~ext_struct~ guarantees declaration order ==
    memory order for the C ABI. This is the "document its contract" half
    of the issue.
  - First special-casing reduction: builder.til had the full primitive
    name list (~U8 I8 I16 U16 I32 U32 U64 I64 F32 Bool~) spelled out four
    times to seed the header-emission "already emitted" sets. Collapsed
    those into one ~seed_primitive_names~ helper, so adding a primitive is
    no longer a four-place edit in that file.

*Remaining.* The bulk of the special-casing is still scattered: hundreds
  of individual ~name.eq("U32")~-style checks and parallel
  ~switch~/~if~-chains across builder, initer, typer, and dispatch. Each
  tests one primitive (or a near-full subset) for a property -- builtin
  scalar, forward-decl skip, C representation, default value, etc. Folding
  those onto a single source of truth needs a shared cross-file helper
  (home TBD: core/ast.til or context.til) and care around the ~USize~ /
  ~UPtr~ aliases that some lists include and others do not. That is the
  incremental work this issue still tracks.

*Rescoped.* The original premise -- "remove or simplify ~ext_struct~,
maybe fold it into ordinary ~struct~" -- is wrong and is dropped.
~ext_struct~ has an irreducible reason to exist, and its ~Str~ argument
is a real feature, not noise.

*** Why ext_struct stays (the layout contract)

  The true, irreducible difference between ~struct~ and ~ext_struct~ is
  field layout:

  - ~struct~ reserves til's right to reorder fields (packing, alignment,
    whatever future layout optimization wants). Source order is not a
    promise.
  - ~ext_struct~ guarantees declaration order == memory order. The
    fields are laid out exactly as written, because the type has to
    match a C struct's ABI for FFI.

  This holds even though the layout-optimization pass for ~struct~ is not
  implemented yet: it is a contract, not a current behavior. You cannot
  express "match this exact C layout" with a type form that is allowed to
  reorder, so ~ext_struct~ cannot fold into ~struct~.

*** The Str tag argument is a real feature

  ~ext_struct~ takes an optional ~Str~ tag because C libraries use
  different conventions for the two names of one thing (the ~struct foo~
  tag vs the ~typedef~ name). Forms:

  - ~ext_struct {}~          -- anonymous (tag ~@anon~), no C struct tag
  - ~ext_struct("foo") {}~   -- custom C struct tag ~foo~

  (The redundant ~ext_struct("") {}~ spelling is being removed in its own
  cleanup issue; see below.)

*** What is actually left to do

  The part of the #176 follow-up that is still real: the compiler carries
  repeated hardcoded primitive-name lists (~U8~, ~I16~, ~I32~, ~U32~,
  ~U64~, ~I64~, ~F32~, ~Bool~, ...) across builder, initer, typer, and
  ~Type~. Those lists decide:

  - whether a type is treated as a builtin scalar
  - whether generated headers skip forward declarations for it
  - whether C emission uses the alias-backed primitive representation or
    falls back to ordinary struct handling

  Every new primitive (most recently ~I8~ via #176) shipped by adding one
  more row to every one of those lists. The goal of this issue is to
  reduce that special-casing to the minimum truly required by runtime
  representation, so adding/representing a primitive is not an N-place
  edit.
