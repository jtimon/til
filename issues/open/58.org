** OPEN #58 First-class functions: Type hierarchy and FuncSig/FuncDef

:DEPENDENCIES: [[./14.org][#14]]
  Based on rstil/doc/first_class.org. Everything in til is a value —
  functions, structs, enums can all be assigned, passed, returned, and
  stored. This issue covers the metatype system that makes it work.

*** Type hierarchy

   ~Type~ is the metatype — the type of all type definitions:

   #+begin_src til
   Type := enum { StructDef, EnumDef, FuncSig, FuncDef }
   #+end_src

   | Metatype    | Defines                      | Instance example                     |
   |-------------+------------------------------+--------------------------------------|
   | ~StructDef~ | Data layout (fields)         | ~p := Point(x=10)~ is a Point        |
   | ~EnumDef~   | Tagged union (variants)      | ~c := Color.Red~ is a Color          |
   | ~FuncSig~   | Calling convention (types)   | ~BinaryOp := func(I64, I64) ret I64~ |
   | ~FuncDef~   | Concrete function (sig+body) | ~add : BinaryOp = (a, b) { ... }~   |

   ~FuncSig~ is a pure type — parameter types, return type, func/proc
   kind. Parameter *names* are NOT part of the signature, only types.

   ~FuncDef~ is a concrete function definition — it has a ~FuncSig~ (its
   type) plus argument names, modifiers, and a body. A FuncDef is to its
   FuncSig what a struct instance is to its StructDef: the signature is
   the type, the definition is the value.

   Function types are *structural*: two signatures with the same parameter
   types, return type, and throws are the same type regardless of name.

*** Explicit type annotations (~: Type =~ syntax)

   Separate the type annotation (~:~) from the value assignment (~=~).
   ~:=~ remains as shorthand for "infer the type":

   #+begin_src til
   // Explicit type + value
   x : I64 = 5
   name : Str = "hello"

   // Type definitions: explicit metatype annotation
   Point : StructDef = struct { mut x: I64 = 0, mut y: I64 = 0 }
   Color : EnumDef = enum { Red, Green, Blue }
   BinaryOp : FuncSig = func(I64, I64) returns I64

   // Inferred (unchanged from today)
   Point := struct { mut x: I64 = 0, mut y: I64 = 0 }   // infers StructDef
   Color := enum { Red, Green, Blue }                      // infers EnumDef
   BinaryOp := func(I64, I64) returns I64                  // infers FuncSig
   #+end_src

*** Type inference rules

   | Declaration                                                | Inferred type |
   |------------------------------------------------------------+---------------|
   | ~x := 5~                                                   | ~I64~         |
   | ~name := "hello"~                                          | ~Str~         |
   | ~Point := struct { ... }~                                  | ~StructDef~   |
   | ~Color := enum { ... }~                                    | ~EnumDef~     |
   | ~BinaryOp := func(I64, I64) returns I64~                   | ~FuncSig~     |
   | ~add := func(a: I64, b: I64) returns I64 { return ... }~   | ~FuncDef~     |

   The distinction mirrors struct/enum:
   - ~Point := struct { ... }~ defines a *type* (infers StructDef)
   - ~p := Point(x=10)~ creates an *instance* (infers Point)
   - ~BinaryOp := func(I64, I64) returns I64~ defines a *type* (infers FuncSig)
   - ~add := func(a: I64, b: I64) returns I64 { ... }~ creates an *instance* (infers FuncDef)

   A bodyless func/proc is a FuncSig (type definition). A bodied func/proc
   is a FuncDef (value). The presence of a body (and parameter names)
   distinguishes them.

*** Parameters always require explicit types

   Lambda and named function parameters always require explicit types.
   ~Dynamic~ and ~Type~ are special parameter types and inferring argument
   types would create ambiguity. Explicit parameter types also make function
   signatures self-documenting.

   #+begin_src til
   // REQUIRED: explicit types in function parameters
   add := func(a: I64, b: I64) returns I64 { return a.add(b) }

   // NOT ALLOWED: inferred parameter types
   add := func(a, b) returns I64 { ... }  // error
   #+end_src

*** FuncSig form vs FuncDef form

   Two equivalent ways to define a function, mirroring the two metatypes.
   Both produce the same FuncDef value:

   #+begin_src til
   // 1. FuncDef form: FuncDef literal (type inferred as FuncDef)
   add := func(a: I64, b: I64) returns I64 { return a.add(b) }

   // 2. FuncSig form: explicit signature as type, names + body as value
   add : func(I64, I64) returns I64 = (a, b) { return a.add(b) }

   // 3. FuncSig form with named alias
   add : BinaryOp = (a, b) { return a.add(b) }

   // 4. Explicit FuncDef metatype
   add : FuncDef = func(a: I64, b: I64) returns I64 { return a.add(b) }
   #+end_src

   Forms 2 and 3 are the same (with or without a type alias). Form 1 is the
   FuncDef literal — the parser extracts argument names and body to produce
   the same internal representation. Form 4 is the explicit-metatype version
   of form 1.

*** Collections of types

   #+begin_src til
   Vec.new(BinaryOp)   // Vec of function pointers with the same signature
   Vec.new(FuncSig)    // Vec of function signatures (different signatures)
   Vec.new(FuncDef)    // Vec of function definitions (different definitions)
   Vec.new(StructDef)  // Vec of struct definitions
   Vec.new(EnumDef)    // Vec of enum definitions
   Vec.new(Type)       // Vec of any type definition
   #+end_src

   Storing function pointers in collections works for create, push, and
   retrieve+call: ~ops := Vec.new(BinaryOp)~ then ~ops.push(own fn_add)~
   stores the function pointer in a Fn-typed slot, and
   ~ref g : BinaryOp = ops[i] ; g(...)~ pulls the slot back out via
   ~reinterpret_ptr_value~'s FuncSig branch and dispatches. The push
   path uses ~write_ptr~ to deposit the raw pointer instead of memcpy +
   free (fn ptrs aren't heap memory); the read path is the existing
   FuncSig-aware deref. The remaining sugar -- inline
   ~cast(ops.get(i), BinaryOp)(3, 4)~ and direct ~ops[i](3, 4)~ -- is
   still TODO (the typer's ~cast_pair_ok~ rejects Dynamic -> FuncPtr,
   and the parser doesn't allow ~<expr>(args)~).  Originally moved here
   from [[./61.org][#61]].

*** Introspection

   First-class types support reflection via the ~func_sig_of~,
   ~struct_def_of~, ~enum_def_of~ helpers (see rstil/std.meta):

   #+begin_src til
   sig := func_sig_of("BinaryOp")
   sig.args.len()             // 2
   sig.return_types.len()     // 1
   sig.throw_types.len()      // 0
   #+end_src

*** Type parameters

   Two-phase plan:

   *Phase now*: ~func(T: Type)~ — no new syntax needed. ~Type~ is already
   a special parameter type. Macros handle the template use case by
   executing arbitrary code at compile time (like Jai's comptime, Zig's
   comptime):

   #+begin_src til
   // Type as a regular parameter
   swap := func(T: Type, mut a: T, mut b: T) { ... }
   swap(I64, x, y)

   Pair := macro(A: Type, B: Type) returns Type {
       return struct { mut first: A, mut second: B }
   }
   IntStr := Pair(I64, Str)
   #+end_src

   *Phase later*: ~func<T>()~ as sugar that desugars to macro calls.
   Use ~<>~ for familiarity (Rust, C++, Java):

   #+begin_src til
   // Sugar (later):
   swap<T> := func(mut a: T, mut b: T) { ... }
   swap(x, y)       // T inferred
   swap<I64>(x, y)  // T explicit

   // Desugars to:
   swap := func(T: Type, mut a: T, mut b: T) { ... }
   swap(I64, x, y)
   #+end_src

*** Unified pattern summary

   | Construct  | Type declaration               | Definition (type = value)              |
   |------------+--------------------------------+----------------------------------------|
   | Variable   | ~x : I64~                      | ~x : I64 = 5~ or ~x := 5~             |
   | Function   | ~F : func(I64) returns I64~    | ~f : func(I64) returns I64 = { ... }~  |
   | Body       | (no separate type decl)        | ~b : Body = { ... }~ or ~b := { ... }~ |
   | Struct     | ~S : struct { ... }~           | (struct decl IS the definition)        |
   | Enum       | ~E : enum { ... }~             | (enum decl IS the definition)          |

*** Progress (rstil-parity arc, 2026-05)

   The first-class-everything machinery feeding #56 (structs) and #57
   (enums) lands a chunk of #58 along the way. Tracking the metatype
   side here:

   What works today:

   - [X] Type hierarchy: ~Type.StructDef~, ~Type.EnumDef~,
     ~Type.FuncDef~, ~Type.FuncPtr~ all round-trip through
     ~dyn_type_to_str~ and accept the bare metatype name in type-
     annotation position (~triple : FuncDef = func(...)~).
   - [X] Type inference rules: ~Point := struct { ... }~ infers
     ~StructDef~, ~Color := enum { ... }~ infers ~EnumDef~,
     ~BinaryOp := func(I64) returns I64~ (bodyless) infers ~FuncPtr~
     ("Fn"), ~add := func(a: I64) returns I64 { ... }~ infers ~FuncDef~.
   - [X] Explicit-type metatype annotation forms (~: StructDef =~,
     ~: EnumDef =~, ~: FuncDef =~, ~: FunctionDef =~) all type-check.
   - [X] Parameters always require explicit types (enforced).
   - [X] FuncSig form: ~add : BinaryOp = (a, b) { return ... }~ and
     ~func(I64) returns I64 = (a, b) { ... }~ both produce the same
     ~FuncDef~ value.
   - [X] Collections of types: ~Vec.new(BinaryOp)~ (Fn-typed slot),
     ~Vec.new(StructDef)~, ~Vec.new(EnumDef)~, ~Vec.new(FuncDef)~
     all create + push (Vec stores ~struct_def_of("X")~ /
     ~enum_def_of("X")~ values; for FuncDef, function-pointer
     slots). Retrieve + call works in both modes via a FuncSig-typed
     ref binding: ~ref g : BinaryOp = ops[i] ; g(3, 4)~. The FFI
     bridge round-trips ~Value.Func~ through ~Vec.push~ so the slot
     gets the real pointer value instead of a NULL written by the
     missing variant in ~ext_dispatch_ffi~.
   - [X] Introspection helpers ~func_sig_of~, ~struct_def_of~,
     ~enum_def_of~ assemble FuncSig / StructDef / EnumDef values from
     the loaded program.
   - [X] Type-parameter macros (~Phase now~): ~func(T: Type)~ and
     ~macro(T: Type)~. Macros taking ~Type~ get a comptime-arg gate
     extension for the ~Type.<Variant>(...)~ constructor form; the
     typer-side rewrite is skipped for macro args so the original
     ident reaches precomp's arg loop, which builds the equivalent
     ~Type.Custom("X")~ ctor.
   - [X] Macros as struct-type generators
     (~Pair := make_pair(I64)~ -> struct decl); see #56.
   - [X] Macros as enum-type generators (~Opt := make_opt(I64)~ ->
     enum decl); see #57.
   - [X] Macros as function generators (~id_i64 := make_id(I64)~ where
     ~make_id := macro(T: Type) returns FuncDef { return func(x: T) returns T { return x } }~).
     initer's pre-pass substitutes type-params in the func literal's
     param types, return type, and body; the rest of the pipeline
     types the expanded decl as an ordinary named function. The
     typer's lambda-lift, throws lowering, body-walk, and unused
     checks all skip *template* macros (returning StructDef / EnumDef
     / FuncDef, or with a ~Type~ parameter); mirrors rstil
     typer.rs:1090-1097.
   - [X] Bodyless ~func(I64, I64) returns I64~ as a Type alias for
     ~Type.FuncPtr~ (the existing "Fn" round-trip).

   Still missing for rstil parity:

   - [X] Inline ~cast(ops.get(0), BinaryOp)(3, 4)~. The parser accepts
     postfix ~(args)~ on any expression; ~cast_pair_ok~ accepts
     Dynamic -> FuncPtr against a user FuncSig name (and FuncPtr ->
     FuncPtr identity); ~infer_funcptr_call_via_cast~ resolves the
     FuncSig through the cast's preserved Ident child and validates the
     outer call; ~h_cast~ adds the Fn branch (two derefs to read the
     fn ptr stored at the slot the Dynamic points to); codegen emits
     ~(*(void **)slot)~ for the dynamic-to-fn-ptr read, then the same
     indirect-call cast (~Ret (*)(Args)~) the FuncSig-typed-struct-field
     path uses.
   - [ ] Direct ~ops[0](3, 4)~ (no cast). The subscript desugars to
     ~ops.get(0)?~ which returns Dynamic; the bang hoist lifts it into
     ~_bang_ret_0 := ops.get(0)?~ and the typer rejects owning Dynamic
     without an explicit type. Making this work needs either a typed
     ~Vec.get~ specialization (return the element type, not Dynamic)
     or context-driven typing of the hoisted temp from the outer call
     site.
   - [X] Macro body cases beyond ~return <single-literal>~: an
     ~if T.is(Type.<Variant>) { return struct {...} } else { ... }~
     macro now expands. init's pre-pass walks the body recursively;
     a small comptime evaluator recognises ~<Ident>.is(Type.<Variant>)~
     against the substituted type-param and picks the matching
     branch's return literal. Else-if chains and StructDef / EnumDef
     / FuncDef returns all work. Conditions beyond that exact shape
     still leave the macro un-expanded (the pre-pass returns NULL,
     caller continues without rewrite).
   - [X] Value-param substitution in function generators:
     ~make_adder := macro(N: I64) returns FuncDef { return func(x: I64) returns I64 { return x.add(N) } }~
     works -- value-param Idents are replaced by the call-site arg
     expression (test_macro_func_gen_value_param).
   - [X] User namespace methods inside type-generator macros coexist
     with the auto-generated clone/delete (test_macro_struct_ns_*).
     to_str is not auto-generated for any struct, so there is nothing
     to merge there.
   - [X] Direct generic functions ~func(T: Type, v: T) returns T~
     (~Phase now~). A top-level func/proc that uses a ~Type~-named
     param as the type of another param (or the return type) is a
     generic; init monomorphizes it per distinct type-arg tuple,
     synthesizes one concrete copy per tuple (deduped), and rewrites
     each call ~gid(I64, 42)~ to the concrete ~gid__I64(42)~ with the
     type-args dropped, then removes the generic decl. Plain
     ~func(t: Type)~ runtime-Type-value params (e.g. dyn_type_to_str,
     til_type_name_c) are left untouched. ~T()~ constructor calls in
     the body resolve at the monomorphized type. init_expand_generic_funcs
     in initer.til; tests test_generic_func_* in first_class.til.
     Composition works: a generic func calling another generic func
     (`g2 := func(T: Type, v: T) returns T { return g1(T, v) }`) is
     fixpoint-expanded -- after substituting the synthesized body the
     expander walks it again, so the inner `g1(T, v)` becomes
     `g1__I64(v)` (test_generic_func_compose).
     Remaining v1 limitation: ~own~ value args aren't preserved across
     the call rewrite, so a generic taking `own v: T` is not yet
     covered (the own-marker would need carrying onto the rewritten
     call without introducing the double-/no-free hazard).
