# priority: high
** OPEN #196 Interpreter mishandles enum values in struct fields and switch hoists

Two interpret-mode-only bugs surfaced while doing #184 in
src/self/dispatch.til. Both compile fine via build/run -- only the
interpreter (eval_expr / read_field / write_field) is broken.

*** Bug A: ~own mut Field : Enum~ struct field write/read

Minimal repro:

#+begin_src til
mode test

E := enum {
    A,
    B: I64
}

W := struct {
    own mut e := E.A
}

test_field_assign_own_enum := test() {
    w := W()
    w.e = E.B(42)
    assert(__LOC__, w.e.is(E.B))   // FAILS in interpret, passes in compile
}
#+end_src

Cause: ~src/self/interpreter.til:write_field~ (line ~1650) and
~read_field~ (line ~1576) take an early ~OwnType.Own~ branch that
treats the field's payload as a raw pointer/struct-data:

- ~write_field~'s ~OwnType.Own~ default fallback does
  ~ref pv : Dynamic = get_payload(val); write_ptr(ptr, deref(pv))~
  which writes the first 8 bytes of an ~EnumInstance~ (the tag) to
  the field, instead of allocating a heap ~EnumInstance~ and storing
  its pointer (the way the no-own branch at line ~1699 does).
- ~read_field~'s ~OwnType.Own~ branch returns ~Value.Ptr(deref(ptr))~
  for an enum-typed field, instead of reconstructing a ~Value.Enum~
  from the heap ~EnumInstance~ at ~ptr~ (the way the no-own branch
  at line ~1612 does).

Workaround: drop the ~own~ keyword on the enum field. The
~mut e := E.A~ form (default ~OwnType.Shallow~) round-trips correctly.

Fix: route enum-typed fields to the same EnumInstance handling
regardless of ~own_type~. Either short-circuit the ~OwnType.Own~
branch when the field type is an enum, or duplicate the enum
heap-alloc path inside the ~OwnType.Own~ branch.

*** Bug B: nested switch case-binding on a hoisted enum subject

Real-world site that triggered this: ~src/self/dispatch.til~ in
~ffi_init_scan_program~. The ~d4a123a~ refactor wrote:

#+begin_src til
switch stmt.node_type {                 // simple lvalue, no hoist
case NodeType.Decl(sdd):
    switch stmt.child(0).node_type {    // FCall in path, hoists
    case NodeType.StructDef(sdd_data):
        for d_j : Declaration in sdd_data.ns_decls {
            ... use sdd.name and sdd_data ...
        }
    case:
        UNREACHABLE(__LOC__)
    }
case:
    UNREACHABLE(__LOC__)
}
#+end_src

Interpret panics with ~null deref~ when the program has any
struct with a namespace section, even if the case body never
executes a problematic operation. Compile path is fine.

Likely cause: the typer's switch desugar
(~src/self/typer.til:make_switch_case_condition~ and the surrounding
~infer_switch_stmt~) hoists the non-simple-lvalue subject into a
~mut _sw_X := <expr>~ local with the default ~OwnType.Shallow~. For
an enum payload that owns heap data (NodeType.StructDef -> StructDef
-> Vec<Declaration>), a shallow assignment aliases the storage. The
case-binding ~ref sdd_data : StructDef = get_payload(_sw_X)~ then
borrows into a payload that is double-owned by the original
expression and the hoisted local; the interpreter hits this
aliasing differently than the compiled path, producing the null
deref.

Workaround in d4a123a / 5446330: collapse to a single outer switch
and keep ~ref sdd_data : StructDef = get_payload(stmt.child(0).node_type)~
inside the case arm. Same shape as the original pre-#184 code.

Fix candidates:
- Make the switch hoist clone the subject deeply (call ~.clone()~
  on enum-typed subjects so the local owns its own payload).
- Or extend ~is_simple_lvalue_expr~ to accept ref-returning FCalls
  so they don't get hoisted at all.
- Or change the hoist Decl to ~ref~ when the subject expression
  returns a ref (covered by ~fcall_returns_ref~ for FCalls).

*** Why it matters for #184

These two bugs put a hard ceiling on how aggressively #184 can use
switch destructure in interpreter-driven code paths. Until #196 is
fixed, switch refactors that introduce ~own~ enum struct fields or
nested switches over hoisted enum-payload subjects must be reverted
to the bare ~get_payload~ form.

Belongs under #109 indirectly (interp leaks/bugs), and blocks the
last few percent of the #184 cleanup.
