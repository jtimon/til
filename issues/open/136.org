# priority: high
** OPEN #136 Precomp doesn't fold pure function calls

Reopened: an earlier "fix" landed (Vec.move_from / Vec.push_take work,
   originally in [[../fixed/][issues/fixed/]]) but didn't address the original issue --
   ~add(1, 2)~ at compile time still reaches runtime. The current
   precomp eval-fold (commit 4b4c2e2) folds simple integer arithmetic
   but bails on the more interesting cases. Goal is rstil parity:
   test/constfold.til ~test_lolalalo~ should compile to a constant
   string the same way rstil's does.

   Status:
   - Folds: ~test_simple_add~ -> ~result = 3~,
     ~test_nested_arithmetic~ -> ~15~, ~test_deeply_nested~ -> ~26~.
   - Does NOT fold: ~test_lolalalo~ -- ~make_poem(...)~ stays as a
     runtime call. The ~lolalolo~ / ~lalei~ strings the test asserts on
     are computed at runtime, not at compile time.

   Status update (post-rstil-parity merges): items 3, 6, 7, 9 below
   are no longer in master. ~test_lolalalo~ AND ~test_string_concat~
   both fold to ~Str_lit("...", N)~ at compile time now (see
   ~test/constfold.c~). What's left is the residual exception
   list, plus one DCE-shaped follow-up surfaced by trying to remove
   item 1 -- documented inline below.

   Bails / exceptions added in src/self/precomp.til (none of these are
   in rstil; remove one by one until the suite still passes with
   fewer of them):

   1. ~try_eval_call~ bails when ~_fd.func_type.is(FuncType.ExtFunc)~
      (gated on ~is_macro_mode.not()~ since #191). Workaround for
      user .so not dlopen'd at precomp time. Built-in ext_funcs
      (I64_add etc.) are statically linked and ~ext_dispatch_ffi~
      would dispatch them gracefully (it returns false on
      unregistered names, so a user .so that wasn't dlopen'd at
      precomp time would just refuse to fold rather than crash).
      Tried removing the bail (master ~7bf0cc9~): folds proceeded
      but ~test_runner.til~ failed compile with
      ~-Werror=unused-variable~ on temps like ~_t_I64_858 = -2~ in
      ~main()~'s top-level init -- the fold replaced
      ~CAP_LIT = something(...)~ with ~CAP_LIT = 4294967295~ and
      orphaned the arg temps. ~ee487a5~ (Body-level DCE for
      post-precomp dead temps) handles function-body Decls but does
      NOT walk ~main()~'s top-level init sequence, so the dead
      temps slip through and gcc rejects them. Real fix: extend
      DCE to the main()/init codegen path before this bail can
      come out.

   2. ~try_eval_call~ FA-callee path bails when the receiver Ident
      isn't a registered struct namespace and isn't a known constant
      (~known.has(_ri).not()~). Excludes value-receiver method calls
      (~p.diff()~) from fold attempts.

   3. (REMOVED in rstil-parity merge.) ~func_uses_unknown_globals~
      no longer flags ~clone~/~delete~/~concat~/~substr~/~format~ as
      unknown -- the old workaround was unblocked by tracking
      ~CAP_VIEW~ / ~CAP_LIT~ in the precomp interpreter scope at
      precomp_imported time (see ~scope_set_owned(global,
      "CAP_VIEW", ...)~ in ~precomp.til~) and by the transitive
      ~func_uses_unknown_globals~ walk via the ~visiting~ Set.
      ~test_lolalalo~ now folds.

   4. ~func_uses_unknown_globals~ flags Assign nodes whose LHS isn't
      a body-local. Workaround for funcs like
      ~build_register_funcsig_names_lp~ where folding would silently
      mutate precomp's interpreter scope (or panic when the global
      isn't bound there).

   5. ~try_eval_call~ bails when ~_fd.return_own_type.eq(OwnType.Ref)~.
      Avoids baking compile-time aliases / dlsym-truncated pointers
      into the AST.

   6. (NOT IN MASTER.) The "skip when parent is NodeType.Body" guard
      isn't in current ~fold_subtree~ -- the loop folds any FCall
      child that ~is_func_call~ / ~is_macro_call~ identifies. Statement-
      level pure-func calls technically would be folded if the args
      are comptime, dropping the call. In practice the codebase
      doesn't have stmt-level pure-func-call-with-comptime-args
      (most stmt-level calls are procs, which ~is_func_call~ skips).
      Worth keeping an eye on if a future fold drops a side effect.

   7. (NOT IN MASTER.) ~is_macro_call~ / ~is_func_call~ both handle
      ~NodeType.FieldAccess~ callees via ~fa_recv_type_name +
      precomp_has_*~. Method-shaped folds like ~Str.size()~ work.

   8. ~value_to_expr~ returns NULL for ~Value.Float~. ~F32_to_str~
      uses ~%g~ which can yield ~6.28318e+06~ that doesn't round-trip
      through ~Str.to_f32~. Until the AST has a faithful F32 literal
      repr, F32-returning calls can't fold.

   9. (PARTIALLY REMOVED.) ~value_to_expr~ now folds
      ~Value.Struct(Str)~ via the ~from_source_form~ /
      ~str_to_source_form~ escape round-trip introduced in the
      rstil-parity merge; ~test_string_concat~ and ~test_lolalalo~
      both produce ~Str_lit("hello world", 11ULL)~ and the 248-byte
      permutation poem at compile time. Other ~Value.Struct(...)~
      shapes (non-Str payloads) still return NULL -- the AST has no
      faithful literal repr for them.

   10. ~fold_subtree~ tracks Decl literals only when the name starts
       with ~_t_~ (typer-hoisted temps). User-named locals
       (~x~, ~result~, ~i~) deliberately not tracked because the
       global ~known~ Map would let one body's value bleed into
       another's fold. rstil avoids this with a scope stack; til
       doesn't have one.

   11. ~process_body~ tracks top-level Decl literals only when
       ~at_global~ (outermost call), so nested function bodies don't
       pollute the global ~known~ map.

   12. ~try_eval_call~ ~can_convert~ list excludes ~Value.Float~ and
       ~Value.Struct~ explicitly (mirrors items 8 and 9 at the call
       site).

   13. ~is_side_effecting_name~ hard-codes a list of ~exit~ /
       ~malloc~ / ~println~ / ~ffi_*~ / ~dlopen~ / ... names. rstil
       has its own version of this list; ours has drifted as I added
       names whenever a fold blew up. Worth re-checking against
       rstil and pruning to the minimum set that's actually unsafe
       to invoke at compile time.

   Pre-existing fix that landed earlier (kept; not part of the
   precomp work): ~Vec.move_from~ / ~Vec.push_take~ in src/core/vec.til
   for in-place body.children rebuilds in the typer desugar passes.
   Reduced the ASAN footprint and unblocked further work, but doesn't
   touch precomp at all. Related: [[./146.org][#146]] (unify desugar/insert passes),
   [[./109.org][#109]] (ASAN leaks).
