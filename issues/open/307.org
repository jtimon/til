** OPEN #307 StructDef/EnumDef name indexes: keep the ordered Vecs, add a Map(Str, USize) per collection

StructDef stores its members as two plain Vecs (`fields`, `ns_decls`), so
every lookup-by-name in the typer/initer is a linear scan, and several
places keep hand-rolled side collections on top (Context layout tables;
initer's gather vecs like `field_names`, a porting artifact rstil does not
have -- its preinit builds the generated clone directly off
`struct_def.members`).

The design (nothing new is introduced -- the existing Vec and the existing
til Map compose):

    fields := Vec(Declaration)      // stays: declaration order = layout truth
    index  := Map(Str, USize)       // added: name -> position in the Vec

one index Map for `fields`, another for `ns_decls`. The Vec remains the
single source of truth and keeps its semantic order; the Map is only an
index over it. til's Map is sorted by key, which is irrelevant for an
index (nobody iterates it) -- what it buys is that lookup-by-name becomes
a map probe instead of a linear scan. Appends to the Vec never invalidate
stored positions.

Why the simpler-looking alternatives fail:

- Replacing the Vec with `Map(Str, Declaration)` outright: the Map's
  iteration order is KEY order, and field order is semantic -- it is the C
  layout order. raylib's ext_struct `Color { r, g, b, a }` iterated in key
  order comes out `a, b, g, r` and scrambles the FFI layout; positional
  construction and the memcpy-layout logic assume source order too.
- Keeping the Map primary plus a rank-index Vec for order: inserting into
  a sorted map shifts the ranks of every later key, and struct/enum defs
  get post-parse insertions (initer pushes the generated clone/delete and
  variant constructors into `ns_decls`), so stored ranks go stale.

EnumDef has the same shape and gets the same treatment: `ns_decls` gains a
name index like StructDef's, and `variants` (a Vec(Str) whose ORDER is the
tag assignment, so it must stay an ordered Vec) gains a
variant-name -> tag-position index to replace the by-name scans. The
existing per-variant parallel structures already key off that position
(`payload_types : Map(I64, Str)` by tag, `payload_consts` indexed by tag),
so they fit unchanged.

Scope notes:

- Mechanical but wide: every `sdd.fields` / `sdd.ns_decls` iteration and
  by-name scan in parser/initer/typer/desugarer/builder touches this.
- Kills redundant gather collections (initer's `field_names`/`field_refs`
  and friends) outright.
- NOT a string-churn fix by itself: index keys are still Strs compared
  byte-wise; the churn question (interning / SSO / hashed tables) stays
  separate and is largely post-#302 territory.
