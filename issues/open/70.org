** OPEN #70 Constfolder: constant folding for all types (not just scalars)

The builder emits top-level constants as heap-allocated runtime
  variables. Constfolder currently folds scalar expressions (I64, Str,
  Bool) but not struct-valued or collection-valued ones.

  Broader scope: constfolder should fold ANY compile-time-known expression
  into a constant value in the generated code. Examples:
  - ~CAP_LIT := max_u64()~ → ~#define CAP_LIT ULLONG_MAX~
  - ~MODE_SCRIPT := Mode(name = "script")~ → ~Mode MODE_SCRIPT = {...}~
  - ~core_modes := {"script": Mode(...), ...}~ → static Map
  - Fibonacci via macro returning Vec → static array

  The syntax already works: ~ref MODE_SCRIPT : Mode = core_modes.get("script")~
  is a typed reference to a compile-time-known value. Constfolder just needs
  to evaluate it and the builder needs to emit the result as a constant.

  CAP_LIT/CAP_VIEW no longer rely on [[file:../../src/c/pre70.h][src/c/pre70.h]]. Handwritten C now
  gets them through the normal generated headers, while the remaining
  broader constant-folding work stays here.

  Progress: constfolder now folds struct-VALUED pure calls. ~value_to_expr~
  (src/self/constfolder.til) reconstructs a non-Str ~Value.Struct~ into a
  positional POD-struct constructor (Ident(name) + one folded arg per
  field, in field order); the builder emits it as a compile-time C
  compound literal via ~emit_pod_compound_literal~. Only POD structs fold
  (no own/ref fields); nested POD structs recurse. Verified in
  test/constfold.til: ~CfVec2.magic()~ -> ~(CfVec2){.x=42,.y=99}~,
  ~CfVec2.at(10,20)~, and nested ~CfRect.sample()~ all fold (see
  test/constfold.c), with stack-value access and call_free=false deletes.
  Progress: constfolder also folds enum-VALUED pure calls (no-payload variants).
  The interpreter resolves a no-payload variant access (~Color.Green~) via
  the EnumDef's ns_decls ~default_value~ to its integer tag (~Value.Int32~),
  so naively converting that to a Num literal baked ~*c = 1~ (an int into
  a Color struct). ~value_to_expr~ now intercepts when the source call's
  return type is ~Type.Enum~ and reconstructs the variant FieldAccess
  ~Color.<variant>~ wrapped in ~Color.clone(...)~ -- mirroring the typer's
  handling of direct ~c := Color.Green~. The builder's
  ~is_pod_enum_clone_call~ path then emits a stack value with
  ~stack_locals~ tracking (~call_free=false~ on the matching delete).
  Verified in test/constfold.til: ~mk_color() -> Color.Green~ folds to the
  same ~Color_clone(&(Color){.tag = Color_TAG_Green})~ codegen as the
  direct variant access (see test/constfold.c), suite green under ASAN.

  Progress: payload-bearing enum-returning pure calls also fold. The
  interpreter returns ~Value.Enum(EnumInstance)~ for an enum-typed call
  when the enum has any payload variants; ~value_to_expr~'s enum
  intercept now decodes the variant tag from ~ei.data~, reads the payload
  bytes at ~ei.data + ENUM_PAYLOAD_OFFSET~ via a fabricated Declaration
  whose ~explicit_type~ is the variant's payload type (from
  ~EnumDef.payload_types[tag]~) and a ~read_field~ call, recurses
  ~value_to_expr~ on the decoded payload with a synthetic non-enum src
  (so the intercept doesn't refire on the scalar), and assembles
  ~Token.Num(<payload>)~ as an FCall. No clone wrap: variant ctors are
  heap-allocating funcs, so the Decl init emits exactly the codegen of a
  direct ~t := Token.Num(...)~. Verified in test/constfold.til:
  ~mk_tok(7) -> Token.Num(7)~ folds to
  ~Token *t = Token_Num(&(I64){7});~ (see test/constfold.c). Nested-enum
  payloads (an enum whose payload is another enum) currently bail; the
  synthetic src skips the intercept and the default ~Value.Enum~ branch
  of the scalar switch returns NULL. Also added ~Value.Enum~ to
  ~can_convert~ in ~try_eval_call~.

  Still TODO under this issue: nested-enum payload folds,
  collection-valued folds (Vec/Map/Array), namespace/static-field globals
  (~Str.CAP_LIT~ -> generated ~Str_CAP_LIT~), and ~const~ emission.

  Current status after the real ~[[./104.org][#104]]~ fix:
  - ordinary top-level globals are now usable from handwritten C through
    generated ~extern~ declarations in ~boot/til_forward.h~
  - but namespace/static-field globals still do not have the same final
    generated-C story
  - so the remaining work to make things like ~Str.CAP_LIT~ / generated
    ~Str_CAP_LIT~ usable through the same final generated-C path still
    belongs here under ~[[./70.org][#70]]~, not under ~[[./104.org][#104]]~

  Also blocks ~const~ in hand-written C: the builder doesn't emit
  ~const~ qualifiers, so C code must drop ~const~ from ~Mode *~ etc.
  to match generated signatures for [[./39.org][#39]] swaps. Once [[./70.org][#70]] lands, the
  builder can emit ~const~ where appropriate and C can use it again.
