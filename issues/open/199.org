# priority: low
** OPEN #199 Guard clauses in switch cases

Allow ~case <pattern> if <cond>:~ in switch arms. Listed as a Phase 5
feature in #36 (now fixed) but deferred to its own issue.

Example::

  test_color_match := func(c: Color) returns Str {
      switch c {
      case Color.Unknown: return "unknown"
      case Color.Green(is_olive):
          if is_olive { return "olive" }
          return "dark green"
      case Color.Number(code) if code.gt(0): return "positive"
      case Color.Number(code): return "non-positive"
      case Color.Name(name): return format("named: ", name)
      }
  }

Sketch:

- Parser: after ~parse_expression~ for the case pattern, if the next
  token is ~KwIf~, advance and parse a second expression as the guard.
  Store both children before the ~Colon~ + body. Today a ~Case~ node
  has exactly ~[pattern, body]~; add an optional third child for the
  guard (or wrap the body in an ~If~).
- Typer: in ~make_switch_case_condition~, AND the existing pattern
  condition with the guard. The guard must reference any bindings
  introduced by the pattern, so it has to be lowered AFTER bindings
  are in scope -- easiest is to emit:

      if <pattern_cond> {
          <pattern_bindings>
          if <guard> { <body> }
      }

  rather than folding the guard into the chained ~else if~ condition.
  Exhaustiveness on a guarded case must NOT mark the outer variant
  covered (the guard could fail at runtime).

Out of scope of #36 because guards interact with both the parser and
the exhaustiveness checker; nested patterns only touched the typer.
