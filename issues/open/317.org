** OPEN #317 MLIR backend alongside the C backend

Add MLIR (Multi-Level Intermediate Representation, part of the LLVM
project, https://mlir.llvm.org/) as a codegen backend next to the C
backend. Filed as a low priority feature to capture the design space;
the C backend stays the default and the bootstrap path.

*** Why MLIR

- MLIR is organized around "dialects" (func, arith, scf/cf, memref,
  vector, gpu, llvm, ...) and progressive lowering: high-level ops stay
  target-independent until late, then lower dialect-by-dialect to LLVM
  IR and finally per-target machine code. One input IR can target x86
  (SSE/AVX/AVX-512), ARM (NEON/SVE), RISC-V vectors, and GPUs
  (NVVM/ROCDL) from the same source.
- Mojo is the proof that a small language can ride this: the whole
  language is a thin frontend over MLIR (Lattner's team; see the 2025
  LLVM devmtg talk "Building Modern Language Frontends with MLIR:
  Lessons from Mojo"). Mojo gets state-of-the-art CPU/GPU codegen and
  performance portability without owning any of that infrastructure.
- For til it opens what C emission cannot express directly: real vector
  types with target legalization (issue #318 is the first customer),
  GPU kernels, and compiler optimization passes below the C level.

*** Fit with existing til structure

- Issue #288 already drew the portability boundary with MLIR explicitly
  in mind: ~core_func~ is the mandatory primitive ISA every backend
  must lower inline (C -> ~a*b~, MLIR -> ~arith.muli~), ~ext_func~ is
  the per-backend symbol layer. That contract is the backend interface.
- ~Lang~ in src/self/targets.til already enumerates codegen languages
  (C, HolyC, TIL). MLIR would be a new ~Lang~ variant plus an emitter
  alongside the C emitter in src/self/builder.til, selected per target
  the same way HolyC is.

*** Implementation plan (staged; C backend stays green at every stage)

  Strategy in one line: emit the textual .mlir format into gen/ the
  way the C backend emits textual C, drive external tools
  (~mlir-translate~, ~clang~/~llc~) to get objects, keep the whole
  ~ext_func~ layer as the already-compiled C runtime linked in, and
  never link MLIR's C++ libraries into til itself. Only ~core_func~
  ops, control flow, and data layout are emitted as MLIR.

  1. Toolchain probe, zero compiler changes: hand-write a hello world
     .mlir (func + arith + llvm dialects, calling a C symbol), compile
     it with ~mlir-translate --mlir-to-llvmir~ + ~clang~, link it
     against the existing C runtime objects, run it. Record the exact
     tool invocations and minimum tool versions in this issue.
  2. Backend plumbing: new ~Lang.MLIR~ variant in src/self/targets.til
     mirroring HolyC (lang_to_str, target/lang selection, output
     paths). Proposed surface, to be approved at implementation time:
     .mlir emitted under gen/ next to the C output, compile driven by
     the existing build entry points, no new top-level commands.
  3. Emitter skeleton alongside the C emitter in src/self/builder.til:
     module header, ~func.func~ signatures using the SAME ABI the C
     backend emits (return value, ~_err_kind~ and payload out-params
     as pointer arguments), plus extern declarations for every
     ~ext_func~ symbol. ABI-compatibility with the C runtime is the
     load-bearing decision: the runtime is not re-implemented, MLIR
     code just calls it (#288's ~ext_func~ layer).
  4. Minimal end-to-end subset: main, scalar locals, ~core_func~ ops
     lowered inline to ~arith.*~ (the #288 contract: C -> ~a*b~,
     MLIR -> ~arith.muli~), direct calls, string literals as globals,
     print via ~ext_func~. A til hello world runs through the MLIR
     pipeline.
  5. Control flow: if/while/for/switch, and the throw/catch tag checks
     after calls, lowered with cf blocks (cf matches the shape the C
     emitter produces more directly than scf; revisit if scf's
     structured ops make the emitter simpler).
  6. Aggregates and memory: structs as ~llvm.struct~ with GEP field
     access, own/ref pointers as ~llvm.ptr~, constructor/clone/delete
     calls into the runtime. Str/Vec/Map/Set work at this point since
     they are field access + ~ext_func~ calls.
  7. Suite burn-down: run the test corpus under the MLIR backend and
     close the gap (dyn/FuncPtr, closures, FFI, gui mode last). Every
     landed stage keeps ~make test~ (C backend) green; the MLIR
     backend is allowed to stay partial between stages.

*** Acceptance

  - A smoke program compiles through gen/ .mlir and runs with output
    identical to its C-backend build.
  - ~Lang.C~ remains the default; boot/ and the self-hosting loop are
    untouched (the MLIR backend never enters the bootstrap).
  - ~core_func~ ops appear inline as arith/llvm dialect ops in the
    emitted .mlir (no calls); ~ext_func~ calls link against the
    unchanged C runtime objects.
  - ~make test~ does not require MLIR tools; the MLIR path has its own
    opt-in target/mode gated on the tools being installed.
  - #318's Simd ops have their vector-dialect emission point in the
    new emitter.
