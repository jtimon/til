# priority: low
** WIP #223 auto-shallow: infer shallow for small value types, forbid explicit shallow where automatic

  Currently ~shallow~ must be written explicitly on args and return values
  for types that are passed/returned by value (primitives, and small
  fieldless/payloadless structs/enums). This is boilerplate: the compiler
  already knows which types are value-sized, so the qualifier could be
  inferred. This issue tracks making ~shallow~ automatic for those types
  and then forbidding the now-redundant explicit ~shallow~ with a clear
  error.

  Steps (do them in order, one commit each):

  1. [DONE] Treat primitives as shallow without the qualifier.
     Primitives (I8/I16/I32/I64, U8/U16/U32/U64, F32/F64, Bool, etc.)
     used as args or return values are passed/returned by value
     regardless of whether ~shallow~ was written. Make the absence of
     ~shallow~ behave exactly like its presence for these types.
     Done: the typer normalizes primitive params/returns to Shallow in
     infer_func_def_expr / infer_func_sig_expr (ordinary func/proc and
     funcptr sigs; NOT ext_func/ext_proc, mut/own params, or a
     `returns ref` function's params).

  2. [DONE] Forbid explicit ~shallow~ on primitives.
     Done in two parts:
     - Removed every now-redundant explicit ~shallow~ on a primitive
       across src/, test/, examples/, lib/.
     - The typer now REJECTS a redundant explicit ~shallow~ on a
       primitive in an ordinary func/proc: "redundant 'shallow' on
       primitive ...; drop it". Auto-generated methods (initer) emit
       primitive params/returns without ~shallow~ so they flow through
       step-1 normalization and never trip the rule. The funcptr-sig
       variant of the check is sketched but not active (infer_func_sig_expr
       is a func with no ctx to raise the diagnostic from); funcptr sigs
       were cleaned, so nothing relies on it yet.

  3. [DONE] Extend step 1 to small value-shaped structs.
     A struct is "shallow" -- safe to copy byte-for-byte -- when ALL of:
       - ~MyStruct.size().lte(UPtr.size())~ (fits in a machine word; 8 on
         the current target), AND
       - it has no ~own~ fields (an ~own~ field is an owned heap pointer;
         copying the bytes would alias/double-free it -> "deep"), AND
       - it has no ~ref~ fields (disqualified for now; revisitable later), AND
       - every field's TYPE is itself shallow-eligible (a primitive, or an
         already-shallow struct). This transitive check closes the hole
         where a size-8 struct embeds a deep struct inline (e.g. a struct
         whose single field is itself a struct holding an ~own~ pointer).
     Such structs are treated as shallow in args/returns without the
     qualifier, exactly like primitives in step 1. (Enums: revisit
     separately; this step is structs.)
     Done: typer predicate is_shallow_struct + the unifying
     is_auto_shallow_type, wired into the step-1 param/return normalization in
     infer_func_def_expr. Builder fixes for value structs: use_dot_access /
     emit_field treat shallow params like stack locals ('.' not '->'); the
     mut-reassign + payload-alias paths assign a shallow-returning call's value
     directly instead of the pointer dance; emit_deref derefs a ref local
     before the shallow-param shortcut (a same-named shallow param of another
     func no longer false-positives). The #168 ref-init check
     (fcall_returns_own -> fdef_returns_unborrowable) now also rejects binding a
     ref to a shallow STRUCT return, since it is a fresh by-value result.

  4. Explicit ~shallow~ on a custom struct = force/override (NOT forbidden).
     Decision (revised from the original "forbid" plan): for custom
     structs we do NOT reject a redundant explicit ~shallow~. Instead:
       - if ~shallow~ is absent, the step-3 rule decides (auto-shallow when
         the struct qualifies, pointer-passed otherwise), AND
       - if ~shallow~ is present, honor it: force by-value passing and SKIP
         the auto-shallow eligibility check entirely.
     Rationale: a user who writes ~shallow~ on a struct is explicitly opting
     into value semantics; we take them at their word rather than second-
     guessing with the size/field heuristic. (This differs from primitives,
     where explicit ~shallow~ stays redundant-and-rejected per step 2,
     because a primitive is unconditionally value-typed.)
     Status: NOT STARTED. Need to confirm the typer already honors an
     explicit ~shallow~ struct param/return (it predates this issue -- e.g.
     ~shallow v: Value~), and that the step-3 normalization leaves an
     already-Shallow qualifier untouched (it only promotes the Ref/Own
     default). No redundant-shallow rejection for structs.

  Notes:
  - See ~doc/qualifiers.org~ for the shallow -> by-value C mapping and
    ~doc/ffi.org~ for how the binder currently emits ~shallow~ for
    primitives by default.
  - The error messages in steps 2 and 4 should point the user to remove
    the qualifier, not just say "invalid".
