** WIP #267 Replace dynamic core containers with templated (monomorphized) ones

   Move the core containers (Array, Vec, Set, Map, ...) from their current
   TYPE-ERASED ("dynamic") implementation -- one struct that stores
   elem_size + elem_clone/elem_delete function pointers and exposes a
   `Dynamic` element api -- to TEMPLATED (monomorphized) versions generated
   per element type, with a fully typed api and no stored type metadata.

   *** Progress

   - Caveat 1 (`struct<T>` sugar): DONE. `Name := struct<T: Type, ...> { BODY }`
     is parsed (parser.til, parse_struct_def + wrap_struct_def_in_macro) and
     desugars to `Name := macro(T: Type, ...) returns StructDef { return
     struct { BODY } }`, so it reuses the whole expansion/instantiation/dedup
     path. Multi-param works (`struct<K: Type, V: Type>`). test/templ_vec.til
     now declares TemplVec with the sugar.
   - Caveat 2 (expression-position): DONE. Inline instantiation works for
     value/receiver positions -- `mut v := TemplVec(Bool).new()` compiles
     in both interpret and build modes (test/templ_vec.til now uses the
     inline form for I64/Str/Bool/U8). The initer (`init_hoist_inline_macros`
     in src/self/initer.til) walks each unit, hoists every inline
     `MACRO(args)` call into a synthesized top-level `MACRO__Args := MACRO(args)`
     decl, and rewrites the use to reference it; the existing expansion loop
     then monomorphizes it. ASan-clean.
   - Caveat 2 (DEDUP): DONE. The synthesized name is derived
     deterministically from (macro, type-args), so N inline uses of
     `TemplVec(Bool)` in one file compile to ONE struct + one method set
     (verified in the generated C). Cross-unit dedup is also done: the
     initer keeps a Context-level set of synthesized monomorphization names,
     so the first unit that sees `Macro(Args)` owns `Macro__Args` and later
     units rewrite to that same name without emitting another type. Imported
     type-gen macros are resolved through scope, so sibling units can both
     instantiate a template defined in a shared dependency.
   - Caveat (type-annotation position): DONE for concrete instantiations.
     `mut v : TemplVec(Bool) = ...`, `func(x: TemplVec(I64))`, `returns
     TemplVec(Bool)` parse (parser.til parse_type_ref accepts `Name(args)`
     and mangles it to the same `Name__Args` the value-position hoist
     generates), so the annotation resolves to that monomorphization.
     Type-only occurrences now synthesize the matching top-level
     `Name__Args := Name(args)` decl during parse, and the initer dedups
     deterministic synthesized decls across units with the same Context-level
     set used by value-position hoisting. Covered by test/templ_type_only.til.
    - Caveat 3 (Dyn* in std): STARTED. The current type-erased `HashMap`
      implementation has been preserved as `DynHashMap` in
      src/std/dyn_hashmap.til, with duplicated regression coverage in
      test/dyn_hashmaps.til. The current type-erased `Array` implementation has
      also been preserved as `DynArray` in src/std/dyn_array.til, with initial
      duplicated coverage in test/dyn_arrays.til, before `Array` is migrated.
      The current type-erased `Vec`, `Set`, and `Map` implementations have also
      been preserved as `DynVec`, `DynSet`, and `DynMap` in src/std/, with
      duplicated regression coverage in test/dyn_vecs.til, test/dyn_sets.til,
      and test/dyn_maps.til.
    - STILL OPEN: 2b (drop `Self`) and 4 (migrate Array/Vec/Set/Map; leave
      `HashMap<K,V>` until typed Vec/buffer support is
     ready). Type-only synthesis is
     enough for concrete annotations, but removing `Self` globally is not
     correct yet: explicit user aliases like `Alias := make_type(I64)` still
     need method bodies to resolve `Self` to the alias LHS, not to a
     deterministic `make_type__I64` name.

   *** Foundation already in place (issue #243)

   `test/templ_vec.til` proves the model end to end: `TemplVec(T)` is a
   `macro(T: Type) returns StructDef` whose instances store only
   `{ data, count, cap }` -- NO elem_size, NO elem_kind, NO clone/delete
   function pointers. Size and element ops resolve directly at compile time
   per instantiation: `T.size()`, `src.clone()`, `T.delete(own ptr, false)`.
   Verified functionally equivalent to core Vec across I64/Str/Bool/U8 and
   leak/UAF-free under ASan. The enabling compiler change was making the
   macro type-param `T` (and the `Self` placeholder) substitute into
   namespace-method signatures AND bodies, including body-local `ref x : T`
   decls (src/self/initer.til).

   So the per-type win is demonstrated. What remains is ergonomics, a
   dedup guarantee, and the actual migration. We are close -- with the
   caveats below.

   *** Caveat 1: friendlier declaration syntax (parser sugar)

   Today a templated type is spelled:

   : TemplVec := macro(T: Type) returns StructDef {
   :     return struct { ... }
   : }

   We want a direct form that the parser DESUGARS to exactly the above:

   : TemplVec := struct<T: Type> {
   :     ...
   : }

   Pure parser-level convenience: `struct<P: Type, ...> { BODY }` lowers to
   `macro(P: Type, ...) returns StructDef { return struct { BODY } }`. No new
   semantics; the existing macro/StructDef machinery does the rest. (Note:
   `<>` here is unambiguous in declaration position; the general `Vec<T>`
   call-site question is a separate discussion -- see #243.)

   *** Caveat 2: inline instantiation must DEDUP to one compiled type

   The end goal is to use a templated type inline, by value, anywhere:

   : mut v  : TemplVec(Bool) = TemplVec(Bool).new()
   : mut v2 : TemplVec(Bool) = TemplVec(Bool).new()

   Two problems, now mostly done:

   1. Inline `TemplVec(Bool)` expands in value/receiver positions and in
      concrete type-annotation positions, including type-only occurrences
      with no value-position `Name(args)` occurrence in that unit.

   2. DEDUP: DONE within and across units. `test/templ_cross_unit.til`
      imports two sibling units that both inline the same imported template
      instantiation; only the first unit emits the monomorphization and the
      second unit refers to it.

   *** Caveat 2b: solving #2 retires the special-cased `Self`

   Templated types use a `Self` placeholder so namespace methods can name
   their own monomorphized type (`self: Self`, `returns Self`, `Self(...)`
   ctor calls). `Self` is a SPECIAL CASE added by the expansion
   (src/self/initer.til): it maps `Self -> <decl-LHS-name>` because aliases
   such as `Alias := make_type(I64)` need the method body to name `Alias`,
   not the deterministic macro-instantiation name.

   With type-only synthesis solved, a method on a deterministically generated
   inline type can write `TemplVec(T)` everywhere `Self` appears -- the SAME
   macro-call with the same `T` already in scope -- and dedup can resolve it
   to the monomorphized type:

   : TemplVec := struct<T: Type> {
   :     ...
   :     new := func() returns TemplVec(T) { ... }
   :     push := func(mut self: TemplVec(T), own val: T) { ... }
   :     clone := func(self: TemplVec(T)) returns TemplVec(T) { ... }
   : }

   But `Self` cannot be removed globally until explicit user aliases have an
   equivalent way to refer to their own LHS type.

   *** Caveat 3: keep the dynamic versions in std as DynArray/DynVec/...

   Copy the current type-erased implementations to std, renamed
   `DynArray`, `DynVec`, `DynMap`, `DynSet`, ... (duplicating their tests as
   `dyn_arrays`, `dyn_vecs`, `dyn_maps`, ...). They stay useful: a single
   shared erased impl is smaller code than N monomorphizations, which may
   matter for binary size on small/embedded targets, and for genuinely
   heterogeneous / reflective storage. So templated vs dynamic is a
   deliberate both-exist choice, not a replacement.

   *** Caveat 4: migrate the core types one at a time

   With Dyn* preserved in std, first migrate `HashMap` as the pilot: copy the
   current dynamic implementation to `DynHashMap`, duplicate its tests, then
   make `HashMap<K, V>` templated. After that, re-implement `Array`, `Vec`,
   `Set`, and `Map` as templated versions, ONE per change -- each is a
   sizable refactor (every call site that relies on the `Dynamic` element
   api, `.new(T)`, `elem_*` fields, etc. has to move to the typed api). Do
   them independently so each lands and stabilizes on its own.

   *** Dependencies

   This builds on the first-class-types / macro work: #55, #56, #57, #58
   (first-class structs/enums/functions, macros returning StructDef/EnumDef/
   FuncDef, FuncSig reflection). Those are well advanced in implementation,
   but some parts of caveats 1-2 may need pieces of them finished first
   (e.g. macro-call-in-type-position, deterministic monomorphization names).

   *** Suggested order

   1. Type-only instantiation synthesis. DONE.
   2. Caveat 2b: drop `Self` after explicit user aliases have an equivalent
      way to refer to their own LHS type.
    3. Caveat 3: copy dynamic implementations to Dyn* names with duplicated
       tests. STARTED with DynHashMap, DynArray, DynVec, DynSet, and DynMap.
   4. Caveat 4 core migration: Array, then Vec, then Set, then Map.
   5. Revisit `HashMap<K, V>` once typed Vec/buffer support is available.
