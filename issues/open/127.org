# priority: high
** WIP #127 Runtime ~Value~/binding ownership cleanup and ~Ptr~ split

The interpreter runtime currently overloads one mechanism for several
  different ownership cases:
  - scoped values live in ~Cell.val~
  - ~Scope~ bindings own some ~Cell*~ values via ~cell_is_local~
  - one runtime variant, current ~Value.Ptr~, is used for both:
    - borrowed/ref-like storage views
    - owning opaque heap pointers

  That causes two classes of problems:
  - real owner cleanup is missing at some scope teardown / overwrite boundaries
  - dispatcher/runtime code grows manual ~free(...)~ workarounds because the
    runtime value model does not distinguish borrowed pointer views from
    owned dynamic pointers cleanly

  Partial fixes landed (9d8a011 and subsequent):
  - ~scope_free~ now frees Enum payload data for owned cells
  - return values are cloned before ~scope_free~ to prevent dangling
  - Struct data in scope_free is still NOT freed (aliasing risk)
  - ~scope_set_owned~ now calls ~free_value~ on old value before overwriting

  Remaining pressure points:
  - current ~Value.Ptr~ is produced by:
    - ~null~
    - ref field reads
    - own field reads exposing owned storage
    - ~ptr_add(...)~
    - FFI ~Dynamic~ returns
    - container data fields like ~Array.data~ / ~Vec.data~
  - current consumers then guess whether that pointer is borrowed or owned

  Cleanup direction:
  - give runtime pointer-like values distinct meanings
  - stop overloading one variant for both ref-like and owning semantics
  - make the real owner delete runtime values at scope teardown and on
    overwrite
  - then simplify dispatcher helpers like ~val_to_ptr~ so they are only
    borrowed pointer-view helpers instead of ownership escape hatches

  Current candidate shape:
  - split current ~Value.Ptr~ into:
    - ~Value.Ref~ for borrowed/nullable pointer views
    - ~Value.Dynamic~ for owning opaque pointers
  - keep ~val_to_ptr := func(v: Value) returns ref Dynamic~
  - repair runtime binding cleanup before relying on translated dispatcher
    helpers as proof of correctness

  Important constraints discovered during analysis:
  - simple removal of ~Cell~ is not obviously safe yet, because current
    borrowed bindings are used for:
    - ref declarations from identifiers
    - borrowed function argument binding paths
  - own field reads cannot simply be classified globally as borrowed or
    owning without checking the typed/lowered move/clone behavior

  This should be tackled as a focused runtime ownership cleanup, not mixed
  into unrelated dispatcher translations.

  Progress:
  - Structural split landed: ~Value.Ptr~ is replaced by two variants,
    ~Value.Ref~ (borrowed/nullable pointer views) and ~Value.Dynamic~
    (owning opaque pointers). Producers are now classified:
    - ~Value.Ref~: ~null~, ~ptr_add(...)~, ref field reads, container
      data fields (e.g. ~Array.data~ in build_argv_array).
    - ~Value.Dynamic~: own field reads exposing owned storage, FFI
      ~Dynamic~ returns, ~cfile_open~ handles.
  - Two read-only view helpers added in interpreter.til:
    ~value_ptr_view(v) returns ref Dynamic~ and ~is_value_ptr(v)~.
    ~val_to_ptr~ in dispatch.til now matches both variants.
  - This step is intentionally behavior-preserving: neither variant is
    auto-freed yet. Auto-freeing ~Value.Dynamic~ at scope teardown /
    overwrite is NOT safe without first adding move/clone tracking --
    e.g. ~p := malloc(...); return p~ (return value is cloned to a
    borrowed view) and explicit ~free(own p)~ would both double-free or
    dangle. That owner-cleanup work is the remaining deferred piece,
    along with simplifying ~val_to_ptr~ to a pure borrowed-view helper
    and repairing ~scope_free~ Struct-data cleanup.
