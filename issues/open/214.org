# priority: low
** WIP #214 Unordered HashMap for big tables where key order does not matter

`Map` in [[file:../../src/core/map.til][src/core/map.til]] is a sorted-array
ordered map: every `set` does a binary search and then a `memmove` of the
suffix to keep keys in `cmp` order, every `get` / `has` is a binary search.
That's O(log n) per lookup and O(n) per insert in the worst case (the
memmove cost dominates once n is large).

For many of `bin/til`'s internal tables the key order is irrelevant -- we
only ever ask "is this name bound?" or "fetch the binding for this name".
The places that actually iterate in key order are rare. On big tables
(typer scope chains, the symbol table, the dyn dispatch tables, etc.) the
binary search + memmove cost is paid on every lookup and every insert,
even though we never observe the ordering it buys us.

*** Status

Initial implementation landed:
- [[file:../../src/std/hashmap.til][src/std/hashmap.til]] -- `HashMap` struct, separate chaining,
  load factor 0.75, no per-entry malloc. Adapted from rstil's
  src/std/hashmap.til (same shape).
- [[file:../../src/core/str.til][src/core/str.til]] -- `Str.hash` method (FNV-1a over content).
- [[file:../../lib/hash.til][lib/hash.til]] -- `hash_bytes(data, n)` primitive, default for
  any key whose in-memory bytes ARE the value.
- [[file:../../test/hashmaps.til][test/hashmaps.til]] -- 6 tests, all green under --asan.

Dispatch matches rstil: bytewise FNV-1a is the default; Str overrides
because its struct {c_str, count, cap} carries a heap pointer so a
bytewise hash would key on pointer identity, not content.

*** Resolved bug workarounds

- 64-bit Knuth / FNV-1a constants in [[file:../../lib/hash.til][lib/hash.til]] and Str.hash
  (issue #219 fix: builder now emits the ULL suffix).
- All `to_ptr(...)` wraps in [[file:../../src/std/hashmap.til][src/std/hashmap.til]] dropped
  (issue #217 fix: scalar args to Dynamic params autobox in interp).
- `ref val_slot : Dynamic = self.values.get(slot_idx)` restored
  (issue #218 fix: function-pointer dispatch no longer applies
  own-semantics across the bridge).

*** Still open

- Migration. No call site has migrated from `Map` to `HashMap` yet.
  Pick a hot table first (typer scope, dyn dispatch) and measure the
  crossover before bulk migrating.
- Robin-hood vs straight linear probing. Robin-hood smooths the worst
  case but adds bookkeeping; not worth it for the first cut.
- Removal. Tombstones vs back-shift; back-shift is simpler for linear
  probing and avoids the load-factor drift that tombstones cause.
- Benchmark target. Build a microbench that fills a Map vs a HashMap
  with 10k, 100k, 1M keys and measures get / set wall time.
