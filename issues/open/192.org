# priority: high
** WIP #192 USize should not be hardwired to U32

Current state mixes two separate problems:

1. ~USize~ is still effectively hardwired to ~U32~, even on 64-bit builds.
2. Code that is really talking about collection sizes/capacities sometimes
   gets routed through ~UPtr~ or explicit ~to_u32()~/~to_u64()~ conversions.

This creates avoidable friction in the compiler and core library:

- alias-resolution bugs around ~USize~ return types and declarations
- noisy conversion chains where a value should stay in one logical size type
- confusion between "count/capacity/len" values and raw pointer-width values

Desired direction:

- Make ~USize~ and ~UPtr~ target-driven in one place.
- On 32-bit targets: ~USize := U32~, ~UPtr := U32~.
- On 64-bit targets: ~USize := U64~, ~UPtr := U64~.
- Keep APIs that mean collection length/capacity/size on ~USize~.
- Use ~UPtr~ only where raw pointer arithmetic / byte offsets really need it.
- Clean up callsites that currently force this manually with things like
  ~to_u32()~ where ~to_usize()~ or no conversion should be needed.

This likely needs a careful pass through:

- builtin type registration / alias normalization in init + typer
- memory and pointer APIs in ~src/core/mem.til~
- collection constructors and helpers
- codegen / interpreter places that currently assume ~USize == U32~ on all targets

The recent ~dyn_size_of~ / collection-constructor work kept tripping over this,
so this cleanup should happen before more size-related refactors pile up.

*** Progress

Step 1: plumb a single source of truth without changing the underlying
width, so the bootstrap stays consistent.

- Added ~target_usize_name(target)~ in ~src/self/targets.til~ alongside
  the existing ~target_uptr_name~. Currently still returns ~U32~ on every
  target -- bumping it to ~U64~ on 64-bit is the next step.
- ~retarget_loaded_aliases~ in ~src/self/loader.til~ now retargets the
  ~USize~ alias the same way it already retargeted ~UPtr~.
- FFI dispatch (~src/self/dispatch.til~): ~shallow_ffi_type~,
  ~field_ffi_type~, and ~ffi_decode_scalar~ now consult the resolved
  ~USize~ alias via a new ~usize_ffi_type~ helper, instead of hardwiring
  ~ffi_type_uint32_ref~.
- ~src/self/builder.til~: replaced four ~last_slash.to_u32()~ /
  ~slash.to_u32()~ callsites that fed ~USize~ slots with ~to_usize()~,
  removing the manual width-conversion noise the issue calls out.

Step 2 (this commit): make ~I64.to_usize~ track the ~USize~ typedef.

- ~src/c/ext.h~ + ~src/c/ext.c~: declare and define ~USize I64_to_usize(I64)~
  as a one-line ~(USize)a~ cast. This stays equivalent to ~I64_to_u32~
  while ~USize~ typedefs to ~U32~, but it will widen to ~U64~ for free
  once ~ext.h~'s ~USize~ typedef goes target-driven.
- ~src/core/i64.til~: convert ~to_usize~ from a TIL wrapper that called
  ~val.to_u32()~ into an ~ext_func~ that resolves to ~I64_to_usize~. The
  TIL wrapper would have kept returning ~U32~ even after the alias flip;
  the ext_func form does not.

*** Remaining (to be done in follow-up commits)

The actual width flip on 64-bit targets is not done yet because it
participates in the C ABI between ~bin/til_boot~ and ~bin/til~. Doing it
safely needs the staged migration described in ~doc/self.org~:

- Ensure every ~USize~ field on Str/Vec/Array/Map/Set/Tuple/etc. and
  every ~USize~ parameter on container APIs is reachable through the
  alias (so flipping ~target_usize_name~ propagates everywhere at once).
- Make ~src/c/ext.h~'s ~USize~ typedef target-driven (mirroring the
  existing ~#if UINTPTR_MAX == 0xffffffff~ block for ~UPtr~), and adapt
  ~USIZE_REF~ accordingly.
- Then flip ~target_usize_name~ to return ~U64~ on 64-bit. This ships as
  a "Two-pass: " commit (or as a preparation+use pair) so ~boot/til.c~
  is regenerated against the new alias.
- After the flip, sweep callsites that route container sizes through
  ~UPtr~ or explicit ~to_u64()~ and route them back through ~USize~.
