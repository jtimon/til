# priority: high
** OPEN #155 Typer infers ~unknown~ for enum-typed struct field access

Spun off from ~[[./113.org][#113]]~. Accessing a struct field whose declared
   default is an enum constructor types as ~unknown~ instead of the
   field's enum type, so UFCS method calls on the field don't resolve.

   Repro in-tree:

   #+begin_src til
   Cell := struct {
       mut val := Value.None
   }
   ref c : Cell = ...
   if c.val.is(Value.Func) { ... }   // type error: no method 'is' for type 'unknown'
   #+end_src

   ~Value~ here is an enum, ~Cell.val~ has no explicit type annotation
   and is initialised to ~Value.None~. The typer should infer the
   field's til_type from the default-value expression
   (~Type.Enum("Value")~) and propagate it through field access so
   ~c.val~ has type ~Value~. Today it leaves the field at
   ~Type.Unknown~, which leaves the UFCS rewrite with no
   ~type_name~ to dispatch on and no namespace-method lookup target.

   Symptom site: [[file:../../src/self/precomp.til][src/self/precomp.til]] ~try_eval_call~ has a
   ~TODO([[./113.org][#113]])~ workaround that calls ~is(fn_cell.val, Value.Func)~ in
   free-function form for exactly this reason. Once the inference is
   fixed the workaround can be reverted to ~fn_cell.val.is(Value.Func)~
   and the TODO removed.

   Likely fix area: field-type inference in [[file:../../src/self/typer.til][src/self/typer.til]] /
   [[file:../../src/self/initer.til][src/self/initer.til]] where struct fields without explicit types
   pick up their type from the default-value expression. The same code
   path probably already handles primitive defaults (~mut x := 0~ ->
   ~I64~); the enum case looks like an oversight.

   Out of scope:
   - Exhaustive-switch checking for enums (~[[./113.org][#113]]~ part d).
   - Widening UFCS itself (already done in ~[[./113.org][#113]]~ follow-up).
