** OPEN #147 AST/Context redesign: lessons from rstil

rstil (the previous Rust implementation, at ~../rstil~) was slower and
   leakier than current til, but several of its architectural choices around
   the AST and Context were cleaner than what til carries today. This issue
   catalogs the ideas worth stealing.

   Root pattern: the AST should be the result of parsing, not a database for
   later phases. rstil keeps cross-phase facts in ~Context~; til grew AST
   fields whenever a pass needed to remember something. Three concrete
   anti-patterns follow from that:

   1. *AST-as-cache*: typer/initer write derived facts onto ~Expr~,
      ~Declaration~, ~FCallData~, ~StructDef~, ~EnumDef~ for later phases
      to read. The parse tree stops being immutable after parse.
   2. *Dangerous refs into the AST*: ~ref mut .. : Expr = NULL~ fields
      stitched onto AST nodes (and onto ~TypeBinding~) point into the
      parse tree from outside. When a pass rebuilds a body via
      clone-and-filter the refs go stale; surviving them depends on every
      pass agreeing not to drop the original node. Owned ~Expr~ fields or
      lookup-by-name in ~Context~ are safer.
   3. *Per-phase state without a home*: pre-rebase, til had no
      ~imports_*_done~ memos, no precomp side-table; whether an import got
      re-walked depended on incidental control flow elsewhere. Per-phase
      Sets in ~Context~ (rstil pattern) make it explicit.

*** Audit (read/write site counts; remaining dangerous refs after ~fn_sig~ removal)

   **** Dangerous ~ref mut .. : Expr = NULL~ refs -- DONE

   All AST-borrowing refs on AST/scope structs are now gone:

   - ~Declaration.field_struct_def~ -- removed in step 4 (~2417f71~);
     the defensive ~resolve_field_struct_defs~ /
     ~reregister_field_struct_defs~ walks went with it.
   - ~FCallData.fn_sig~ -- removed (step 7); ~resolve_fn_sig~ /
     ~fcall_fn_sig~ name+lookup replacements were the template.
   - ~TypeBinding.func_def~ -- removed (commit ~1c9265a~). The FuncDef
     Expr lives in ~scope.func_defs~ (name -> Dynamic cell); readers
     use ~scope.lookup_func(name)~; ~func_type~ caches the FuncType.
   - ~TypeBinding.struct_def~ -- removed (commit ~7c5280c~). The
     StructDef/EnumDef Expr lives in ~scope.struct_defs~; type-name
     resolution uses ~scope.get_struct(name)~ (fall-through walk),
     exact-binding "is THIS binding a type?" uses
     ~scope.struct_def_of(name)~ (gated, no fall-through). The two
     lookups are distinct on purpose -- collapsing them miscompiled
     the self-host compiler (a shadowing local read as a type), and a
     latent ~type_from_name~ alias-return UAF surfaced and was fixed
     in the same commit.

   ~TypeBinding~ now carries no ~ref Expr~ into the parse tree at all.
   The only remaining AST-as-cache item is step 5
   (~Declaration.explicit_type~), which is blocked on the FuncPtr
   name-loss described above (#58 / step 8).

   Read characterization:

   - Most non-typer reads are simple pivots: ~e.til_type.is(Type.Struct)~,
     ~til_type_name(e.til_type)~, ~til_type_to_c(e.til_type)~. They
     want a concrete ~Type~ value, not a re-derivation walk.
   - Builder + interpreter together: 119 reads, *zero* writes. They're
     pure consumers running after typer.
   - Writes outside typer: literals (parser/precomp -- trivially
     re-derivable from ~node_type~), and a few synthesized-node writes
     in initer/precomp where the synthesizer already knows the type.

   Implications for removal:

   - *Pure re-derivation* in hot paths (path 2 from earlier): the
     builder reads ~til_type~ ~97 times during code emission. Each
     read becoming a scope walk is a real cost. Probably not viable
     without caching.
   - *Side table* (path 1, step 11): would require some form of Expr
     identity to key the table, and til has explicitly decided
     against adding node IDs / Expr.id. So this path is *off the
     table*; do not pursue. (rstil also has no Expr identity --
     ~Expr~ in rstil is ~{node_type, params, line, col}~. It works
     without a side table by computing types on demand from
     ~Context~ + AST shape; see below.)
   - *rstil-style derive-from-context+AST*: rstil's
     ~init.rs:get_value_type(context, e)~ is a pure function of
     (Context, Expr) that walks the AST + scope_stack to produce a
     ~ValueType~. No per-Expr storage, no node IDs. Idents resolve
     by name lookup in ~scope_stack~; literals are implied by their
     ~NodeType~ payload; FCall return types come from the resolved
     callee. Narrowing-as-mutation (~narrow_dynamic~ rewriting
     ~Expr.til_type~ in place) does not exist there: when til-style
     narrowing would be needed, rstil either uses an explicit
     ~.dereference(T, &out)~ (e.g. ~Array.get~ returns ~Ptr~, not a
     bare ~Dynamic~) or hoists into a typed binding so the lookup
     returns the narrowed type for the new name.
   - *Hybrid* (intermediate step toward the rstil pattern): keep
     ~til_type~ on selected node types where consumers pivot on it
     (FCall return type, FieldAccess result type, Ident resolved
     type), and trim the ~Expr~ field into a NodeType-payload thing
     for those cases. Smaller win, less invasive than full removal,
     compatible with the no-node-id constraint.

   Bottom line: ~Expr.til_type~ removal landed via #206 (hybrid
   payload-storage path; full side-table-with-node-id rejected, as
   was direct rstil-style derive-from-Context+AST in hot paths). Step
   7's FCallData purge is now also done. The remaining dangerous-ref
   work (~field_struct_def~, then ~TypeBinding.struct_def~ /
   ~func_def~) is the actionable next structural move. Both also help
   leak reduction (cloners no longer need to keep referenced nodes
   live just for the ref).

   ~Context.struct_layouts~ (~88723f1~), step 4 dropped
   ~field_struct_def~ (~2417f71~) and the rest (~5372e3c~). Readers
   route through ~ctx.struct_layouts~ via the
   ~ctx_total_size~ / ~ctx_field_layout~ helpers; the defensive
   ~resolve_field_struct_defs~ / ~reregister_field_struct_defs~
   walks went with the field. See [[./247.org][#247]] for the optional
   step 5 (drop the stored map entirely, compute layout on demand
   in the rstil-pure shape).

   **** FCallData resolution artifacts (step 7) -- DONE

   All four typer-resolution count/index fields are gone (~b1161e7~,
   ~13816f5~, ~30dfe64~, ~e296692~). The original "six fields, none
   are parse facts" framing was incorrect: ~is_splat~ (from ~..expr~
   call syntax) and ~own_args~ (parser sets bits from ~own expr~
   argument syntax, then the typer syncs/shifts) are parse facts, in
   the same keep-bucket as ~does_throw~ (#198) and ~is_bang~ (#202).
   What got removed:

   - ~kwargs_count~ -- derived structurally via ~fcall_kwargs_count~
     as the trailing-NamedArg count (NamedArg only survives the
     desugar's name-to-fixed-slot matching when its name matches the
     kwargs param).
   - ~variadic_count~ -- derived via ~fcall_variadic_count~ from the
     callee's nparam + has-kwargs flag + children layout. Two helpers:
     a bare derivation for callers holding ~fdef_data~, and a
     scope-resolving wrapper for sites that don't.
   - ~kwargs_index~ -- derived via ~fcall_kwargs_index_via_scope~ as
     ~nparam~ (kwargs is always the last param). The body-pre-pass
     uses only the >= 0 sign as a "has kwargs block" predicate; the
     position itself is correct only post-variadic-rewrite, which is
     when the only positional consumer (~process_call_desugars~'s
     kwargs branch) reads it.
   - ~variadic_index~ -- derived via ~fcall_variadic_index_via_scope~
     as ~1 + fdef_data.variadic_index~. The "is the variadic block
     already rewritten" state previously encoded by
     ~variadic_index = -1~ post-rewrite is handled by trusting the
     ~process_call_desugars~ pipeline order: each FCall is processed
     once, with variadic rewrite running before kwargs derivation, so
     no caller needs to distinguish pre- vs post-rewrite states.

   The scope-resolving helpers (~fcall_*_via_scope~) all mirror the
   same Ident-binding / Type.method namespace lookup that
   ~resolve_variadic_elem_type~ uses, and ~resolve_fn_sig~ /
   ~fcall_fn_sig~ (the originals that replaced ~fn_sig~) remain the
   template the rest of the dangerous-ref purge follows.

   **** explicit_type / til_type duplication (step 5 finish)

   ~Declaration~ now carries both ~explicit_type : Str~ and
   ~til_type : Type~ kept in sync at write sites (~5f822fc~,
   ~8e72cd3~). ~243~ ~explicit_type~ read sites across
   ~src/self/*~ still pivot on the string.

   The naive plan ("migrate readers to ~til_type~, drop
   ~explicit_type~") is the wrong direction. ~explicit_type~ is the
   parse-time fact; ~til_type~ is filled in by the typer. Dropping
   the parse fact while keeping the typer-mutated cache makes the
   cache the database -- the AST-as-cache antipattern. It also makes
   step 11 (~Expr.til_type~ removal) /harder/, because step 11 wants
   re-derivation from parse facts + scope, not trust in a cache.

   The correct shape, matching rstil's single-field
   ~Declaration.value_type~: have the parser write
   ~til_type = Type.Custom(name)~ at parse time (the unresolved form),
   stop writing ~explicit_type~, then migrate readers and drop
   ~explicit_type~. The typer's job becomes refining ~Type.Custom~ ->
   ~Type.Struct~ / ~Type.Enum~ / ... where it can, not filling an
   empty slot. This is larger than a mechanical reader migration and
   is best done as part of the step-11 push, not before.

   **** Expr.til_type per-node cache (step 11, stretch) -- DONE

   Removed via ~[[./206.org][#206]]~. ~expr_til_type~ is now a
   pure function of ~node_type~ + payload (no per-Expr storage):
   per-variant type storage lives on the NodeType payloads
   (~IdentData~, ~LiteralNumData~, ~FCallData~, ~FieldAccessData~,
   ~ForInData~) and trivially-derivable variants derive on demand.
   The rstil-style derive-from-Context+AST shape (no payload
   storage at all) was not pursued.

   **** Dangerous ~ref mut .. : Expr = NULL~ refs (cross-cutting)

   Raw refs into the parse tree, scattered across structs that outlive
   or pre-date the nodes they point at. When any pass clones-and-filters
   ~body.children~ the refs can dangle. Catalog:

   - ~Declaration.field_struct_def~ -- *removed* (~2417f71~). The
     defensive ~resolve_field_struct_defs~ /
     ~reregister_field_struct_defs~ walks went with it.
   - ~FCallData.fn_sig~ -- *removed*. Was the original motivating
     case; the "name + lookup" replacement
     (~resolve_fn_sig~ / ~fcall_fn_sig~) is now the template for the
     remaining purges.
   - ~TypeBinding.struct_def~, ~TypeBinding.func_def~
     ([[file:../../src/self/initer.til:19-20][src/self/initer.til:19-20]]) -- these live in ~scope.bindings~,
     not on the AST, but they reference into the AST and have the same
     dangling-on-rebuild risk. Same fix: store name + look up the
     authoritative payload via ~scope.get_struct~ /
     ~scope.lookup_func~.
   - Per-walk ~ref mut current_*_def : Expr = NULL~ thread-locals in
     ~typer.til~ / ~builder.til~ / ~interpreter.til~ -- *migrated to
     Context fields* per #177; module-scope-only ref-NULL carve-out is
     gone. Listed here for completeness.

*** Things til does better than rstil (honest balance, kept)

   - ~OwnType~ 3-way enum (Own/Ref/Shallow) beats rstil's two bools
     (~is_own~/~is_ref~).
   - Tagged-union ~NodeType~ with per-variant payload types is cleaner
     than rstil's manually-unrolled match arms; recent ~switch~
     destructuring migrations (~ae8b4ec~, ~b96bd5e~, ~57696a4~,
     ~ab843fe~ and friends) make this even nicer to read.
   - ~FCallData.own_args~ U64 bitmask avoids the tree restructuring
     rstil's ~OwnArg~ wrapper node forces.
   - Per-file ~ImportUnit~ + ~Context.imports_*_done~ Sets mirror
     rstil's design without rstil's ~Ptr~+~malloc~+~memcpy~ Box<Expr>
     dance.
   - ~[[./130.org][#130]]~ itself: rstil had no ongoing simplification tracking.
