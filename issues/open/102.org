** OPEN #102 Auto-generate eq and to_str for structs and enums

Auto-generated namespace helpers are synthesized during init
  (~init_declarations~, implemented in ~src/self/initer.til~). Structs
  currently get ~clone~, ~delete~, ~size~, and sometimes ~hash~. Enums
  currently get constructors/variant constants, ~is~, ~eq~, ~delete~,
  ~to_str~, ~clone~, and ~size~.

  The missing feature is one capability-gated rule for generated ~eq~
  and ~to_str~ across both structs and enums:

  - for structs, generate ~eq~ when every instance field is comparable
  - for structs, generate ~to_str~ when every instance field is printable
  - for enums, apply the same rule to payload variants
  - tag-only enum variants are always comparable and printable
  - existing user-written namespace methods still win
  - if a generated method would need unsupported field/payload methods,
    omit it and let the user write an explicit method

  Relationship to ~[[./125.org][#125]]~:

  ~#125~ is the concrete enum-payload failure: enum generation currently
  emits ~eq~ / ~to_str~ bodies that call payload methods without first
  proving those methods exist. This issue should fix that by using the
  same support check for struct fields and enum payloads. Once this is
  implemented, ~#125~ should be closed by the same commit.

  - [ ] Struct field-wise ~eq~ generation.
