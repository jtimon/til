   --- Original issue below ---

   A top-level type keyword, ~interface~, modelled on Java interfaces and
   C++ abstract classes. An interface is a NAMED, fieldless contract of
   methods. It DESUGARS to a struct with EMPTY fields; the methods live in
   the struct's namespace. Two hard prohibitions: an interface cannot have
   instance fields, and it cannot carry a ~namespace:~ marker (there are no
   fields to separate from the methods).

   A member is one of two kinds:

   1. *Abstract* -- declared with ~ext_func~ / ~ext_proc~ and an empty
      ~{}~ body. It has no implementation here; a type that declares it
      implements the interface MUST provide it. This reuses the
      ~ext_func(...) returns T {}~ shape (a declaration whose body is
      supplied elsewhere), so abstract interface methods read exactly like
      external declarations.

   2. *Concrete (default)* -- an ordinary ~func~ / ~proc~ WITH a real body.
      The interface supplies the implementation itself; conforming types
      inherit it (like a non-pure method on a C++ abstract class). These
      are regular namespace methods.

*** Example

    : Drawable := interface {
    :     // abstract: every implementer must provide `draw`
    :     draw := ext_func(self: Drawable) returns Str {}
    :     // concrete default: provided by the interface
    :     label := func(self: Drawable) returns Str { return "shape" }
    : }
    :
    : Collection := interface {
    :     get := ext_func(self: Collection, i: U32) returns Dynamic {}
    :     len := ext_func(self: Collection) returns USize {}
    : }

    ~Collection~ names the ~get~/~len~ pair a ~for ... in~ loop relies on
    (both abstract). ~Drawable~ shows the mix: ~draw~ is abstract, ~label~
    a concrete default.

*** Conformance (`implements`) -- next step

    A type declares it satisfies an interface with ~implements~:

    : Circle := struct implements Drawable {
    :     r := 1.0
    : namespace:
    :     draw := func(self: Circle) returns Str { return "circle" }
    : }

    The typer verifies ~Circle~ provides every ABSTRACT (ext_func/ext_proc)
    method ~Drawable~ lists, with a matching signature; concrete defaults
    are inherited and need not be re-declared. A missing or mis-typed
    abstract method is a type error. Conformance checking is the NEXT step;
    the interface-definition side -- the two member kinds plus the
    no-fields / no-namespace rules below -- comes first.

*** Semantics

    - ~Name := interface { ... }~ desugars to a ~StructDef~ flagged
      ~is_interface~, with EMPTY ~fields~ and ~ns_decls~ = the members.
    - Members must be ~ext_func~/~ext_proc~ (abstract) or ~func~/~proc~
      (concrete). All of these are reported by the TYPER (not a parser
      panic) so the diagnostics aggregate in test/typer_errors.til:
      - a non-func member (a "field") is a type error;
      - a ~test~/~macro~/~lazy_func~/~lazy_proc~ member is a type error;
      - a ~namespace:~ marker is a type error.
    - The interface type is usable anywhere a (fieldless) struct type is:
      as a parameter / field / variable type annotation.

*** Non-goals (future issues)

    - No dynamic dispatch / vtables / interface values that erase the
      concrete type.
    - Edge case: an interface that re-declares ~clone~/~delete~/~size~
      suppresses the compiler's auto-generated versions for that empty
      struct, so INSTANTIATING such an interface leaves it without real
      clone/delete/size bodies. Only instantiate interfaces whose method
      names do not collide with the auto-gen trio.

*** Implementation (touch points)

    1. *Lexer* (src/self/lexer.til): ~TokenType.KwInterface~ keyword.

    2. *Parser* (src/self/parser.til): ~parse_interface_def~ produces a
       ~NodeType.StructDef(is_interface = true)~ with empty fields; each
       member parsed like a namespace decl, NO validation here (a
       ~namespace:~ marker sets ~StructDef.interface_ns_marker~ for the
       typer to reject). Hook ~KwInterface~ into ~parse_primary~.

    3. *Typer* (src/self/typer.til, ~infer_type_def_expr~): enforce the
       member-kind rules and the no-namespace rule as aggregatable type
       errors. Auto-generated clone/delete/size members are skipped.

    4. *Pipeline guards.* Abstract (ext_func/ext_proc) and bodyless
       members must not crash the namespace-method walkers
       (ffi_init_scan_program, emit_ns_method, the interpreter call path);
       each guards the no-body case.

*** Tests

    - Positive (test/interfaces.til): ~Collection~ (abstract ext_func
      methods) and a ~Drawable~ mixing an abstract method and a concrete
      default; compiles and runs on BOTH backends.
    - Negative (test/typer_errors.til): a field, a ~namespace:~ marker,
      and a non-func-kind member (e.g. ~test()~) are each rejected.
