** #173 Vec.new(Expr) syntax: copy rstil's runtime-type-name trick

Goal:

   - Replace ~Vec.new("Foo", Foo.size())~ with ~Vec.new(Foo)~ across
     the codebase (and similarly for Map/Set/Array/List/Tuple).
   - Mirrors the rstil trick where the codegen converts a type
     identifier passed to a ~Type~-typed parameter into a string
     literal at the call site, and runtime size is looked up via a
     generated ~til_size_of(name)~ switch.

   Plan (incremental commits):

   1. [X] Add ~dyn_size_of(type_name: Str) returns USize~ as a new
      ext_func, paralleling ~dyn_call~/~dyn_fn~/~dyn_has_method~.
      Interpreter handler reuses ~get_elem_size~. Codegen emits a
      ~U32 dyn_size_of(Str *)~ function with a ~memcmp~ switch over
      every struct/enum returning ~sizeof(<TypeName>)~.
   2. [X] Slim ~Vec.new~ to a single ~elem_type: Str~ arg and call
      ~dyn_size_of~ internally. Rewrite all ~Vec.new~ callsites
      (~50 sites).
   3. [X] Same treatment for ~Array.new~, ~Map.new~, ~Set.new~,
      and any ~Tuple~ fields that pass ~T.size()~.
   4. [X] Add ~dyn_type_to_str(T)~ ext_func: at the call site, the
      codegen converts the type identifier ~T~ to the string literal
      ~"T"~. Special-cased in the typer like ~dyn_call~ requires its
      method arg to be a literal.
   5. [X] Rewrite all ~Vec.new("Foo")~ callsites to ~Vec.new(Foo)~
      and equivalents for the other collections. End state: no
      ~T.size()~ noise at any callsite, no string-literal type names
      either.

   Resolution summary:

   - Step 1 added ~dyn_size_of(type_name: Str)~ and the generated C
     runtime/type-size switch.
   - Step 2 removed explicit ~T.size()~ noise from ~Vec.new~ and
     rewrote the original string-based callsites.
   - Step 3 completed the remaining collection/container surface work:
     ~Tuple.push(T, val)~, ~Set.new(T)~, ~Map.new(K, V)~,
     ~Vec.new(T)~, and ~Array.new(T, cap)~.
   - For ~Set~, ~Map~, ~Vec~, and ~Array~, the migration used staged
     bootstrap-safe commits with temporary ~new_newi~ aliases (and for
     ~Array~, one extra ~new_type_name~ preparation commit for raw C
     builder emission).
   - Step 4 added ~dyn_type_to_str(T)~ plus generic Type-token lowering
     in typer / builder / interpreter so bare type identifiers can be
     passed through ordinary function parameters.
   - Step 5 is complete: source callsites now use type tokens directly,
     including ~Vec.new(Foo)~, ~Set.new(Foo)~, ~Map.new(K, V)~,
     ~Array.new(Foo, cap)~, and ~Tuple.push(Foo, val)~.
   - Final cherry on top: public source code can now spell the compiler's
     type object as ~Type~ via ~Type := TilType~, while the bootstrap
     compiler and generated C continue to use ~TilType~ internally.

   Final state:

   - ~vec(...)~ and ~array(...)~ are desugared in the typer now, not
     lowered specially by builder / dispatch.
   - There is no core ~List~ type in this tree.
   - Call sites no longer need explicit ~T.size()~ or string literal
     type names for the supported collection constructors.

   Side issue uncovered (may be subsumed by master's [[./165.org][#165]]):

   - When the ~dyn_size_of~ ext_func declaration was placed at the
     bottom of [[file:../../src/core/dyn.til][src/core/dyn.til]] (after the ~CloneFn~/~DeleteFn~/...
     ~default_clone~/... block), an unrelated assert in
     ~misc.til~'s ~Bool.and()~ chain would fail. Reproducible only
     with this exact name + that exact position; renaming at the
     same position passed. Workaround in step 1: place the
     declaration with the other ~dyn_*~ funcs. Master's [[./165.org][#165]]
     ("Declaration order should not matter") looks like the same
     family.
