** FIXED #282 Throwing call in a switch/match case guard runs unconditionally

*** Post-mortem

    Root cause: the pre-infer throw/bang pass hoisted a guard's throwing
    (bang) call out to a temp *before* the whole switch -- because
    ~hoist_walk_consolidated~ recursed into a ~Case~ node's guard child (a
    bare expression, not a ~Body~). So the call ran once, unconditionally,
    and an empty ~v[0]~ panicked ~OutOfBounds~ even when the pattern never
    matched. (~match~ guards have the identical ~Case~ shape and bug.) The
    conditional position that should gate the guard only exists *after*
    inference, where ~infer_switch_stmt~'s #199 flag desugar lowers a
    guarded case to ~if _sw_matched.not().and(pattern) { bindings; if
    <guard> { body; _sw_matched = true } }~. ~?~-guards were already fine:
    a ~?~ call is never hoisted by ~hoist_walk_consolidated~ (it only
    handles bangs), so it stayed inside the guard and rode the desugar's
    pattern gate.

    Fix (all in src/self/typer.til):
    - ~hoist_walk_consolidated~ now returns on a ~Case~ node, so a guard's
      bang is no longer hoisted before the switch (case bodies are still
      lowered via the step-2 recursion; patterns hold no calls).
    - A new ~register_bang_guard_throws~ walk runs in the pre-infer pass
      (from the ~Switch~ step-2 handler and ~process_match_arms_in~). It
      discovers each guard bang's throw types and registers them into
      ~types_to_declare~/~seen~ (and ~throw_type_registry~) so the
      function-level ~_err_kind~/~_err_<T>~ slots are still declared --
      without hoisting.
    - The validate+inject step skips a bang guard (in both the ~Switch~
      step-4 handler and ~validate_and_inject_expr_consolidated~'s ~Case~
      recursion) so the surviving ~!~ isn't flagged "function does not
      throw".
    - ~infer_switch_stmt~ now lowers the guard's bang at infer time:
      ~lower_guard_bang_calls~ seeds the bang callee's receiver type from
      the live typer scope (~seed_guard_receiver_types~ into
      ~lowering_param_types~), then reuses ~hoist_walk_consolidated~ to
      hoist the bang into the pattern-gated case body, before the inner
      guard ~if~. The throwing call (and its panic-on-throw check) now run
      only on a pattern hit -- short-circuit preserved.

    Regression tests: test/enums.til
    (~test_guard_throwing_call_short_circuits~,
    ~test_guard_throwing_call_runs_on_hit~,
    ~test_guard_throwing_call_false_guard_falls_through~) and test/match.til
    (~test_match_guard_throwing_call~). Verified in interp and compiled.

   A throwing call in a case *guard* is evaluated even when the case's
   pattern does not match, so it can panic on a value the guard was never
   meant to see. Minimal repro (panics ~OutOfBounds~ in both interp and
   compiled, despite the guard's pattern never matching):

   : t := test() {
   :     v := Vec(I64).new()       # empty
   :     mut r := 0
   :     switch v.len() {          # 0
   :     case 1 if v[0].gt(0): r = 1   # pattern 1 != 0; guard v[0] must NOT run
   :     case: r = 2
   :     }
   :     expect(__LOC__, r.eq(2), "guard not evaluated when pattern misses")
   : }

   It compiles cleanly; at run time ~v[0]~ (a throwing ~get(0)!~) is
   evaluated unconditionally and the empty ~v~ panics. A non-throwing guard
   is fine; this only bites when the guard contains a throwing call
   (subscript, ~f()?~, ~f()!~).

*** Root cause

    A guard is the middle child of a 3-child ~Case~ ~[pattern, guard,
    body]~ -- a bare expression. The throw/bang lowering pass
    (~process_throw_catch_in_body~ -> ~hoist_walk_consolidated~) recurses
    into it (it is not a ~Body~) and hoists the guard's throwing call out
    to a temp *before* the whole switch statement, so it runs once,
    unconditionally. The case body is fine -- step 2 of the pass recurses
    into switch case ~Body~ nodes -- but the guard is not.

*** Why it is not a one-liner (the infer-timing wall)

    The guard's conditional position only exists *after* inference:
    ~infer_switch_stmt~'s flag-based desugar (#199) lowers a guarded case to
    ~if _sw_matched.not().and(pattern) { bindings; if <guard> { body;
    _sw_matched = true } }~. The throw/bang pass runs *before* inference, so:

    - Just skipping ~Case~ in ~hoist_walk~ (stop the premature hoist) leaves
      the guard's call unlowered; inference then fails with "missing
      argument for parameter '_err_kind'".
    - Running the throw pass on the case body *during* inference (after the
      inner ~if <guard>~ is built) fails because the throws-resolver
      (~find_callee_throws_key~ -> ~callee_owning_type~ -> ~resolve_value_type~)
      determines a method's throw set by scanning the body AST for the
      receiver's declaration -- the case body does not contain it, and the
      receiver is not inferred yet at that point either. Attempting this
      produces "function does not throw, remove the '!'" / "may throw
      'OutOfBounds' but it is not in the throws clause".

    The #262 match-arm fix (~process_match_arms_in~) sidestepped exactly
    this by processing un-lowered ~match~ arm ~Body~ nodes *during* the
    pre-infer throw pass, where the enclosing function body (and so the
    receiver's decl) is in scope. A guard's inner ~if~ does not exist
    pre-infer, so it cannot use the same trick.

*** Fix direction

    Give the throw machinery a way to lower throwing calls in
    conditionally-evaluated positions that are *created during inference*:

    - Make throws-resolution consult the typer scope / the receiver's
      inferred ~til_type~ instead of only scanning the body AST, so it can
      run after inference; OR thread the enclosing function body plus an
      explicit ~root_body~ argument into the guard lowering so the existing
      pass resolves against the function body, not the case body. Then hoist
      the guard's throwing call to the top of the (pattern-gated) case body
      with local ~_err_kind~/~_err_<T>~ slots, so it runs only on a pattern
      hit -- short-circuit preserved.

    This is the same root cause as #262 (throwing calls in
    conditionally-evaluated positions lowered during inference); a unified
    infer-time lowering would cover both match arms and case guards.

   Cross-refs: [[./262.org][#262]] (throwing calls in ~match~ arms -- the
   sibling case, fixed), #199 (switch guards / flag-based desugar).
