** #153 Interpreter reinterpret_ptr_value dropped enum tag

  ~ref va : E = m.get("a")~ always read back tag 0 in the interpreter
  regardless of the stored value, so any pointer-returning API that
  hands out an enum (Map.get, FFI return, heap-held enum field)
  produced the wrong variant. Compiled mode was correct because the C
  read path reads the tag straight from the pointer.

  Root cause: ~reinterpret_ptr_value~ in [[file:../../src/self/interpreter.til][src/self/interpreter.til]]
  had a stub for ~NodeType.EnumDef~ that hardcoded ~etag = 0~ and
  copied 8 bytes from the enum pointer into a new ~Value.Enum~
  payload, ignoring the actual tag at ~*p~ entirely. Integer branches
  in the same function all read from ~p~ correctly; the enum branch
  was an outlier.

  Fix:
  - Read the 4-byte tag at ~*p~ into ~etag~.
  - If the enum has no payloads, return ~val_i32(etag)~ to match the
    representation ~FieldAccess~ already uses at line 441 for bare
    variant constructors (~val_i32(tag)~).
  - If the enum has payloads, copy the 8 payload bytes from
    ~p + ENUM_PAYLOAD_OFFSET~ (not ~p~ itself) and build a
    ~val_enum_flat~ with the actual tag.

  Discovered while fixing [[./150.org][#150]] (module-level enum Map literal in
  compiled mode) and initially registered with ~-no-interp~ as a
  workaround; now [[file:../../test/bug150.til][test/bug150.til]] exercises both fixes on the
  default ~flags~ (interpret + compile + run).
