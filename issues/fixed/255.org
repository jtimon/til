** FIXED #255 lexer: richer number literals (hex/binary/octal, digit separators, exponents)

*** Post-mortem
Root cause: this was a missing feature, not a bug -- the lexer's Number
case only scanned decimal digits plus a single optional decimal point,
and the value parsers (~Str.to_i64~/~Str.to_f32~) only understood base-10
with no separators.

Fix (all in one commit):
- ~src/self/lexer.til~: added ~is_hex_digit~/~is_bin_digit~/~is_oct_digit~
  helpers and rewrote the Number scanning case to recognise base prefixes
  (~0x~/~0b~/~0o~, case-insensitive), ~_~ digit separators in every base,
  and float exponents (~1e9~, ~1.5e-3~). ~is_digit~ stays decimal-only.
- ~src/core/str.til~: ~Str.to_i64~ now parses an optional ~-~, a base
  prefix, and ~_~ separators (shared ~digit_value~ helper); ~Str.to_f32~
  strips ~_~ before handing the text to strtof (which already handles
  exponents). These cover the typer (~literal_in_range~), interpreter,
  and constfolder paths since they all route through these two parsers.
- ~src/self/builder.til~: added ~num_lit_to_c~ to normalise a literal's
  text into a valid C constant before emit -- binary/octal become a
  decimal constant, ~_~ separators are dropped, hex/decimal/float pass
  through (preserving the #219 huge-U64 hex/decimal path, which never
  routes through to_i64). A first attempt reassigned a ~mut s~ binding to
  a substring view of itself, freeing the buffer the view pointed into
  and emitting garbage bytes for every negative literal; rewritten to
  use index offsets and owned copies.

Regression test: ~test_number_literals~ in ~test/operators.til~ (runs in
both interp and compile modes) covers hex/binary/octal, digit separators,
negative base-prefixed literals, float exponents, typed contexts
(U8/U32/I32/F32), and ~Str.to_i64~ at runtime.

The lexer only recognises base-10 integers and a single optional decimal
point (~tokenize~ Number case: digits, then ~.~ + digits). Production
lexers for similar languages (Zig, Odin) also accept:

- hex ~0x1F~, binary ~0b1010~, octal ~0o17~ (and/or ~0~-prefixed),
- digit separators ~1_000_000~, ~0xff_ff~,
- exponents ~1e9~, ~1.5e-3~ (and hex-float forms).

*** Scope
This is a *feature*, not an optimisation -- only worth doing if/when the
language wants these literal forms.

Touch points:
- ~src/self/lexer.til~: extend the Number scanning (and add the relevant
  digit-class helpers; today ~is_digit~ is decimal-only).
- The number *value* is parsed later from the token text (~literal_num_node~
  -> ~Str.to_i64~ / ~Str.to_f32~), so those parsers must learn the new
  bases/separators too, or the lexer must normalise the text.
- ~F32~/integer ~to_str~/~to_repr~ round-tripping should stay consistent.
- Add lexer + parser tests (~test/parsing.til~, a lexer test) covering each
  new form and rejecting malformed ones (e.g. ~0x~ with no digits,
  trailing ~_~).

*** Why medium
Pure additive feature with a clear spec; no urgency, but a natural gap
versus comparable languages and self-contained once the value parsers are
extended.
