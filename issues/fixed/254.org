** FIXED #254 codegen: emit a real C switch (jump table) for dense int/char/enum switches

  POST-MORTEM
  Implemented the revised-scope plan in two commits. The typer's
  switch_is_clike detects a C-like switch and infer_clike_switch
  type-checks it in place, leaving Switch/Case nodes for the backends
  instead of desugaring to if/elif; infer_switch_stmt returns whether it
  lowered so the caller skips the re-infer pass for kept switches.

  C-like = scrutinee is an integer/char, OR an enum whose scrutinee is a
  simple lvalue, AND every case is a bare constant: an int/char literal,
  or a bare `Type.Variant` (FieldAccess, no payload). No payload binding,
  range, guard, string compare, duplicate label, or loop-targeting `break`
  (a til `break` would be captured by the C switch; til `continue` maps to
  C `continue`, which a C switch does not capture). Enum switches reuse the
  desugar's coverage machinery for exhaustiveness; an exhaustive enum
  switch with no `case:` turns its last case into the `default:` so the C
  switch has a terminal arm (no -Wreturn-type).

  Emission: builder.emit_stmt_switch writes `switch (expr) { case K: {..}
  break; }` for int/char and `switch ((expr).tag) { case Enum_TAG_V: ... }`
  for enums; interpreter.eval_switch dispatches on the scrutinee value
  (value_switch_key reduces enum values to their tag). The garbager
  recurses into surviving switch case bodies for exit-edge cleanup.

  Verified: make test 154/154 (both backends). boot/til.c now contains 22
  int/char C switches (e.g. the lexer's lookup_single_char). The full
  suite (test/enums.til, test/strings.til, lexer/typer tests) exercises
  both backends as the regression guard.

  Known limitation (by design -- listed as a fallback in the original
  scope): a switch that binds a payload in ANY case -- the bulk of the hot
  enum switches in typer/builder/dispatch (`case NodeType.Foo(x):`) -- is
  mixed and stays desugared, so it does NOT get a jump table yet. Only
  all-bare-variant enum switches and int/char switches convert. A future
  enhancement could split a mixed switch into a C switch on the tag with
  payload extraction inside each case.

The typer's ~infer_switch_stmt~ (src/self/typer.til) currently desugars
EVERY ~switch~ (and ~match~) into a nested if/elif chain of equality tests
before codegen. The builder and interpreter therefore never see a real
switch node -- only if/elif ~Body~ trees -- so an N-case switch costs O(N)
comparisons at runtime and the C compiler can never build a jump table
(see ~boot/til.c~, e.g. ~lookup_single_char~: each case is an ~I8_eq~ +
~if/else~).

This shows up directly in the lexer: ~tokenize~'s ~switch c~ dispatch,
~lookup_single_char~, ~lookup_two_char~, ~tok_name~, etc. are all
sequential comparisons. It matters far more in the hot ~switch~ statements
all over ~typer.til~ / ~builder.til~ / ~dispatch.til~ that run once per AST
node per phase.

*** Why keep both lowerings
til's ~switch~/~match~ is far richer than C's: payload binding, ranges,
guards, string compares, exhaustiveness. Desugaring all of that to if/elif
in the typer (as today) is correct but throws away the cheap case. The
answer to "why support emission for both?" is this issue: efficiency. So
we split by complexity -- the typer keeps owning the rich lowering, but a
C-like subset survives untouched to the backends.

*** Goal (revised scope)
Keep the typer as the single owner of switch lowering, but teach
~infer_switch_stmt~ to detect a *C-like* switch and NOT desugar it:
- scrutinee is an integer/char, or an enum *tag* (switching on ~.tag~), and
- every case label is a constant, non-overlapping literal, with
  - NO payload binding (~case NodeType.Ident(id):~),
  - NO range (~case 'a'..'z':~),
  - NO guard (~case X if cond:~),
  - NO string compare.

A C-like switch is left as canonical ~Switch~/~Case~ nodes. Everything
else keeps desugaring to the if/elif chain exactly as today.

Backends then gain a switch path that did not exist before:
- the builder emits a real C ~switch (expr) { case K: ...; default: ... }~
  for the surviving ~Switch~ nodes; the C compiler picks a jump table or
  binary search.
- the interpreter gains a direct ~Switch~ dispatch (previously unreachable
  because everything was pre-desugared).

*** Notes / risks
- ~continue~/~break~ inside a til ~switch~ must keep their current meaning
  (continue/break the ENCLOSING loop). A C ~switch~ captures ~break~, so
  the lowering must route til ~break~/~continue~ past the C switch (a label
  + ~goto~, or by keeping the body out of the C switch's ~break~ scope).
- Needs constant, non-overlapping integer labels. Ranges either expand to
  many C ~case~ labels or stay desugared.
- BOTH backends change now -- the interpreter is no longer a no-op. Verify
  with two-pass ~make test~ and check ~doc/totals.csv~ for a build-time win.

*** Related
See #261 (category: experiments) for the inverse direction -- desugaring
if/else INTO switch, inspired by Gleam. A 2-case boolean switch produced
that way would be trivially C-like and reuse this issue's emission path.

*** Why medium
Broad, compiler-wide performance lever with no behaviour change, but it
touches both backends' control-flow lowering (the ~break~/~continue~
interaction needs care), so it is a real piece of work rather than a quick
win.
