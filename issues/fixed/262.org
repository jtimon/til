Introduce ~lazy_func~ and ~lazy_proc~ as new callable kinds (alongside
  ~func~/~proc~ and ~ext_func~/~ext_proc~). When you call a lazy callable,
  its arguments are NOT evaluated eagerly at the call site; each argument
  expression is treated as a thunk and is only evaluated if and when the
  body uses it. Most of the work should live in the desugarer.

Motivation:
  - It unblocks short-circuiting ~and~/~or~ (#117). With lazy args, ~and~
    is just:
        and := lazy_func(a: Bool, b: Bool) returns Bool {
            if a { return b } else { return false }
        }
    and ~b~ is never evaluated when ~a~ is false. ~or~ is the dual.
  - It enables a variadic, Lisp-style ~and~/~or~ that short-circuits across
    N operands, e.g. ~and(p, q, r, ...)~ folding to nested ~if~s, instead of
    the chained ~.and(...).and(...)~ method form.

Sketch of the desugaring approach:
  - At a lazy call site, the desugarer inlines the callee's body and splices
    each argument expression in at the points where the corresponding
    parameter is used (call-by-name / macro-style expansion). For ~and~/~or~
    this collapses to the ~if~ forms above with no thunk allocation.
  - Variadic lazy and/or desugar by right-folding the operand list into
    nested conditionals.

Open questions / honest caveats (these are why this is its own issue, not a
  trivial rewrite):
  - "All in the desugarer" only holds when the lazy callee can be inlined at
    the call site. A first-class lazy func (stored in a var, passed as a
    funcptr) or a recursive one cannot be fully desugared by inlining; those
    cases need real thunks/closures or must be disallowed initially.
  - Side-effect / evaluation-order semantics change by design: an unused lazy
    argument's side effects never run. This needs to be documented clearly.
  - Each lazy parameter that is used more than once in the body would
    re-evaluate its argument unless we bind it once; decide whether lazy means
    call-by-name (re-eval) or call-by-need (memoized once).

Design options considered (no approach picked -- documented for the human to
decide):

  Core constraint. Ordinary variadic args become a runtime collection of
  VALUES (today an ~Array(Bool)~ built in ~desugar_user_func_fcall_args~).
  Building that collection evaluates every operand, which destroys
  short-circuiting. Switching ~Array~ -> ~Vec~ does NOT help: the trap is
  storing values, not the container. To defer evaluation an operand must be
  stored as something that is not yet the value -- either no runtime
  collection at all (compile-time expansion, Design 1) or a collection of
  thunks (Design 2).

  Design 1 -- compile-time unroll (generalize the current fold; zero runtime
  cost). Keep the variadic as a compile-time list of argument ASTs; the
  author writes the fold as head/tail recursion and the desugarer unrolls it
  by the call's known arity:
      and := lazy_func(xs: ..Bool) returns Bool {
          return match xs {
          case []:        true                                  // identity
          case [x, ..tl]: match x { case true: and(tl...) case: false }
          }
      }
  ~and(a, b, c)~ expands to nested ~match~es bottoming at ~true~: no
  allocation, full short-circuit, and the AUTHOR (not the compiler) defines
  the fold shape and base case, so associativity is no longer assumed.
  Needs new surface features: (a) a spread in arg position (~and(tl...)~),
  (b) compile-time head/tail list patterns (~[]~, ~[x, ..tl]~), (c)
  recursion-unrolling in the expander. Could stay a pre-typer structural
  pass. Limits: arity must be statically known (fine for direct calls, not
  first-class); generated code grows ~linearly with operand count. Cannot be
  done with ~match~/~switch~ today -- the patterns and spread are genuinely
  new features (worth their own issue, useful beyond lazy_func).

  Design 2 -- array of thunks (call-by-name; first-class; runtime cost).
  Make the variadic ~Array(() -> T)~: each operand is wrapped in a zero-arg
  closure at the call site (no evaluation, just closure construction), and
  the body forces on demand:
      and := lazy_func(xs: ..Bool) returns Bool {
          for x in xs { if x.not() { return false } }   // x auto-forced
          return true
      }
  Fully general: runtime arity, first-class (storable/passable), no
  recursion needed. Reuses til closures (#54; ~typer_lift_lambdas~ in
  src/self/desugarer.til). Costs: one closure allocation per operand plus
  indirect calls; must live in desugarer.til (type-directed -- it builds
  typed closures). Needs auto-force (the compiler inserts the call at each
  use of a lazy param so the body reads ~x~, not ~force(x)~) and a thunk
  ownership/lifetime story (who owns the closure + captured env, when freed).

  Design 3 -- recursion with pop. Same REPRESENTATION as Design 2, just a
  recursive body instead of a loop: ~match force(xs.pop()) { case true:
  and(xs) case: false }~ with ~xs~ a ~Vec~ of thunks. NOT a separate option:
  a ~Vec~ of VALUES would be eager (see core constraint); only a ~Vec~ of
  thunks works, which is Design 2 with a different body shape.

  Promises vs thunks. A Scheme-style ~promise~ (~delay~ / ~force~) is exactly
  a MEMOIZED thunk (call-by-need): the first force caches the result, later
  forces reuse it. So "array of promises" = Design 2 + memoization, not a
  separate mechanism. (JS / std::future async promises are orthogonal AND
  wrong here: they start eagerly and defer only delivery, giving no
  short-circuit.) Memoization only pays off when a lazy param is used MORE
  THAN ONCE in a body; ~and~/~or~ use each operand at most once, so plain
  call-by-name thunks already evaluate each operand 0-or-1 times and a
  promise's cache cell would never earn its keep there. Call-by-need also
  adds the "not-forced-yet" representation + cached-value ownership cost.

  One candidate -- "inline when possible, thunk otherwise", with semantics
  = call-by-name:
    - Pick call-by-name. Under it, inline-splice and thunk-call are
      OBSERVATIONALLY IDENTICAL (an unused arg never runs; an arg used N
      times runs N times), so the compiler may choose either lowering per
      call site invisibly to the user. Call-by-need would break this
      equivalence (inline re-evaluates per use, a promise evaluates once),
      which is a further reason to defer it.
    - Mental model: a lazy_func IS a function whose args are thunks (the
      always-correct general lowering); INLINING is a pure optimization
      applied where the body is inlinable and arity is statically known.
      ~and~/~or~ keep the zero-cost inline path; everything else falls back
      to thunks (still lazy, still short-circuit).
    - Benefits: (a) generic lazy_func / lazy_proc with no user-visible
      special cases; (b) ~and~/~or~ stay binary and inlined -- no
      ~var_and~/~var_or~ split needed; (c) it fixes the silent-eager bug
      below for free (a non-inlinable body falls back to THUNKS, not eager).
    - Gotchas to nail down: thunk ownership/lifetime in til's manual model;
      auto-force insertion; and capture-by-REFERENCE so a thunk that reads a
      variable the body mutates before forcing agrees with the inline splice
      (matters only for side-effecting / mutating captures; moot for pure
      args like and/or operands). The mechanism spans two phases: inline
      stays pre-typer (needs raw bodies), thunk-lowering is type-directed
      (desugarer.til).

  Silent-eager fallback bug (FIXED). A lazy_func whose body was not inlinable
  used to be silently left un-inlined and degrade to an EAGER call (right
  value, but all args evaluated, no diagnostic -- a correctness bug, since a
  guarded throwing / out-of-bounds operand would panic). Now: the if/else
  single-return shape inlines (see v1 status), and ANY other body shape
  (multi-statement, loops, ...) is rejected with a HARD ERROR at the lazy
  definition -- ~lazy_body_inlinable~ (src/core/ast.til) is the shared shape
  predicate; the typer checks it in ~infer_func_def_expr~ before the body is
  lowered. Negative test: test/typer_errors.til (terr_bad_lazy). When the
  thunk fallback lands, this hard error becomes a THUNK lowering instead, so
  those bodies start working rather than erroring.

  A possible ordering (for the human to decide, not chosen): (1) DONE --
  stopped the silent-eager fallback (hard error) and widened the inline path
  (return-expr, if/else, guard chains, N-ary fold, UFCS); (2) Design 2 thunks
  as the generic fallback, keeping the inline path as the fast path; (3)
  Design 1's compile-time list-patterns + spread; (4) call-by-need / promises.

Note: the binary short-circuit ~and~/~or~ from #117 could also be done as a
  narrow, special-cased desugar rewrite (~a.and(b)~ -> ~if a { b } else
  { false }~) without any of this general machinery. This issue is the
  general path that additionally buys user-defined lazy callables and the
  variadic form; #117 can be satisfied by either.
