** #157 Rust-style ~///~ doc comments and ~help()~ builtin
  Replace the emacs-mode-only ~/** ... */~ doc-comment placeholder with
  real Rust-style ~///~ doc comments. The text rides through the
  lexer and parser, lands on ~Declaration.doc~, and is exposed to the
  user interactively in the REPL via a ~help(name : Str)~ builtin so
  ~help("Vec")~ prints the registered doc + signature/fields/methods
  info. Scope reduction: ~help()~ is REPL-only. Non-REPL pipelines
  (~til interpret/build/run/test/translate~) deliberately do NOT
  populate the runtime doc registry, so ~help(name)~ from a script or
  compiled binary always prints "undefined symbol: <name>". Compiled
  binaries stay lean and source-level browsing happens through Phase
  2's dedicated ~til doc <path>~ generator.

** Phase 1 -- doc comments + REPL ~help()~ builtin

  - ~editor/til-mode.el~: highlight ~///~ as ~font-lock-doc-face~;
    drop the legacy ~/** ... */~ regex.
  - [[file:../../src/self/lexer.til][src/self/lexer.til]]: add ~TokenType.DocComment~. Recognise
    ~///~, capture the rest of the line (stripped of the leading
    ~///~ and one optional space), emit one DocComment token per
    line. Plain ~//~ and ~/* */~ continue to be skipped as before.
  - [[file:../../src/core/ast.til][src/core/ast.til]]: add ~mut doc := ""~ to ~Declaration~.
  - [[file:../../src/self/parser.til][src/self/parser.til]]: at the top of ~parse_statement~ drain a
    run of leading DocComment tokens, joining them with ~\n~. After
    the inner statement parses, if the result is a ~NodeType.Decl~
    attach the accumulated text to its ~Declaration.doc~. Doc
    comments that don't precede a Decl are discarded. Inside
    ~parse_struct_def~ / ~parse_enum_def~ the same drain runs on the
    namespace blocks so ~Declaration.doc~ is attached to ns_decls
    and struct fields too.
  - [[file:../../src/core/doc.til][src/core/doc.til]] (new): two packed runtime registries
    ~__til_docs_blob__~ and ~__til_info_blob__~ encoded as
    ~name<0x01>value<0x02>...~; ~__doc_init__(info, docs)~ entry
    point that just stores both blob references; linear-scan
    ~__doc_lookup__~ helper; ~help(name : Str)~ that prints the
    doc text and the rich info string (kind + signature for funcs;
    fields + namespace methods for structs/enums; "<name> is a
    <Type>" for variables -- inferred type comes from the typer
    scope).
  - [[file:../../src/core/core.til][src/core/core.til]]: ~import("doc.til")~ so the registries and
    ~help~ are visible to user code.
  - [[file:../../src/self/loader.til][src/self/loader.til]]: ~doc_init_src_line(core_ast, user_ast,
    scope)~ helper that returns a ~__doc_init__("...", "...")~
    source line populated with packed blobs built from the typed
    core+user ASTs. Top-level Decls register under their bare name
    and struct/enum members register under their qualified name
    (~Vec.pop~, ~Foo.bar~) so ~help("Vec.pop")~ finds the method's
    doc and signature. Non-REPL pipelines do not call this.
  - ~src/til.til~: REPL prepends ~doc_init_src_line~ to each turn's
    subprocess source so ~help(name)~ resolves against the freshly
    typed program on every input. ~repl_typecheck~ mirrors master's
    post-rstil ~prepare_program~ pipeline (~init_file~ ->
    ~init_declarations_global~ -> ~normalize_scope_bindings~ ->
    ~type_file~) so the REPL's typed AST stays in sync with what
    ~til interpret~ sees.
  - On-demand smoke test: [[file:../../test/repl_help.til][test/repl_help.til]] (a TIL ~mode cli~
    script) pipes ~help("println")~ into ~bin/til~, captures the
    output, and asserts the expected core doc text appears. Wired
    via the ~make test_repl_help~ Makefile target -- NOT into
    ~make test~. Spawning ~bin/til~ + a subprocess ~til interpret~
    for a single help() turn costs 30-80 s of wall time depending
    on the host (the longest single test by far), so the cost
    would dominate ~make test~. Run on demand alongside changes
    that touch the REPL or doc registry.

** Phase 1 follow-up -- speed up ~doc_init_src_line~

  The first implementation walked every core ns_decl on every REPL
  turn, rebuilding a ~30 KB info blob from scratch (~80 s on this
  host for a single ~help()~ turn). Fix landed: cache the core
  packed info/docs blobs in ~run_repl_session~ on the first
  successful typecheck and reuse them forever after; per-turn work
  only walks the user units. [[file:../../src/self/loader.til][src/self/loader.til]] now exports
  ~build_doc_init_line(packed_info, packed_docs)~ so the REPL can
  stitch cached core + fresh user blobs without re-walking core.
  The fix amortizes well over many turns but the first turn still
  pays the cache-fill cost, which is why the single-turn
  ~repl_help~ smoke test was eventually pulled too (see Phase 1
  Tests bullet above).

** Phase 2 -- ~til doc <path>~ generator

  New top-level subcommand ~til doc <path>~ runs init + typer
  (skipping precomp + scavenge -- those are codegen prep), then
  walks each loaded ProgramUnit's typed AST and emits one org-mode
  page per source file under ~doc/gen/<source-path>.org~. Each
  page lists every top-level ~Decl~ with its ~///~ doc text (if
  any) followed by a kind + signature info block (reusing
  ~format_decl_info~ from Phase 1). Struct/enum bodies render
  their fields and namespace methods inline, including the
  field/method-level ~///~ comments.

  Implementation in [[file:../../src/self/builder.til][src/self/builder.til]]:
  - ~format_unit_doc_org(unit_path, ast, scope) returns Str~
    (per-decl chunks accumulated then concatenated; sized
    correctly so push_str doesn't overflow on big files like
    [[file:../../src/self/initer.til][src/self/initer.til]])
  - ~til_doc_out_path(unit_path) returns Str~ (strips leading
    ~./~, ~/foo.til/~ -> ~/foo.org/~)
  - ~ensure_parent_dir(path)~ -- ~mkdir -p~ for the output dir
  - ~emit_unit_doc(u, lp)~ + ~cmd_doc(lp) returns I32~

  [[file:../../src/self/loader.til][src/self/loader.til]]: ~prepare_program~ split into
  ~init_and_type_program~ (init + typer, used by ~til doc~) plus
  the existing ~precomp_imported~ + ~scavenge_imported~ tail
  (used by ~translate~/~build~/~run~/~interpret~/~test~).

  ~src/til.til~: dispatch ~doc~ before ~prepare_program~ via
  ~init_and_type_program~ + ~cmd_doc~; usage banner.

  Workflow: ~make doc~ regenerates ~doc/gen/~ on demand. Not wired
  into ~make test~ -- a full doc gen on ~src/til.til~ runs the
  init+typer pipeline end-to-end and adds ~1 minute of wall time,
  which would dominate the test suite. The first version of this
  fix DID wire a ~doc_cmd~ smoke test through ~test/doc_cmd.sh~
  but it pushed totals.csv up by ~60 s, so it was pulled out and
  replaced with the ~doc~ Makefile target. Refresh ~doc/gen/~
  alongside notable doc-affecting changes; the ~.gitignore~
  exception (~!doc/gen/~) lets the generated tree be tracked
  despite the ~gen/~ rule.
