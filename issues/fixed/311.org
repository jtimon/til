** FIXED #311 Enum payload narrower than 8 bytes mis-extracts in COMPILE mode

Post-mortem
-----------
Root cause: two separate spots hardcoded a pointer-sized (8) payload offset
instead of the payload union's true offset.

1. Compile mode (the reported bug). The get_payload lowering in
   src/self/builder.til emitted a fixed `sizeof(void *)` byte offset for the
   payload read. The enum lowers to `struct E { <tag> tag; union {...} data; }`,
   so `data` sits at whatever offset the payload's alignment forces -- 1 for a
   byte-aligned Bool, 8 only for pointer/I64/Str/struct payloads. Reading offset
   8 of a 2-byte struct returned garbage. Fix: emit `offsetof(<enum>, data)` --
   the constructor's own offset (it stores `r->data.V = *val;`), so read and
   write always agree -- and add `#include <stddef.h>`. The enum name comes from
   the get_payload receiver's til_type. The auto-generated `<enum>_delete`
   method passes get_payload as a pre-typed own-arg, which the typer's arg pass
   skips (it only infers Unknown-typed args), so its `self` receiver arrived
   untyped and produced `offsetof(, data)`; src/self/initer.til now sets the
   receiver's enum type at the source in the delete generator.

2. Interpret mode (latent, surfaced by the regression test under ASAN). The
   interpreter's flat enum value stores the payload at a fixed
   ENUM_PAYLOAD_OFFSET (8), decoupled from the tighter C layout, but
   val_enum_flat sized the buffer by the C struct size (2 for a Bool payload).
   Writing the payload at offset 8 then overflowed the buffer -- a heap-buffer-
   overflow ASAN caught (the value was still correct: write@8, read@8). Fix:
   size the buffer to ENUM_PAYLOAD_OFFSET + payload_size (floor 8 for tag-only
   values) in src/self/interpreter.til.

Regression tests: test/enums.til `test_switch_payload_bool` (Bool payload via
`match`) and `test_switch_payload_i32` (I32 payload via `switch`) assert the
extracted value under both interpret and compile;
`test_narrow_enum_struct_field`, `test_narrow_enum_struct_clone`, and
`test_narrow_enum_by_value_and_ref` cover a narrow-payload enum stored inline
BY VALUE (struct field, struct clone, by-value / ref-bound parameter), which the
offset-8 buffer overran in the interpreter until the pinning below made every
side agree.

Update -- read/write ABI unified via pointer-pinning (commit 77e1b89)
---------------------------------------------------------------------
The `offsetof(<enum>, data)` read above made compile-mode extraction agree with
the C struct's NATURAL layout, but three other readers of the enum ABI still
assumed a FIXED offset of sizeof(void *) (8 on a 64-bit host): ext.c's
`get_payload()`, the interpreter's `ENUM_PAYLOAD_OFFSET`, and `X.size()` /
`ctx.struct_layouts` (used for container striding). A narrow payload (Bool,
I8/U8, I16/U16, I32/U32, F32, or a USize payload under `--usize=32`) sits below
offset 8 in the natural layout, so those readers missed it -- and, in the
interpreter, the offset-8 flat value overran the tighter C-sized struct field it
was copied into (an ASAN heap-buffer-overflow; the value survived because write
and read both used 8).

Rather than teach every reader the per-enum natural offset, 77e1b89 pins the
payload to sizeof(void *) on BOTH sides so the fixed offset is always correct
(and supersedes the `offsetof` read):
  - `emit_enum_struct_body` emits a `void *_til_payload_align` sentinel in the
    payload union, forcing `data` to pointer alignment and >= pointer size, so
    it always starts at offset 8; the pattern-match binds
    `(U8 *)self + sizeof(void *)` again.
  - `compute_enum_layout` places the payload at `UPtr.size()` with a slot of at
    least `UPtr.size()`, so a folded `X.size()` / container stride matches the
    emitted `sizeof`.
Enums that already carry a pointer/Str/U64/struct payload are byte-for-byte
unchanged (their `data` was pointer-aligned and >= 8 already). This is the
current implementation; the `offsetof` read and the offset-8 buffer sizing
described above were interim steps.


A `switch`/`match` case that binds an enum payload reads the WRONG bytes in the
C backend whenever the payload's natural alignment is under 8. The value comes
out garbage (often reads as a neighbouring/padding byte), silently. The
interpreter is correct -- it does not use byte offsets -- so this only bites
compiled programs.

Minimal repro (compile mode returns the wrong Bool; interpret is fine):

    E := enum { V: Bool, W }
    pick := func(t: E, dflt: Bool) returns Bool {
        return match t { case E.V(x): x  case: dflt }
    }
    # pick(E.V(true), false)  -> interpret: true (correct); compile: wrong

Root cause (confirmed by reading the generated C):
  The enum lowers to
      struct E { U8 tag; union { Bool V; } data; };
  so `data.V` sits at offset 1 (U8 tag, then a byte-aligned Bool). But the
  get_payload extraction emits a FIXED byte offset of `sizeof(void *)` (8):
      Bool *x = ((void *)((U8 *)(t) + sizeof(void *)));
  Reading offset 8 of a 2-byte struct yields garbage.

  It works by luck for payloads whose alignment forces the union to offset 8 --
  I64, pointers, Str, and struct payloads (8-aligned) -- which is why every
  existing enum-payload test (Token.Num(I64), Shape.Dot(Point), ...) passes and
  this stayed latent. It is broken for any payload narrower/less-aligned than a
  pointer: Bool, I8/U8, I16/U16, I32/U32, F32.

  The fix is to use the payload union's TRUE offset (an offsetof-style value, or
  the tag size rounded up to the payload's alignment) instead of the hardcoded
  `sizeof(void *)`. The constructor already stores at the right place
  (`r->data.V = *val;`), so only the read side (get_payload emission) is wrong;
  they must agree.

Where: the get_payload byte-offset codegen (search the builder / get_payload
lowering for `sizeof(void *)` used as the payload offset). Cross-check against
`enum_tag_size` -- the tag is U8 here, not pointer-sized, so any offset keyed on
pointer size is suspect.

Discovered while finishing #262 item 2 (a lazy body binding a param name in a
case pattern); it is unrelated to lazy_func -- reproduces on the plain non-lazy
`pick` above. #262's tests deliberately use an I64 payload to dodge this bug.

Regression test when fixed: add a Bool-payload (and ideally an I32-payload)
enum match to test/enums.til asserting the extracted value under BOTH interpret
