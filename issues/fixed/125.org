** FIXED #125 Enum payload method generation should not assume ~eq~/~to_str~

Post-mortem:

The root cause was that ~gen_enum_nonclone_methods_for_stmt~ always
  generated enum ~eq~ and ~to_str~ when the enum namespace did not already
  define them. For payload enums, those generated bodies called payload
  ~.eq(...)~ and ~.to_str()~ without proving the payload type supported
  those methods, so otherwise-valid payload enums failed later during
  generated helper typing.

The fix adds conservative capability checks in ~src/self/initer.til~ and
  generates enum ~eq~ / ~to_str~ only when every payload variant supports
  the method that the generated body would call. Tag-only variants and
  function-pointer payloads remain supported. ~src/self/builder.til~ now
  also emits enum ~_eq~ C/Python binding declarations only when an enum
  actually has an ~eq~ method.

Regression test: ~test/regression.til:bug125~.

This is the concrete bug case for ~[[./102.org][#102]]~. Enum helper
  generation currently emits ~eq~ and ~to_str~ for payload enums without
  first checking that every payload type supports the methods the generated
  body will call.

  Current behavior:
  - ~eq~ by calling payload ~.eq(...)~
  - ~to_str~ by calling payload ~.to_str()~

  That fails for payload types like ~Expr~, which currently do not expose
  those methods, even though the enum itself may still be useful for
  internal compiler bookkeeping.

  Example that currently fails during self-hosting:
  - ~CtorArg := enum { Unfilled, Filled: Expr }~

  Expected rule:
  - generate enum ~eq~ only when all payload variants are comparable
  - generate enum ~to_str~ only when all payload variants are printable
  - tag-only variants do not block generation
  - existing user-written namespace methods still win
  - if the check fails, omit the generated method instead of emitting a
    body that will fail later in the pipeline

  This should be fixed as part of ~#102~. If ~#102~ implements the shared
  support-gated generation rule for structs and enum payloads, this issue
  becomes a duplicate symptom and should be closed by that implementation
  commit.
