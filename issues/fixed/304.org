** FIXED #304 Upgrade vendored libffi (3.4.6 -> latest)

:RELATED: [[./25.org][#25]]

   POST-MORTEM: vendored libffi is now v3.7.1, taken from the upstream
   git tag (the release-tarball hosts, github release assets and
   sourceware, are unreachable from bot sessions) with configure and
   friends generated by autogen.sh -- see the REM note by the
   Makefile's vendor/libffi/.built recipe for the exact toolchain.
   The 3.4.6-era local backport (upstream 8308bed, sysv.S cfi order)
   is upstream since 3.4.7, so no local patches remain in the tree.
   vendor/bindings/libffi.til was regenerated with `til bindings` off
   3.7.1's ffi.h and re-audited per doc/ffi.org: the mutating calls
   flipped back to ext_proc (incl. the new 3.7.1 call-plan API's
   allocator), the issue-#25 hand-edits restored (ffi_cif
   extra_cif_fields slot, ffi_abi warning comment) and both marked
   keep-across-regens; doc/ffi.org gained a libffi hand-edits section
   mirroring the nng one. The new generator types ffi_type's
   alignment/type fields as U16, so the interpreter's two
   FFI_TYPE_STRUCT casts moved to to_u16(). Regression coverage: the
   FFI suite tests (ffi_interp, ffi_run, ffi_scavenge_link_c_run,
   ext_libc_run) via make test. The commented "uncoment when upgrading
   dependency libraries" block under `make clean` became a real `make
   clean_vendor` target (git clean of the raylib/tinyfd/nng/libffi
   build artifacts): after merging a vendor upgrade a stale
   vendor/libffi/.built stamp would keep linking the old archive, and
   the old version's leftover generated Makefile cannot be trusted to
   clean the new tree.

   The vendored libffi is release 3.4.6. It carries one local backport
   (issue #25 macOS bring-up): upstream commit 8308bed ("Move
   cfi_startproc after CNAME(label)") in src/aarch64/sysv.S, without
   which Apple clang's assembler rejects the file with "invalid CFI
   advance_loc expression". There is a REM note next to the
   vendor/libffi/.built recipe in the Makefile.

   Upgrading to any release newer than 3.4.6 includes that fix upstream,
   so the local patch disappears naturally in the re-vendor -- nothing
   to re-apply.

   Upgrade checklist (see also the commented "upgrading dependency
   libraries" block under the Makefile clean target):

   - Replace the vendor/libffi tree with the new release tarball.
   - Rebuild from scratch (the .built stamp and configure output are
     per-tree): rm vendor/libffi/.built, then make.
   - Regenerate vendor/bindings/libffi.til (make update_c_libs) and
     redo the hand-audit documented in doc/ffi.org before committing --
     the committed bindings are generator output plus manual
     proc/func + own/ref qualifier fixes that a regen blows away.
   - Confirm the interpreter's ffi dispatch still passes the FFI tests
     on linux-x64 and the macos CI job.
