** FIXED #210 Eliminate the per-literal malloc in dyn_type_to_str returns

*** Post-mortem

Root cause: ~emit_dyn_type_to_str_body~ returned literal Type names
through per-use literal-wrapper expressions.  The literal bytes were
static, but each emission site still produced its own wrapper expression
instead of sharing the builder's file-scope literal pool.

The public signature remains owned:

  dyn_type_to_str := ext_func(type: Type) returns Str {}

Payload variants (~Type.Struct~, ~Type.Enum~, ~Type.Custom~, and named
FuncSig payloads) still use ~Str_clone~ because their names live inside
the runtime ~Type~ value and callers own the result.  Tag-only variants
and nested enum payload names are static strings, so they can use the
shared literal pool safely.

*** What changed

The builder now collects unique string literal contents before emitting
C and writes one file-scope wrapper per unique value:

  static __attribute__((unused)) Str _til_str_lit_N = ...;

~emit_str_lit_expr~ returns a pointer to the pooled wrapper when the
string was registered, and falls back to an inline static only for
unregistered literals.  ~emit_dyn_type_to_str_body~ registers its
tag-only and nested-enum names, so those return pooled ~_til_str_lit_N~
wrappers.  The owned ~Str_clone~ cases are unchanged.

Hoisted immutable string-literal identifiers (~hoisted__Str_N~) are also
resolved to the pooled ~_til_str_lit_N~ symbol for their contents, so the
generated C does not keep duplicate wrappers for those literals.

*** Regression test

Existing ~test/dynamic.til~ (test_dyn_type_to_str) and all container
tests exercise ~dyn_type_to_str~ through ~Vec.new~ / ~Array.new~ /
~Map.new~ / ~Set.new~ / ~Tuple.push~ path.  No new test file needed;
the behaviour is already covered.
