** FIXED #266 Flaky build_checkers (xvfb-run tmpdir cleanup race in parallel GUI suite)

*** Post-mortem

Root cause: each of the 16 GUI tests wrapped its game in its own
`xvfb-run --auto-servernum`, spawning 16 X servers whose EXIT-trap
teardowns ran concurrently. xvfb-run's trap does `xauth remove` (which
churns transient `Xauthority-c`/`-l`/`-n` lock files in its private
/tmp dir) followed by a bare `rm -r` of that dir; if the dir mutates
under rm (ENOTEMPTY/ENOENT), the trap's `exit 5` overwrites the saved
healthy exit status (124 from `timeout`), so a passing test reported
`exit=5 expected=124`.

Fix: the Makefile already runs the whole suite under ONE outer
`xvfb-run --auto-servernum bin/tests` (the standard one-X-server,
many-clients CI pattern), so every test inherits a live DISPLAY. The
per-test inner `xvfb-run` was pure redundancy; the fix drops it, so
GUI commands are now `til build ... && timeout 1 bin/<name>`. One
server, one teardown, nothing left to race. The stagger idea sketched
below was never implemented and is superseded. Side benefit for #25:
`bin/tests` itself no longer invokes the Linux-only `xvfb-run` (on mac
the inner wrapper exited 127; `make test` there runs the suite bare).

Regression test: the 16 `build_*` GUI tests themselves -- the flake
was intermittent, so the signal is the absence of spurious
`exit=5 expected=124` failures across full-suite runs.

`make test` intermittently fails one GUI test with `exit=5 expected=124`
(seen most recently as `build_checkers`, but any of the build_* GUI
tests can be the unlucky one). The program under test runs fine; the
failing exit code comes from `xvfb-run` itself.

*** Root cause

`/usr/bin/xvfb-run` ends with::

    if [ -n "$XVFB_RUN_TMPDIR" ]; then
        if ! rm -r "$XVFB_RUN_TMPDIR"; then
            exit 5

When 15 GUI tests all `xvfb-run --auto-servernum` at once, their
teardowns race on /tmp -- one of them fails to `rm -r` its
`$XVFB_RUN_TMPDIR` (or the auth cookie file under it), and xvfb-run
exits 5, masking the real `timeout 1`-driven 124 exit from
`bin/<example>`.

Confirmed in isolation: ASAN-built `checkers` exits 124 (the healthy
"timeout reached" status) on 5/5 single runs and 5/5 parallel runs.
The exit=5 only ever surfaces inside the full GUI suite.

*** Fix idea: stagger xvfb-run starts in tests.til

Each `build_<name>` test is added in `register_tests` in `src/tests.til`
(the `if en_gui.eq(1)` block) by `add_single(...)` with a shell command
that begins `bin/til build ... && xvfb-run --auto-servernum timeout 1
bin/<name>`. They all hit `xvfb-run` simultaneously when the parallel
runner releases them.

Give each GUI test a 0..N id (its index in `gui_files`) and prefix the
command with `sleep <id * 0.1>` (or similar):

#+begin_src
for i in 0..gui_files.len() {
    gf := gui_files[i]
    bn := test_basename(gf)
    ...
    stagger := format("sleep ", (i.mul(100)).to_str(), "e-3 '&&' ")
    add_single(tests, format("build_", bn), "build", 124, "",
               format(stagger, til_bin, " build", asan_flag, " ", gf,
                      " '&&' xvfb-run --auto-servernum timeout ",
                      GUI_TIMEOUT, " bin/", bn, run_args))
}
#+end_src

15 GUI tests staggered 100ms apart costs <1.5s of wall-clock (and the
`til build` step in front of `xvfb-run` is many seconds anyway, so the
effective spread on the actual xvfb-run calls stays sub-second).

*** Alternatives if staggering isn't enough

  - `xvfb-run -e /dev/stderr` and inspect: maybe the rm failure is a
    diagnosable EBUSY on a leftover Xauthority lock.
  - Wrap the GUI test command so a 5 exit gets rewritten to 124 when
    the inner `timeout` actually ran. Fragile (loses real failures).
  - Roll our own minimal xvfb wrapper that skips the cleanup
    (`Xvfb :N -screen 0 ...` + `DISPLAY=:N`). Avoids xvfb-run entirely.
    Heaviest fix; only justified if the stagger doesn't take.

*** Why low

Cosmetic: the program under test passes; only the test harness
mis-reports it as a failure. Annoying because it shows up as a real
red `FAIL` in `make test` logs and can mask actual regressions in the
same run.
