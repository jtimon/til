** FIXED #245 Value memcpy semantics block owned heap fields on StructInstance/EnumInstance

*** Post-mortem

    Root cause: ~StructInstance.struct_name~ / ~EnumInstance.enum_name~
    were ~ref Str~ borrows pointed at AST ~Expr~ storage. The AST is
    freed at program teardown before the Context's cached Values, so the
    borrow dangled. The natural remediation -- make the field an owned
    ~Str~ (#239 option 1) -- is what this issue investigated, and it
    reintroduced reproducible UAFs: ~shallow_copy_value~ is a literal
    ~memcpy~ of the Value, so copying a ~Value.Struct~/~Value.Enum~
    duplicates the owned name's ~c_str~ pointer; the first destructor
    frees it and every aliased copy dangles (the two failure modes
    documented below).

    Fix (the issue's recommended path 1): keep the field a ~ref Str~ but
    canonicalise every name at construction to program-lifetime storage
    via ~stable_type_name~ (~ctx.scope.get_binding(name).name~ -- a
    TypeBinding name that lives for the whole program, never the dying
    AST). With the borrow rooted in stable scope storage instead of the
    AST, it never dangles and no per-instance ownership/cloning is
    needed, so the memcpy-aliasing class cannot arise. This is landed at
    every construction site: ~make_struct_value~ and ~make_enum_value~
    (~stable_sname~ / ~stable_enum_name~), the ~StructInstance.clone~
    nested-field paths (~stable_ftype~ / ~stable_ftype2~), ~read_field~,
    and ~resolve_ptr_to_struct~ (which routes through
    ~make_struct_value~). The deeper Value-model rework (clone-aware
    copy, or interning type names into a Context table and dropping the
    per-instance field) is tracked as cleanup under [[./147.org][#147]]
    (dangerous AST refs) and [[./109.org][#109]] (interpreter leaks);
    it is not required to close the UAFs this issue surfaced.

    Regression tests (both pass under the ref+canonicalise model;
    both were the failing cases under the abandoned owned-Str attempt):
    - Failure 2 (nested enum payload mutation):
      ~test/enums.til~ ~test_mut_nested_enum_payload_binding~.
    - Failure 1 (owned-fields teardown):
      ~test/own_fields.til~.
    - Type-name stability across nested struct/enum copies:
      ~test/regression.til~ ~bug239~.

   Attempted #239 option 1 (change ~StructInstance.struct_name~ and
   ~EnumInstance.enum_name~ from ~ref Str~ to owned ~Str~).
   Compiles, but breaks two interpreter tests with reproducible UAFs
   under ASAN. The failure modes pinpoint a deeper ownership-model
   problem that #239 alone can't fix.

   Cross-refs: [[./147.org][#147]] (AST-as-cache antipattern: the
   ~ref Str~ field is one of the dangerous AST-borrowing refs),
   [[./109.org][#109]] (some interpreter "leaks" today are values
   that can't be safely freed because shallow-copy aliasing means
   freeing one would dangle the others).

    Translation of the trace into source terms:

    2. That Str eventually lands in an ~EnumInstance.enum_name~
       (an owned field, post-patch).
    3. ~Str_delete~ runs on a sibling Str through ~eval_expr~ ->

       ~shallow_copy_value~ memcpy along the way aliased the

    reused for an unrelated allocation, and its byte layout (in
    particular the ~count~ field) is no longer a valid string

    Source-level: the test mutates an enum payload via
    ~case NestedOuter.Wrap(p): p = NestedInner.A(99)~. The write
    constructs a temporary ~EnumInstance(enum_name = "NestedInner".clone(), ...)~
    and memcpys it into the parent enum's payload slot. The two
    EnumInstance bytes (temporary and destination) now share the
    same ~c_str~ pointer in ~enum_name~ via the memcpy. When the
    enclosing scope ends, ~scope_free~ frees the temporary's
    storage (including ~enum_name.c_str~). The next read of the
    destination's ~enum_name~ (via ~reinterpret_ptr_value~ ->
    ~val_enum_flat~) sees a freed Str, the inner ~switch q { case
    NestedInner.A(n): ... }~ fails to match, and the test reports
    "wrong inner variant".

*** Why the model can't accommodate owned fields here

    Two interacting pieces of the interpreter machinery make
    owned heap fields on ~StructInstance~ / ~EnumInstance~
    unsafe:

    1. *~shallow_copy_value~ is a literal ~memcpy~.* It's used
       all over the interpreter to byte-transfer Values across
       returns, payload writes, and enum reinterpret paths. The
       generated code looks like:

       : Value hoisted__Value_X = val_enum_flat(ei->enum_name, ...);

       with ~ei~ a borrowed ~EnumInstance~ pointer. ~val_enum_flat~
       does ~enum_name = enum_name~ assignment internally; the
       net effect is a memcpy that duplicates the ~Str~ struct
       (including ~c_str~). With owned ~Str~ semantics, the two
       resulting Values both "own" the same ~c_str~. The first
       destructor frees it, the rest dangle. (Failure mode 2.)

    2. *Stale ~ref EnumInstance~ borrows already exist.* Failure
       mode 1's allocation site
       (~ident_name~) and free site (~eval_expr~ via ~eval_call~)
       are in the same call chain; the dereference site
       (~read_field~) reads through a borrowed
       ~ref ei : EnumInstance = deref(ptr)~ whose underlying
       storage was freed by call-scope teardown. Before this
       patch the field was a ~ref Str~ rooted at long-lived
       scope-binding storage, so even when ~ei~ was stale the
       Str read was still valid. Owning the field exposes the
       underlying stale-~ei~ UAF that was previously masked.

*** Relationship to #109

    Some of the interpreter leaks tracked under #109 are
    suspected to be Values that *should* be freed but currently
    aren't, precisely because freeing them would dangle aliased
    copies under (1). Said differently: today's intentional
    leaks are how the interpreter avoids today's would-be UAFs.
    Fixing the ownership model unlocks freeing those Values,
    which feeds back into closing #109.

*** Relationship to #147

    #147's "dangerous AST refs" catalog (in
    [[./147.org][#147]]'s Audit section) names
    ~Declaration.field_struct_def~ (already removed),
    ~TypeBinding.struct_def~, and ~TypeBinding.func_def~ as the
    next purge targets. ~StructInstance.struct_name~ /
    ~EnumInstance.enum_name~ are the same shape -- a borrow
    field with no checker support -- and #239 was meant to be
    the small win that proved out the "own it" remediation
    template before the ~TypeBinding~ work. The investigation
    here shows the small win is not in fact small: it requires
    fixing the Value-level ownership model first.

*** Path forward (in increasing scope)

    1. *Land #239 option 2 (audit + canonicalise via
       ~ctx.scope.get_binding(name).name~)* at every
       StructInstance / EnumInstance construction site.
       Keeps the ~ref Str~ field; enforces a written rule
       that authors pass program-lifetime Strs. ~d6c3554~
       did this for ~read_field~'s own / ref / enum branches;
       extend to ~resolve_ptr_to_struct~ (still uses
       ~obj_sname~ from a dying case payload), ~make_struct_value~
       internals, and the direct ~StructInstance(...)~ /
       ~EnumInstance(...)~ constructor calls. Mechanical;
       leaves the field type alone; doesn't fix the broader
       Value-model gap but does close the surfaced UAFs.

    2. *Replace ~shallow_copy_value~ with a clone-aware copy
       for Value.Struct / Value.Enum* -- deep-clone the
       struct_name / enum_name Str on copy. Then the owned-
       field model from #239 option 1 can land safely. Cost:
       audit every ~returns shallow Value~ caller to see
       which ones still need byte-transfer vs deep clone;
       some legitimately want byte-transfer (move semantics).
       Probably wants a separate ~clone_value~ primitive and
       the call sites get retargeted.

    3. *Intern type names in a single Context-side table*,
       drop the per-instance ~struct_name~ / ~enum_name~ field
       entirely, store an index instead. Type names are
       already routinely interned in ~ctx.scope.bindings~ via
       string equality lookup; promoting that to an explicit
       intern table is mostly bookkeeping. Largest scope but
       eliminates the field, not just makes it safer.

    Recommend (1) for the immediate close of the surfaced
    UAFs (and as the natural extension of master's ~d6c3554~
    template), with (2) or (3) tracked as the longer-term
    fix that #239 option 1 would actually depend on.
