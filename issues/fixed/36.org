** #36 Nested pattern matching in switch

Patterns of the form ~case Outer.WrapperA(Inner.ValueA(val)):~ now
desugar in the typer to a conjunction:

  sw.is(Outer.WrapperA).and(get_payload(sw, Inner).is(Inner.ValueA))

and the binding ~val~ resolves to a nested ~get_payload~ chain rooted
at the outer subject. ~.and~ is eager, but ~get_payload~ is just
pointer arithmetic over the enum's union storage, so reading the
inner operand when the outer tag is wrong is harmless (the outer
~false~ short-circuits the body either way).

Supported forms inside a single case arm:

- Bare inner variant: ~case Outer.WrapperA(Inner.ValueC):~
- Inner variant with binding: ~case Outer.WrapperA(Inner.ValueA(v)):~
- Arbitrary nesting depth (~case Deep.O(Mid.M(Inner.X(n))):~)

Exhaustiveness still tracks only the outer enum: a nested case
~Outer.WrapperA(Inner.ValueA(v))~ marks ~WrapperA~ covered without
requiring every ~Inner~ variant to appear. Non-matching inner tags
fall through to the next case (or the default).

Other Phase 5 features from #35 (wildcard ~_~ as a non-binding,
struct field matching, guard clauses) are not part of this change.
Wildcard ~_~ already works as a payload binding name because ~_~
is a normal identifier; the leading-underscore convention discards
unused bindings without a separate codepath.

See ~src/test/enums.til~ test_nested_payload_*, test_nested_bare_inner,
test_nested_outer_b, test_nested_triple, test_nested_triple_fallthrough.
