# priority: low
** FIXED #252 Derive Declaration.til_type from explicit_type at construction

  POST-MORTEM
  Root cause: not a bug -- 49 inline ~Declaration(...)~ constructor sites
  across the self layer (parser/initer/typer/context) each repeated the
  mechanical ~til_type = til_type_from_explicit_type(explicit_type)~
  derivation by hand, duplicating the same call.

  Fix: added a ~decl_typed~ factory that derives the initial ~til_type~
  from ~explicit_type~ and forwards the remaining fields via named-arg
  defaults mirroring ~Declaration~'s field defaults. The 49 sites now
  call ~decl_typed(...)~ and drop the explicit ~til_type~ argument (and
  the now-redundant ~.clone()~ on the forwarded name/explicit_type, which
  the factory clones internally).

  Deviation from the recommended plan: the plan proposed moving
  ~til_type_from_explicit_type~ into ~src/core/ast.til~ and putting the
  factory there too. That is not viable: since #249 the mapping is backed
  by ~builtin_type_table~, a module-level ~Map~ global. ~src/core/map.til~
  imports ~ast~, so ~ast~ is compiled before ~Map~ is defined, and a
  ~Map~-typed global in ~ast.til~ fails to compile ("unknown type name
  'Map'"). The factory does not need to be in core: every one of the 49
  call sites is already a self-layer module that sees
  ~til_type_from_explicit_type~, so ~decl_typed~ lives next to the mapping
  in ~src/self/parser.til~. Smaller change, no layering/ordering issue.

  Regression coverage: the existing suite (137/137) exercises every
  migrated site -- the self compiler is rebuilt from this source and
  every test compiles declarations through these paths. No observable
  output change (behaviour-preserving refactor).

  Follow-up from #249. At construction time, ~49 sites across the self
  compiler build a ~Declaration~ passing BOTH ~explicit_type~ and the
  redundant ~til_type = til_type_from_explicit_type(explicit_type)~:

    Declaration(name = f_name, ...,
        explicit_type = f_explicit_type,
        til_type = til_type_from_explicit_type(f_explicit_type), ...)

  The ~til_type~ value is mechanically derived from ~explicit_type~, so
  every site repeats the same call. #249 added ~set_decl_type(mut d,
  name)~ for the handful of mutate-after-construction sites, but the
  inline-constructor sites still pass both by hand.

  Counts (at time of writing):
    - 49 ~til_type = til_type_from_explicit_type(...)~ occurrences
    - across src/self/parser.til, initer.til, typer.til, context.til

*** Why it is not a one-liner
  Two real constraints:

  1. Layering. ~til_type_from_explicit_type~ lives in
     ~src/self/parser.til~ (the self layer). ~Declaration~ lives in
     ~src/core/ast.til~ (core). Core cannot import self, so the
     ~Declaration~ constructor cannot call the mapping where it
     currently sits.
  2. The field must stay mutable/stored. The typer REFINES
     ~til_type~ after parsing (e.g. resolving ~Custom(name)~ to
     ~Struct(name)~/~Enum(name)~), so ~til_type~ cannot be dropped or
     made a pure read-only accessor over ~explicit_type~. The
     derivation only describes the INITIAL value.

  til struct literals run no logic, so a field default cannot compute
  ~til_type~ from a sibling field -- a factory or post-build step is
  required.

*** Recommended plan
  1. Move ~til_type_from_explicit_type~ from ~src/self/parser.til~ into
     ~src/core/ast.til~, next to ~Type~/~Declaration~. It is a pure
     name -> Type mapping that only references ~Type~/~Primitive~ (both
     already in ast.til); it has zero parser dependencies, so it
     belongs in core anyway. The ~49 call sites keep calling it by the
     same name, so this step alone changes no call site (just the
     definition's home) and should pass ~make clean && make test~ on
     its own. Land it first.

  2. Add a factory in core that derives ~til_type~ from
     ~explicit_type~. Two candidate shapes -- pick one when
     implementing:
     a. Free function ~decl_typed(explicit_type: Str, name := "",
        is_mut := false, own_type := OwnType.Shallow, doc := "",
        is_priv := false, ...) returns Declaration~ that forwards the
        named args to ~Declaration(...)~ and fills
        ~til_type = til_type_from_explicit_type(explicit_type)~. Uses
        til's named-arg defaults so callers only pass what they need.
        Cost: the factory's parameter list mirrors ~Declaration~'s
        fields and must be kept in sync when a field is added.
     b. A ~Declaration~ namespace method
        ~with_explicit(mut self, name) returns ref Declaration~ that
        sets both fields, used as a post-construction step. Avoids
        mirroring the field list, but turns one expression into a
        two-step build at each site and the ref-return ownership at
        call sites needs care.
     Recommendation: (a) -- it keeps each site a single expression and
     reads closest to today's code; the field-list coupling is a small,
     localized maintenance cost.

  3. Migrate the ~49 sites to the factory, dropping the explicit
     ~til_type = ...~ argument. Do it file by file
     (parser -> initer -> typer -> context) so each step stays
     reviewable.

  4. ~make clean && make test~ (137/137 at time of writing). This
     touches compiler source, so confirm a plain commit stands on its
     own per doc/self.org; only reach for the two-pass build if
     ~make test~ surfaces a bootstrap mismatch.

*** Alternatives considered
  - Leave it: the redundancy is harmless and uniform, but it is 49
    copies of the same derivation and was explicitly flagged in #249.
  - Construct-then-~set_decl_type~ everywhere (reuse the #249 helper):
    works without moving the mapping, but makes every site a two-step
    build and keeps ~set_decl_type~ in the self layer. Rejected in
    favour of the core factory.

  Behaviour-preserving throughout; no observable output change.
