# priority: high
** FIXED #208 FFI: manual proc/own/ref/shallow on regenerated C bindings + noop_proc escape hatch

The ~precomp~ workaround in commit ~e7b4580~ for "pure-looking
wrappers around ext_funcs fold to compile-time zero" was a blunt
fix in the wrong layer: it made ~func_uses_unknown_globals~ return
true whenever it reached any ~ext_func~ body, blocking the
cross-call fold optimization (#198) for genuinely pure C functions
too.

Root cause: the auto-generated bindings under ~vendor/bindings/~
spelled every C function as ~ext_func~, including the ones that
read or mutate process / library / OS state (~GetScreenWidth~,
~GetMouseX~, ~time~, ...).  C has no syntax for any of:

- proc vs func (purity / side effects),
- ~own~ vs default for pointer args (does the callee take ownership?),
- ~ref~ vs ~own~ for pointer returns (does the caller own the result?),
- ~shallow~ is the only piece the generator gets right unambiguously.

So the generator cannot infer the right qualifiers from the header
alone -- a human curates the output after each regeneration.

*** What got done

1. **Manual-only regeneration.**  ~make bindings~ was already
   gated to manual runs (the default ~make~ / ~make test~ never
   touches ~vendor/bindings/~), but the name was renamed to
   ~make update_c_libs~ and the Makefile comment was rewritten to
   call out the audit-after-regeneration step explicitly.
   Routine builds keep consuming the committed files.

2. **Manual review pass after regeneration.**  All three
   regenerated binding files were walked end-to-end:

   - ~vendor/bindings/raylib.til~: ~145 entries flipped from
     ~ext_func~ to ~ext_proc~ -- input pollers (keys / mouse /
     gamepad / touch / gestures), window + monitor queries,
     screen<->world transforms (they depend on screen dimensions),
     file/dir I/O, PRNG (~GetRandomValue~ + ~LoadRandomSequence~),
     ~MemAlloc~/~MemRealloc~, all ~Load*~/~Save*~/~Export*~ on
     images / textures / fonts / models / waves / sounds / music /
     audio streams, plus the audio playback queries (~IsSoundPlaying~,
     etc).  Pure ext_funcs (color math, geometry checks, string
     ops, mesh generators, camera matrix math, ~IsXxxValid~ struct
     checks) stayed ~ext_func~ -- 143 remain.

   - ~vendor/bindings/tinyfd.til~: all 7 entries flipped to
     ~ext_proc~ -- every tinyfd entry point opens a modal OS
     dialog, so all are side-effecting.

   - ~vendor/bindings/libffi.til~: all 12 entries flipped to
     ~ext_proc~ -- ~ffi_prep_cif~, ~ffi_closure_alloc~, the
     ~ffi_prep_*_closure*~ family and ~ffi_get_struct_offsets~
     all initialize CIF / closure structures (write through
     pointer args); the ~ffi_*raw_size~ readers read fields off
     a CIF that itself is initialized by side-effecting calls,
     so the conservative call is ~ext_proc~ too.

   The own / ref / shallow audit was lighter: the binder already
   emits ~shallow~ for by-value scalars / small structs (correct),
   ~ref T~ for raw pointer returns (the safe default -- caller
   does not own; flipping to owning-return is the rare case), and
   ~T~ (= owned pointer) for boxed compound returns.  No flips
   needed in this pass; the audit checklist in ~doc/ffi.org~
   documents what to look for next time.

   Wrappers in ~src/modes/gui.til~ propagated to procs --
   ~window_should_close~, ~get_screen_width~, ~get_screen_height~,
   ~get_frame_time~, ~get_mouse_x~, ~get_mouse_y~,
   ~is_window_resized~, ~is_key_pressed~, ~is_key_down~,
   ~is_key_repeat~, ~get_char_pressed~,
   ~is_mouse_button_pressed~, ~is_mouse_button_down~,
   ~get_random_value~, ~load_font~, ~make_tone~,
   ~save_file_dialog~, ~open_file_dialog~.  Internal compiler
   helpers around libffi (~ffi_register~, ~ffi_init_scan_program~
   in ~src/self/dispatch.til~) flipped to ~proc~ as well, since
   they now transitively call ~ext_proc~s.

3. **~noop_proc~ escape hatch.**  Added a single new symbol:

   - C side: ~void noop_proc(void) {}~ in ~src/c/ext.c~ (plus
     declaration in ~src/c/ext.h~).
   - til side: ~noop_proc := ext_proc() {}~ in ~src/core/sys.til~
     (alongside ~sleep_ms~ -- the previous "no-op-with-side-
     effects" precedent).
   - precomp: ~noop_proc~ added to ~is_side_effecting_name~ in
     ~src/self/precomp.til~ so calls through user funcs are
     recognized as impure even without going through the FuncDef
     walker.

   Drop a ~noop_proc()~ call anywhere inside a func body to mark
   it impure (no compile-time fold).  Zero runtime cost; the
   call exists purely as a fold-blocker tag.

4. **~doc/ffi.org~.**  New file.  Documents:

   - Why C headers can't express til's qualifier set and the
     generator therefore cannot infer them.
   - The audit checklist after running ~make update_c_libs~
     (proc vs func, own vs ref on args, own vs ref on returns).
   - A quick-reference table mapping "what the C function does"
     to the right binding shape.
   - The ~noop_proc~ idiom for user wrappers.
   - The regeneration workflow (diff before/after to preserve
     prior audit decisions).

5. **Precomp workaround removed.**  The
   ~func_uses_unknown_globals~ ~ext_func~ check from ~e7b4580~
   is gone.  ~ext_func~ goes back to meaning "pure, safe to
   fold" -- exactly what precomp wants it to mean, and the
   cross-call fold optimization works for pure helpers again
   (~sinf~, ~memcmp~, etc).  Verified ~bin/til build
   examples/shootil.til~ emits real ~get_screen_width()~ calls
   in the generated C (not the hard-coded ~0~ that motivated
   this whole exercise), and all games (shootil / chess / go /
   piano / music / popcorn / minesweeper / snake / game_of_life /
   editor / tilminal / gui_hello / checkers / font_test) still
   build and pass the test suite.

*** Why this is the right shape

- The information lives where the human knows it (the binding
  file), not buried in compiler heuristics.
- ~ext_func~ stays useful as a purity claim that precomp can
  trust, which keeps the cross-call fold optimization (#198)
  working for the cases it was designed for.
- The escape hatch is one symbol, not a new keyword or a typer
  change -- low blast radius, easy to find with ~grep~.
- The hand-edited bindings file is fine to check in; we already
  treat ~vendor/~ as linguist-vendored and already commit
  generated artifacts (~boot/til.c~, ~gen/til/*.c~ in tests).
  Issue #203 documents that the hand-written bindings "still
  ship correct" while the auto-gen path has bugs of its own.

*** Out of scope / follow-ups

- Issue #203 (bindings_gen ~parse_decl_shape~ corruption) is
  independent -- it's about the generator itself producing
  garbage Strs.  Both need fixing; neither blocks the other.
- The own/ref pointer-ownership audit on the bindings is
  cursory in this pass.  A deeper sweep -- especially around
  raylib's ~Load*~/~Unload*~ pairs and the various ~Get*Name~
  returns -- can land later when an actual ownership bug
  surfaces; the audit checklist in ~doc/ffi.org~ is the
  handover.
