** FIXED #277 Const-folder cannot fold variadic (`..T`) calls

*** Post-mortem

Root cause was two-layered, both in ~src/self/constfolder.til~:

1. ~body_has_local~ only recognised ~NodeType.Decl~ names, not ~ForIn~
   loop variables (which live in the ~ForIn~ payload). So
   ~func_uses_unknown_globals~ flagged a ~for~-in loop variable as an
   "unknown global".

2. The dominant blocker: the typer lowers a ~for s in parts~ loop over a
   ~..T~ array into a bounds-checked ~Array.get~ whose error branch calls
   ~panic~. ~func_uses_unknown_globals~ (the static reachability pre-filter
   for folds) treats ~panic~ as a hard side-effecting blocker, so any
   wrapper that forwards comptime args into ~format(..Str)~ (or any free
   function that iterates a collection) was rejected before the interpreter
   ever ran. That ~panic~ sits on a failure path valid comptime inputs never
   reach, so the fold eval would have run cleanly.

Fix: added the ~ForIn~ case to ~body_has_local~, and introduced
~is_fold_abort_name~ (panic/TODO/UNREACHABLE). These abort builtins are now
allowed inside a transitively *called* function's body (threaded via a new
~top_level~ parameter on ~func_uses_unknown_globals~), where they are dead
on the fold path, but stay hard blockers inside the directly-folded body --
so ~assert~ (whose own body is ~if cond.not() { panic(...) }; return true~)
still never folds. Without that distinction ~assert(loc, true)~ folded to a
bare ~true~ literal statement and broke the whole suite.

Regression test: ~test_variadic_fold~ in ~test/constfold.til~ (folds
~vfmt(a,b)~ / ~vfmt2(a,b,c)~ wrappers around ~format(..Str)~ to baked
string literals; the helpers are eliminated from the checked-in
~test/constfold.c~ snapshot).

*** Follow-up: folding a DIRECT variadic call (route B)

The wrapper fix above only covered folding a wrapper *call*
(~concat("x","y")~), because a wrapper presents a non-variadic call
boundary whose body the interpreter handles internally. A DIRECT
~format("a","b","c")~ at the use site is different: the typer lowers it
*before* const-fold into a build region --
~_va_Array_N := Array.new(Str, cap)~, ~<va>_ek~/~<va>_eo~ err-slot decls,
a throwing ~Array.set(va, j, <elem>, &_ek, _eo)~ per element, then a
consumer ~format(_va_Array_N)~ -- so the const-folder no longer sees a
single foldable call, just a runtime-built array fed to ~format~.

Fix (all in ~src/self/constfolder.til~): when ~fold_subtree~ finishes a
body, for each ~_va_Array_~ decl whose every element is comptime-known
(~va_elements_known~), ~materialize_va~ interprets ONLY that array's own
build statements (its decl, ~_ek~/~_eo~ decls, and ~Array.set~ calls --
not unrelated sibling hoists, which would make ~eval_expr~ resolve a
runtime var) into the fold scope; ~fold_va_calls~ then evals the consumer
(after binding its other comptime-known args -- e.g. a positional arg's
~hoisted__I64 := 100~ -- into the scope) and bakes the literal; finally
~va_build_dce~ drops the whole machinery all-or-nothing (decl, err-slots,
sets, cleanup deletes), aborting for a given array if any live use of
~va~/~_ek~/~_eo~ remains, so a partial removal can never dangle a
reference. Consumers are only folded when every arg is the materialised
~va~ or comptime-known (~va_consumer_args_ok~), mirroring the normal
~all_comptime~ gate, so ~expect(runtime_cond, "msg")~ and proc consumers
(~println~) are left alone.

Regression test: ~test_variadic_direct_fold~ in ~test/constfold.til~
(direct ~format("x","-","y","-","z")~ / ~format("a","b","c")~ and a
positional+variadic ~dsum(100,1,2,3)~ all bake to literals; no
~_va_Array_~ build remains in the ~test/constfold.c~ snapshot).

The compile-time constant folder (~src/self/constfolder.til~, formerly
~precomp.til~) never folds a call that routes through a variadic
(~..T~) function, even when every argument is a compile-time constant.
Concretely, ~format(..Str)~ is not const-foldable: a pure wrapper such
as

   concat := func(a: Str, b: Str) returns Str { return format(a, b) }
   result := concat("hello", " world")   // emits a runtime call, not "hello world"

emits a runtime ~concat(...)~ + ~format(...)~ instead of baking the
literal. Before ~Str.concat~ was removed in favour of ~format~, the same
wrapper folded to the ~"hello world"~ literal (~Str.concat~ is a plain
2-arg function). The string-concat cases in ~test/constfold.til~
(~test_string_concat~, ~test_lolalalo~) therefore still pass but now
exercise runtime concatenation rather than the folding they were written
to verify.

*** Root cause (verified by isolation)

Experiments compiling ~test/constfold.til~ and checking whether the call
folds to a baked literal (and the helper gets eliminated):

  - non-variadic 2-arg helper (inline body)            -> folds
  - non-variadic helper with a range ~for~ loop        -> folds
  - local variadic helper ~myfmt(..Str)~               -> does NOT fold
  - core ~format(..Str)~                               -> does NOT fold

So for-loops fold fine and ordinary functions fold fine; the single
variable that turns folding off is the variadic ~..T~ parameter.
~try_eval_call~ (constfolder.til) rebuilds the call with literal args and
runs it through the interpreter, but cannot bind N args into a ~..T~ param
/ construct the splat array at fold time, so it bails (returns NULL) and
the call is emitted at runtime.

*** Secondary latent bug found along the way

~body_has_local~ (constfolder.til) only recognizes ~NodeType.Decl~ names,
not ~for~-in loop variables (those live in the ~ForIn~ payload, not as a
~Decl~ child). So ~func_uses_unknown_globals~ flags a loop variable as an
"unknown global" and refuses to fold any foldable *free* function whose
body loops. It is masked today by the variadic limitation above (and by
the fact that core free functions like ~format~ are not registered in the
fold scope, so the recursion never reaches their body), but it would bite
any non-variadic foldable free function that uses a ~for~ loop. A
one-line fix restores it:

   case NodeType.ForIn(fd):
       if fd.name.eq(name) { return true }

*** Fix direction

Teach the const-folder's eval path to handle variadic calls: pack the
comptime splat arguments into the ~..T~ array value before interpreting
the body (mirroring the typer's ~desugar_variadic_calls~), and add the
~ForIn~ case to ~body_has_local~. A regression test would assert that a
~format(...)~ / variadic-wrapper call with comptime args folds to a baked
literal (the helper eliminated from the generated C).
