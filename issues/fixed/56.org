*** Post-mortem

   Long-running feature rollout closed alongside #57; #58 (its
   dependency) closed the metatype foundation, and every
   "Still missing for rstil parity" bullet here either had already
   landed in the #58 arc or just lacked a dedicated regression test in
   ~test/first_class.til~. Verified-and-tested by this commit:

   - Struct-type macros merging a user ~namespace:~ block with the
     auto-generated ~clone~ / ~delete~ -- ~make_box56~ /
     ~test_macro_struct_ns_method~. ~to_str~ is not auto-generated for
     any struct, so there is nothing to merge there.
   - INFER_TYPE resolution: a field with no explicit ~: Type~ but a
     default value pulled from a macro arg -- ~make_holder56~ /
     ~test_macro_struct_infer_type~.
   - Precomp of default-value expressions inside an expanded struct --
     ~make_sized56~ / ~test_macro_struct_precomp_default~ uses
     ~size: I64 = N.add(1)~ and the expression evaluates at compile
     time.
   - Calling a struct *type* as a generic constructor:
     ~make_default := func(T: Type) returns T { return T() }~ -- this
     landed in the #58 ~Phase now~ direct-generic-functions arc as
     ~gdefault~ / ~test_generic_func_constructor~.

   Side effect: the new tests shifted heap layout in the typer enough
   to deterministically trigger a latent use-after-free in
   ~typer_lift_lambdas~ (it registered the scope's func_def pointer
   against a temporary Expr, then pushed only a CLONE into the AST --
   the pointer dangled once the temporary Vec was freed and
   ~func_def_type~ later read freed memory). Fix: push the clone
   first, register against the pushed copy. Surfaced as ASAN
   heap-use-after-free at ~src/c/ext.c:587~ on this commit and was a
   real bug on master, not test churn -- the test additions just made
   the heap layout reproduce it.

   "Beyond rstil parity" items are deliberately out of scope for the
   close, exactly as the issue text already labelled them:

   - ~func<T>(...)~ sugar -- landed under #58 (curried macro form).
   - Conditional macro bodies for generic types whose shape depends
     on macro args -- also already landed under #58
     (~make_tagged~ / ~test_cond_macro_struct_else_if~).
   - Anonymous struct literals in niche positions (~cast(e, struct {
     ... })~, ext_func param/return types) -- parser-side work,
     dedicated future issue if a real user case shows up.

   Regression tests: ~test_macro_struct_ns_method~,
   ~test_macro_struct_infer_type~, ~test_macro_struct_precomp_default~
   (plus the pre-existing ~test_macro_struct_gen~ /

*** Original issue body

:DEPENDENCIES: [[./58.org][#58]]
  Based on rstil/doc/first_class.org. Struct definitions as first-class
  values that can be passed to functions, returned from macros, and stored
  in collections.

*** Passing struct types as arguments (templates/generics)

   #+begin_src til
   // A function that takes a struct type and returns an instance
   make_default := func(T: struct) returns T {
       return T()  // call default constructor
   }

   p := make_default(Point)  // Point(x=0, y=0)
   #+end_src

   Open question: is ~T: struct~ the right syntax for "any struct type"?
   Or should it be ~T: Type~ for any type?

*** Anonymous structs (struct literals as values)

   #+begin_src til
   // Create a value with an anonymous struct type
   p := struct { x: I64 = 10, y: I64 = 20 }

   // Instantiate an anonymous struct
   p := struct { mut x := 0 }(x = 10)
   #+end_src

*** Returning struct types from functions (macros)

   #+begin_src til
   // Already works via macros:
   make_point := macro() returns Type {
       return struct { mut x: I64 = 0, mut y: I64 = 0 }
   }
   Point := make_point()

   // With namespace: keyword, macros return complete types with methods:
   make_point := macro() returns Type {
       return struct {
           mut x: I64 = 0
           mut y: I64 = 0

       namespace:
           origin := func() returns Point { return Point() }
       }
   }
   #+end_src

*** Storing in collections

   Struct definitions are values, so they can be stored:

   #+begin_src til
   // Vec of struct instances (same type, different values)
   mut points := Vec.new(Point)
   points.push(own Point(x=10, y=20))
   points.push(own Point(x=30, y=40))

   // Vec of struct *definitions* (different types stored in same Vec)
   mut defs := Vec.new(StructDef)
   defs.push(own struct_def_of("Point"))
   defs.push(own struct_def_of("Color"))

   // Anonymous struct literal pushed directly
   defs.push(own struct { mut x: I64 = 0, mut y: I64 = 0 })
   #+end_src

*** Comparison

   | Language   | Struct literal syntax     | Anonymous structs?              |
   |------------+---------------------------+---------------------------------|
   | Rust       | ~Point { x: 10, y: 20 }~ | No (tuples only)                |
   | Go         | ~Point{X: 10, Y: 20}~    | Yes: ~struct{X int}{10}~        |
   | TypeScript | ~{ x: 10, y: 20 }~       | Yes (object literals)           |
   | Zig        | ~.{ .x = 10, .y = 20 }~  | Yes (anonymous structs)         |
   | TIL        | ~Point(x=10, y=20)~      | Yes: ~struct{mut x := 0}(x=10)~ |

*** Progress (rstil-parity arc, 2026-05)

   What works today:

   - [X] Anonymous struct instantiation: ~struct { ... }(args)~ -- init
     lifts to a top-level ~AnonStructN~ decl shared per-file with the
     dedup cache (anon_def_key folds in field/variant names, types,
     mut/own, default-value trees, and namespace-method bodies).
   - [X] Anonymous struct in type positions: ~func(p: struct { ... })~,
     ~returns struct { ... }~, ~mut p: struct { ... } = ...~, and
     ~field: struct { ... } = ...~. Cross-position unification: a type-
     position literal and a structurally identical value-position one
     resolve to the same generated type, so the value flows in.
   - [X] Bare anonymous struct as a value:
     ~defs.push(own struct { ... })~ produces a runtime ~StructDef~
     value (the parser emits ~struct_def_of("AnonStructN")~ for the
     ~own~-prefixed case). The non-own arg path stays as a type ref
     so ~count_variants(struct { ... })~-style introspection still
     works.
   - [X] Anonymous struct as a ~func(T: Type)~ / ~macro(T: Type)~ arg
     (introspection style). Macros taking ~Type~ now bypass the typer's
     comptime-arg gate via expr_is_comptime's ~Type.<Variant>(...)~ case
     plus the precomp arg-loop's type-ident handling. The scavenger
     follows ~Type.Struct("X")~ / ~Type.Custom("X")~ so the underlying
     decl isn't pruned.
   - [X] Macros as struct-type generators:
     ~Pair := make_pair(I64)~ where ~make_pair := macro(T: Type) returns StructDef~
     desugars to a plain ~struct { ... }~ decl at the call site
     (init_expand_type_gen_macros in initer.til; mirrors rstil
     precomp.rs ~309 + substitute_type_params_in_struct). Multi-param
     supported.
   - [X] ~Vec.new(StructDef)~ + ~struct_def_of("Name")~ round-trip,
     including ~defs.push(own struct { ... })~ producing a runtime
     value.
   - [X] Returning a ~Struct~ instance from an anonymous-struct return
     type (see test_anon_return_type in test/first_class.til).

   Still missing for rstil parity:

   - [ ] Struct-type macros merging a body-supplied ~namespace:~ block
     with the auto-generated ~delete~ / ~clone~ / ~to_str~ methods
     (rstil precomp.rs ~349-385). Today macro_pair := macro(T: Type)
     returns StructDef can only return a fields-only literal.
   - [ ] INFER_TYPE resolution: a struct member with no explicit type
     but a default value pulled from the macro body (rstil ~328-336).
   - [ ] Precomp of default-value expressions inside an expanded struct
     (rstil ~342-348). Today the substituted literal carries the
     parser-shaped defaults unchanged.
   - [ ] Calling a struct *type* as a generic constructor:
     ~make_default := func(T: Type) returns T { return T() }~. The
     typer doesn't yet rewrite ~T()~ to a constructor call at the
     param's instance type.

   Beyond rstil parity (not in scope for #56 alone):

   - [ ] ~func<T>(...)~ sugar that desugars to a macro call (see #58).
   - [ ] Conditional macro bodies (not just ~return <literal>~) for
     generic types whose shape depends on macro args.
   - [ ] Anonymous struct literals in niche positions: ~cast(e, struct { ... })~,
     ext_func param/return types.
