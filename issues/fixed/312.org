** FIXED #312 Top-level numeric for-loop in a `mode script` panics OutOfBounds in the typer

*** Post-mortem

Root cause: `infer_body_unit` (the top-level file-unit walk, used for
`mode script`/`mode test`/`mode lib` units) read `body.children[i]` AFTER
`infer_body_stmt` returned, to feed `constfolder_classify_stmt`:

  #+begin_src til
  while i < body.children.len() {
      infer_body_stmt(scope, body, i, ...)          // mut i BY REFERENCE
      constfolder_classify_stmt(body.children[i], ctx)   // re-reads children[i]
      i++
  }
  #+end_src

`infer_body_stmt` takes `mut i` by reference and rewinds it (`i--`) to
reprocess a statement it desugared in place. A top-level `for i in 0..N`
is a single ForIn at index 0; `desugar_for_in_range_stmt` replaces it with
a Body block and the case arm does `i--`, underflowing the USize index to
its max value. The trailing `body.children[i]` read then went OutOfBounds
(typer.til:5029). `infer_body` (function bodies) has no trailing
`children[i]` read -- its `i--` is cancelled by the following `i++`, which
wraps the underflowed index back to 0 -- which is why only top-level units
crashed and moving the loop into a func typed fine.

Fix: capture the statement index BEFORE `infer_body_stmt` runs
(`classify_idx := i`) and classify `body.children[classify_idx]`. The
pre-call index is always in bounds and names the statement processed this
iteration. `constfolder_classify_stmt` only records top-level def Decls
(FuncDef/StructDef/EnumDef), so the desugared Body block it may now see is
a harmless no-op, and the set of classified foldables is unchanged.

An earlier attempt that moved the classify to a separate post-walk loop
over `body.children` produced byte-identical `constfolder_foldables` yet
regressed `test/first_class.til` (a macro Type-arg leaked a `hoisted__Type`
temp), so the classify was kept interleaved with the walk; only the
OutOfBounds index was corrected.

Follow-up (#315): that post-walk regression was not a mystery -- it exposed
a real order coupling. The desugarer's arg hoister skips a macro call's args
based on `is_macro_call` (`constfolder_foldables`), and nested bodies are
desugared mid-walk via `body_pre_passes`, so classifying too late leaves
forward-referenced macros invisible to the hoister. The same coupling was
independently reachable on master via a forward reference (a body calling a
macro declared later at top level), filed and fixed as #315. As part of that
fix `infer_body_unit` now classifies every top-level def in ONE pass BEFORE
the walk (not per-statement inside it), which both fixes #315 and subsumes
this #312 OutOfBounds -- there is no longer any `body.children[i]` read in
the walk to go out of bounds. The `classify_idx` shape described above was
the interim #312 fix; see issues/fixed/315.org for the final form.

Regression test: `test/bug312.til` (a `mode script` fixture with a
top-level ascending and descending `for _ in a..b` loop plus a trailing
statement), registered via `add_cli_pair` in src/tests.til so it runs both
interpreted and compiled.

A `for i in 0..N { ... }` written at the TOP LEVEL of a `mode script` crashes
the typer with an OutOfBounds panic, instead of type-checking. It fails
identically whether interpreted or compiled, and independently of `--usize`
(reproduces at both U32 and U64 -- it predates and is unrelated to the #309
default flip; the flip work merely surfaced it).

*** Reproduction

  #+begin_src til
  mode script
  for i in 0..3 { println(i.to_str()) }
  #+end_src

    $ bin/til repro.til
    ./src/self/typer.til:5029:48: panic: OutOfBounds
    $ bin/til build -o /tmp/repro repro.til
    ./src/self/typer.til:5029:48: panic: OutOfBounds

A statement after the loop does not help (still panics). Moving the loop into
a function body types fine -- function bodies go through `infer_body`, not the
top-level `infer_body_unit`.

*** Workarounds

- Put the loop inside a `func`/`proc` and call it.
- Use a `while` loop at the top level instead of `for-in`.

*** Likely root cause (starting point, not verified as the fix)

`infer_body_unit` (src/self/typer.til:5021):

  #+begin_src til
  mut i : USize = 0
  while i < body.children.len() {
      infer_body_stmt(scope, body, i, ...)          // may advance `i`
      constfolder_classify_stmt(body.children[i], ctx)   // 5029: children[i]
      i++
  }
  #+end_src

`infer_body_stmt` advances `i` when it lowers/splices statements (e.g.
`i = i + inserted` at ~typer.til:4867 for match lowering, and the for-in
desugaring inserts a loop scaffold). After it returns with `i` advanced,
`constfolder_classify_stmt(body.children[i], ctx)` at line 5029 re-reads
`body.children[i]` with the advanced index, which can point at or past the end
of `body.children` -> OutOfBounds. `infer_body` (5006), used for function
bodies, does not have this trailing `children[i]` read, which is why only
top-level units crash.

Candidate fixes to evaluate: re-read `body.children.len()` / bounds-check
before the classify call, capture the pre-advance index for the classify, or
move `constfolder_classify_stmt` to run on the statement `infer_body_stmt`
actually processed rather than `children[i]` after the advance.

*** Regression test

Add a `mode script` fixture with a top-level `for i in 0..N` loop (both
interpret and compile) to the suite once fixed.
