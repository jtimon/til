** #104 Module-level mutable globals
  Landed once [[./50.org][#50]] had made ordinary values stack-backed.

  What was changed:

  - top-level value declarations are now pre-registered in
    ~init_declarations~, so functions can reference module globals before
    the declaration appears later in the file
  - builder now emits module globals as value storage
    (~static T x~ / ~static void *x~), not heap pointers
  - global reads, writes, returns, ~ref~ borrows, and ~mut~ calls now use
    value-global semantics consistently (the same ~&x~ / direct assignment
    model as stack locals)
  - generated headers now declare real source-level top-level globals for
    handwritten C (~extern T x~ in ~modes.h~ / ~*_forward.h~)
  - generated root translation units now own those global definitions, so
    handwritten C modules can link directly against til-defined globals
  - compiler-generated lowering temporaries (~_t_*~) stay internal and are
    not leaked into the generated C ABI
  - bootstrap / debug / ASAN direct-C builds were aligned with the same
    root-unit ownership rule by compiling the root generated unit
    (~til.c~), not ~til.c~ + ~modes.c~ together
  - this was verified by moving simple typer counters like
    ~hoist_counter~, ~_va_counter~, and ~_kw_counter~ from C statics to
    til globals and rebuilding cleanly
  - globals tests were strengthened to cover forward references to globals
    declared later
  - the typer error fixture was updated so its first undefined-symbol case
    remains undefined under the new forward-reference behavior

  Validation:
  - ~make clean~
  - ~systemd-run --user --scope -p MemoryMax=64G make test~
  - result: ~82/82~ passed
  - direct C consumption of til globals now works through generated
    declarations in ~boot/modes.h~

  Remaining work:
  - namespace/static-field globals still need the same treatment
  - that is now part of ~[[./70.org][#70]]~, not a remaining ~[[./104.org][#104]]~ blocker
