** FIXED #192 USize should not be hardwired to U32

*** Post-mortem (2026-06-24)

USize is now target-dependent like UPtr: U64 on 64-bit, U32 on 32-bit.
`make test_two_pass` is green (162/162) with the flip in.

Root cause of why this was hard: USize is the width of every `Str`'s
`count`/`cap`, and the compiler constant-folds core code *by interpreting
it at compile time*. A til_boot built at U32 therefore had to manipulate
U64 `Str` values in its own interpreter -- but it read `Str` fields through
its OWN compiled `Str` type (cap@12 in a 16-byte struct), while the folded
U64 values laid cap at offset 16 in a 24-byte struct. UPtr never hit this
because it was born U64 and never migrated underneath a running compiler.

What landed it, in two merged commits:
1. Preparation (width-neutral at U32): emit the USize/UPtr typedef NAME for
   struct fields (ext.h is the ABI source of truth); resolve the width from
   the source alias (lag-free) so flipping usize.til + ext.h is consistent
   in pass 1; make the interpreter's Str builders AND every compile-time
   `cap` read ctx-driven (str_write_count_cap / str_write_cap_at /
   str_cap_at / ctx_total_size) so a U32 til_boot folds U64 core without
   corruption -- `count`@8 and `c_str`@0 are width-invariant, only `cap`'s
   offset moves, so routing cap through ctx is the key; exempt
   .cast(U64/U32) and .to_u64/to_u32 self-casts from the redundant-cast
   lints (target-dependent values need them on one width); widen the whole
   self-compiler's bare-U32 indices/counts/lengths to USize.
2. The flip (Two-pass): usize.til USize:=U64; ext.h USize typedef
   target-driven; target_usize_name -> U64 on 64-bit; context defaults ->
   U64; ffi_prep_cif arg narrowed with np.to_u32(); size-assertion tests
   updated (Str.size 16->24 etc.); the GUI examples + lib/music.til swept
   off the legacy `.to_u32()`/`.cast(I32)` U32-era index/width idioms.

Regression coverage: the existing test/vecs.til, test/dynamic.til,
test/tuples.til, test/dyn_vecs.til size assertions now pin Str/Vec/DynVec
at their U64 widths; the full two-pass build is the end-to-end check that a
U32-built til_boot can produce and self-apply a U64 compiler.



Current state mixes two separate problems:

1. ~USize~ is still effectively hardwired to ~U32~, even on 64-bit builds.
2. Code that is really talking about collection sizes/capacities sometimes
   gets routed through ~UPtr~ or explicit ~to_u32()~/~to_u64()~ conversions.

This creates avoidable friction in the compiler and core library:

- alias-resolution bugs around ~USize~ return types and declarations
- noisy conversion chains where a value should stay in one logical size type
- confusion between "count/capacity/len" values and raw pointer-width values

Desired direction:

- Make ~USize~ and ~UPtr~ target-driven in one place.
- On 32-bit targets: ~USize := U32~, ~UPtr := U32~.
- On 64-bit targets: ~USize := U64~, ~UPtr := U64~.
- Keep APIs that mean collection length/capacity/size on ~USize~.
- Use ~UPtr~ only where raw pointer arithmetic / byte offsets really need it.
- Clean up callsites that currently force this manually with things like
  ~to_u32()~ where ~to_usize()~ or no conversion should be needed.

This likely needs a careful pass through:

- builtin type registration / alias normalization in init + typer
- memory and pointer APIs in ~src/core/mem.til~
- collection constructors and helpers
- codegen / interpreter places that currently assume ~USize == U32~ on all targets

The recent ~dyn_size_of~ / collection-constructor work kept tripping over this,
so this cleanup should happen before more size-related refactors pile up.

*** Remaining (revised plan, 2026-06-05)

The earlier step-4/5/6 split assumed the width flip (step 5) could land
green on its own, with the bare-~U32~ cleanup (step 6) deferred to a
later commit. A spike showed that assumption is wrong:

- ~USize~ is a SOURCE-LEVEL alias (~src/core/usize.til~). The moment the
  alias resolves to ~U64~ (whether via ~USize := U64~ in source or via
  ~target_usize_name~), the compiler doing the typecheck -- INCLUDING the
  ~bin/til_boot~ baked from master -- resolves ~USize~ to ~U64~ on a
  64-bit host. ~target_usize_name~'s retarget to ~U32~ does not save us:
  the host target is 64-bit, so the alias is ~U64~.
- That immediately surfaces ~958~ type errors across ~18 files at PASS 1
  (~bin/til_boot build~), so the flip cannot compile at all until the
  whole self-compiler is width-correct. There is no green intermediate
  with the flip in and the cleanup out. Flip and cleanup are coupled.

The errors fall into four mechanical buckets:
1. ~argument type mismatch ... expected U32/U64~ (~816): a ~USize~ value
   meets a bare-~U32~ counter/param, or a ~U32~ counter meets a ~USize~
   operand (e.g. ~usize_val.lt(u32_counter)~ -- ~U64.lt~ wants a ~U64~
   second arg, no implicit coercion for a NAMED ~U64~ param).
2. ~redundant self-cast: cast(U64)~ (~106): ~usize_val.cast(U64)~ is a
   no-op once ~USize == U64~.
3. ~cannot cast U64 to I32~ (~24): ~usize_val.cast(I32)~ stops being a
   same-size reinterpret once ~USize == U64~.
4. ~no method to_i64 for U64~ (a few): the ~to_*~ conversion APIs on
   ~U32~ and ~U64~ are asymmetric (~U32~ has ~to_i64~, ~U64~ does not;
   ~U64~ has ~to_i32~, ~U32~ does not), so code written against ~USize~
   only typechecks on the width whose underlying type happens to carry
   that method.

Key consequence: nearly all of this cleanup is WIDTH-NEUTRAL and can be
done while ~USize~ is still ~U32~, where it typechecks against the
current ~boot/~ and ~make test~ stays green. So the build can be split
into a green preparation commit plus the flip:

**** Commit 1 -- preparation (plain ~make test~, ~USize~ stays ~U32~)

Make the source width-agnostic without changing any width:

- Symmetric numeric conversions so a ~USize~ value converts portably on
  either width: add the missing ~to_*~ ext_funcs to ~src/core/u32.til~
  and ~src/core/u64.til~ (at least ~U64.to_i64~, ~U64.to_u64~,
  ~U32.to_i32~, ~U32.to_u32~, ~U32.to_u64~) plus their ~src/c/ext.h~ +
  ~src/c/ext.c~ plumbing.
- Sweep the self-compiler (~src/self/*.til~) and any core/std callers:
  change bare-~U32~ length/index/count/offset/capacity locals and params
  that consume ~.len()~ / ~.count~ / size values to ~USize~.
- Replace ~usize_val.cast(U64)~ with a portable conversion: drop the cast
  where the target param is ~UPtr~/~USize~-aliased (the existing
  ~can_implicit_usize_coerce~ handles it), else ~.to_u64()~ for a bare
  ~U64~ target.
- Replace ~usize_val.cast(I32)~ with ~.to_i32()~ and ~usize_val.to_i64()~
  stays valid once ~U64.to_i64~ exists.
- Find the exact site list by flipping on a throwaway branch and reading
  the ~bin/til_boot build~ errors, then apply those fixes on the
  UNflipped base. Verify ~make clean && make test~ (green: ~USize == U32~
  so every change is a no-op width-wise).

**** Commit 2 -- the flip (~Two-pass:~ commit)

With the source already width-clean, widen in one shot:

- ~src/core/usize.til~: ~USize := U64~ (import ~u64~).
- ~src/self/targets.til~: ~target_usize_name~ returns ~U64~ on 64-bit
  (keep ~U32~ on 32-bit), mirroring ~target_uptr_name~.
- ~src/self/context.til~ + ~src/self/initer.til~: init-side ~USize~
  hardcode -> ~U64~ (default; loader still retargets to ~U32~ on 32-bit).
- ~src/c/ext.h~: make the ~USize~ typedef target-driven (mirror the
  ~#if UINTPTR_MAX == 0xffffffff~ block already used for ~UPtr~).
- ~src/self/typer.til~: ~can_implicit_usize_coerce~ gains a ~U64~ case
  for ~USize~ (mirror the ~UPtr~ arm).
- ~src/self/interpreter.til~: make the four ~USize~ scalar/FFI decode
  sites alias-aware (mirror the existing ~UPtr~ arms). ~dispatch.til~ is
  already alias-aware from step 1.
- Ships as ~Two-pass:~ so ~boot/til.c~ regenerates against the widened
  alias. Verify ~make clean && make test_two_pass~. If commit 1 was
  complete this commit should produce no NEW type errors -- only the
  boot regen.

**** Commit 3 -- optional post-flip polish

Sweep any remaining container sizes still routed through ~UPtr~ or
explicit ~to_u64()~ and put them back on ~USize~; remove now-dead
width-bridging helpers. Plain ~make test~.

*** Progress 2026-06-24: revised plan -- emit the USize/UPtr typedef name

The 958-error spike from the 2026-06-05 analysis predated the per-target
authority routing (~ctx.target_usize_pname~ / ~scope_target_usize_pname~
in ~context.til~ / ~typer.til~ / ~initer.til~ / ~loader.til~), which
already makes ~type_from_name~, ~resolve_type_alias~, ~primitive_type_info~
and the USize/UPtr ~TypeBinding~ resolve to the loader-stamped target
width. With that routing in place the flip type-checks cleanly; the only
remaining blocker is the *C ABI*: ~til_boot~ (built from master) emits
struct fields with the *resolved concrete* type (~U32 cap~) while a flipped
~ext.h~ typedefs ~USize~ as ~U64~, so ~TIL_CAP_LIT ((USize)-1)~ and every
size assignment overflow at the C-compile of pass 1. Two-pass cannot help:
pass 1 itself fails to compile.

Root cause: the struct-field emitter resolved the alias away. Fix
(approach used here): make ~ext.h~'s ~USize~/~UPtr~ typedef the single
source of truth for the field ABI by emitting the *alias name* for those
fields, exactly as function params already do (~param_ctype~ ->
~type_name_to_c_value(explicit_type)~ already spells ~USize~/~UPtr~).

Split into:

**** Commit A -- emit-side preparation (plain ~make test~, ~USize~ stays ~U32~)

- ~src/self/initer.til~ ~compute_struct_layout~: keep ~USize~/~UPtr~ on the
  field's ~explicit_type~ (do NOT overwrite it with the resolved concrete
  primitive); resolve only a local copy through the authority for the
  size/align math, and set ~til_type~ to the resolved primitive for
  type-checking.
- ~src/self/builder.til~ ~field_emit_ctype~: emit ~USize~/~UPtr~ (the
  typedef name) for those fields in ~emit_struct_typedef~ instead of the
  concrete primitive from ~til_type~.
Width-neutral at ~USize == U32~ (the typedef is ~U32~, so the emitted C is
byte-identical). Must land before the flip so ~til_boot~ bakes the
alias-name emission in.

**** Commit B -- the flip (~Two-pass:~ commit), after Commit A merges

- ~src/c/ext.h~: ~USize~ typedef target-driven (mirror ~UPtr~'s
  ~#if UINTPTR_MAX==0xffffffff~ block).
- ~src/core/usize.til~: ~USize := U64~.
- ~src/self/targets.til~: ~target_usize_name~ -> ~U64~ on 64-bit.
- ~src/self/context.til~: ~TypeScope~/~Context~ ~target_usize_pname~ default
  -> ~U64~ (loader still retargets to ~U32~ on 32-bit).
- ~src/self/typer.til~: ~can_implicit_usize_coerce~ gains a ~U64~ arm.
- ~src/self/interpreter.til~ + ~src/self/constfolder.til~: USize scalar/FFI
  decode and the ~CAP_LIT~ sentinel become alias-aware (target width).

*** Progress 2026-06-24 (cont.): full type-sweep done; constfold blocker found

With commits A (emit-prep) and the source-driven resolution merged into
~til_boot~, the flip was applied and the entire self-compiler width sweep
completed: every bare-~U32~ index/length/count/offset local, param, loop
var, and ~Vec(U32)~-of-indices was widened to ~USize~, and ~usize.cast(I32)~
/ ~i32.cast(U32)~-as-index idioms were converted to ~.to_i32()~ /
~.to_usize()~. Result: ~bin/til_boot build~ reaches ZERO type errors on the
flipped source (down from 655). The full sweep is preserved on this branch
as a WIP commit (does NOT build -- see below).

REMAINING BLOCKER (architectural, not mechanical): ~bin/til_boot~ (built at
~USize == U32~ from the prep commits) SEGFAULTS while compiling the
~USize == U64~ source, before emitting any C. gdb backtrace:

  free(mem=0x36)  <- free_container_str_data
  <- struct_deep_free  <- free_temp_arg_value  <- scope_free
  <- constfolder fold_subtree ...  <- constfolder_imported  <- prepare_program

The compile-time constfold INTERPRETER inside ~til_boot~ tears down ~Str~
Values using the SOURCE's field offsets (~ctx_field_layout~ -> ~U64~: c_str@0,
count@8, cap@16) while the Value data was built by ~til_boot~'s own
hardcoded ~U32~ ~make_str_value~ (count@8, cap@12, 16-byte). Reading the
~c_str~ at the wrong offset yields garbage (0x36) that ~free~ then crashes
on. In other words, ~til_boot~'s compile-time interpreter is COUPLED to the
source's ~USize~ width, but ~til_boot~ itself was built at the old width, so
it cannot interpret core code of the new width -- and constfolding runs core
code at compile time on every build.

So the flip needs ANOTHER preparation step before it can pass pass 1:
decouple the constfold interpreter's runtime layout from the source's
emitted-code width (make the interpreter's ~Str~/~Vec~/~Array~ layout
helpers -- ~make_str_value~ et al -- compute offsets/sizes the same way the
struct-walk teardown does, i.e. from ONE consistent layout source, so a
~U32~-built ~til_boot~ can fold ~U64~ core code without corruption). This is
an interpreter refactor of unknown tail length (the segfault is only the
first hardcoded-layout site); it is the real remaining work for #192.
