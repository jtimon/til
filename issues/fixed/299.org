** FIXED #299 Fix non-leak ASAN failures in test_asan_full

Follow-up (2026-07-02): a new non-leak failure surfaced and was fixed HERE
(per the standing instruction at the end of this file), no re-open.
~bin/til_asan build test/structs.til~ aborted with a ~heap-use-after-free~
in ~Str_len~ (via ~ctx_field_layout~ <- ~str_count_at~ <- ~str_owns_c_str~
<- ~StructInstance.delete~) during ~Context_delete~.

Root cause: Context's auto-generated delete frees fields in declaration
order, so the AST (~imported~, a ~Map(Str, ImportUnit)~) and the layout
cache (~struct_layouts~) are reclaimed BEFORE the constfolder/interpreter
Value caches (~ns_fields~, ~constfolder_known~), which are declared after
them. Tearing down a cached Str Value runs ~StructInstance.delete~ ->
~str_owns_c_str~ -> ~str_count_at~ -> ~ctx_field_layout(ctx.cached_str_name,
...)~. ~cached_str_name~ is a borrowed ~ref~ into the AST (and, for a
program whose typer inlined lazy calls, into the ~lazy_stmt_inline_body~
~Vec(Expr)~ buffer that ~ImportUnit.delete~ already freed), so the layout
lookup read freed memory. The interpret path avoids this by clearing
~ns_fields~ via ~interp_session_finish_context~; the build/translate/run
paths had no equivalent, and ~constfolder_known~ was cleared only at the
START of ~prepare_program~ (~constfolder_reset_state~), never at the end.

Fix: reclaim both Value caches at the end of ~run~ in ~src/til.til~ (after
the command dispatch, before the ~LoadedProgram~ is torn down) for EVERY
command -- ~interpreter_free_ns(lp.ctx)~ + ~constfolder_clear_known(lp.ctx)~
-- while the AST, ~struct_layouts~ and ~cached_str_name~ they reference are
still alive. ~interpreter_free_ns~ was made non-priv for this. Both clears
are idempotent (an ~ns_inited~ guard and ~Map.clear~ on an empty map), so
the interpret path (which already clears ~ns_fields~) is unaffected.
Verified: ~bin/til_asan build test/structs.til~ plus interpret/translate/
build of structs/tuples/kwargs are non-leak-clean (~detect_leaks=0~);
~make test~ 164/164. Regression coverage: the ~test_asan_full~ compile
phase (~--til-bin bin/til_asan~) exercises this exact path.

Post-mortem (2026-06-29): #299 is the umbrella for every ~test_asan_full~
test that fails AddressSanitizer for a reason OTHER than a leak
(heap-buffer-overflow, heap-use-after-free, bad-free, stack/global
overflow, ...), so that [[../open/109.org][#109]] stays leak-only. All such
failures are fixed AND the whole suite was confirmed non-leak-clean: on
master 1760943 ~make test_asan_full~ = 144/164, and every one of the 20
failures is a LEAK (#109), not a non-leak error -- see "Verification".

History: originally filed for just the ~translate src/til.til~ use-after-
free, then correctly re-scoped to the whole non-leak category. It was
closed after the first batch of fixes, briefly re-opened to confirm the
umbrella really covered ALL non-leak failures (not only the four documented
up front), and closed again once ~test_asan_full~ was verified to have zero
non-leak failures.

*** Fixes

1. ~test/enums.til~ + ~test/regression.til~ -- no-payload enum tag width
   (heap-buffer-overflow in ~src/c/ext.c:is_n~ and in ~Value_Int32~ via
   ~reinterpret_ptr_value~). Root cause: a no-payload enum's discriminant
   was read/stored as a fixed 4-byte I32 regardless of the declared tag
   width (1/2/4/8 bytes, #264). For a U64 enum the cached variant value
   was narrowed to a 4-byte Int32 box at ns-registration, then the
   auto-generated ~is~/~eq~ passed ~enum_tag_size~ (=8) to ~is_n~, which
   read 8 bytes off the 4-byte box. For a default (U8) enum read back
   through a Map slot (~bug150~), ~reinterpret_ptr_value~ read a fixed I32
   (4 bytes) off a 1-byte region. Fix: a width-aware ~read_enum_tag_value~
   helper (reads exactly ~enum_tag_size~ bytes; boxes a U64 tag as the full
   8-byte ~Value.Int~) at both reinterpret sites, plus a ~enum_tag_size <= 4~
   guard on the ns-registration narrowing. Regression:
   ~test/enums.til::test_enum_u64_high_value~, ~test/regression.til::bug150~.

2. ~ffi_interp~ (~examples/ffi/ext_module.til~) -- ~link_c~ FFI allocator
   mismatch (bad-free in the ASAN free wrapper). Root cause: ~bin/til_asan~
   compiled the user ~.so~ WITHOUT ~-fsanitize=address~, so an
   interpreter-side ~calloc~ (e.g. ~Array.new~ for a variadic ~println~)
   that libffi routes through the non-ASAN ~.so~'s libc allocator returned
   brk-heap memory, which the interpreter's ASAN ~free~ then rejected as a
   bad-free. Any variadic call in a ~link_c~ program tripped it. Fix: a new
   ~til_built_with_asan~ ext_proc (C ~__SANITIZE_ADDRESS__~ probe) gates
   ~-fsanitize=address -fno-omit-frame-pointer~ onto the FFI ~.so~ compile
   in ~ffi_init_user_so~, so the whole process shares one allocator. It is
   an ext_PROC, not an ext_func, on purpose: as a func the constfolder
   folds it at build time, and bin/til_asan is built by the non-ASAN
   bin/til, so the fold would bake in ~false~ and the flag would never
   reach the compile.

3. ~translate src/til.til~ -- constfolder heap-use-after-free in
   ~needs_source_escape~ (the original framing of this issue). Root cause:
   when the constfolder eval'd ~EnumType.Variant(own struct)~,
   ~enum_method_dispatch~ byte-copied the WHOLE struct payload
   (~psz = ctx_total_size~) into the enum's own buffer, sharing every
   inlined nested heap pointer (a nested Str's c_str). The own-arg cleanup
   in ~eval_call~ then did ~oc.val = val_none()~, whose deep
   ~struct_deep_free~ recursed and freed those shared pointers; the enum
   (carried up as the fold result) dangled, and ~value_to_expr~ ->
   ~str_to_source_form~ -> ~needs_source_escape~ read the freed c_str. The
   old ~is_str = false~ hack only neutralized a DIRECT Str own arg (whose
   c_str ~val_enum_flat~ moves out). Fix: free only the now-redundant
   wrapper buffer (the bytes already live in the enum's own copy) via a
   shallow ~EvalHeap.heap_free(own osi.data)~ and leave the shared inlined
   pointers for the enum to own.

4. ~test/aliases|dynamic|errors|tuples.til~ -- ~callable_box_is~
   heap-buffer-overflow. Already fixed on master before this issue's work.

*** Verification (master 1760943)

~make test_asan_full~ redirects each spawned test's output to ~/dev/null~
(see ~test_runner.til~), so its log only has PASS/FAIL -- the ASAN text is
NOT in it -- and it runs every interpret/translate under the default
~detect_leaks=1~, so a FAIL can be a leak OR a non-leak error. Classify by
re-running the failing test directly:

    ulimit -d unlimited
    ASAN_OPTIONS=detect_leaks=0 bin/til_asan <interpret|translate> <file>   # non-leak only
    ASAN_OPTIONS=detect_leaks=1 bin/til_asan <interpret|translate> <file>   # +leaks

Clean under ~detect_leaks=0~ but aborting under ~detect_leaks=1~ = a LEAK
(#109). A non-leak abort under ~detect_leaks=0~ = a #299 item.

~make test_asan_full~ = 144/164. All 20 failures classified this way exit 0
with NO non-leak ASAN error under ~detect_leaks=0~, aborting only on a
LeakSanitizer block under ~detect_leaks=1~ -- i.e. all 20 are LEAKS in #109:

  interpret: kwargs, dyn_hashmaps, enums, lazy, dynamic, dyn_maps, asting,
  structs, eval_heap, templ_vec, tuples, errors, closures, regression,
  dyn_sets, own_fields, dyn_arrays; plus ffi_interp, closures_interp,
  self_translate (translate src/til.til, compiler-side leak).

Zero non-leak failures remain. If a new non-leak ASAN failure surfaces in
~test_asan_full~ later, re-open this and fix it here (NOT in #109).
