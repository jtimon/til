** FIXED #225 Subscript assignment a[b] = c -> a.set(b, own c)!

   *** Post-mortem

   Shipped the write-side mirror of the read-side ~a[b] -> a.get(b)!~
   desugar. ~a[b] = c~ now desugars to ~a.set(b, own c)!~ generically,
   so any type whose ~set~ throws is subscript-assignable. ~Vec.set~
   and ~Array.set~ throw ~OutOfBounds~; ~vec[i] = x~ and ~arr[i] = x~
   work. ~Map.set~ inserts-or-overwrites (never fails) so it stays
   non-throwing and ~map[k] = v~ does not compile -- the same
   asymmetry as a ~str[i]~ write.

   The bulk of the work was bootstrap sequencing: ~til_boot~ (the prior
   master compiler) compiles the migrated sources, so each piece of new
   syntax / changed codegen had to land in ~boot/~ before the next
   could use it. ~two_pass~ does not help -- pass 1 fails to parse or
   to compile the emitted C. Hence three preparation commits before the
   feature:

   1. Parser: ~a[b] = c~ desugar (~parse_subscript_assign~) + statement-
      level method ~?~/~!~ (the lowered ~a.set(..)!~ is a statement-
      position throwing method call, which the statement parser did not
      previously accept).
   2. Builder: the variadic-argument codegen collected ~..args~ into an
      ~Array~ via ~Array_set~ in two string-write sites (main()'s entry
      wrapper and the ~array_of_<T>~ helper); rewrote both to ~memcpy~,
      decoupling them from ~Array.set~'s arity.
   3. Typer: the variadic lowering ALSO emits an inline ~Array.set~ per
      arg (~build_variadic_array_set~), and it runs after throws-
      lowering so the call never gets its err-args. Added a non-throwing
      ~Array.set_init~ (panic-on-OOB) and pointed the variadic codegen
      at it.

   Feature commits: ~Vec.set~ throws + migrate ~Vec.set~ call sites
   (~src/self~, ~src/std~ HashMap internals -> subscript; ~test/~ kept
   in the explicit ~vec.set(i, own x)!~ form); then ~Array.set~ throws +
   migrate the two ~done.set~ sites in ~builder.til~ -> subscript and
   the ~slots.set~ sites in ~src/tests.til~.

   Regression tests: ~test_vec_subscript_assign~ / ~_str~ / ~_oob~ in
   ~test/vecs.til~ and ~test_array_subscript_assign~ / ~_str~ / ~_oob~
   in ~test/arrays.til~ -- literal and computed indices, I64 and Str
   elements, round-trip under interp and compile, and the OOB-throws-
   via-~catch~ path (Str element, so the owned value's throw-path
   cleanup is exercised under the suite's ASAN).

   *** Follow-ups (post-merge, tracked here for traceability)

   - DynVec / DynArray subscript assignment: ~DynVec.set~ and
     ~DynArray.set~ now throw ~OutOfBounds~ too, so ~dynvec[i] = x~ and
     ~dynarr[i] = x~ work -- their ~get~ already threw, so this restores
     read/write symmetry. Clean (no prep): the Dyn* types are not in the
     variadic-arg codegen. Their owned ~val~ uses an explicit ~free~
     (not Vec's memset trick), so the throw path disposes ~val~ via
     ~elem_delete~ + ~free~ inside an ~if/else~ -- the garbager must see
     ~val~ consumed on exactly one path. Call sites:
     ~src/std/dyn_hashmap.til~ (DynVec buckets/nexts) -> subscript;
     ~test/dyn_vecs.til~ / ~test/dyn_arrays.til~ kept explicit. Tests:
     ~test_dyn_{vec,array}_subscript_assign~ (+ ~_oob~). ~DynMap.set~
     stays non-throwing (like ~Map~).

   - Drop the ~Array.set_init~ workaround (prep 3 above): the variadic-arg
     codegen no longer needs a non-throwing setter. ~build_variadic_array_set~
     now emits the throwing ~Array.set~ and hand-wires its err out-args,
     because the variadic desugar runs inside ~infer_body~ -- after throws-
     lowering -- so the generated call is never auto-lowered. For each temp
     ~Array~, ~build_variadic_err_slots~ declares two adjacent local slots,
     ~<va>_ek : I64~ and ~own <va>_eo : OutOfBounds~, and the ~set~ call takes
     them as its ~_err_kind~ / ~_err_OutOfBounds~ args (the in-bounds fill
     never throws, so they are write-only padding). The slots are built fully
     resolved (til_type set + a resolved default ctor via
     ~build_resolved_struct_default~, which reads ~OutOfBounds~'s fields from
     scope) because post-inference nodes are not re-inferred -- an un-typed
     decl would fall back to I64 and an un-resolved ~OutOfBounds()~ init would
     emit a bogus C call. Slot names are forced to a leading underscore
     (~variadic_err_slot_name~) so the unused-variable check skips them, the
     same exemption the ~_array_N~ temps already get. This is a staged
     migration: this commit keeps ~Array.set_init~ (the prior ~til_boot~ still
     emits ~set_init~ calls for its own variadics until the new typer is baked
     into ~boot/~); the immediate follow-up commit deletes ~set_init~ from
     ~src/core/array.til~ once master's ~til_boot~ generates the throwing form.

   Mirror of the shipped read-side desugaring: the parser now turns
   ~a[b]~ into ~a.get(b)!~ (generic over any struct/enum whose
   ~get~ throws). Do the same for the write side so

     a[b] = c

   desugars to

     a.set(b, own c)!

   generically -- no type-name special-casing, exactly like
   ~a + b -> a.add(b)~ and ~a[b] -> a.get(b)!~. The ~own~ on the
   third arg matches ~Vec.set~'s and ~Map.set~'s ~own val~
   parameter, so the desugar transfers the RHS into the container
   the same way an explicit hand-written ~a.set(b, own c)!~ would.

   *** Blocked on: set must throw

   ~Array.set~ / ~Vec.set~ / ~Map.set~ currently ~panic~ on a bad
   index/key; they do not throw. The ~!~ in ~a.set(b, own c)!~
   needs a declared throw (e.g. ~OutOfBounds~ / ~KeyNotFound~,
   matching what ~get~ throws on the same container) to
   propagate/panic on, so the collection ~set~ methods must throw
   first. That method-side change is tracked in [[./201.org][#201]]
   (built-in error types) and noted in [[../fixed/198.org][#198]].
   Once ~set~ throws, this issue is unblocked.

   *** Parser work

   The read-side lives in ~parse_primary~: postfix ~[index]~ in the
   field-access chain builds ~e.get(index)!~. For the write side the
   subscript appears on the LHS of an assignment, which the postfix
   expression parser does not own -- ~a[b]~ already parses as the
   rvalue ~a.get(b)!~ (an FCall), so a bare ~a[b] = c~ would try to
   assign to a call result.

   So the assignment-statement parser must recognise an
   ~EXPR[INDEX] = RHS~ shape and emit ~EXPR.set(INDEX, RHS)!~
   instead of routing ~EXPR[INDEX]~ through ~get~. Compound forms
   (~a[b] += c~ etc.) can be left out of scope for a first cut, or
   desugared to ~a.set(b, a.get(b)!.add(c))!~ if wanted.

   *** Call-site migration (after the desugaring lands)

   Once ~set~ throws and ~a[b] = c~ works, do the same mechanical
   migration we did for ~get~: replace ~a.set(b, c)!~ with
   ~a[b] = c~ across ~src/~, ~examples/~, and ~lib/~. Leave
   non-throwing ~.set(...)~ calls (no ~!~) alone, and leave ~test/~
   in the explicit ~.set(b, c)!~ form so the desugaring stays
   exercised both ways. A context-aware script (skip strings and
   comments, balance parens for the args) is the right tool -- the
   read-side migration was 312 call sites across 21 files.

   *** Test

   Add a ~test_array_subscript_assign~ alongside
   ~test_array_subscript~ in ~test/arrays.til~: literal and computed
   indices, under both interp and compile, asserting ~a[i] = v~ then
   ~a[i]~ round-trips. Once ~set~ throws, also cover the OOB-throws
   path via ~catch~.

   *** Recipe

   The throwing-method side of the work (making ~Vec.set~ throw) is
   a copy of the worked example in [[../../doc/throws.org][doc/throws.org]]
   "Adding a new throwing container method" -- same shape as
   ~Vec.get~. The parser-side desugar mirrors ~a[b] -> a.get(b)!~
   from ~parse_primary~; the assignment branch is the new code
   path that needs to recognise an ~EXPR[INDEX] = RHS~ shape
   before routing through the rvalue rule.
