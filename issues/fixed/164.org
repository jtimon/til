# priority: low
** DONE #164 Diagnose `mut` qualifiers that are never used to write

   Step 1 (LANDED): the typer flags redundant `mut` on locals via
   ~check_unused_mut_locals~. A binding's ~written~ flag is set when:
   the local appears on the LHS of a plain assignment, on the LHS of a
   field assignment (`name.f = ...`, including nested chains), as the
   receiver root of a UFCS method call (`name.method(...)`), as an
   Ident or FieldAccess argument passed to a callee parameter declared
   `mut`, or as the source of a `case V(p):` ref-mut payload binding.
   `mut_explicit` is snapshotted from the source declaration so
   typer-internal auto-promotions (`arr := array(...)`,
   payload-alias decls) don't yield false positives. ~is_alias~
   bindings are skipped because the `mut` keyword is synthesised by
   the typer, not written by the user.

   Step 2 (LANDED): the typer flags redundant `mut` on parameters via
   ~check_unused_mut_params~, called once per function right after
   ~check_unused_params~ (params live on the func_scope only, not on
   the inner if/while/block scopes). It uses the same ~written~
   tracking as the locals check; ext_func/ext_proc and auto-generated
   bodies are skipped. Three write-detection gaps in the step-1
   machinery were closed so genuine mutations aren't misflagged:
   (a) args passed to a ~Dynamic~ parameter (raw-pointer write-through:
   ~move~/~memcpy~/~memset~/~write_ptr~) now mark the arg's root
   binding written, in both the regular-call and namespace-method-call
   paths; (b) the namespace-method-call path (~Type.method(arg)~) now
   mirrors the regular path's mut-arg marking, which it previously
   lacked. The self-host sweep dropped ~mut~ from 22 parameters
   (chain-walker cursors in garbager.til, ~infer_map_lit_expr~/
   ~infer_set_lit_expr~, ~eval_body~, etc.) -- exactly the laundering
   "small lies" this issue set out to remove.

   Step 3 (LANDED): namespace declarations -- but only the ~priv mut~
   shape. ~check_unused_mut_ns_decls_in_program~ walks every loaded
   file's program right after the priv-member walker; for each
   StructDef / EnumDef ns_decl that is both ~priv~ and ~mut~ and
   whose ~written~ flag (a new field on ~Declaration~, set at the
   FieldAssign-to-ns_decl site in ~infer_field_assign_stmt~) was
   never flipped, the diagnostic fires.

   Public ~mut~ ns_decls are NOT flagged: they describe an exposed
   mutation point (e.g. ~HashMap.hasher~, where the library leaves a
   knob open even when it doesn't turn the knob itself), and the
   single-file view a typer walk has can't see writes from downstream
   consumers. ~priv mut~ has no such ambiguity -- ~priv~ blocks all
   downstream access, so absence of an in-file write proves the
   ~mut~ is dead.

   Self-host sweep: zero -- no ~priv mut~ ns_decls were found.

   Done. Closing the issue.

After [[./161.org][#161]] the typer rejects `ref mut x : T = src` when `src` is
   immutable. The simplest way to fix the existing chain-walker
   patterns (cursors that only rebind themselves to walk a chain, like
   `scope_get` in src/self/interpreter.til, `extend_ref_local_lifetimes`
   in src/self/typer.til, and `eval_body`'s nested-FieldAssign walker)
   is to mark the source param as `mut` so the cursor's `ref mut` is
   legal. That's a small lie: the param is declared writable but the
   body never writes through it.

   Goal: have the typer warn (later, error) when a `mut` param or
   `mut` / `ref mut` local is declared but never used as a write
   target -- neither `x = ...`, nor `x.f = ...`, nor passed as the
   `mut`/`own` argument of a callee that writes through it. Then the
   chain-walker params can drop the `mut` qualifier without losing
   the laundering check.

   Notes:

   - Rebind-only cursors (`x = x.next`) currently require `ref mut`,
     which forces the source to be mut. A separate language-level
     fix would be to distinguish "rebindable local pointer" from
     "write-through ref mut" so a chain walker doesn't need to lie
     about its source. Worth considering when picking up this issue.
   - The diagnostic should ignore `_`-prefixed names (consistent
     with [[./86.org][#86]] unused-local handling).
