* OPEN #8 Variadic args for funcs, procs, and macros
  Currently `..Type` variadic parameters only work in ext_func and ext_proc
  (format, println, print). Regular funcs, procs, and macros can't use them.

  What's needed:
  - Parser: allow `..Type` in func/proc/macro parameter lists
  - Typer: validate variadic calls, ensure args match the element type
  - Interpreter: collect variadic args into an array-like structure
  - ccodegen: emit variadic C calls or array-based passing

  This blocks several improvements:
  - Desugaring println/print in the typer (the replacement needs to be variadic)
  - User-defined variadic functions (e.g. custom logging, string builders)
  - Variadic macros for compile-time code generation

* OPEN #7 Precomp phase: compile-time macro evaluation
  A new function kind, macro, that executes at compile time. Macros look like
  procs but run during a precomp phase (between typer and scavenger) using the
  interpreter. The macro call is replaced in the AST with its result — a numeric
  literal, string literal, or struct literal with all fields explicitly provided.

  Macros can be called inside funcs (unlike procs). Internally, macros can call
  both funcs and procs. The key constraint: all arguments to a macro call must
  be known at precomp time (literals or constants). If a runtime variable is
  passed, the typer (or precomp phase) emits an error — we can't run the whole
  program just to resolve one macro argument.

  The precomp phase uses the interpreter itself to evaluate macro bodies. This
  creates a circular dependency (interpreter is part of the compiler, and is
  also used during compilation), but C header forward declarations handle this.

  Example:

    macro_add := macro(a: I64, b: I64) returns I64 {
        return a.add(b)
    }

    # OK — both args are literals, known at precomp time
    x := macro_add(3, 4)    # replaced with: x := 7

    # OK — constants are known at precomp time
    N := 10
    y := macro_add(N, 5)    # replaced with: y := 15

    # ERROR — v is a runtime variable, not known at precomp time
    v := read_input()
    z := macro_add(v, 1)    # error: macro arg must be known at precomp time

  Pipeline after this: lex → parse → type check → precomp → scavenge → interpret/codegen

** Phase 2: constant folding for pure funcs
   Once macro evaluation works, extend precomp to also evaluate func calls
   (not procs — they're not pure) when all arguments are known at precomp time.
   Same mechanism: the interpreter evaluates the call and replaces it with a
   literal in the AST.

     double := func(x: I64) returns I64 { return x.add(x) }

     a := double(5)          # folded to: a := 10
     b := double(double(3))  # folded to: b := 12
     c := double(v)          # v is runtime — left as a call

   The scavenger then cleans up anything no longer referenced after precomp:
   macros are always removed (they can't be called at runtime), and funcs are
   removed if all their call sites were folded away.

* OPEN #6 Scavenger phase: remove unused declarations from the AST
  A post-typer pass that walks the AST and removes declarations not reachable
  from the program's entry point. Saves work for both interpreter and ccodegen
  — no need to process unused structs, functions, or core library methods.

  In mode cli, main is the entry point: only what main calls (transitively) is
  kept. In mode script, all top-level statements are entry points.

  Currently the full core library is prepended to every program, so most
  programs carry dozens of unused namespace methods, type definitions, and
  builtins. The scavenger would trim this down to only what's actually used.

* OPEN #5 Pointers and arrays: unlock C-level memory access
  :DEPENDENCIES: #0 #2 #3
  After #2 (ref) and #3 (own fields), til will have single-pointer semantics
  for structs, args, and returns — but still no way to work with multi-element
  allocations. C has malloc(sizeof(Elem)*n), pointer[i] indexing, pointer
  arithmetic, and dereference. Some of this is needed for self-hosting since
  the current C implementation uses these features (e.g. Expr.children, field
  arrays).

  own and ref will replace C's raw * for single pointers, giving the typer
  enough information for compile-time garbage collection (no explicit free).
  Arrays need the same treatment: the user gets power without the responsibility
  of manual free — ASAP and auto-generated delete handle cleanup.

  What's needed:
  - Array type: own arr: [I64] or similar — contiguous heap allocation of N elements
  - Indexed access: arr[i] or arr.get(i)
  - Length: arr.len()
  - Construction: [1, 2, 3] or Array(I64, size=10)
  - Ownership: arrays are owned by default, clone deep-copies, delete frees

  What's probably NOT needed (at least initially):
  - Raw pointer arithmetic (arrays with bounds checking are safer)
  - Manual dereference (* operator)
  - Void pointers

* OPEN #4 FFI: call external C functions from til
  The ext_func/ext_proc syntax already works for builtins. Extend it so users
  can call arbitrary C functions by providing a .c file alongside their .til
  program. The user declares signatures in til, implements them in C following
  the til pointer convention, and the compiler links them.

  First milestone — user-provided ext file:

    # square.c
    #include <stdlib.h>
    long long *til_square(long long *x) {
        long long *r = malloc(sizeof(long long));
        *r = (*x) * (*x);
        return r;
    }

    # program.til
    square := ext_func(x: I64) returns I64 {}
    assert_eq(square(5), 25)

  Later — mode-specific imports:
  A mode like "gui" could auto-link a library (e.g. raylib) and auto-import
  a .til file with ext_func/ext_proc signatures and wrapper structs, so the
  user just writes `mode gui` and gets access to the API.

* OPEN #3 own keyword for struct fields: heap-allocated owned pointers
  :DEPENDENCIES: #0
  Struct fields currently are always inline (contiguous in parent allocation).
  An own field stores a heap pointer instead. Unlike ref (#2), own implies
  ownership: clone recursively clones it, delete recursively deletes it.

  Three modes for struct fields:
  - inline (default): value stored in the struct. Clone deep-copies, delete recurses.
  - own: heap pointer. Clone recursively clones, delete recursively deletes. Owned.
  - ref (#2): heap pointer. Clone copies the pointer, delete skips. Borrowed.

  own and ref replace C's * syntax with explicit ownership semantics.

    Node := struct {
        value := 0
        own left: Node     # owned heap pointer, recursively managed
        own right: Node    # owned heap pointer, recursively managed
        ref parent: Node   # borrowed pointer, not managed
    }

* OPEN #2 ref keyword: non-owning references for returns, declarations, and struct fields
  :DEPENDENCIES: #0
  Currently all returns are owned by the caller (heap-allocated, caller must
  delete). This breaks when a function wants to return one of its args:

    min := func(a: I64, b: I64) returns I64 {
        if a.lt(b) { return a } else { return b }
    }

  Either a or b would be owned twice and deleted twice (double-free).

** ref returns
   returns ref I64 — the caller does NOT own the returned value and must
   not delete it. ASAP destruction skips ref return values.

    min := func(a: I64, b: I64) returns ref I64 {
        if a.lt(b) { return a } else { return b }
    }

** ref declarations
   Required when binding the result of a ref-returning function:
   ref my_min := min(34, 35). ASAP destruction skips ref variables.

** ref struct fields
   A ref field stores a pointer to an existing value (not inline).
   clone does not deep-copy it, delete does not recurse into it —
   just copies/ignores the pointer.

** args are already ref by default
   Function args are borrowed (ref) unless explicitly marked own.
   No change needed for args.

* OPEN #1 ccodegen/interpreter consistency: eliminate unnecessary special cases
  The ccodegen and interpreter should be as similar and minimal as possible.
  The typer/initer should do more AST rewriting so backends can be dumb and uniform.

** FIXED ccodegen: free() special case
   Moved to ext.c as til_free(). ccodegen generic call path handles it.
   Interpreter also uses til_free() from ext.c.

** FIXED ccodegen: exit() special case
   Moved to ext.c as til_exit(). ccodegen generic call path handles it.
   Interpreter also uses til_exit() from ext.c.

** FIXED ccodegen: i64_to_str, u8_to_str, str_eq emitted inline
   Moved to ext.c. Both backends now share the same C implementations.
   Interpreter calls ext.c functions directly instead of reimplementing.

** FIXED bool_to_str was a needless builtin
   Replaced with pure til: if b { return "true" } else { return "false" }.
   Removed ext_func, C wrapper from ccodegen, and strcmp case from interpreter.

** ccodegen: println/print special case (45 lines of type-aware printf)
   ccodegen has ~45 lines (ccodegen.c:81-126) of type-specific printf optimization:
   unwraps to_str() calls, emits bool ternaries, format specifiers per type, etc.
   The interpreter has its own ~30-line version (interpreter.c:231-263).
   The typer could desugar println so both backends handle it uniformly — e.g.
   ensure all args are Str before reaching the backends, or desugar to per-arg
   print calls.

** ccodegen: println/print unwrap_to_str optimization
   The unwrap_to_str() helper (ccodegen.c:17-36) detects to_str() calls inside
   println/print args and bypasses the string conversion to print values directly.
   This is an optimization that adds complexity. If println is desugared in the
   typer to always receive Str args, this optimization becomes unnecessary.

** ccodegen: format() special case
   ccodegen special-cases format() to prepend the arg count (ccodegen.c:131-137):
   format(a, b) -> til_format(2, a, b). The typer could insert the count as a
   literal first arg during desugaring, eliminating the special case.

** ccodegen: struct constructors (til_new_X functions)
   ccodegen emits per-struct til_X_new() constructor functions and special-cases
   struct instantiation calls to route to them. These could be replaced with C99
   compound literals at the call site: &(til_Point){.x = *x, .y = *y} instead
   of til_Point_new(x, y). This would eliminate both the constructor functions
   and the special-case dispatch.

   Progress: ccodegen now generates _new for ALL structs including fieldless
   ones (void params, malloc only). Empty struct typedefs use char padding to
   avoid zero-size issues. Previously fieldless structs had no _new, which
   broke fieldless struct clone in ccodegen.

** ccodegen: literal return values need heap allocation
   When a function returns a literal (ccodegen.c:290-306), the codegen must
   malloc+copy because literals aren't heap-allocated. The hoisting pass could
   convert all return-value literals into temporaries beforehand, making return
   emission uniform.

** ccodegen: declaration emission branches on RHS type
   Declaration emission (ccodegen.c:250-266) branches: if RHS is NODE_FCALL,
   just assign (already a pointer); otherwise malloc+copy. The hoisting pass
   could normalize this so all RHS values are either fcalls or idents, making
   declaration emission uniform.

** interpreter: ~30 remaining strcmp checks for ext_func/ext_proc builtins
   The interpreter still has ~30 strcmp(name, ...) checks in eval_call for
   builtins: i64_add, i64_sub, u8_eq, bool_not, etc. Some now delegate to
   ext.c (i64_to_str, u8_to_str, str_eq, exit, free), but the strcmp dispatch
   remains. Could be organized better — e.g. a dispatch table, or move more
   arithmetic/comparison builtins to ext.c.

** interpreter: free() accesses variable cell directly
   The interpreter's free() handler (interpreter.c:508-524) assumes the argument
   is always a NODE_IDENT and directly accesses the variable's Cell via scope_get.
   This is the only builtin that reaches into the scope like this. It would crash
   on free(some_expr()). Now that ASAP uses delete() instead, user code rarely
   calls free directly — it lives inside auto-generated delete bodies where the
   arg is always "self" (an ident). Still fragile.

** interpreter: value_deep_copy on NODE_IDENT breaks pass-by-reference semantics
   The interpreter's eval_expr for NODE_IDENT (interpreter.c:617) calls
   value_deep_copy(cell->val), so every variable read produces a copy. Arguments
   passed via scope_set_borrowed share the Cell, but the deep copy on read means
   mutations to the read value don't affect the original. ccodegen passes all args
   as C pointers (true by-reference). The interpreter should match.

   This is a bandaid for the shallow copy problem (see #0): since declarations
   and assignments shallow-copy Values, the deep copy on read prevents aliasing.
   Once the typer inserts explicit clone calls at copy sites, NODE_IDENT should
   return the value directly without deep copying, matching ccodegen.

** namespace call path: no named arg resolution
   The typer's namespace call path (typer.c:193-228) doesn't resolve named args
   into positional — it only fills defaults for missing trailing args (added in
   commit 2eb2601). If a namespace method is called with named args, they pass
   through unresolved and the interpreter crashes (NODE_NAMED_ARG not handled
   in eval_expr). Currently works because auto-generated calls use positional args,
   but would break for user code like Point.delete(call_free=false).

* OPEN #0 Complete basic memory management before extending pointer semantics
  :DEPENDENCIES: #1
  Before #2 (ref), #3 (own fields), and #5 (arrays), the existing memory
  management needs to be solid. The user should never need to call free or
  delete explicitly — ASAP handles it. But there are gaps.

  Current state:
  - ASAP inserts Type.delete(own var) after last use of each local. Works.
  - Auto-generated delete calls field deletes then free(own self). Works.
  - Auto-generated clone deep-copies fields. Works.
  - own at call sites transfers ownership, ASAP skips cleanup. Works.

  What's missing or fragile:

** Eliminate shallow copies: typer-inserted clone calls
   Both backends have shallow copies in the same 4 places — the typer should
   insert explicit Type.clone(x) calls so backends never need copy logic:

   | Operation              | ccodegen             | interpreter                  | Status |
   | Declaration (y := x)   | *dest = *src         | scope_set_owned copies Value | FIXED  |
   | Assignment (y = x)     | *dest = *src         | cell->val = val              | FIXED  |
   | Field assign (o.f = x) | field = deref(val)   | field_values[i] = val        |        |
   | Struct construction    | _r->field = *param   | field_values[fi] = eval(arg) |        |

   Declarations and assignments: the typer now inserts Type.clone(x) when the
   RHS is a NODE_IDENT. Works for ALL types — primitives (I64, U8, Bool, Str)
   and user-defined structs (with or without instance fields). Primitive clone
   is defined in pure til in core.til (e.g. I64.clone via val.add(0)), so no
   new C special cases were added. The initer's has_clone check skips
   auto-generation when a user-defined or core.til clone already exists.

   Fieldless struct clone: the initer now generates `return Type()` instead of
   `other := self; return other`, avoiding infinite recursion from clone
   insertion. ccodegen now supports fieldless struct constructors (_new with
   void params, empty struct typedef with char padding).

   Still TODO: field assignments and struct constructor args.

   Related: the interpreter's value_deep_copy on NODE_IDENT (see #1) is
   redundant now for declarations/assignments (clone handles the copy), but
   removing it requires handling ALL 4 copy sites first. Once all are done,
   NODE_IDENT should return the value directly (by reference), matching
   ccodegen's pointer semantics.

** FIXED Custom clone/delete: user-defined override for auto-generated
   The initer checks for existing clone/delete in the struct namespace before
   auto-generating. If the user defines their own, the initer skips generation.
   Works for both clone (func) and delete (proc). Tested in structs.til.

** Use-after-own error: catch transferred variables at compile time
   After `own p.delete()` or `foo(own p)`, p is no longer owned by the current
   scope. Using p after that is a bug. The typer should emit an error, ideally
   suggesting clone: "p was transferred via own — did you mean to clone it first?"

** Clone responsibility: clarify when the user must clone explicitly
   ASAP handles deletion automatically, but cloning is the user's responsibility
   in some cases. If a function needs to keep a value that was passed by ref,
   it must clone it. The rules should be clear and enforced:
   - Passing by ref (default): caller keeps ownership, callee borrows
   - Passing with own: caller gives up ownership
   - If callee wants to store a ref arg beyond its scope, it must clone

** ASAP edge cases
   ASAP works for straightforward code but may have edge cases:
   - Variables used in both branches of if/else
   - Variables captured in closures (not yet supported, but eventually)
   - Variables returned from deeply nested scopes
