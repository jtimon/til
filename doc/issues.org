** OPEN #169 Typer panics with Vec.get out-of-bounds, suppressing later type errors
   Symptom:

   The end of every ~bin/til translate~ / ~bin/til interpret~ run on
   ~src/test/typer_errors.til~ ends with a panic:

   #+begin_example
   ./src/test/typer_errors.til:295:40: error: undefined symbol 'urmom'
   ./src/test/typer_errors.til:299:17: error: proc has no proc/ext_proc calls; should be a func
   ./src/core/vec.til:97:19: panic: Vec.get: index 0 out of bounds (len 0)
   #+end_example

   The panic is silent in the test infrastructure: the typer_errors test
   counts ~error:~ lines (currently expected: 67) and the panic comes
   after they are all printed, so the test still passes -- but every
   type error that *would* have fired after the panic is suppressed.

   Minimal reproducer (2 lines):

   #+begin_src til
   break
   s := {1, 2}
   #+end_src

   Cause:

   ~desugar_set_literals~ (~src/self/typer.til:3782~) and
   ~desugar_map_literals~ (~src/self/typer.til:3816~) both use this
   pattern in their per-stmt loop:

   #+begin_src til
   if stmt.node_type.is(NodeType.Decl)
       .and(stmt.children.len().gt(0))
       .and(stmt.child(0).node_type.is(NodeType.SetLit)) {
       ...
   }
   #+end_src

   ~.and()~ is a regular method on ~Bool~, so both arguments are
   evaluated eagerly. When ~stmt~ is a ~break~ or ~continue~ (or any
   stmt with zero children), the third clause calls ~stmt.child(0)~ on
   an empty ~children~ Vec and ~Vec.get(0)~ panics. The earlier
   ~stmt.children.len().gt(0)~ check is correct but doesn't guard the
   later access because it doesn't short-circuit.

   Concrete impact:

   In ~src/test/typer_errors.til~ the ~break~ at line 145 followed
   later by SetLit/MapLit decls at lines 210/213 triggers this. The
   panic happens during the desugar pass, which suppresses at least 4
   real errors that the file's comments document as expected:

   - ~:123:16: use of 'doomed' after ownership transfer~
   - ~:142:9: use of 'ctor_name' after ownership transfer~
   - ~:210:13: map literal: key type 'NoCmp' must implement cmp~
   - ~:213:13: set literal: element type 'NoCmp' must implement cmp~

   Run those 4 expected cases in isolation and they fire correctly --
   the panic is the only thing keeping them quiet in the full file.

   Fix:

   Replace the ~.and()~ chain with nested ~if~ in
   ~desugar_set_literals~ and ~desugar_map_literals~ (and any other
   place using the same eager-and-guards-a-Vec.get pattern). After
   the fix, the typer_errors test count needs to be updated (likely
   to 71).

   This is a concrete crash caused by the broader short-circuit gap
   tracked in #117 (~Bool and/or do not short-circuit~). Fixing #117
   would prevent this kind of panic by construction. In the
   meantime, every existing call site that pairs ~.and()~ with a
   dereference whose safety depends on the earlier clause is a
   latent crash. The same shape bit ~init_declarations~ during the
   #158 refactor:
   ~rhs.node_type.is(NodeType.FuncDef).and(rhs.child(0).children.len().eq(0))~
   crashed because ~rhs.child(0)~ ran for non-FuncDef rhs with no
   children.

** OPEN #165 Declaration order should not matter inside a file or across imports
   Symptom:

   ~src/core/core.til~ contains the alias ~USize := U32~ wedged between two
   groups of ~import("...")~ calls. Moving that single line either to the
   top of the file (before the imports) or to the bottom (after every
   import) breaks the build. Today the only "safe" position is right after
   ~import("u32.til")~ and before ~import("i64.til")~. That is forward-
   declaration discipline by another name. til is supposed to have an init
   pass precisely so source order does not matter.

   Why this still happens (analysis):

   - The loader pre-extracts ~import("...")~ calls from each file's AST
     (see ~extract_imports~ in ~src/self/loader.til:97~) and builds an
     import list per file.
   - ~init_file~ (~src/self/initer.til:2319~) recurses into the dep list
     first, then runs ~init_declarations_unit~ on the importing file.
     The dep list is iterated in source order.
   - Sibling imports inside the same file are therefore order-sensitive:
     ~core.til~ relies on ~u32.til~ being init'd before any later code
     that mentions ~U32~. The ~USize := U32~ alias only works because it
     sits between ~import("u32.til")~ and the next group; if it moves
     above ~u32.til~, ~scope.find("U32")~ in
     ~register_type_alias_for_stmt~ (~src/self/initer.til:1097~) returns
     ~ScopeFind.NotFound~ and the alias is silently dropped (or produces a
     downstream type error). If it moves below, then ~i64.til~ /
     ~range.til~ / ~bool.til~ / ~mem.til~ / ... are init'd before USize is
     in scope, so any of their decls that reference USize through derived
     methods (size methods, ~for~ desugaring against ~Vec.len()~, etc.)
     resolve USize as ~TilType.Unknown~ at init time.
   - Even within a single file, ~init_declarations_unit~ does the AST walk
     in source order: ~register_struct_def_for_stmt~ runs for struct decls
     immediately during the walk, but alias decls (~NodeType.Ident~ on the
     RHS) are bucketed and processed only after the walk. That asymmetry
     means a struct that mentions an alias defined later in the same file
     sees ~Unknown~ during ~register_struct_def_for_stmt~ /
     ~gen_struct_clone_delete_for_stmt~ / ~compute_struct_layout~ -- those
     all read types via ~type_from_name_init~ before any alias bucket
     drains.

   Root cause in one line:

   ~initer.til~ does three jobs at once -- declaration indexing, struct
   layout computation, and clone/delete/size/eq/to_str method synthesis --
   and the last two need scope-resolved types, which forces the first
   into a load-bearing source order. The init pass is supposed to be
   pure name indexing; everything that needs to know "what is the type of
   X?" belongs in the typer.

   Architectural goal:

   - Make declaration order inside a single file and across imports
     irrelevant by moving every type-resolving job out of initer and
     into typer, in line with #158 (one pass per phase). Initer keeps a
     single AST walk per file that registers names by kind (struct,
     enum, func, alias, value); typer keeps its existing walk and folds
     in method synthesis and layout computation while it has full scope.
     No new phase. Net pass count goes from "init + sub-walks-of-init +
     typer" to "init + typer".

   How later phases see auto-generated methods:

   - When typer visits a struct decl it: type-checks fields, synthesizes
     any missing ~clone~ / ~delete~ / ~size~ / ~eq~ / ~to_str~ methods
     into ~ns_decls~, type-checks their bodies in the same recursion,
     registers ~Type.method~ in scope (the same ~scope.set~ that initer
     would have called), and writes the layout into
     ~Context.struct_layouts~. After the visit, the Context looks
     identical to one in which the methods had been hand-written and
     initer had registered them. Precomp / scavenger / builder look up
     ~Type.clone~ in scope and never need to know whether the user wrote
     it or the typer synthesized it -- Context is the cross-phase memo,
     same pattern #147 calls out as one of rstil's wins.

   How rstil informs the design (~../rstil/~):

   - rstil's ~init.rs:init_context~ walks each file's body once and, on
     hitting ~import("path")~, recurses via
     ~init_import_declarations~ at that point in the walk. Imports and
     local declarations interleave in source order; ~imports_init_done~
     handles cycles and re-imports. There is no separately pre-computed
     import list driving a different order than the AST itself.
   - rstil's ~preinit~ phase synthesizes ~clone~ / ~delete~ as a pure
     AST-to-AST rewrite using only field shape (mut?, primitive name?,
     FuncSig?). til does not need this as a separate phase: typer
     already knows field types and is the natural home, so we collapse
     it in rather than introducing a third pass.
   - In rstil, init never asks "what is the type of X?". That is what
     makes USize position irrelevant. til adopts the same rule but
     keeps two passes total instead of three.

   Plan (incremental, smallest landable steps first; each lands green):

   1. [PARTIAL] Strip every ~type_from_name_init~ call out of
      ~initer.til~. ~register_struct_def_for_stmt~,
      ~register_enum_def_for_stmt~, ~register_type_alias_for_stmt~,
      ~register_function_def_for_stmt~ and
      ~register_top_level_value_for_stmt~ should record names and
      shapes only. The alias case stores the target name as a Str on
      the binding without resolving it; the typer dereferences on use.
      After this step, initer is one walk per file with no scope-typed
      reads.

      Done in c4b8064: ~register_type_alias_for_stmt~ no longer
      silently drops on NotFound for builtin-primitive targets.
      Stores ~alias_target~ + ~is_type_alias = true~ when
      ~til_type_from_explicit_type(target)~ returns a non-Custom
      TilType (so ~USize := U32~ works wherever it is in the file).
      ce0a584 moved ~USize := U32~ to the top of core.til to validate
      end-to-end. Boot regenerated identically -- the alias resolves
      to U32 throughout codegen regardless of position.

      Still TODO: the same NotFound-permissive treatment for
      user-defined type aliases (target is ~TilType.Custom~), and
      stripping ~type_from_name_init~ from
      ~register_function_def_for_stmt~ and ~infer_top_level_decl_type~.

   2. [BLOCKED ON #162] Move ~gen_struct_clone_delete_for_stmt~,
      ~gen_enum_nonclone_methods_for_stmt~,
      ~gen_missing_enum_clone_for_stmt~,
      ~gen_struct_size_method_for_stmt~,
      ~gen_enum_size_method_for_stmt~, ~gen_cmp_derived_for_stmt~ and
      ~gen_unity_derived_for_stmt~ into the typer's struct/enum visit.
      The typer pushes synthesized methods into ~ns_decls~ and calls
      the same ~scope.set~ initer would have called for ~Type.method~
      bindings, so subsequent phases see no difference vs. user-written
      methods.

      Attempted as a smallest-possible-step (move just the two

      recursion-with-memo, so once ~#162~ adds explicit per-file
      imports the recursion produces a topo sort and step 2 unblocks

      Sub-finding: ~gen_cmp_derived_for_stmt~ and
      ~gen_unity_derived_for_stmt~ are already scope-free pure AST

   3. Replace ~compute_struct_layout~ + ~compute_enum_layout~ with a
      ~Context.struct_layouts~ map populated by the typer on first
      visit (subsumes ~#147~ steps 3-4: ~Declaration.field_offset~,
      ~field_size~, ~field_struct_def~ and
      ~StructDefData.total_struct_size~ all migrate to that Context
      side table).

   4. Drop ~init_declarations_global~ entirely. Once steps 1-3 land,
      every job it does (layouts, size methods, derived methods, late
      func/value re-registration) lives inside the typer's per-decl
      visit. Initer is one driver and one walk per file.

   5. Replace ~extract_imports~ + the pre-built ~iu.imports~ list with
      rstil-style in-walk recursion. ~init_file~ becomes ~init_body~:
      walk the AST in source order, and on ~import("...")~ FCalls
      recurse through ~ctx.imports_init_done~. Removes the second
      source of ordering (sibling import order driven by
      extract_imports rather than the AST itself) and is a
      precondition for circular imports (see ~#163~).

   6. [PARTIAL] Alphabetize ~src/core/core.til~'s imports and put
      ~USize := U32~ wherever (its own file, top of core.til,
      anywhere). Add regression tests covering both directions: (a)
      two files where the importer declares ~MyAlias := U32~ and an
      importee struct uses ~MyAlias~ in a field type; (b) USize-style
      alias declared above every ~import("...")~ in the importing
      file. Both must build.

      Done: ~core.til~ alphabetized; ~USize := U32~ at the top.
      Discovery: alphabetizing was a no-op on the build because
      today's machinery (init_file recursion + init_declarations_global)

** OPEN #162 Core files should import semantic dependencies explicitly
   We still rely on ~src/core/core.til~ import order for some dependencies.
   This is fragile and makes refactors (like changing Map internals) break in
   non-obvious ways.

   Goal:

   - Each ~src/core/*.til~ file should explicitly ~import("...")~ the other core
     files it depends on.
   - Then ~src/core/core.til~ can be ordered consistently (alphabetically is
     fine) without encoding hidden dependency order.

   Status:

   - ~src/core/core.til~ alphabetized (the "order consistently" bullet).
     Today's machinery does not depend on core.til's order at all:
     ~init_file~ recursively registers every name in scope across every
     imported file before the typer runs anywhere, so cross-file references
     resolve regardless of where the importing file appears in core.til's
     list. Alphabetizing is therefore a no-op on the build today.
   - The wedged ~USize := U32~ position that originally motivated this
     issue was a separate problem (alias registration silently dropped on
     NotFound), already addressed in #165 step 1.

   Remaining: per-file explicit imports.

   - Each ~src/core/*.til~ file still needs to add ~import("...")~ for
     the types it actually uses. Today this works without explicit
     imports because ~init_declarations_global~ runs across every file
     before any typer work; cross-file references thus see a fully
     populated scope.
   - This becomes a real constraint the moment ~#165~ step 2 lands
     (move method synthesis from ~init_declarations_global~ into the
     typer's per-decl visit). At that point, file A's typer visit must
     come after file B's whenever A calls auto-generated methods on a
     B-typed value, and that ordering is exactly what per-file imports
     give us via the existing ~init_file~ / ~type_file~ recursion.
   - Concrete example that bit this directly: today
     ~src/core/ast.til~ has function bodies calling ~U32.lt~ on
     ~Expr.line~, but no ~import("u32.til")~. With ~gen_cmp_derived~
     still in init_declarations_global the typer sees ~U32.lt~
     regardless of order; with it moved to typer-time the build breaks
     at ast.til:431 unless ast.til imports u32.til (or the recursion
     reaches u32.til through some other dep).
   - Short list of dependencies that need to be added (not exhaustive --
     full audit is part of the work):
     - ~src/core/map.til~ depends on ~vec.til~.
     - ~src/core/dyn_map.til~ depends on ~map.til~ and ~tuple.til~.
     - ~src/core/ast.til~ depends on ~bool~, ~str~, ~vec~, ~map~,
       ~dyn~, ~u8~, ~u32~, ~u64~, ~i32~, ~i64~ (everything its struct
       fields and function bodies reference).
     - ~src/core/str.til~ uses ~Vec~ in ~Str.split~; ~src/core/vec.til~
       uses ~Str~ in error messages and elem-type fields. Real cycle --
       resolve by picking one direction (likely ~vec~ imports ~str~,
       and ~str.til~ uses ~Vec~ via builtin name resolution rather
       than explicit import) or by extracting the shared bit.

** OPEN #158 One pass per phase across the whole pipeline
   Each phase listed in doc/pipeline.org (lex, parse,
   init_declarations, type_check, precomp, scavenge, interpret /
   build) should walk the AST exactly once. At each statement, all
   the work that phase needs to do should happen before moving on.

   Today this is not true. Concrete evidence read directly from the
   source:

   - ~tokenize~ (src/self/lexer.til:259) -- single pass over the
     source string. OK.

   - ~parse~ (src/self/parser.til:1319) -- single pass over the
     token stream. OK. After the main loop it appends pre-collected
     ~fn_sig_decls~ and ~lambda_decls~ -- no extra AST walks.

   - ~init_declarations~ (src/self/initer.til:1870-1887) calls 12
     separate sub-procs, each taking ~program~ and walking it:
       register_struct_definitions, register_funcsig_aliases,
       generate_missing_struct_clones, generate_missing_struct_deletes,
       register_enums_and_generate_nonclone_methods,
       generate_missing_enum_clones, register_type_aliases,
       compute_all_struct_layouts, generate_size_methods,
       generate_derived_methods, register_function_definitions,
       register_top_level_values.

   - ~type_check~ -> ~infer_body~ runs 7 sub-passes per body
     (already covered in #146): desugar_set_literals,
     desugar_map_literals, desugar_variadic_calls,
     desugar_kwargs_calls, hoist_fcall_args, insert_field_deletes,
     insert_free_calls.

   - ~precomp~ (src/self/precomp.til:506) has 3 numbered sub-passes
     (collect names, set up scope, fold calls).

   - ~scavenge~ (src/self/scavenger.til:212) has 5 numbered
     sub-passes (build top maps, seed worklist, BFS, filter
     top-level decls, filter ns methods).

   - ~interpret~ (src/self/interpreter.til:1622) walks core and user
     programs through 5 sub-procs each before evaluation:
     ~ffi_init_scan_program~, ~interpret_register_defs~,
     ~interpret_register_aliases~, ~interpreter_init_ns~,
     ~interpret_copy_alias_ns~, then ~eval_body~. So core/user are
     each walked ~6 times before execution.

   - ~build~ (src/self/builder.til:3617-3699) calls roughly a dozen
     emit_* helpers, each walking core and user programs:
     build_register_lookups, build_register_funcsig_names,
     emit_monolithic_header, emit_ext_func_declarations,
     emit_all_forward_declarations, emit_global_declarations,
     emit_function_bodies, emit_dyn_call_bodies,
     emit_dyn_fn_wrappers, emit_dyn_has_bodies,
     emit_collection_helpers, plus emit_test_main /
     emit_script_main / emit_lib_init at the end.

   What I want: one pass per phase. Not one pass for some sub-passes
   inside the typer (#146's scope). Every phase: lexer, parser,
   initer, typer, precomp, scavenger, builder, interpreter -- one
   walk over the AST (or token stream / source for lexer/parser).

   Related:
   - #146 -- partial step inside the typer only: merge the
     desugar/insert sub-passes
   - #144 -- per-pass clone-and-assign that this would also
     eliminate
   - #109 -- the leaks dominated by these repeated rebuilds

** OPEN #157 Rust-style ~///~ doc comments and ~help()~ builtin
   Replace the emacs-mode-only ~/** ... */~ doc-comment placeholder with
   real Rust-style ~///~ doc comments. The text rides through the
   lexer and parser, lands on ~Declaration.doc~, and is exposed to user
   code via a ~help(name : Str)~ builtin so ~help("Vec")~ works the
   same in the REPL, in ~til interpret~, and in compiled ~til run~
   binaries. Done in two phases.

*** Phase 1 -- doc comments + ~help()~ builtin (this commit)

   - ~editor/til-mode.el~: highlight ~///~ as ~font-lock-doc-face~;
     drop the legacy ~/** ... */~ regex.
   - ~src/self/lexer.til~: add ~TokenType.DocComment~. Recognise
     ~///~, capture the rest of the line (stripped of the leading
     ~///~ and one optional space), emit one DocComment token per
     line. Plain ~//~ and ~/* */~ continue to be skipped as before.
   - ~src/core/ast.til~: add ~mut doc := ""~ to ~Declaration~.
   - ~src/self/parser.til~: at the top of ~parse_statement~ drain a
     run of leading DocComment tokens, joining them with ~\n~. After
     the inner statement parses, if the result is a ~NodeType.Decl~
     attach the accumulated text to its ~Declaration.doc~. Doc
     comments that don't precede a Decl are discarded. Inside struct
     and enum bodies DocComment tokens are skipped silently in Phase
     1 (no attachment to namespace decls; see Phase 1.5 below).
   - ~src/core/doc.til~ (new): module-global
     ~mut __til_docs__ : Map<Str, Str>~; ~__doc_register__(name, text)~
     proc; ~help(name : Str)~ proc that prints the doc or
     "no documentation for: <name>".
   - ~src/core/core.til~: ~import("doc.til")~ after ~io.til~ so the
     map and ~println~ are both already in scope.
   - ~src/self/loader.til~: after parsing core + user, walk both
     ASTs, collect every top-level Decl with a non-empty ~doc~,
     synthesise ~__doc_register__("name", "text")~ FCalls and append
     them to ~lp.core_ast.children~ so they run before user
     top-level code.
   - Tests: a small ~src/test/doc/~ .til file with ~///~ on a
     handful of top-level decls, asserting ~help("name")~ prints the
     expected text.

*** Phase 1.5 -- namespace methods (later commit)

   Phase 1 only collects top-level Decls. Extend the parser drain
   into ~parse_struct_def~ / ~parse_enum_def~ namespace blocks so
   ~Declaration.doc~ is attached to ns_decls too, then teach the
   loader synthesis to register them under the qualified name
   (~Vec.pop~, ~Foo.bar~). Also covers struct fields and enum
   variants if cheap.

*** Phase 2 -- ~til doc <path>~ generator (later commit)

   Add a new top-level subcommand ~til doc <path>~ that runs the
   same loader pipeline as ~til run~ but, instead of building or
   interpreting, walks the merged AST and emits a doxygen-style
   cross-referenced documentation tree from the collected ~///~
   text. Reuses the Phase 1 ~Declaration.doc~ and the registry
   walker -- no new lexer/parser work expected.

   Default proposed output: one ~.org~ file per source file under
   ~doc/api/<source>.org~, because emacs already renders org
   natively and ~org-export~ then provides HTML as a secondary
   target. ~.html~-direct emission is an open question -- decide
   when implementing.

*** Out of scope (note for Phase 1)

   - ~help(Vec)~ / ~help(Vec.pop)~ taking the value rather than the
     name. Phase 1 is Str-only.
   - ~//!~ inner doc comments (Rust's module/crate docs).
   - Doc comments on struct fields and enum variants (covered in
     Phase 1.5).
   - Cross-linking between docs / type names in rendered output
     (Phase 2 concern).

** OPEN #155 Typer infers ~unknown~ for enum-typed struct field access
   Spun off from ~#113~. Accessing a struct field whose declared
   default is an enum constructor types as ~unknown~ instead of the
   field's enum type, so UFCS method calls on the field don't resolve.

   Repro in-tree:

   #+begin_src til
   Cell := struct {
       mut val := Value.None
   }
   ref c : Cell = ...
   if c.val.is(Value.Func) { ... }   // type error: no method 'is' for type 'unknown'
   #+end_src

   ~Value~ here is an enum, ~Cell.val~ has no explicit type annotation
   and is initialised to ~Value.None~. The typer should infer the
   field's til_type from the default-value expression
   (~TilType.Enum("Value")~) and propagate it through field access so
   ~c.val~ has type ~Value~. Today it leaves the field at
   ~TilType.Unknown~, which leaves the UFCS rewrite with no

** OPEN #147 AST/Context redesign: lessons from rstil
   rstil (the previous Rust implementation, at ~../rstil~) was slower and
   leakier than current til, but several of its architectural choices around
   the AST and Context were cleaner than what til carries today. This issue
   catalogs the ideas worth stealing.

   Root pattern: the AST should be the result of parsing, not a database for
   later phases. rstil keeps cross-phase facts in ~Context~; til grew AST
   fields whenever a pass needed to remember something. Three concrete
   anti-patterns follow from that:

   1. *AST-as-cache*: typer/initer write derived facts onto ~Expr~,
      ~Declaration~, ~FCallData~, ~StructDef~, ~EnumDef~ for later phases
      to read. The parse tree stops being immutable after parse.
   2. *Dangerous refs into the AST*: ~ref mut .. : Expr = NULL~ fields
      stitched onto AST nodes (and onto ~TypeBinding~) point into the
      parse tree from outside. When a pass rebuilds a body via
      clone-and-filter the refs go stale; surviving them depends on every
      pass agreeing not to drop the original node. Owned ~Expr~ fields or
      lookup-by-name in ~Context~ are safer.
   3. *Per-phase state without a home*: pre-rebase, til had no
      ~imports_*_done~ memos, no precomp side-table; whether an import got
      re-walked depended on incidental control flow elsewhere. Per-phase
      Sets in ~Context~ (rstil pattern) make it explicit.

** OPEN #146 Unify typer desugar/insert passes into a single walk
   ~infer_body~ in typer.til (lines 2489-2495) runs 7 separate passes in
   sequence over every body, each rebuilding ~body.children~ with a full
   deep clone of kept children:
   - ~desugar_set_literals~
   - ~desugar_map_literals~
   - ~desugar_variadic_calls~
   - ~desugar_kwargs_calls~
   - ~hoist_fcall_args~
   - ~insert_field_deletes~
   - ~insert_free_calls~

   Plus ~insert_exit_deletes~ runs from other call sites.

   Consequences:
   - every body pays 7x (or more) the cost of a single rebuild
   - each pass clones every kept child into its own ~new_ch~ (see ~#144~)
   - the clones cascade through nested bodies recursively
   - most of the remaining ASAN leaks (~#109~) trace back to these
     rebuilds

   Desired direction:
   - merge the desugar/insert passes into one walk per body
   - at each statement, apply all relevant transformations before moving on
   - would also interact with ~#144~: a single-pass rebuild lets
     ~take_children~ / move-based reconstruction work without fighting
     seven separate consumers of the same vector

   Related:
   - ~#144~ clone-and-assign in each pass -- one pass would reduce but
     not eliminate the cost
   - ~#109~ the leaks around ~body.children = new_ch~ are dominated by
     the number of times each body is rebuilt

** OPEN #143 Avoid unnecessary heap round-trips for Expr construction
   ~Expr.new~ itself constructs on the stack, but the generated C for
   callers that store the result does a heap round-trip: the return value
   is heap-allocated, copied to the stack, then freed (~Expr x = *_hp;
   free(_hp)~). Similarly, ~try_eval_call~ in precomp.til allocates
   folded constants on the heap via ~malloc~ + ~memcpy~ just to return
   them. Could be avoided if the codegen supported returning structs by
   value without the heap detour.

   Progress (args side):
   - Commit 4392524: typer's ~hoist_to_temp~ now takes ~is_own~; when
     a hoisted arg flows into an own parameter slot the synthesized
     temp's Declaration is marked ~own_type = Own~, so the builder's
     existing own-local path emits it directly on the heap and skips
     the stack-then-~_oa~-malloc-copy wrap at the call site.
   - Commit 8e8763a: ~insert_free_calls~ in typer.til now runs a
     ~promote_own_transferred_locals~ step after
     ~collect_scope_locals~: user-declared locals with
     ~own_transfer >= 0~ get their Decl's ~own_type~ flipped to Own,
     so they're emitted as heap pointers too.
   - This commit: early-exit deletes emitted by ~insert_exit_deletes~
     now use ~arg_is_own=false, call_free=false~ instead of
     ~true, true~. The ~live~ filter ensures these deletes only fire
     for Shallow (stack) locals that haven't been promoted, so
     ~call_free=false~ is the right semantic (fields cleaned, stack
     container left alone for the frame to unwind). With the own bit
     cleared, emit_as_ptr no longer wraps the arg in the ~_oa~ malloc.

   - Commit 1f42959: variadic desugar's ~_va_Array_N~ temp now carries
     ~own_type = Own~ on both the Declaration and the scope binding,
     so the builder emits the Array directly on the heap and the
     variadic call site stops wrapping it in ~_oa~.
   - This commit: hoisted temps for compound (Struct/Enum) types get
     promoted to ~own_type = Own~ unconditionally (not just when they
     flow into an own arg slot). ~LocalInfo~ gains an ~is_heap~ bool,
     populated in ~collect_scope_locals~ and updated in
     ~promote_own_transferred_locals~. The five ~make_delete_call~
     sites in ~insert_free_calls~ (early-exit / nested / post-stmt /
     scope-end / assign-replace) now pass ~arg_is_own = is_heap,
     call_free = is_heap~, so heap locals are properly freed at exit
     and stack locals stay borrow-style. Eliminates the ~Expr x =
     *_hp; free(_hp)~ return round-trip for hoisted compound temps
     (the main pattern called out in the issue body).

   - This commit: ~promote_own_transferred_locals~ also promotes any
     user local of ~Struct~ type (even when ~own_transfer < 0~), so
     ~mut x := Str_lit("...")~ etc. are emitted as ~Str *x = ...~
     directly and skip the ~Str x; { Str *_hp = ...; x = *_hp;
     free(_hp); }~ round-trip. ~Enum~-typed user locals are NOT
     promoted: doing so trips an interpreter-mode regression in
     ~Token.Num(42).to_str()~ (returns ~Num(0)~) where the FFI path
     for ~get_payload~ on a heap-promoted enum local mishandles the
     payload. Compiled-C run mode is fine. Investigation deferred.

   ~_oa = malloc~ wrap sites in boot/til.c: 2472 (pre-fix) -> 26.
   ~_hp = ~ return round-trip sites in boot/til.c: 7909 (pre-fix) -> 381.

   Still TODO: remaining ~_hp~ round-trips for non-Struct compound
   user locals (Enum, blocked on the interp regression above) and a
   handful of edge cases; final ~_oa~ wraps on module-global delete
   paths.

   1. ~try_eval_call~ bails when ~_fd.func_type.is(FuncType.ExtFunc)~.

      ext_funcs (I64_add etc.) are statically linked and *would* be
      dispatchable; this bail is a sledgehammer over a finer-grained
      "is this ext_func resolvable right now?" check.

   2. ~try_eval_call~ FA-callee path bails when the receiver Ident
      isn't a registered struct namespace and isn't a known constant
      (~known.has(_ri).not()~). Excludes value-receiver method calls
      (~p.diff()~) from fold attempts.

      ~clone~/~delete~/~concat~/~substr~/~format~ as "unknown",
      bailing the whole fold. Workaround for precomp's interpreter
      scope not having top-level constants like ~CAP_VIEW~ that
      ~Str.delete~ reads on scope exit. Same check on FieldAccess
      method names. This is what blocks ~make_poem~ from folding --
      its body is built around ~concat~.

   4. ~func_uses_unknown_globals~ flags Assign nodes whose LHS isn't
      a body-local. Workaround for funcs like

   5. ~try_eval_call~ bails when ~_fd.return_own_type.eq(OwnType.Ref)~.

** OPEN #130 AST / ~Expr~ representation cleanup
  The current AST shape is confusing and duplicates state in ways that make
  parser and ownership bugs harder to reason about.

  Current pressure points:
  - ~ExprData~ is still acting as a giant catch-all payload enum
  - node-kind-specific data is split between ~Expr~, ~ExprData~, and
    sometimes child lists
  - that makes ownership/lifetime of payload pointers much harder to see
  - parser code is forced into fragile payload-rebuild patterns around
    ~ExprData.FuncDef~, ~Decl~, and related nodes

  Intended direction:
  - move toward a clearer node model where node-kind payload data lives in
    the node-kind payload itself

** OPEN #127 Runtime ~Value~/binding ownership cleanup and ~Ptr~ split
  The interpreter runtime currently overloads one mechanism for several
  different ownership cases:
  - scoped values live in ~Cell.val~
  - ~Scope~ bindings own some ~Cell*~ values via ~cell_is_local~
  - one runtime variant, current ~Value.Ptr~, is used for both:
    - borrowed/ref-like storage views
    - owning opaque heap pointers

  That causes two classes of problems:
  - real owner cleanup is missing at some scope teardown / overwrite boundaries
  - dispatcher/runtime code grows manual ~free(...)~ workarounds because the
    runtime value model does not distinguish borrowed pointer views from
    owned dynamic pointers cleanly

  Partial fixes landed (9d8a011 and subsequent):
  - ~scope_free~ now frees Enum payload data for owned cells
  - return values are cloned before ~scope_free~ to prevent dangling
  - Struct data in scope_free is still NOT freed (aliasing risk)
  - ~scope_set_owned~ now calls ~free_value~ on old value before overwriting

  Remaining pressure points:
  - current ~Value.Ptr~ is produced by:
    - ~null~
    - ref field reads
    - own field reads exposing owned storage
    - ~ptr_add(...)~
    - FFI ~Dynamic~ returns
    - container data fields like ~Array.data~ / ~Vec.data~
  - current consumers then guess whether that pointer is borrowed or owned

  Cleanup direction:
  - give runtime pointer-like values distinct meanings
  - stop overloading one variant for both ref-like and owning semantics
  - make the real owner delete runtime values at scope teardown and on
    overwrite
  - then simplify dispatcher helpers like ~val_to_ptr~ so they are only
    borrowed pointer-view helpers instead of ownership escape hatches

  Current candidate shape:
  - split current ~Value.Ptr~ into:
    - ~Value.Ref~ for borrowed/nullable pointer views
    - ~Value.Dynamic~ for owning opaque pointers
  - keep ~val_to_ptr := func(v: Value) returns ref Dynamic~
  - repair runtime binding cleanup before relying on translated dispatcher
    helpers as proof of correctness

  Important constraints discovered during analysis:
  - simple removal of ~Cell~ is not obviously safe yet, because current
    borrowed bindings are used for:
    - ref declarations from identifiers
    - borrowed function argument binding paths
  - own field reads cannot simply be classified globally as borrowed or
    owning without checking the typed/lowered move/clone behavior

  This should be tackled as a focused runtime ownership cleanup, not mixed
  into unrelated dispatcher translations.

** OPEN #124 Raw `Dynamic` ownership should not require explicit `free(...)` in til
  Today, typed owned values and containers mostly get their cleanup inserted
  automatically by typer, but raw ~Dynamic~ buffers do not. That leaks the
  low-level memory model into ordinary til code.

  Current symptom:
  - code that takes ownership of raw memory from helpers like:
    - ~malloc(...)~
    - ~realloc(...)~
    - ~Vec_take(...)~
  - still needs explicit:
    - ~free(own buf)~

  That is inconsistent with the rest of til ownership:
  - owned ~Str~, ~Expr~, ~Vec~, ~Map~, etc. are cleaned up by generated
    delete/free insertion
  - but owned raw ~Dynamic~ values are effectively second-class and require
    handwritten cleanup in til source

  Desired end state:
  - if til code owns a ~Dynamic~, the ownership system should be able to
    release it automatically without handwritten ~free(...)~
  - or the language should grow a distinct raw-owned buffer type whose
    destructor is unambiguously ~free~
  - this should be the default behavior, not a ban on explicit manual
    cleanup in edge cases like:
    - custom/manual ~delete~ implementations
    - foreign ownership
    - custom allocators
    - raw buffers that intentionally should not go through plain ~free~

  This matters for self-hosting cleanup too:
  - current translated code still contains explicit ~free(own locals)~ style
    calls for buffers returned from helpers like ~Vec_take~
  - that should ideally become unnecessary once raw owned memory has a proper
    automatic cleanup story

  Likely directions:
  - teach typer/runtime that owned ~Dynamic~ defaults to ~free~
  - or introduce a dedicated raw-buffer type separate from generic ~Dynamic~
  - make the final rule explicit in ~qualifiers.org~ and ~c2til.org~
** OPEN #116 Imported files are parsed in isolation
  Imported files are currently lexed and parsed as standalone units, then
  merged later. They do not see declarations from the importing file during
  parse/bootstrap time.

  Current behavior in ~src/til.til~:
  - parse one file
  - extract ~import("...")~ calls from its AST
  - recursively read, lex, and parse each imported file separately
  - only after that merge:
    - core
    - core imports
    - mode imports
    - user imports
    - user file

  parsing imported files. Imported files are self-contained until the later
  AST merge.

    context

  - relying on importer-local predeclarations to shape imported AST types

  Likely directions:
  - keep such canonical payload types local to the imported file for now

  - or change import/bootstrap semantics so imported files can see a shared

  - treat the failed ~StructDef~ unification as a symptom of the import model,

** OPEN #91 Containers: store clone/delete as function pointers instead of dyn_call
  Vec/Map/Set currently dispatch clone/delete by type name string via
  ~dyn_call~, which does a linear scan of if/memcmp chains every time.
  Instead, store clone/delete function pointers as fields in the
  container struct. ~Vec.new("Str", Str.size())~ resolves the function
  pointers once at construction, then all subsequent clone/delete calls
  go through the stored pointers directly.

  Proven pattern: ~AlphaBeta~ struct in ~src/lib/alphabeta.til~ uses
  the same function-pointer-on-struct approach — ~h~ and ~nexts~ fields
  called via ~self.h(board)~ and ~self.nexts(board)~. All three game
  examples (checkers, chess, go) use it successfully.

  Design:

  New ext_func to resolve function pointers from type name + method:
  : dyn_get_fn_ptr := ext_func(type_name: Str, method: Str) returns Dynamic {}

  Vec struct gains function pointer fields:
  #+begin_src til
  Vec := struct {
      own mut data : U8 = 0
      mut count : USize = 0
      mut cap : USize = 0
      elem_size : USize = 0
      elem_type := ""
      elem_clone : func(Dynamic) returns Dynamic = default_clone
      elem_delete : proc(Dynamic, Bool) = default_delete
  }
  #+end_src

  Vec.new resolves pointers once at construction:
  #+begin_src til
  new := func(elem_type: Str, elem_size: USize) returns Vec {
      return Vec(
          data = malloc(elem_size),
          cap = 1,
          elem_size = elem_size,
          elem_type = elem_type,
          elem_clone = dyn_get_fn_ptr(elem_type, "clone"),
          elem_delete = dyn_get_fn_ptr(elem_type, "delete"),
      )
  }
  #+end_src

  Container methods use the fields directly:
  #+begin_src til
  // Before (dyn_call):
  dyn_call(self.elem_type, "delete", 2, ptr_add(...), false)
  own cloned : Dynamic = dyn_call_ret(self.elem_type, "clone", 1, ptr_add(...))

  // After (function pointer):
  self.elem_delete(ptr_add(...), false)
  own cloned : Dynamic = self.elem_clone(ptr_add(...))
  #+end_src

  Same pattern for Map (key_clone, key_delete, key_cmp, val_clone,
  val_delete), Set (elem_clone, elem_delete, elem_cmp), Array, Tuple.

  No workarounds for syntax issues — if function pointers on structs
  don't work for some case, fix the blocker first, then come back.

  Done:
  - ~dyn_fn~ ext_func: resolves function pointer by type+method name
  - ~dyn_fn~ renamed from ~dyn_get_fn_ptr~ for brevity
  - Builder generates ~dyn_fn~ dispatch (independent from dyn_call)
  - Builder generates wrapper functions (~Type_method_dyn~) for methods
    with shallow params/returns, fixing calling convention mismatch
  - ~CloneFn~, ~DeleteFn~, ~CmpFn~ type aliases + defaults in core.til
  - Builder indirect call codegen uses fn_sig for shallow detection
  - AlphaBeta refactoring (all 3 games) as proof of function pointer
    pattern on structs
  - Interpreter blocker FIXED: FuncPtr field calls for ext_funcs
  - FuncPtr fields added to Vec, Array, Set, Map in core.til
  - Constructors resolve fn ptrs via ~dyn_fn~ (with is_empty guards
    for empty elem_type/val_type)
  - ~dyn_call~ replaced with direct field calls in delete/clone/cmp
  - C-side handlers (~h_array~, ~h_vec~, ~build_argv_array~) populate
    FuncPtr fields via ~ns_get~
  - FuncPtr fields propagated in clone returns

  Tuple still uses ~dyn_call~ (per-element types, different pattern).

  Performance: no measurable improvement. The dyn_call string matching
  overhead was replaced by larger struct sizes (Vec +16 bytes, Map +40,
  Set +24, Array +16) which increase memory pressure and cache misses.
  The tradeoff is roughly neutral -- fewer comparisons but more bytes
  per container instance throughout the program.

  Decision needed: since no performance improvement was observed, decide
  whether to:
  - finish the job (extend to Tuple with per-element FuncPtr vectors)
  - or revert the whole thing and go back to dyn_call everywhere
  The architectural benefits (no "unknown type for delete" failures,
  simpler generated code) may justify keeping it even without a perf win,
  but the extra struct size is a real cost.

  Keep ~dyn_call~ as a general-purpose runtime dispatch mechanism
  (it's useful for other things), but containers no longer use it.

** OPEN #168 Typer: reject ~ref~ bindings initialized from owned values
   Background:

   The leaks closed for ~FCallData.fn_sig~ in ~src/self/typer.til~
   (under #109) all had the same shape:

   #+begin_src til
       ref mut fn_sig : Expr = NULL
       ...
       fcd.fn_sig = src.clone()
   #+end_src

   ~src.clone()~ returns an owned heap allocation. ~ref~ is a
   non-owning pointer slot. Storing the owned value into a ~ref~
   slot leaves the allocation with no owner: the auto-generated
   delete skips ~ref~ fields, the function's stack frame holds no
   handle, and the heap copy leaks. In every shape of this pattern
   -- decls, assignments, struct field defaults, ref-typed
   function returns -- the predicate is the same: a ~ref~ binding
   cannot legally hold an owned value. If the caller wanted
   ownership they would have written ~own~; if they wanted a
   borrow they would not have produced an owned value to begin
   with.

   Plan:

   Have the typer reject ~ref~ / ~ref mut~ bindings whose
   initializer is an owned value, growing the rule incrementally
   so each step lands, surfaces its own audit, and informs the
   next. The issue stays open until the user decides the rule is
   broad enough.

   Incremental steps (each is its own commit):

   1. ~ref X := Y.clone()~ in a local decl. Most mechanical case:
      look at the ~Decl~ initializer, if it is an ~NodeType.FCall~
      whose callee child is ~NodeType.FieldAccess("clone")~,
      emit a ~type_error~. No existing source is expected to use
      this shape after #109's bucket of ~fn_sig~ clones; if any
      remain they are bugs in the same class.

   2. Generalize the predicate from "literal ~.clone()~" to "any
      FCall whose callee returns ~OwnType.Own~". Constructors
      (~Foo()~), ~take_*~ helpers, factory funcs, etc. The typer
      already tracks ~return_own_type~; the new piece is per-
      callsite resolution through ~Ident~ / ~FieldAccess~ /
      namespace-method shapes (similar logic to the
      ~callee_returns_shallow~-style helpers in builder.til,
      hoistable into a typer helper).

   3. Apply the rule to struct field defaults:
      ~ref mut x : T = Y.clone()~ as a struct field declaration.
      Same predicate, but the check site is the parser/initer
      pass that records ~Declaration.default_value~, not the
      typer's local-decl path. Same audit afterwards.

   4. Apply the rule to assignments: ~ref_lvalue = X.clone()~
      (and the generalized owned RHS). Harder than decls because
      the typer must resolve the lvalue's declared ~own_type~
      from scope rather than reading it off the ~Decl~ node.
      This is the shape that originally bit ~FCallData.fn_sig~,
      so it is the highest-value step -- worth keeping as a
      named step rather than folding into one of the earlier
      ones.

   5. Apply the rule to ref-typed function returns:
      ~func f() returns ref T { return Y.clone() }~. The
      returned value has nowhere to live in the caller. Check
      site is the ~Return~-statement inference path, gated on
      the enclosing func's declared ~return_own_type == Ref~.

   6. Codebase audit pass after each step lands: run the typer
      over every ~src/**/*.til~ file, fix any new violation the
      step surfaced (or open a follow-up issue if a violation
      reveals a deeper structural fix). Some violations will be
      genuine bugs in the same class as #109; others may be
      false positives that the predicate needs to refine away.

   Notes:

   - Suggested error message: "ref binding cannot be initialized
     from an owned value (~.clone()~ / constructor / function
     returning ~own~) -- use ~own~ to keep the value, or borrow
     the source directly". Each step refines what counts as
     "owned value".

   - Each step's acceptance: the typer rejects every shape it
     covers, existing tests still pass, and a new typer error
     test pins the diagnostic for the new shape.

   - More steps may be added as the audit surfaces patterns; the
     list above is a starting point, not a closed catalog.

** OPEN #152 True cross-compilation to macOS (and other host-restricted targets) via precompiled ext.o
  :DEPENDENCIES: #150 (unrelated, just handy to mention)
  :RELATED: rstil Issue #135

  Cross-compiling til programs to macOS from Linux or Windows currently
  is not feasible the way linux-arm64 / linux-riscv64 / windows-x64 are:
  Apple's SDK headers and frameworks aren't redistributable, and
  mingw-style freely available toolchains don't exist for Mach-O. Our
  current cross-compile pipeline assumes each host has a working

     a small fixed set of ~til_*~ runtime symbols. The #ifdef _WIN32
     blocks added for windows-x64 are a decent starting audit of which
     functions need this treatment.

     the resulting ~lib/ext-{target}.o~ (e.g. ~ext-macos-x64.o~,
     ~ext-macos-arm64.o~) alongside the sources. For macOS that means a
     ~runs-on: macos-latest~ / ~macos-15-intel~ job whose only output
     is the committed ext object.
  3. Teach ~src/self/builder.til~ to link against the precompiled
     ~ext-{target}.o~ when cross-compiling instead of compiling ext.c
     in-tree. Non-macOS targets (linux-*, windows-x64) keep the current
     compile-ext.c-together flow; macOS uses the precompiled object.
  4. CI adds a ~macos-x64~ / ~macos-arm64~ cross-compile job that
     downloads the committed ext-macos-*.o artifacts and links.

  ~claude/fix-issue-135-SMlEh~ in rstil) that removed ~static inline~

  ccodegen. That approach is too brittle; the duplication in both

  edits in three places. Whatever til does here needs a lighter coupling

  ~ext_forward.h~ emitted once and consumed by both the generated C and
  the precompiled ext object.

  Open questions (same as rstil's):
  - How are platform-specific ext.o files versioned / regenerated when
    ext.c changes? Do we fail the main CI if ext-macos-*.o is stale?
  - Ship ~zig cc~ as an alternative? Single toolchain that can target
    everything, at the cost of a ~300MB dependency.
  - Do we keep the current compile-ext.c-together path forever for the

** OPEN #115 Remove or simplify ~ext_struct~
  Revisit whether ~ext_struct~ should exist as a separate concept at all.

  Today many uses are really just opaque or fieldless types with methods, for
  example the primitive types split out of ~core.til~. Conceptually, these are
  still just types. The main special behavior ~ext_struct~ is standing in for
  is:

  - "the caller should not assume fields/layout from source syntax"
  - "the type may still define methods in its namespace"
  - "size/clone/delete may need custom behavior"

  That may not justify a dedicated type form.

  Possible direction:

  - use ordinary ~struct {}~ (or fieldless struct declarations) for opaque
    external types too
  - if the type is opaque, do not rely on autogenerated size/clone/delete
  - instead, require explicit methods where the default would be wrong

  In particular, if a type wants to be opaque but still copyable/deletable
  correctly, it should be able to provide:

  - ~size() returns USize~
  - ~clone(...)~
  - ~delete(...)~

  rather than needing a distinct ~ext_struct~ concept.

  Questions to resolve:

  - should opaque external types just be ordinary structs with no fields?
  - should initer only autogenerate ~size/clone/delete~ when the layout is
    fully known from source fields?
  - is there any remaining semantic difference that truly requires
    ~ext_struct~?

  Recent evidence:

  - a dormant ~src/core/i8.til~ plus matching ~ext.c/ext.h~ support was easy
    to add
  - but importing ~I8~ immediately exposed that the current compiler still
    has repeated hardcoded primitive-name lists (~U8~, ~I16~, ~I32~, ~U32~,
    ~U64~, ~I64~, ~F32~, ~Bool~) across builder, initer, typer, and
    ~TilType~
  - those lists decide things like:
    - whether a type is treated as a builtin scalar
    - whether generated headers skip forward declarations for it
    - whether C emission uses the alias-backed primitive representation or
      falls back to ordinary struct handling
  - so the current ~I8~ blocker is not the lack of another ~ext_struct~
    mechanism, but the broader primitive special-casing model

  That pushes this issue in a clearer direction:

  - simplify/remove ~ext_struct~
  - reduce hardcoded primitive special cases to the minimum truly required by
    runtime representation
  - make ordinary type declarations plus explicit methods carry more of the
    behavior today encoded by compiler categories

  This issue is partly about simplifying the language surface and partly about
  reducing one more C-leaning special case in the core model.

** OPEN #58 First-class functions: Type hierarchy and FuncSig/FuncDef
  :DEPENDENCIES: #14
  Based on rstil/doc/first_class.org. Everything in til is a value —
  functions, structs, enums can all be assigned, passed, returned, and
  stored. This issue covers the metatype system that makes it work.

*** Type hierarchy

   ~Type~ is the metatype — the type of all type definitions:

   #+begin_src til
   Type := enum { StructDef, EnumDef, FuncSig, FuncDef }
   #+end_src

   | Metatype    | Defines                      | Instance example                     |
   |-------------+------------------------------+--------------------------------------|
   | ~StructDef~ | Data layout (fields)         | ~p := Point(x=10)~ is a Point        |
   | ~EnumDef~   | Tagged union (variants)      | ~c := Color.Red~ is a Color          |
   | ~FuncSig~   | Calling convention (types)   | ~BinaryOp := func(I64, I64) ret I64~ |
   | ~FuncDef~   | Concrete function (sig+body) | ~add : BinaryOp = (a, b) { ... }~   |

   ~FuncSig~ is a pure type — parameter types, return type, func/proc
   kind. Parameter *names* are NOT part of the signature, only types.

   ~FuncDef~ is a concrete function definition — it has a ~FuncSig~ (its
   type) plus argument names, modifiers, and a body. A FuncDef is to its
   FuncSig what a struct instance is to its StructDef: the signature is
   the type, the definition is the value.

   Function types are *structural*: two signatures with the same parameter
   types, return type, and throws are the same type regardless of name.

*** Explicit type annotations (~: Type =~ syntax)

   Separate the type annotation (~:~) from the value assignment (~=~).
   ~:=~ remains as shorthand for "infer the type":

   #+begin_src til
   // Explicit type + value
   x : I64 = 5
   name : Str = "hello"

   // Type definitions: explicit metatype annotation
   Point : StructDef = struct { mut x: I64 = 0, mut y: I64 = 0 }
   Color : EnumDef = enum { Red, Green, Blue }
   BinaryOp : FuncSig = func(I64, I64) returns I64

   // Inferred (unchanged from today)
   Point := struct { mut x: I64 = 0, mut y: I64 = 0 }   // infers StructDef
   Color := enum { Red, Green, Blue }                      // infers EnumDef
   BinaryOp := func(I64, I64) returns I64                  // infers FuncSig
   #+end_src

*** Type inference rules

   | Declaration                                                | Inferred type |
   |------------------------------------------------------------+---------------|
   | ~x := 5~                                                   | ~I64~         |
   | ~name := "hello"~                                          | ~Str~         |
   | ~Point := struct { ... }~                                  | ~StructDef~   |
   | ~Color := enum { ... }~                                    | ~EnumDef~     |
   | ~BinaryOp := func(I64, I64) returns I64~                   | ~FuncSig~     |
   | ~add := func(a: I64, b: I64) returns I64 { return ... }~   | ~FuncDef~     |

   The distinction mirrors struct/enum:
   - ~Point := struct { ... }~ defines a *type* (infers StructDef)
   - ~p := Point(x=10)~ creates an *instance* (infers Point)
   - ~BinaryOp := func(I64, I64) returns I64~ defines a *type* (infers FuncSig)
   - ~add := func(a: I64, b: I64) returns I64 { ... }~ creates an *instance* (infers FuncDef)

   A bodyless func/proc is a FuncSig (type definition). A bodied func/proc
   is a FuncDef (value). The presence of a body (and parameter names)
   distinguishes them.

*** Parameters always require explicit types

   Lambda and named function parameters always require explicit types.
   ~Dynamic~ and ~Type~ are special parameter types and inferring argument
   types would create ambiguity. Explicit parameter types also make function
   signatures self-documenting.

   #+begin_src til
   // REQUIRED: explicit types in function parameters
   add := func(a: I64, b: I64) returns I64 { return a.add(b) }

   // NOT ALLOWED: inferred parameter types
   add := func(a, b) returns I64 { ... }  // error
   #+end_src

*** FuncSig form vs FuncDef form

   Two equivalent ways to define a function, mirroring the two metatypes.
   Both produce the same FuncDef value:

   #+begin_src til
   // 1. FuncDef form: FuncDef literal (type inferred as FuncDef)
   add := func(a: I64, b: I64) returns I64 { return a.add(b) }

   // 2. FuncSig form: explicit signature as type, names + body as value
   add : func(I64, I64) returns I64 = (a, b) { return a.add(b) }

   // 3. FuncSig form with named alias
   add : BinaryOp = (a, b) { return a.add(b) }

   // 4. Explicit FuncDef metatype
   add : FuncDef = func(a: I64, b: I64) returns I64 { return a.add(b) }
   #+end_src

   Forms 2 and 3 are the same (with or without a type alias). Form 1 is the
   FuncDef literal — the parser extracts argument names and body to produce
   the same internal representation. Form 4 is the explicit-metatype version
   of form 1.

*** Collections of types

   #+begin_src til
   Vec.new(BinaryOp)   // Vec of function pointers with the same signature
   Vec.new(FuncSig)    // Vec of function signatures (different signatures)
   Vec.new(FuncDef)    // Vec of function definitions (different definitions)
   Vec.new(StructDef)  // Vec of struct definitions
   Vec.new(EnumDef)    // Vec of enum definitions
   Vec.new(Type)       // Vec of any type definition
   #+end_src

   Storing function pointers in collections currently fails:
   ~ops.push(own fn_add)~ where ~ops := Vec.new("BinaryOp", ...)~ panics
   with ~dyn_fn: unknown BinaryOp.clone~. Vec.push assumes values are
   heap-allocated boxes and calls elem_clone/elem_delete, but a function
   pointer is a plain value (not a struct). Needs Vec to support
   FuncSig-typed elements via a dedicated clone/delete path. Moved here
   from #61 (where it appeared as "storing function pointers in
   collections") since fixing it requires type-hierarchy work, not
   lambda-lifting changes.

*** Introspection

   First-class types support reflection via the ~func_sig_of~,
   ~struct_def_of~, ~enum_def_of~ helpers (see rstil/std.meta):

   #+begin_src til
   sig := func_sig_of("BinaryOp")
   sig.args.len()             // 2
   sig.return_types.len()     // 1
   sig.throw_types.len()      // 0
   #+end_src

*** Type parameters

   Two-phase plan:

   *Phase now*: ~func(T: Type)~ — no new syntax needed. ~Type~ is already
   a special parameter type. Macros handle the template use case by
   executing arbitrary code at compile time (like Jai's comptime, Zig's
   comptime):

   #+begin_src til
   // Type as a regular parameter
   swap := func(T: Type, mut a: T, mut b: T) { ... }
   swap(I64, x, y)

   Pair := macro(A: Type, B: Type) returns Type {
       return struct { mut first: A, mut second: B }
   }
   IntStr := Pair(I64, Str)
   #+end_src

   *Phase later*: ~func<T>()~ as sugar that desugars to macro calls.
   Use ~<>~ for familiarity (Rust, C++, Java):

   #+begin_src til
   // Sugar (later):
   swap<T> := func(mut a: T, mut b: T) { ... }
   swap(x, y)       // T inferred
   swap<I64>(x, y)  // T explicit

   // Desugars to:
   swap := func(T: Type, mut a: T, mut b: T) { ... }
   swap(I64, x, y)
   #+end_src

*** Unified pattern summary

   | Construct  | Type declaration               | Definition (type = value)              |
   |------------+--------------------------------+----------------------------------------|
   | Variable   | ~x : I64~                      | ~x : I64 = 5~ or ~x := 5~             |
   | Function   | ~F : func(I64) returns I64~    | ~f : func(I64) returns I64 = { ... }~  |
   | Body       | (no separate type decl)        | ~b : Body = { ... }~ or ~b := { ... }~ |
   | Struct     | ~S : struct { ... }~           | (struct decl IS the definition)        |
   | Enum       | ~E : enum { ... }~             | (enum decl IS the definition)          |

** OPEN #55 Lambda functions and first-class bodies
  :DEPENDENCIES: #14 #54
  Based on rstil/doc/first_class.org. Anonymous inline functions (lambdas)
  and first-class Body values (blocks as values).

*** Lambda functions (anonymous inline functions)

   Lambda functions are anonymous FuncDef literals passed directly as
   arguments. Same syntax as named definitions, just without a name:

   #+begin_src til
   Predicate := func(I64) returns Bool
   Reducer := func(I64, I64) returns I64

   // filter with inline lambda
   evens := numbers.filter(func(a: I64) returns Bool { return a.mod(2).eq(0) })
   // [2, 4]

   // reduce with inline lambda
   sum := numbers.reduce(0, func(acc: I64, x: I64) returns I64 { return acc.add(x) })
   // 15

   // FuncSig form lambda (type separated from param names)
   doubled := numbers.map(func(I64) returns I64 = (a) { return a.mul(2) })
   #+end_src

*** First-class bodies (blocks as values)

   A Body is a sequence of statements — the simplest unit of executable
   code. Simpler than a function: no signature, no parameters, no return
   type. A Body is essentially a zero-argument, void closure.

   #+begin_src til
   // Store in a variable
   cleanup := { file.close(); println("done") }

   // Pass as argument
   run_twice := proc(b: Body) {
       b()
       b()
   }
   run_twice({ println("hello") })

   // Store in a struct field
   Task := struct {
       mut on_complete: Body
   }
   t := Task(on_complete = { println("finished") })
   t.on_complete()

   // Store in a collection
   steps := Vec.new(Body)
   steps.push(own { println("step 1") })
   steps.push(own { println("step 2") })
   for step : Body in steps {
       step()
   }
   #+end_src

*** Relationship to functions and closures

   | Construct         | Has params | Has return type | Has captures | Has body |
   |-------------------+------------+-----------------+--------------+----------|
   | Body              | No         | No              | No*          | Yes      |
   | func (no closure) | Yes        | Yes             | No           | Yes      |
   | func (closure)    | Yes        | Yes             | Yes (#54)    | Yes      |

   *Without captures (#54), a Body can only access globals. With captures,
   a Body becomes a zero-argument closure:

   #+begin_src til
   mut total := 0
   data := vec("I64", 1, 2, 3)

   // Body with capture list
   summer : Body = [mut total, data] { for item : I64 in data { total = total.add(item) } }
   summer()
   // total is now 6
   #+end_src

   A Body is to ~proc()~ what a literal is to a named constant — the
   anonymous, inline version. The distinction is that a Body has no
   function overhead (no frame push/pop in the interpreter, no C function
   call in codegen) — it executes in the caller's scope.

*** Use cases

   #+begin_src til
   // Deferred execution (like Go's defer)
   cleanup := { file.close() }
   // ... do work ...
   cleanup()

   // Callbacks and hooks
   Widget := struct {
       mut on_click: Body
       mut on_hover: Body
   }

   // Builder patterns
   with_transaction := proc(db: Database, action: Body) {
       db.begin()
       action()
       db.commit()
   }
   with_transaction(db, { db.insert(record1); db.insert(record2) })

   // Test setup/teardown
   run_test := proc(name: Str, setup: Body, test_body: Body, teardown: Body) {
       println("Running:", name)
       setup()
       test_body()
       teardown()
   }
   #+end_src

*** Scope semantics: inline vs isolated

   Open question: when a Body executes, does it share the caller's scope
   (like an inlined block) or get its own scope (like a function call)?

   - *Option A: Inline* — shared scope (how bare ~{ }~ blocks work today).
     Simple, powerful, zero overhead. But passing a Body as argument is
     weird — which scope does it execute in?
   - *Option B: Isolated* — own scope, captures only (#54). Safe and
     predictable for stored/passed bodies. But needs captures to be useful.
   - *Option C: Both* — bare ~{ }~ blocks are inline (current behavior),
     Body values (stored, passed, returned) are isolated and require
     captures. This is probably the right answer.

*** Comparison

   | Language   | Block-as-value          | Notes                              |
   |------------+-------------------------+------------------------------------|
   | Ruby       | ~{ ... }~ / ~do...end~  | Blocks and Procs are first-class   |
   | Smalltalk  | ~[ ... ]~               | Blocks are objects                 |
   | Swift      | ~{ () -> Void in ... }~ | Closures (always have captures)    |
   | Kotlin     | ~{ ... }~ (lambda)      | Trailing lambda syntax             |
   | Rust       | closure only            | No bare blocks as values           |
   | Go         | ~func() { ... }()~      | Must wrap in function              |
   | TIL        | ~Body = { ... }~        | Simpler than closure               |

   Ruby is the closest precedent. Ruby blocks can be passed to methods
   implicitly (~yield~) or explicitly (~&block~), and ~Proc.new~ wraps a
   block into a first-class value.

** OPEN #54 Explicit capture syntax (closures)
  :DEPENDENCIES: #61
  :DEPENDENCIES_OF: #12 #13
  Add syntax for listing captured variables from outer scopes, both for
  function definitions and anonymous blocks. Based on rstil/doc/first_class.org.
  Split from #17 (global variable access, now fixed).

  Closures = lambdas (#61) + captures. #61 provides anonymous inline
  functions (lambdas); this issue adds the bracket capture list that turns
  a lambda into a closure by binding outer-scope variables.

  til requires *explicit capture lists* — if a function references an outer
  variable, it MUST be listed in the capture list. This makes dependencies
  visible and refactoring mechanical. Inspired by Jai (Jonathan Blow).

*** Capture syntax

   The capture list ~[name1, name2, ...]~ appears after ~returns Type~
   (or after the parameter list when there is no return type) and before
   the body in FuncDef form, or after the argument names in FuncSig form:

   #+begin_src til
   scale := 10

   // FuncDef form: captures after returns, before body
   f := func(a: I64) returns I64 [scale] { return a.mul(scale) }

   // FuncSig form: captures after argument names
   f : func(I64) returns I64 = (a) [scale] { return a.mul(scale) }

   // With a named type alias
   Transformer := func(I64) returns I64
   f : Transformer = (a) [scale] { return a.mul(scale) }
   #+end_src

*** Capture modifiers

   Captures follow the same rules as function parameters — no new rules
   to learn:

   | Capture syntax | Semantics                     | Same as param mode |
   |----------------+-------------------------------+--------------------|
   | ~[x]~          | Const by reference (default)  | ~x: T~             |
   | ~[mut x]~      | Mutable reference             | ~mut x: T~         |
   | ~[copy x]~     | Deep copy (snapshot)          | ~copy x: T~        |
   | ~[own x]~      | Ownership transfer (moved in) | ~own x: T~         |

   #+begin_src til
   mut total := 0
   data := vec("I64", 1, 2, 3)

   // mut capture: closure can modify total
   data.for_each(func(x: I64) [mut total] { total = total.add(x) })

   // copy capture: closure gets a snapshot, original unchanged
   data.map(func(x: I64) returns I64 [copy total] { return x.add(total) })

   // own capture: moves data into the closure, caller can't use it after
   consume := proc() [own data] { data.clear() }
   #+end_src

*** Block captures (scope restriction)

   Even without making a function, a capture list on a bare code block
   limits what it can see — an "insurance policy" that the block only
   touches specified state:

   #+begin_src til
   mut total := 0
   mut count := 0
   data := vec("I64", 1, 2, 3)

   // Only total and data are accessible inside this block
   [total, data] {
       for item : I64 in data {
           total = total.add(item)
       }
   }
   // count is guaranteed untouched
   #+end_src

*** Jai-inspired refactoring progression

   Each step is mechanical. The capture list tells you exactly what needs
   to become a parameter when you promote to a pure function:

   #+begin_src til
   // Step 1: inline code
   total = total.add(x)

   // Step 2: block with captures (limit scope, verify dependencies)
   [total, x] {
       total = total.add(x)
   }

   // Step 3: promote to lambda (add params, same captures)
   adder : func(a: I64) returns I64 = [total] { return total.add(a) }

   // Step 4: promote to pure function (captures become params, no closure)
   adder := func(a: I64, b: I64) returns I64 { return a.add(b) }
   #+end_src

*** Why explicit captures

   Implicit captures (Python, JavaScript, Rust) feel magical:
   - *Reading*: you see ~scale~ and have to hunt for where it's defined.
     Is it a global? A local three scopes up? A typo for a parameter?
   - *Refactoring*: extracting a block into a function means figuring out
     ALL the implicit dependencies. Miss one and it breaks.
   - *Inlining*: moving a closure back inline means understanding what it
     captures to know where it can safely go.

   Explicit captures solve all three:
   - *Reading*: ~[scale]~ tells you exactly what comes from outside.
   - *Refactoring*: captures become parameters. Mechanical transformation.
   - *Inlining*: captures tell you exactly what the block depends on.

*** Implementation

   - Parser: recognize ~[name, ...]~ capture list in func defs (after
     returns Type, before body) and bare blocks (before ~{~)
   - AST: store capture list on NODE_FUNC_DEF and NODE_BODY
   - Typer: validate captures exist in outer scope, enforce modifier rules
   - Interpreter: bind captured variables in call_scope from definition scope
   - Codegen: emit closure structs or pass captures as hidden parameters

** OPEN #25 Support for the 4 main operative systems
  :DEPENDENCIES: #12
  til currently only builds and runs on GNU/Linux. Should support the 4
  main operative systems, in priority order:
  1. GNU/Linux/BSD (current, done)
  2. TempleOS
  3. macOS
  4. Windows

  This is not just "til builds natively on each OS." The goal is full
  cross-compilation: build a til binary for any target from any host.
  The matrix is 3×4 + 1×1 = 13 combinations: Linux, macOS, and Windows
  can each target all 4 platforms, but TempleOS as a host only targets
  itself (cross-compiling from TempleOS would mean replacing gcc/clang,
  which is out of scope).

  ctil (the C bootstrap compiler) stays Linux-only. Once til is
  self-hosting (#12), til replaces ctil and handles all platforms.
  Bootstrap path: ctil on Linux compiles self-hosted til → til can
  then cross-compile itself and any .til program to any target.

  The C codegen path helps: emit C + invoke the target's C compiler
  works for most POSIX targets. But TempleOS breaks this pattern
  (no C compiler), requiring a dedicated HolyC codegen backend.

  The builder must select the appropriate codegen backend per target.

  Common areas affected across platforms:
  - Build system (builder that selects backend per target)
  - FFI / dlsym (platform-specific dynamic loading)
  - System primitives (spawn_cmd, readfile, etc. — POSIX vs platform APIs)
  - GUI mode (raylib on POSIX/Windows, native primitives on TempleOS)
  - Audio (miniaudio on POSIX/Windows, native on TempleOS)

*** TempleOS
   :DEPENDENCIES: #12
   The most interesting and challenging target. TempleOS has no C compiler,
   no POSIX, no libc — just HolyC and its own kernel/graphics/audio APIs.

   Requires a new holybuilder backend that emits HolyC instead of C.
   HolyC is C-like enough that the translation should be mostly mechanical
   (similar types, similar control flow), but differences in calling
   conventions, memory management, and string handling will need attention.

   For mode gui: raylib doesn't exist on TempleOS, so we need to recreate
   raylib's interface using TempleOS graphics primitives (GrBlot, GrLine,
   GrRect, etc.) and audio primitives. This is likely easier than most
   people would expect — TempleOS has direct framebuffer access and simple
   audio APIs, and raylib's 2D API surface is not that large. The hard
   part is the impedance mismatch (TempleOS is cooperative multitasking,
   640x480 16-color default, etc.), not the drawing primitives themselves.

   Running the same mode gui program (e.g. music.til) on both GNU/Linux
   via raylib AND TempleOS via native HolyC primitives would be a serious
   language flex — proving til programs are truly platform-independent.

   For mode server: TempleOS docker containers could be extremely small
   (no libc, no userland, minimal kernel), making til a compelling choice
   for lightweight server deployments on TempleOS-based containers.

   Self-hosting (#12) is a hard dependency: TempleOS has no C compiler,
   so til must be able to compile itself to bring its own compiler onto
   the platform. There's no way to cross-compile C for TempleOS from Linux.

   Can test from Linux using QEMU with a TempleOS VM image. Copy til
   files in via a virtual disk, run them, capture output. This is the
   standard way to develop for TempleOS from a modern host.

*** macOS
   Closest to the current GNU/Linux target. All C code (ext.c,
   dispatch.c) is already POSIX-portable — fork, execl, waitpid, dlopen,
   dlsym all work the same on macOS. No cross-compilation needed: ship
   source, user runs `make` with clang natively.

   Makefile changes needed:
   - `-ldl` not needed on macOS (dlopen is in libc)
   - `-lffi` needs `brew install libffi` + Homebrew `-I`/`-L` paths
   - `-rdynamic` works with clang
   - Platform detection to conditionally apply the above

   Runtime differences:
   - `.dylib` instead of `.so` for dynamic libraries (mode gui, FFI)
   - Homebrew paths for raylib headers and libs

   No `#ifdef __APPLE__` needed in the C source itself — only in the
   Makefile and possibly in the codegen output (linker flags when
   compiling .til programs that use mode gui).

*** Windows
   Furthest from POSIX, but possibly dying — don't wait for self-hosting.
   Options: MSVC, MinGW, or cross-compile with zig cc. Dynamic loading
   via LoadLibrary/GetProcAddress instead of dlopen/dlsym. Process
   spawning via CreateProcess instead of fork/exec. raylib supports
   Windows natively, so mode gui should work once the build system
   handles .dll and Windows linking.

   Can test from Linux using Wine as a first step before touching a
   real Windows machine.
