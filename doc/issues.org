** OPEN #87 ~priv~ keyword for private declarations
  :DEPENDENCIES: #86
  Everything in til is public by default. Add ~priv~ keyword to
  mark declarations as module-private (not visible to importers).

  #+begin_src til
  priv helper := func(x: I64) returns I64 { x.mul(2) }
  do_thing := func(x: I64) returns I64 { helper(x).add(1) }
  #+end_src

  Importing this module gives access to ~do_thing~ but not ~helper~.

  Depends on #86 (unused declarations): private functions that are
  only called internally would trigger unused errors without proper
  visibility tracking. The typer needs to know that ~priv~ functions
  are "used" if called within their own module.

  Like Rust's ~pub~ but inverted — til defaults to public, you opt
  into private. Simpler for small programs (everything just works),
  explicit when you need encapsulation.

** OPEN #86 Typer error for unused declarations
  Unused variables, function parameters, and imports should be
  type errors — not warnings. til has no warnings, only errors.

  Escape hatch: prefix with ~_~ to suppress. This lets WIP code
  compile without noise while keeping the codebase clean.

  #+begin_src til
  x := 42          // error: unused variable 'x'
  _x := 42         // ok: _ prefix suppresses

  proc foo(a: I64, b: I64) {   // error: unused parameter 'b'
      println(a.to_str())
  }
  proc foo(a: I64, _b: I64) {  // ok
      println(a.to_str())
  }
  #+end_src

  Exempt from this rule:
  - Struct fields (part of a layout, removing one changes the struct)
  - Enum variants (part of a type, removing one changes the enum)
  - Namespace functions (may be called externally)

  Implementation: typer tracks which bindings are read. After
  type-checking a scope, any unread non-~_~ binding is an error.

** OPEN #85 C code must use enum structs, not raw tags
  :DEPENDENCIES_OF: #39
  Hand-written C code uses raw enum tags (~TokenType_tag~,
  ~ExprData_tag~, etc.) directly — passing them as function args,
  comparing with ~==~. But til enums are tagged unions: a struct
  with a ~.tag~ field (and optionally a ~union .data~ for payload
  variants). The builder already generates this layout (see
  bootstrap/ast.h), but C code bypasses the struct and uses the
  raw int tag.

  This is a cross-cutting blocker for #39: every C module that
  touches enums needs to use the struct form to match what til
  generates. Not just parser — typer, interpreter, builder,
  initer, scavenger all use raw tags.

  Example (parser.c):
  #+begin_src c
  // Current: raw tag
  I32 check(Parser *p, TokenType_tag type) {
      return peek(p)->type.tag == type;
  }
  check(p, TokenType_TAG_Ident);

  // Target: struct (matches til-generated code)
  I32 check(Parser *p, TokenType *type) {
      return peek(p)->type.tag == type->tag;
  }
  check(p, &(TokenType){TokenType_TAG_Ident});
  #+end_src

  Scope: ~100 sites in parser.c alone (check/expect_token and all
  callers). Similar patterns in typer.c, interpreter.c, etc.
  Each module can be converted independently as part of its #39 step.

  The struct wrapper is always the same shape:
  - No-payload enum: ~struct { Tag tag; }~ (e.g. TokenType, TilType)
  - With-payload enum: ~struct { Tag tag; union { ... } data; }~
    (e.g. ExprData)

** OPEN #80 USize type alias for container sizes
  :DEPENDENCIES: #79
  ~USize~ — semantic type for container sizes: Vec count/cap, Str
  count/cap, Array cap, Map/Set internals. Currently all use U64.

  Default ~USize := U32~ — 4 billion elements is enough for almost
  everything, saves memory in every container. User-configurable via
  compiler flag for programs that need larger containers.

  Distinct from UPtr (#81) which is platform-determined pointer size.
  USize is about "how many elements" (user's choice), UPtr is about
  "how big is an address" (platform's choice).

  Phase 1 (pre, Linux only):
  1. Create ~src/core/aliases.til~ with ~USize := U64~, imported by core.til
  2. Replace U64 → USize in core.til containers (Str, Vec, Array, Map, Set)
  3. Replace U64 → USize in C modules simultaneously (single commit,
     same bootstrap philosophy — .til + C + bootstrap all at once)
  4. Then switch alias to ~USize := U32~ — everything gets smaller/faster.
     The alias approach avoids the bootstrap chicken-and-egg that blocked
     the ~wip_u32~ branch (tried U32 directly, precomp returned wrong types).

  Phase 2 (post, depends on #25): USize maps to platform native pointer
  size — U64 on x86_64/arm64, U32 on x86/arm32. Initially just
  Linux 64 vs 32. Cross-platform builds (rstil showed gcc handles most
  targets, clang for mac, wine for windows testing from linux).

  Reference: ~wip_u32~ branch has prior work on the U32 migration.

** OPEN #50 Shallow types: stack allocation for structs/enums without own fields
  Currently only primitive ext_structs (I64, U8, Bool, etc.) get the stack
  allocation optimization — declared as stack variables instead of heap-allocated
  pointers. This can be extended to any struct or enum that has no ~own~ fields
  or payloads, recursively.

*** What makes a type "shallow"

   A type is "shallow" (in the copy-safety sense) if:
   - It has no ~own~ fields (structs) or ~own~ payloads (enums)
   - All its inline (non-ref, non-own) fields/payloads are themselves shallow,
     recursively
   - ~ref~ fields are fine — they're borrowed pointers, shallow-copying them
     is safe since we don't manage their lifetime

   The check is mutually recursive: structs can have enum fields, enums can
   have struct or enum payloads, so both must be checked together.

   #+begin_src til
   // SHALLOW — all fields are inline primitives, no own
   Point := struct {
       mut x: I64 = 0
       mut y: I64 = 0
   }

   // SHALLOW — ref fields are fine (borrowed pointers, not owned)
   Slice := struct {
       ref data: U8 = null
       mut len: I64 = 0
   }

   // SHALLOW — simple enum (no payloads = just an integer tag)
   Color := enum { Red, Green, Blue }

   // SHALLOW — all payloads are themselves shallow types
   Shape := enum {
       Circle: I64,
       Rect: Point,
   }

   // DEEP — has an own field (Str contains own c_str: U8)
   Person := struct {
       name: Str = ""
       age: I64 = 0
   }

   // DEEP — Vec has own internal buffer
   Container := struct {
       items: Vec = Vec.new("I64")
   }

   // DEEP — payload contains a deep type (Str)
   Token := enum {
       Num: I64,
       Name: Str,
       Eof,
   }
   #+end_src

*** What changes for shallow types

   Shallow types can be:
   - Stack-allocated instead of heap-allocated (no malloc/free)
   - Shallow-copied safely (memcpy, no deep clone needed)

   #+begin_src til
   // Currently: Point is heap-allocated like everything else
   //   Point *p = (Point *)malloc(sizeof(Point));
   //   p->x = 10; p->y = 20;
   //   // ... use p ...
   //   free(p);

   // With this optimization: Point lives on the stack
   //   Point p = { .x = 10, .y = 20 };
   //   // ... use p ...
   //   // no free needed

   // Clone becomes memcpy instead of malloc+field-by-field copy
   p := Point(x = 10, y = 20)
   q := p  // memcpy(&q, &p, sizeof(Point)) — no heap allocation
   #+end_src

*** Delete is still called

   Important: shallow types still need ~delete~ called (unlike primitives
   whose delete is a no-op). Custom delete methods may do non-memory work
   (close file descriptors, connections, etc.). The guarantee to the user
   is that delete is called ASAP after the last use of an instance. What
   changes is: no ~free~ at the end (since it wasn't malloc'd), and no
   deep-copy clone (just memcpy).

   #+begin_src til
   // Even though FileHandle is shallow (only inline fields),
   // delete must still be called to close the fd
   FileHandle := struct {
       fd: I64 = 0

   namespace:
       delete := proc(own self: FileHandle) {
           close_fd(self.fd)
           // no free(self) — it was stack-allocated
       }
   }
   #+end_src

*** Distinction from ~shallow~ keyword

   This is a different notion from the ~shallow~ keyword on args/returns,
   which is about passing by value vs pointer based on the size of the type
   relative to the platform pointer size. The "shallow type" concept here
   is about copy semantics — whether a type can be safely shallow-copied
   without ownership issues. In the future, the ~shallow~ keyword could
   be computed automatically from this property plus size considerations.

** OPEN #46 Optimize Str with with_capacity in concat, join, etc.

*** Architecture: core.til as ctil's runtime library

   core.til (which imports str.til, vec.til, map.til, set.til, array.til)
   is the single source of truth for Str, Array, Vec, Map, Set and all
   their methods. Built into bootstrap via ~ctil translate~, these
   generated C files ARE the implementation — no parallel C versions.

   ext.c is the canonical source for irreducible primitives ONLY:
   scalar arithmetic (I64_add, U8_eq, etc.), system calls (readfile,
   writefile, spawn_cmd), pointer ops (ptr_add, is_null), CLI parsing.
   Everything else comes from til.

   The builder emits ~#include "ext.h"~ in generated headers AFTER
   forward declarations, so ext.h can reference any til-defined type
   (Str, Array, Vec, etc.) in its function signatures without needing
   manual forward declarations. As more types move from C to til,
   ext.h naturally shrinks with no coordination needed.

   Bootstrap: generated files are copied to bootstrap/ and checked
   into the repo. People who clone the repo get pre-generated files.
   Two-pass build (cc → ctil_core → cc) handles breaking changes.

*** Duplication and --allow-multiple-definition

   During the transition, ctil links BOTH bootstrap code (e.g.
   bootstrap/lexer.c with core.til types) AND hand-written C modules
   that define the same functions. ~--allow-multiple-definition~ lets
   the linker pick one (first in SRCS order wins).

   This duplication is purely transitional. Once #39 is complete,
   til.til IS the compiler — it imports everything as til files, gets
   compiled into one bootstrap .c file with all definitions. No
   duplication, no ~--allow-multiple-definition~ needed.

   During the transition, forward.h (#72) helps: the builder emits a
   header with ALL forward declarations (structs, enums, functions).
   ext.h includes forward.h, so link_c files and ext.c see all types.
   Each type/function is defined once and forward-declared everywhere.

*** Calling convention: C must match til ownership semantics

   The C code must follow the same patterns that til generates.
   Reference: qualifiers.org for arg/return/field mapping.

   Vec of struct values (e.g. Expr children):
   #+begin_src til
   // til: Vec stores inline values, push takes ownership
   children := Vec.new("Expr", Expr.size())
   add_child := proc(mut self: Expr, own child: Expr) {
       self.children.push(own child)    // copies + frees child
   }
   child := func(parent: Expr, i: I64) returns ref Expr {
       ref c : Expr = parent.children.get(i.to_u64())
       return c                         // pointer into Vec buffer
   }
   #+end_src

   #+begin_src c
   // C equivalent (matching til):
   // Vec_new with sizeof(Expr) — value storage, NOT pointer storage
   Vec_new(&(Str){"Expr", 4, CAP_LIT}, &(U64){sizeof(Expr)})

   // add_child: push copies sizeof(Expr) bytes, frees original
   void Expr_add_child(Expr *self, Expr *child) {
       Vec_push(&self->children, child);  // copies + free(child)
   }

   // child: single cast, returns pointer INTO Vec buffer
   Expr *Expr_child(Expr *parent, I64 *i) {
       return (Expr *)Vec_get(&parent->children, ...);
   }
   #+end_src

   #+begin_src c
   // WRONG (old C pattern — pointer-based, not matching til):
   Vec_new(&(Str){"", 0, CAP_LIT}, &(U64){sizeof(Expr *)})  // WRONG
   Expr **_p = malloc(sizeof(Expr *)); *_p = child;
   Vec_push(&self->children, _p);                             // WRONG
   *(Expr **)Vec_get(...)                                     // WRONG
   #+end_src

   Moving children between Vecs (e.g. ctil.c merge pattern):
   #+begin_src til
   // til: clone before push (Expr_child returns ref into source Vec,
   // push would free it — can't free a pointer into another Vec's buffer)
   mut merged := Vec.new("Expr", Expr.size())
   for i in 0..ast.children.len() {
       ref ch : Expr = ast.children.get(i)
       merged.push(own ch.clone())  // clone → new heap alloc → push copies + frees
   }
   #+end_src

   #+begin_src c
   // C equivalent:
   Vec merged; { Vec *_vp = Vec_new(..., &(U64){sizeof(Expr)}); merged = *_vp; free(_vp); }
   for (U32 i = 0; i < ast->children.count; i++) {
       Expr *ch = Expr_child(ast, &(I64){i});  // ref into ast's Vec buffer
       Expr *cloned = Expr_clone(ch);           // new heap allocation
       Vec_push(&merged, cloned);               // copies sizeof(Expr), frees cloned
   }
   #+end_src

   Assigning to an existing children slot:
   #+begin_src c
   // C: copy into slot, free the source
   Expr *new_val = Expr_new(...);
   memcpy(Vec_get(&e->children, &(U64){idx}), new_val, sizeof(Expr));
   free(new_val);
   #+end_src

  variables. Precomp currently folds scalar expressions (I64, Str,
  Bool) but not struct-valued or collection-valued ones.

** OPEN #58 First-class functions: Type hierarchy and FuncSig/FuncDef
  :DEPENDENCIES: #14 #61
  Based on rstil/doc/first_class.org. Everything in til is a value —
  functions, structs, enums can all be assigned, passed, returned, and
  stored. This issue covers the metatype system that makes it work.

*** Type hierarchy

   ~Type~ is the metatype — the type of all type definitions:

   #+begin_src til
   Type := enum { StructDef, EnumDef, FuncSig, FuncDef }
   #+end_src

   | Metatype    | Defines                      | Instance example                     |
   |-------------+------------------------------+--------------------------------------|
   | ~StructDef~ | Data layout (fields)         | ~p := Point(x=10)~ is a Point        |
   | ~EnumDef~   | Tagged union (variants)      | ~c := Color.Red~ is a Color          |
   | ~FuncSig~   | Calling convention (types)   | ~BinaryOp := func(I64, I64) ret I64~ |
   | ~FuncDef~   | Concrete function (sig+body) | ~add : BinaryOp = (a, b) { ... }~   |

   ~FuncSig~ is a pure type — parameter types, return type, func/proc
   kind. Parameter *names* are NOT part of the signature, only types.

   ~FuncDef~ is a concrete function definition — it has a ~FuncSig~ (its
   type) plus argument names, modifiers, and a body. A FuncDef is to its
   FuncSig what a struct instance is to its StructDef: the signature is
   the type, the definition is the value.

   Function types are *structural*: two signatures with the same parameter
   types, return type, and throws are the same type regardless of name.

*** Explicit type annotations (~: Type =~ syntax)

   Separate the type annotation (~:~) from the value assignment (~=~).
   ~:=~ remains as shorthand for "infer the type":

   #+begin_src til
   // Explicit type + value
   x : I64 = 5
   name : Str = "hello"

   // Type definitions: explicit metatype annotation
   Point : StructDef = struct { mut x: I64 = 0, mut y: I64 = 0 }
   Color : EnumDef = enum { Red, Green, Blue }
   BinaryOp : FuncSig = func(I64, I64) returns I64

   // Inferred (unchanged from today)
   Point := struct { mut x: I64 = 0, mut y: I64 = 0 }   // infers StructDef
   Color := enum { Red, Green, Blue }                      // infers EnumDef
   BinaryOp := func(I64, I64) returns I64                  // infers FuncSig
   #+end_src

*** Type inference rules

   | Declaration                                                | Inferred type |
   |------------------------------------------------------------+---------------|
   | ~x := 5~                                                   | ~I64~         |
   | ~name := "hello"~                                          | ~Str~         |
   | ~Point := struct { ... }~                                  | ~StructDef~   |
   | ~Color := enum { ... }~                                    | ~EnumDef~     |
   | ~BinaryOp := func(I64, I64) returns I64~                   | ~FuncSig~     |
   | ~add := func(a: I64, b: I64) returns I64 { return ... }~   | ~FuncDef~     |

   The distinction mirrors struct/enum:
   - ~Point := struct { ... }~ defines a *type* (infers StructDef)
   - ~p := Point(x=10)~ creates an *instance* (infers Point)
   - ~BinaryOp := func(I64, I64) returns I64~ defines a *type* (infers FuncSig)
   - ~add := func(a: I64, b: I64) returns I64 { ... }~ creates an *instance* (infers FuncDef)

   A bodyless func/proc is a FuncSig (type definition). A bodied func/proc
   is a FuncDef (value). The presence of a body (and parameter names)
   distinguishes them.

*** Collections of types

   #+begin_src til
   Vec.new("BinaryOp")   // Vec of function pointers with the same signature
   Vec.new("FuncSig")    // Vec of function signatures (different signatures)
   Vec.new("FuncDef")    // Vec of function definitions (different definitions)
   Vec.new("StructDef")  // Vec of struct definitions
   Vec.new("Type")       // Vec of any type definition
   #+end_src

*** Type parameters

   Two-phase plan:

   *Phase now*: ~func(T: Type)~ — no new syntax needed. ~Type~ is already
   a special parameter type. Macros handle the template use case by
   executing arbitrary code at compile time (like Jai's comptime, Zig's
   comptime):

   #+begin_src til
   // Type as a regular parameter
   swap := func(T: Type, mut a: T, mut b: T) { ... }
   swap(I64, x, y)

   Pair := macro(A: Type, B: Type) returns Type {
       return struct { mut first: A, mut second: B }
   }
   IntStr := Pair(I64, Str)
   #+end_src

   *Phase later*: ~func<T>()~ as sugar that desugars to macro calls.
   Use ~<>~ for familiarity (Rust, C++, Java):

   #+begin_src til
   // Sugar (later):
   swap<T> := func(mut a: T, mut b: T) { ... }
   swap(x, y)       // T inferred
   swap<I64>(x, y)  // T explicit

   // Desugars to:
   swap := func(T: Type, mut a: T, mut b: T) { ... }
   swap(I64, x, y)
   #+end_src

*** Unified pattern summary

   | Construct  | Type declaration               | Definition (type = value)              |
   |------------+--------------------------------+----------------------------------------|
   | Variable   | ~x : I64~                      | ~x : I64 = 5~ or ~x := 5~             |
   | Function   | ~F : func(I64) returns I64~    | ~f : func(I64) returns I64 = { ... }~  |
   | Body       | (no separate type decl)        | ~b : Body = { ... }~ or ~b := { ... }~ |

** OPEN #55 Lambda functions and first-class bodies
  :DEPENDENCIES: #14 #54
  Based on rstil/doc/first_class.org. Anonymous inline functions (lambdas)
  and first-class Body values (blocks as values).

*** Lambda functions (anonymous inline functions)

   Lambda functions are anonymous FuncDef literals passed directly as
   arguments. Same syntax as named definitions, just without a name:

   #+begin_src til
   Predicate := func(I64) returns Bool
   Reducer := func(I64, I64) returns I64

   // filter with inline lambda
   evens := numbers.filter(func(a: I64) returns Bool { return a.mod(2).eq(0) })
   // [2, 4]

   // reduce with inline lambda
   sum := numbers.reduce(0, func(acc: I64, x: I64) returns I64 { return acc.add(x) })
   // 15

   // FuncSig form lambda (type separated from param names)
   doubled := numbers.map(func(I64) returns I64 = (a) { return a.mul(2) })
   #+end_src

*** First-class bodies (blocks as values)

   A Body is a sequence of statements — the simplest unit of executable
   code. Simpler than a function: no signature, no parameters, no return
   type. A Body is essentially a zero-argument, void closure.

   #+begin_src til
   // Store in a variable
   cleanup := { file.close(); println("done") }

   // Pass as argument
   run_twice := proc(b: Body) {
       b()
       b()
   }
   run_twice({ println("hello") })

   // Store in a struct field
   Task := struct {
       mut on_complete: Body
   }
   t := Task(on_complete = { println("finished") })
   t.on_complete()

   // Store in a collection
   steps := Vec.new("Body")
   steps.push(own { println("step 1") })
   steps.push(own { println("step 2") })
   for step : Body in steps {
       step()
   }
   #+end_src

*** Relationship to functions and closures

   | Construct         | Has params | Has return type | Has captures | Has body |
   |-------------------+------------+-----------------+--------------+----------|
   | Body              | No         | No              | No*          | Yes      |
   | func (no closure) | Yes        | Yes             | No           | Yes      |
   | func (closure)    | Yes        | Yes             | Yes (#54)    | Yes      |

   *Without captures (#54), a Body can only access globals. With captures,
   a Body becomes a zero-argument closure:

   #+begin_src til
   mut total := 0
   data := vec("I64", 1, 2, 3)

   // Body with capture list
   summer := captures mut total, data { for item : I64 in data { total = total.add(item) } }
   summer()
   // total is now 6
   #+end_src

   A Body is to ~proc()~ what a literal is to a named constant — the
   anonymous, inline version. The distinction is that a Body has no
   function overhead (no frame push/pop in the interpreter, no C function
   call in codegen) — it executes in the caller's scope.

*** Use cases

   #+begin_src til
   // Deferred execution (like Go's defer)
   cleanup := { file.close() }
   // ... do work ...
   cleanup()

   // Callbacks and hooks
   Widget := struct {
       mut on_click: Body
       mut on_hover: Body
   }

   // Builder patterns
   with_transaction := proc(db: Database, action: Body) {
       db.begin()
       action()
       db.commit()
   }
   with_transaction(db, { db.insert(record1); db.insert(record2) })

   // Test setup/teardown
   run_test := proc(name: Str, setup: Body, test_body: Body, teardown: Body) {
       println("Running:", name)
       setup()
       test_body()
       teardown()
   }
   #+end_src

*** Scope semantics: inline vs isolated

   Open question: when a Body executes, does it share the caller's scope
   (like an inlined block) or get its own scope (like a function call)?

   - *Option A: Inline* — shared scope (how bare ~{ }~ blocks work today).
     Simple, powerful, zero overhead. But passing a Body as argument is
     weird — which scope does it execute in?
   - *Option B: Isolated* — own scope, captures only (#54). Safe and
     predictable for stored/passed bodies. But needs captures to be useful.
   - *Option C: Both* — bare ~{ }~ blocks are inline (current behavior),
     Body values (stored, passed, returned) are isolated and require
     captures. This is probably the right answer.

*** Comparison

   | Language   | Block-as-value          | Notes                              |
   |------------+-------------------------+------------------------------------|
   | Ruby       | ~{ ... }~ / ~do...end~  | Blocks and Procs are first-class   |
   | Smalltalk  | ~[ ... ]~               | Blocks are objects                 |
   | Swift      | ~{ () -> Void in ... }~ | Closures (always have captures)    |
   | Kotlin     | ~{ ... }~ (lambda)      | Trailing lambda syntax             |
   | Rust       | closure only            | No bare blocks as values           |
   | Go         | ~func() { ... }()~      | Must wrap in function              |
   | TIL        | ~Body = { ... }~        | Simpler than closure               |

   Ruby is the closest precedent. Ruby blocks can be passed to methods
   implicitly (~yield~) or explicitly (~&block~), and ~Proc.new~ wraps a
   block into a first-class value.

** OPEN #54 Explicit capture syntax (closures)
  :DEPENDENCIES: #61
  :DEPENDENCIES_OF: #12 #13
  Add syntax for listing captured variables from outer scopes, both for
  function definitions and anonymous blocks. Based on rstil/doc/first_class.org.
  Split from #17 (global variable access, now fixed).

  Closures = lambdas (#61) + captures. #61 provides anonymous inline
  functions (lambdas); this issue adds the ~captures~ keyword that turns
  a lambda into a closure by binding outer-scope variables.

  til requires *explicit capture lists* — if a function references an outer
  variable, it MUST be listed in the capture list. This makes dependencies
  visible and refactoring mechanical. Inspired by Jai (Jonathan Blow).

*** Capture syntax

   The ~captures~ keyword appears after the parameter list (and before
   ~returns~ if present) in FuncDef form, or after the argument names in
   FuncSig form:

   #+begin_src til
   scale := 10

   // FuncDef form: captures between params and returns
   f := func(a: I64) captures scale returns I64 { return a.mul(scale) }

   // FuncSig form: captures after argument names
   f : func(I64) returns I64 = (a) captures scale { return a.mul(scale) }

   // With a named type alias
   Transformer := func(I64) returns I64
   f : Transformer = (a) captures scale { return a.mul(scale) }
   #+end_src

*** Capture modifiers

   Captures follow the same rules as function parameters — no new rules
   to learn:

   | Capture syntax        | Semantics                     | Same as param mode |
   |-----------------------+-------------------------------+--------------------|
   | ~captures x~          | Const by reference (default)  | ~x: T~             |
   | ~captures mut x~      | Mutable reference             | ~mut x: T~         |
   | ~captures own x~      | Ownership transfer (moved in) | ~own x: T~         |

   #+begin_src til
   mut total := 0
   data := vec("I64", 1, 2, 3)

   // mut capture: closure can modify total
   data.for_each(func(x: I64) captures mut total { total = total.add(x) })

   // own capture: moves data into the closure, caller can't use it after
   consume := proc() captures own data { data.clear() }
   #+end_src

*** Block captures (scope restriction)

   Even without making a function, a capture list on a bare code block
   limits what it can see — an "insurance policy" that the block only
   touches specified state:

   #+begin_src til
   mut total := 0
   mut count := 0
   data := vec("I64", 1, 2, 3)

   // Only total and data are accessible inside this block
   captures total, data {
       for item : I64 in data {
           total = total.add(item)
       }
   }
   // count is guaranteed untouched
   #+end_src

*** Jai-inspired refactoring progression

   Each step is mechanical. The capture list tells you exactly what needs
   to become a parameter when you promote to a pure function:

   #+begin_src til
   // Step 1: inline code
   total = total.add(x)

   // Step 2: block with captures (limit scope, verify dependencies)
   captures total, x {
       total = total.add(x)
   }

   // Step 3: promote to lambda (add params, same captures)
   adder := func(a: I64) captures total returns I64 { return total.add(a) }

   // Step 4: promote to pure function (captures become params, no closure)
   adder := func(a: I64, b: I64) returns I64 { return a.add(b) }
   #+end_src

*** Why explicit captures

   Implicit captures (Python, JavaScript, Rust) feel magical:
   - *Reading*: you see ~scale~ and have to hunt for where it's defined.
     Is it a global? A local three scopes up? A typo for a parameter?
   - *Refactoring*: extracting a block into a function means figuring out
     ALL the implicit dependencies. Miss one and it breaks.
   - *Inlining*: moving a closure back inline means understanding what it
     captures to know where it can safely go.

   Explicit captures solve all three:
   - *Reading*: ~captures scale~ tells you exactly what comes from outside.
   - *Refactoring*: captures become parameters. Mechanical transformation.
   - *Inlining*: captures tell you exactly what the block depends on.

*** Implementation

   - Parser: recognize ~captures~ keyword in func defs (between params
     and returns/body) and bare blocks (before ~{~)
   - AST: store capture list on NODE_FUNC_DEF and NODE_BODY
   - Typer: validate captures exist in outer scope, enforce modifier rules
   - Interpreter: bind captured variables in call_scope from definition scope
   - Codegen: emit closure structs or pass captures as hidden parameters

** OPEN #25 Support for the 4 main operative systems
  :DEPENDENCIES: #12
  til currently only builds and runs on GNU/Linux. Should support the 4
  main operative systems, in priority order:
  1. GNU/Linux/BSD (current, done)
  2. TempleOS
  3. macOS
  4. Windows

  This is not just "til builds natively on each OS." The goal is full
  cross-compilation: build a til binary for any target from any host.
  The matrix is 3×4 + 1×1 = 13 combinations: Linux, macOS, and Windows
  can each target all 4 platforms, but TempleOS as a host only targets
  itself (cross-compiling from TempleOS would mean replacing gcc/clang,
  which is out of scope).

  ctil (the C bootstrap compiler) stays Linux-only. Once til is
  self-hosting (#12), til replaces ctil and handles all platforms.
  Bootstrap path: ctil on Linux compiles self-hosted til → til can
  then cross-compile itself and any .til program to any target.

  The C codegen path helps: emit C + invoke the target's C compiler
  works for most POSIX targets. But TempleOS breaks this pattern
  (no C compiler), requiring a dedicated HolyC codegen backend.

  The builder must select the appropriate codegen backend per target.

  Common areas affected across platforms:
  - Build system (builder that selects backend per target)
  - FFI / dlsym (platform-specific dynamic loading)
  - System primitives (spawn_cmd, readfile, etc. — POSIX vs platform APIs)
  - GUI mode (raylib on POSIX/Windows, native primitives on TempleOS)
  - Audio (miniaudio on POSIX/Windows, native on TempleOS)

*** TempleOS
   :DEPENDENCIES: #12
   The most interesting and challenging target. TempleOS has no C compiler,
   no POSIX, no libc — just HolyC and its own kernel/graphics/audio APIs.

   Requires a new holybuilder backend that emits HolyC instead of C.
   HolyC is C-like enough that the translation should be mostly mechanical
   (similar types, similar control flow), but differences in calling
   conventions, memory management, and string handling will need attention.

   raylib's interface using TempleOS graphics primitives (GrBlot, GrLine,

   via raylib AND TempleOS via native HolyC primitives would be a serious
   language flex — proving til programs are truly platform-independent.

   For mode server: TempleOS docker containers could be extremely small
   (no libc, no userland, minimal kernel), making til a compelling choice
   for lightweight server deployments on TempleOS-based containers.

   Self-hosting (#12) is a hard dependency: TempleOS has no C compiler,

** OPEN #22 Operator syntax: desugar operators to method calls
  :DEPENDENCIES: #12
  Syntactic sugar so a + b desugars to a.add(b), a < b to a.lt(b), etc.
  Purely a parser transform — no changes needed in typer, interpreter, or
  codegen. The parser rewrites operator expressions into method call AST
  nodes before anything downstream sees them.

  Operator → method mapping:
  - Comparison: < → lt, <= → lte, > → gt, >= → gte, == → eq, != → neq
  - Arithmetic: + → add, - → sub, * → mul, / → div, % → mod

  Precedence (low to high):
  - == != < <= > >=
  - + -
  - * / %

  Parenthesized expressions (a + b) * c override precedence. The parser
  must disambiguate grouping parens from function call parens: ( after an
  identifier/field-access is a call, otherwise it's grouping.

  Parsing approach: Pratt parsing (precedence climbing) or one function
  per precedence level (mutually recursive descent). Both work; Pratt is
  more compact and easier to extend.

  The cmp/initer synergy makes this especially nice for comparisons:
  implement cmp once, the initer auto-generates eq, neq, lt, gt, lte, gte,
  and operator desugaring gives you <, <=, >, >=, ==, != for free. If you
  override eq with a faster implementation, neq (and !=) automatically
  benefits since neq is generated as a.eq(b).not().

*** Phase 1: Comparison operators
   < <= > >= == !=
   All same precedence, binary, no chaining ambiguity.
   Desugars: a < b → a.lt(b), a != b → a.neq(b), etc.

*** Phase 2: Arithmetic operators
   + - * / %
   Introduces real precedence: * / % bind tighter than + -.
   Desugars: a + b * c → a.add(b.mul(c)).
   Requires grouping parentheses support.

   Also: ~++~ / ~--~ as unary sugar for inc/dec (methods added in #34 Phase 2).
   ~i++~ → ~i = i.inc()~, ~i--~ → ~i = i.dec()~.
   ~+=~ / ~-=~ as compound assignment sugar: ~i += 1~ → ~i = i.add(1)~.
