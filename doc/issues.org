** OPEN #131 `mode` syntax: keyword vs special identifier/directive
  The language currently treats ~mode~ as a dedicated keyword/token, even
  though it is only meaningful in the file-header position.

  This should be decided explicitly instead of remaining an inherited parser
  special-case.

  Current question:
  - should ~mode~ remain a reserved keyword
  - or should it become a normal identifier that is recognized specially only
    in the file-header directive position

  Why this matters:
  - it affects lexer complexity and reserved-word surface
  - it affects whether ~mode~ can be used as an ordinary identifier elsewhere
  - it changes how much of the current parser logic needs to stay keyword-based

  This is design-only for now.
  It is tracked separately from the current ~Context~ / mode-plumbing refactor.

** OPEN #130 AST / ~Expr~ representation cleanup
  The current AST shape is confusing and duplicates state in ways that make
  parser and ownership bugs harder to reason about.

  Current pressure points:
  - ~ExprData~ is still acting as a giant catch-all payload enum
  - node-kind-specific data is split between ~Expr~, ~ExprData~, and
    sometimes child lists
  - that makes ownership/lifetime of payload pointers much harder to see
  - parser code is forced into fragile payload-rebuild patterns around
    ~ExprData.FuncDef~, ~Decl~, and related nodes

  Intended direction:
  - move toward a clearer node model where node-kind payload data lives in
    the node-kind payload itself
  - keep structural subexpressions in children, instead of duplicating
    state between children and payload structs
  - reduce the amount of duplicated / mirrored metadata spread across
    ~Expr~ and embedded payload structs

  Expected payoff:
  - simpler parser code
  - fewer use-after-free / stale-payload bugs during AST rewrites
  - easier future cleanup of clone/delete semantics around AST nodes

** OPEN #129 Runtime ~Value~ instance payload fields still use ~ref~ where they should own
  This is the runtime ~Value~ / interpreter-side ownership issue, separate
  from ~#128~'s auto-generated enum methods in ~initer.til~.

  Current verified pressure points:
  - ~StructInstance~ / ~EnumInstance~ are the runtime headers for
    ~Value.Struct~ / ~Value.Enum~
  - ~StructInstance.data~ already moved to owned storage, but surrounding
    runtime lifecycle code still behaves as if these payload buffers were
    borrowed
  - ~EnumInstance.data~ is still declared as ~ref mut data : U8 = NULL~
  - those choices force manual ~free(...)~ / clone workarounds in
    interpreter and dispatch code instead of making wrapper ownership explicit

  Consequences:
  - runtime ownership is encoded in ad hoc C paths instead of the type shape
  - translated til code keeps tripping over borrowed-vs-owned mismatches
    around struct/enum payload buffers
  - cleanup of ~Value.Struct~ / ~Value.Enum~ remains harder than it should be

  Desired direction:
  - make the runtime instance wrappers own the payload buffers they are
    responsible for
  - then simplify the runtime clone/delete/free paths around those wrappers

** OPEN #128 Auto-generated enum clone/delete in ~initer.til~ is wrong for managed payloads
  This issue is specifically about enum methods auto-generated by
  ~src/self/initer.til~, not about the separate runtime ~Value.Enum~ wrapper
  problem tracked in ~#129~.

  Auto-generated enum lifecycle must be fully correct on its own:
  - generated enum ~clone~ must correctly duplicate managed payload ownership
  - generated enum ~delete~ must correctly release managed payload ownership

  Current verified problem:
  - generated enum ~delete~ only frees ~self~ and does not recurse into
    payload-owned data

  This matters for payload enums carrying managed values like:
  - ~Str~
  - structs with owned internals
  - containers such as ~Vec~

  Verified examples already allowed by the language/tests:
  - ~Token.Name: Str~
  - ~Shape.Dot: Point~
  - ~Shape.Line: Str~

  Consequences:
  - generated enum lifecycle is not a correct ownership model for managed
    payloads
  - generated/builder code cannot rely on enum methods to manage payload
    ownership correctly
  - manual cleanup workarounds become tempting elsewhere, instead of fixing
    enum generation itself

  Desired end state:
  - auto-generated enum ~clone/delete~ are correct for managed payload enums
  - no workaround semantics, temporary exceptions, or caller-side cleanup are
    needed to make payload enum ownership correct

** OPEN #127 Runtime ~Value~/binding ownership cleanup and ~Ptr~ split
  The interpreter runtime currently overloads one mechanism for several
  different ownership cases:
  - scoped values live in ~Cell.val~
  - ~Scope~ bindings own some ~Cell*~ values via ~cell_is_local~
  - ~scope_free~ frees owned ~Cell*~, but does not delete the runtime
    ~Value~ stored inside them
  - one runtime variant, current ~Value.Ptr~, is used for both:
    - borrowed/ref-like storage views
    - owning opaque heap pointers

  That causes two classes of problems:
  - real owner cleanup is missing at scope teardown / overwrite boundaries
  - dispatcher/runtime code grows manual ~free(...)~ workarounds because the
    runtime value model does not distinguish borrowed pointer views from
    owned dynamic pointers cleanly

  Current verified pressure points:
  - ~scope_free~ frees local cells but not their values
  - ~scope_set_owned~ overwrites owned bindings without deleting the old
    value first
  - current ~Value.Ptr~ is produced by:
    - ~null~
    - ref field reads
    - own field reads exposing owned storage
    - ~ptr_add(...)~
    - FFI ~Dynamic~ returns
    - container data fields like ~Array.data~ / ~Vec.data~
  - current consumers then guess whether that pointer is borrowed or owned

  Cleanup direction:
  - give runtime pointer-like values distinct meanings
  - stop overloading one variant for both ref-like and owning semantics
  - make the real owner delete runtime values at scope teardown and on
    overwrite
  - then simplify dispatcher helpers like ~val_to_ptr~ so they are only
    borrowed pointer-view helpers instead of ownership escape hatches

  Current candidate shape:
  - split current ~Value.Ptr~ into:
    - ~Value.Ref~ for borrowed/nullable pointer views
    - ~Value.Dynamic~ for owning opaque pointers
  - keep ~val_to_ptr := func(v: Value) returns ref Dynamic~
  - repair runtime binding cleanup before relying on translated dispatcher
    helpers as proof of correctness

  Important constraints discovered during analysis:
  - simple removal of ~Cell~ is not obviously safe yet, because current
    borrowed bindings are used for:
    - ref declarations from identifiers
    - borrowed function argument binding paths
  - own field reads cannot simply be classified globally as borrowed or
    owning without checking the typed/lowered move/clone behavior

  This should be tackled as a focused runtime ownership cleanup, not mixed
  into unrelated dispatcher translations.

** OPEN #126 Auto-generated enum delete does not delete payload-owned data
  Auto-generated enum ~delete~ currently only does:
  - ~if call_free { free(own self) }~

  It does not:
  - inspect the active variant
  - call ~delete~ on payloads
  - free payload-owned memory for variants whose payload is itself owning

  That makes generated enum deletion too weak for enums like runtime
  ~Value~, whose payload variants can carry owned data indirectly via:
  - structs
  - enums
  - raw dynamic pointers

  Consequences:
  - runtime code is forced to keep manual side cleanup helpers instead of
    relying on generated delete
  - hand-written empty ~delete~ methods can end up suppressing generation
    without replacing it with correct behavior

  Desired end state:
  - generated enum ~delete~ should recursively delete payloads when the
    payload type is owning
  - tag-only / borrowed payload variants should remain cheap and non-freeing
  - manual enum ~delete~ should only be needed for truly custom behavior

** OPEN #125 Enum payload method generation should not assume ~eq~/~to_str~
  Related to ~#102~, but separate: even if structs later auto-generate
  ~eq~ and ~to_str~, enum auto-generation still needs a clear rule for
  payload variants.

  Right now payload enums auto-generate:
  - ~eq~ by calling payload ~.eq(...)~
  - ~to_str~ by calling payload ~.to_str()~

  That fails for payload types like ~Expr~, which currently do not expose
  those methods, even though the enum itself may still be useful for
  internal compiler bookkeeping.

  Example that currently fails during self-hosting:
  - ~CtorArg := enum { Unfilled, Filled: Expr }~

  - should enums auto-generate ~eq~/~to_str~ only when all payload types
    support them?
  - should those methods be skipped entirely unless explicitly requested?
  - should payload and non-payload enums follow different defaults?

** OPEN #124 Raw `Dynamic` ownership should not require explicit `free(...)` in til
  Today, typed owned values and containers mostly get their cleanup inserted
  automatically by typer, but raw ~Dynamic~ buffers do not. That leaks the
  low-level memory model into ordinary til code.

  Current symptom:
  - code that takes ownership of raw memory from helpers like:
    - ~malloc(...)~
    - ~realloc(...)~
    - ~Vec_take(...)~
  - still needs explicit:
    - ~free(own buf)~

  That is inconsistent with the rest of til ownership:
  - owned ~Str~, ~Expr~, ~Vec~, ~Map~, etc. are cleaned up by generated
    delete/free insertion
  - but owned raw ~Dynamic~ values are effectively second-class and require
    handwritten cleanup in til source

  Desired end state:
  - if til code owns a ~Dynamic~, the ownership system should be able to
    release it automatically without handwritten ~free(...)~
  - or the language should grow a distinct raw-owned buffer type whose
    destructor is unambiguously ~free~
  - this should be the default behavior, not a ban on explicit manual
    cleanup in edge cases like:
    - custom/manual ~delete~ implementations
    - foreign ownership
    - custom allocators
    - raw buffers that intentionally should not go through plain ~free~

  This matters for self-hosting cleanup too:
  - current translated code still contains explicit ~free(own locals)~ style
    calls for buffers returned from helpers like ~Vec_take~
  - that should ideally become unnecessary once raw owned memory has a proper
    automatic cleanup story

  Likely directions:
  - teach typer/runtime that owned ~Dynamic~ defaults to ~free~
  - or introduce a dedicated raw-buffer type separate from generic ~Dynamic~
  - make the final rule explicit in ~qualifiers.org~ and ~c2til.org~

** OPEN #122 Mode-sensitive type checks should be per module
  Mode-sensitive typing rules are currently applied with one global
  ~current_mode~ for the whole compilation unit. That is wrong for
  imported modules.

  Concrete failure that exposed this:
  - [src/core/str.til](/home/jt/sync/data_common/code/jt/til/src/core/str.til)
    was temporarily switched from ~mode lib~ to ~mode liba~ for debug
  - ~liba~ sets ~debug_prints = true~ in
    [src/self/modes.til](/home/jt/sync/data_common/code/jt/til/src/self/modes.til)
  - but typing still rejected ~println(...)~ inside a ~func~ in that file with:
    ~func cannot call proc 'println'~

  Why this happens:
  - [src/til.til](/home/jt/sync/data_common/code/jt/til/src/til.til) resolves one
    top-level ~cur_mode~
  - [src/c/typer.c](/home/jt/sync/data_common/code/jt/til/src/c/typer.c)
    stores that once in the global ~current_mode~
  - mode-sensitive checks like the ~print/println~ debug exemption then
    consult only that one global value
  - imported files are therefore type-checked as if they had the top-level
    program mode, not their own declared ~mode~

  This is not just about debug prints. Any mode-sensitive typing rule is
  suspect under the current architecture, including at least:
  - ~func~ calling ~proc~ exemptions in debug modes
  - pure/pura restrictions
  - any future rule that depends on per-module mode

  Desired end state:
  - each parsed module keeps its own resolved mode
  - typer applies mode-sensitive checks according to the module that owns
    the AST node being checked
  - imported ~mode lib~, ~liba~, ~pure~, ~pura~, etc. stop inheriting the
    caller/top-level mode for typing purposes

  Likely implementation directions:
  - store resolved module mode on the parsed module/root body and propagate
    it to descendant nodes
  - or thread current module mode through typer recursion when descending
    across merged module boundaries
  - avoid keeping one process-global ~current_mode~ as the only source of
    truth for semantic checks

** OPEN #117 Bool and/or do not short-circuit
  ~and~ and ~or~ currently behave like ordinary Bool calls rather than
  C-style short-circuit operators.

  Current consequence:
  - translations that are mechanically correct in C can become wrong in til
    if they rely on ~&&~ / ~||~ evaluation order
  - example from the ~initer.c~ translation:
    - C: ~if (!target->struct_def && !target->is_builtin &&
      (!target->func_def || target->func_def->children.count != 0))~
    - naive til translation using ~.and(...)~ / ~.or(...)~ dereferenced
      ~target.func_def~ even when it was null

  This matters for ~c2til~ directly:
  - mechanical C-to-til translation is harder and more error-prone than it
    should be
  - translators must currently rewrite many conditions into nested ~if~s
    just to preserve C semantics

  Desired end state:
  - a Bool-level surface that preserves short-circuit semantics where the
    source expects them
  - or, at minimum, a clearly documented distinction and a dedicated
    short-circuiting form for translated code

  Possible directions:
  - make ~and~ / ~or~ short-circuit in the language
  - or introduce explicit short-circuiting operators/forms distinct from the
    current Bool methods
  - update ~c2til.org~ once the final rule is chosen

  For now:
  - keep translating such conditions carefully with nested ~if~ statements
    when preserving C behavior matters

** OPEN #116 Imported files are parsed in isolation
  Imported files are currently lexed and parsed as standalone units, then
  merged later. They do not see declarations from the importing file during
  parse/bootstrap time.

  Current behavior in ~src/til.til~:
  - parse one file
  - extract ~import("...")~ calls from its AST
  - recursively read, lex, and parse each imported file separately
  - only after that merge:
    - core
    - core imports
    - mode imports
    - user imports
    - user file

  This means import order in the source tree does not imply visibility while
  parsing imported files. Imported files are self-contained until the later
  AST merge.

  Concrete blocker example:
  - we tried to unify ~StructDefData~ with the existing ~StructDef~ type
  - the natural cleanup would be to define one canonical ~StructDef~ in
    ~core.til~ and use it as the ~ExprData.StructDef~ payload type in
    ~ast.til~
  - that fails during bootstrap, because ~ast.til~ is parsed in isolation and
    cannot see ~StructDef~ from ~core.til~ at the time ~ExprData~ is declared
  - moving the ~import("ast.til")~ line within ~core.til~ does not help,
    because imported files are not parsed in the importer’s declaration
    context

  What this blocks:
  - unifying local AST payload helper types with canonical types defined in
    importer files
  - some circular/meta-type cleanups that look valid after merge but fail
    during isolated parse/bootstrap
  - relying on importer-local predeclarations to shape imported AST types

  Likely directions:
  - keep such canonical payload types local to the imported file for now
  - or introduce a true predeclared/builtin meta-type path
  - or change import/bootstrap semantics so imported files can see a shared
    prelude / prior declarations before parsing

  Immediate outcome:
  - keep ~StructDefData~ for now
  - treat the failed ~StructDef~ unification as a symptom of the import model,
    not just a naming conflict

** OPEN #101 Deterministic temp variable names in generated C
  :DEPENDENCY_OF: #77
  The builder/typer generates temp variable names (e.g. ~_t4253~,
  ~_rc4617~, ~_wcond4618~) using counters that depend on processing
  order or memory layout, making the output non-deterministic.
  Two identical ~til build~ runs on the same source can produce
  different C files (same logic, different temp names).

  This blocks #77: ~til build src/til.til~ must be idempotent --
  the self-built binary must produce identical output to the
  bootstrap binary. Without deterministic names, we can't diff
  generated C to verify correctness.

  Fix: ensure temp variable counters are reset or seeded
  deterministically per translation unit, so the same input
  always produces the same output.

** OPEN #91 Containers: store clone/delete as function pointers instead of dyn_call
  Vec/Map/Set currently dispatch clone/delete by type name string via
  ~dyn_call~, which does a linear scan of if/memcmp chains every time.
  Instead, store clone/delete function pointers as fields in the
  container struct. ~Vec.new("Str", Str.size())~ resolves the function
  pointers once at construction, then all subsequent clone/delete calls
  go through the stored pointers directly.

  Proven pattern: ~AlphaBeta~ struct in ~src/lib/alphabeta.til~ uses
  the same function-pointer-on-struct approach — ~h~ and ~nexts~ fields
  called via ~self.h(board)~ and ~self.nexts(board)~. All three game
  examples (checkers, chess, go) use it successfully.

  Design:

  New ext_func to resolve function pointers from type name + method:
  : dyn_get_fn_ptr := ext_func(type_name: Str, method: Str) returns Dynamic {}

  Vec struct gains function pointer fields:
  #+begin_src til
  Vec := struct {
      own mut data : U8 = 0
      mut count : USize = 0
      mut cap : USize = 0
      elem_size : USize = 0
      elem_type := ""
      elem_clone : func(Dynamic) returns Dynamic = default_clone
      elem_delete : proc(Dynamic, Bool) = default_delete
  }
  #+end_src

  Vec.new resolves pointers once at construction:
  #+begin_src til
  new := func(elem_type: Str, elem_size: USize) returns Vec {
      return Vec(
          data = malloc(elem_size),
          cap = 1,
          elem_size = elem_size,
          elem_type = elem_type,
          elem_clone = dyn_get_fn_ptr(elem_type, "clone"),
          elem_delete = dyn_get_fn_ptr(elem_type, "delete"),
      )
  }
  #+end_src

  Container methods use the fields directly:
  #+begin_src til
  // Before (dyn_call):
  dyn_call(self.elem_type, "delete", 2, ptr_add(...), false)
  own cloned : Dynamic = dyn_call_ret(self.elem_type, "clone", 1, ptr_add(...))

  // After (function pointer):
  self.elem_delete(ptr_add(...), false)
  own cloned : Dynamic = self.elem_clone(ptr_add(...))
  #+end_src

  Same pattern for Map (key_clone, key_delete, key_cmp, val_clone,
  val_delete), Set (elem_clone, elem_delete, elem_cmp), Array, Tuple.

  No workarounds for syntax issues — if function pointers on structs

** OPEN #86 Typer error for unused declarations
  Unused variables, function parameters, and imports should be
  type errors — not warnings. til has no warnings, only errors.

  Escape hatch: prefix with ~_~ to suppress. This lets WIP code
  compile without noise while keeping the codebase clean.

  #+begin_src til
  x := 42          // error: unused variable 'x'
  _x := 42         // ok: _ prefix suppresses

  proc foo(a: I64, b: I64) {   // error: unused parameter 'b'
      println(a.to_str())
  }
  proc foo(a: I64, _b: I64) {  // ok
      println(a.to_str())
  }
  #+end_src

  Exempt from this rule:
  - Struct fields (part of a layout, removing one changes the struct)
  - Enum variants (part of a type, removing one changes the enum)
  - Namespace functions (may be called externally)

  Implementation: typer tracks which bindings are read. After
  type-checking a scope, any unread non-~_~ binding is an error.

*** What changes for shallow types

   Shallow types can be:
   - Stack-allocated instead of heap-allocated (no malloc/free)
   - Shallow-copied safely (memcpy, no deep clone needed)

   #+begin_src til
   // Currently: Point is heap-allocated like everything else
   //   Point *p = (Point *)malloc(sizeof(Point));
   //   p->x = 10; p->y = 20;
   //   // ... use p ...
   //   free(p);

   // With this optimization: Point lives on the stack
   //   Point p = { .x = 10, .y = 20 };
   //   // ... use p ...
   //   // no free needed

   // Clone becomes memcpy instead of malloc+field-by-field copy
   p := Point(x = 10, y = 20)
   q := p  // memcpy(&q, &p, sizeof(Point)) — no heap allocation
   #+end_src

*** Delete is still called

   Important: shallow types still need ~delete~ called (unlike primitives
   whose delete is a no-op). Custom delete methods may do non-memory work
   (close file descriptors, connections, etc.). The guarantee to the user
   is that delete is called ASAP after the last use of an instance. What
   changes is: no ~free~ at the end (since it wasn't malloc'd), and no
   deep-copy clone (just memcpy).

   #+begin_src til
   // Even though FileHandle is shallow (only inline fields),
   // delete must still be called to close the fd
   FileHandle := struct {
       fd: I64 = 0

   namespace:
       delete := proc(own self: FileHandle) {
           close_fd(self.fd)
           // no free(self) — it was stack-allocated
       }
   }
   #+end_src

*** Distinction from ~shallow~ keyword

   This is a different notion from the ~shallow~ keyword on args/returns,
   which is about passing by value vs pointer based on the size of the type
   relative to the platform pointer size. The "shallow type" concept here
   is about copy semantics — whether a type can be safely shallow-copied
   without ownership issues. In the future, the ~shallow~ keyword could
   be computed automatically from this property plus size considerations.

*** Calling convention: C must match til ownership semantics

   The C code must follow the same patterns that til generates.
   Reference: qualifiers.org for arg/return/field mapping.

   Vec of struct values (e.g. Expr children):
   #+begin_src til
   // til: Vec stores inline values, push takes ownership
   children := Vec.new("Expr", Expr.size())
   add_child := proc(mut self: Expr, own child: Expr) {
       self.children.push(own child)    // copies + frees child
   }
   child := func(parent: Expr, i: I64) returns ref Expr {
       ref c : Expr = parent.children.get(i.to_u64())
       return c                         // pointer into Vec buffer
   }
   #+end_src

   #+begin_src c
   // C equivalent (matching til):
   // Vec_new with sizeof(Expr) — value storage, NOT pointer storage
   Vec_new(&(Str){"Expr", 4, CAP_LIT}, &(U64){sizeof(Expr)})

   // add_child: push copies sizeof(Expr) bytes, frees original
   void Expr_add_child(Expr *self, Expr *child) {
       Vec_push(&self->children, child);  // copies + free(child)
   }

   // child: single cast, returns pointer INTO Vec buffer
   Expr *Expr_child(Expr *parent, I64 *i) {
       return (Expr *)Vec_get(&parent->children, ...);
   }
   #+end_src

   #+begin_src c
   // WRONG (old C pattern — pointer-based, not matching til):
   Vec_new(&(Str){"", 0, CAP_LIT}, &(U64){sizeof(Expr *)})  // WRONG
   Expr **_p = malloc(sizeof(Expr *)); *_p = child;
   Vec_push(&self->children, _p);                             // WRONG
   *(Expr **)Vec_get(...)                                     // WRONG
   #+end_src

   Moving children between Vecs (e.g. ctil.c merge pattern):
   #+begin_src til
   // til: clone before push (Expr_child returns ref into source Vec,
   // push would free it — can't free a pointer into another Vec's buffer)
   mut merged := Vec.new("Expr", Expr.size())
   for i in 0..ast.children.len() {
       ref ch : Expr = ast.children.get(i)
       merged.push(own ch.clone())  // clone → new heap alloc → push copies + frees
   }
   #+end_src

   #+begin_src c
   // C equivalent:
   Vec merged; { Vec *_vp = Vec_new(..., &(U64){sizeof(Expr)}); merged = *_vp; free(_vp); }
   for (U32 i = 0; i < ast->children.count; i++) {

       Expr *cloned = Expr_clone(ch);           // new heap allocation
       Vec_push(&merged, cloned);               // copies sizeof(Expr), frees cloned
   }
   #+end_src

** OPEN #115 Remove or simplify ~ext_struct~
  Revisit whether ~ext_struct~ should exist as a separate concept at all.

  Today many uses are really just opaque or fieldless types with methods, for
  example the primitive types split out of ~core.til~. Conceptually, these are
  still just types. The main special behavior ~ext_struct~ is standing in for
  is:

  - "the caller should not assume fields/layout from source syntax"
  - "the type may still define methods in its namespace"
  - "size/clone/delete may need custom behavior"

  That may not justify a dedicated type form.

  Possible direction:

  - use ordinary ~struct {}~ (or fieldless struct declarations) for opaque
    external types too
  - if the type is opaque, do not rely on autogenerated size/clone/delete
  - instead, require explicit methods where the default would be wrong

  In particular, if a type wants to be opaque but still copyable/deletable
  correctly, it should be able to provide:

  - ~size() returns USize~
  - ~clone(...)~
  - ~delete(...)~

  rather than needing a distinct ~ext_struct~ concept.

  Questions to resolve:

  - should opaque external types just be ordinary structs with no fields?
  - should initer only autogenerate ~size/clone/delete~ when the layout is
    fully known from source fields?
  - is there any remaining semantic difference that truly requires
    ~ext_struct~?

  Recent evidence:

  - a dormant ~src/core/i8.til~ plus matching ~ext.c/ext.h~ support was easy
    to add
  - but importing ~I8~ immediately exposed that the current compiler still
    has repeated hardcoded primitive-name lists (~U8~, ~I16~, ~I32~, ~U32~,
    ~U64~, ~I64~, ~F32~, ~Bool~) across builder, initer, typer, and
    ~TilType~
  - those lists decide things like:
    - whether a type is treated as a builtin scalar
    - whether generated headers skip forward declarations for it
    - whether C emission uses the alias-backed primitive representation or
      falls back to ordinary struct handling
  - so the current ~I8~ blocker is not the lack of another ~ext_struct~
    mechanism, but the broader primitive special-casing model

  That pushes this issue in a clearer direction:

  - simplify/remove ~ext_struct~
  - reduce hardcoded primitive special cases to the minimum truly required by
    runtime representation
  - make ordinary type declarations plus explicit methods carry more of the
    behavior today encoded by compiler categories

  This issue is partly about simplifying the language surface and partly about
  reducing one more C-leaning special case in the core model.

** OPEN #58 First-class functions: Type hierarchy and FuncSig/FuncDef
  :DEPENDENCIES: #14 #61
  Based on rstil/doc/first_class.org. Everything in til is a value —
  functions, structs, enums can all be assigned, passed, returned, and
  stored. This issue covers the metatype system that makes it work.

*** Type hierarchy

   ~Type~ is the metatype — the type of all type definitions:

   #+begin_src til
   Type := enum { StructDef, EnumDef, FuncSig, FuncDef }
   #+end_src

   | Metatype    | Defines                      | Instance example                     |
   |-------------+------------------------------+--------------------------------------|
   | ~StructDef~ | Data layout (fields)         | ~p := Point(x=10)~ is a Point        |
   | ~EnumDef~   | Tagged union (variants)      | ~c := Color.Red~ is a Color          |
   | ~FuncSig~   | Calling convention (types)   | ~BinaryOp := func(I64, I64) ret I64~ |
   | ~FuncDef~   | Concrete function (sig+body) | ~add : BinaryOp = (a, b) { ... }~   |

   ~FuncSig~ is a pure type — parameter types, return type, func/proc
   kind. Parameter *names* are NOT part of the signature, only types.

   ~FuncDef~ is a concrete function definition — it has a ~FuncSig~ (its
   type) plus argument names, modifiers, and a body. A FuncDef is to its
   FuncSig what a struct instance is to its StructDef: the signature is
   the type, the definition is the value.

   Function types are *structural*: two signatures with the same parameter
   types, return type, and throws are the same type regardless of name.

*** Explicit type annotations (~: Type =~ syntax)

   Separate the type annotation (~:~) from the value assignment (~=~).
   ~:=~ remains as shorthand for "infer the type":

   #+begin_src til
   // Explicit type + value
   x : I64 = 5
   name : Str = "hello"

   // Type definitions: explicit metatype annotation
   Point : StructDef = struct { mut x: I64 = 0, mut y: I64 = 0 }
   Color : EnumDef = enum { Red, Green, Blue }
   BinaryOp : FuncSig = func(I64, I64) returns I64

   // Inferred (unchanged from today)
   Point := struct { mut x: I64 = 0, mut y: I64 = 0 }   // infers StructDef
   Color := enum { Red, Green, Blue }                      // infers EnumDef
   BinaryOp := func(I64, I64) returns I64                  // infers FuncSig
   #+end_src

*** Type inference rules

   | Declaration                                                | Inferred type |
   |------------------------------------------------------------+---------------|
   | ~x := 5~                                                   | ~I64~         |
   | ~name := "hello"~                                          | ~Str~         |
   | ~Point := struct { ... }~                                  | ~StructDef~   |
   | ~Color := enum { ... }~                                    | ~EnumDef~     |
   | ~BinaryOp := func(I64, I64) returns I64~                   | ~FuncSig~     |
   | ~add := func(a: I64, b: I64) returns I64 { return ... }~   | ~FuncDef~     |

   The distinction mirrors struct/enum:
   - ~Point := struct { ... }~ defines a *type* (infers StructDef)
   - ~p := Point(x=10)~ creates an *instance* (infers Point)
   - ~BinaryOp := func(I64, I64) returns I64~ defines a *type* (infers FuncSig)
   - ~add := func(a: I64, b: I64) returns I64 { ... }~ creates an *instance* (infers FuncDef)

   A bodyless func/proc is a FuncSig (type definition). A bodied func/proc
   is a FuncDef (value). The presence of a body (and parameter names)
   distinguishes them.

*** Collections of types

   #+begin_src til
   Vec.new("BinaryOp")   // Vec of function pointers with the same signature
   Vec.new("FuncSig")    // Vec of function signatures (different signatures)
   Vec.new("FuncDef")    // Vec of function definitions (different definitions)
   Vec.new("StructDef")  // Vec of struct definitions
   Vec.new("Type")       // Vec of any type definition
   #+end_src

*** Type parameters

   Two-phase plan:

   *Phase now*: ~func(T: Type)~ — no new syntax needed. ~Type~ is already
   a special parameter type. Macros handle the template use case by
   executing arbitrary code at compile time (like Jai's comptime, Zig's
   comptime):

   #+begin_src til
   // Type as a regular parameter
   swap := func(T: Type, mut a: T, mut b: T) { ... }
   swap(I64, x, y)

   Pair := macro(A: Type, B: Type) returns Type {
       return struct { mut first: A, mut second: B }
   }
   IntStr := Pair(I64, Str)
   #+end_src

   *Phase later*: ~func<T>()~ as sugar that desugars to macro calls.
   Use ~<>~ for familiarity (Rust, C++, Java):

   #+begin_src til
   // Sugar (later):
   swap<T> := func(mut a: T, mut b: T) { ... }
   swap(x, y)       // T inferred
   swap<I64>(x, y)  // T explicit

   // Desugars to:
   swap := func(T: Type, mut a: T, mut b: T) { ... }
   swap(I64, x, y)
   #+end_src

*** Unified pattern summary

   | Construct  | Type declaration               | Definition (type = value)              |
   |------------+--------------------------------+----------------------------------------|
   | Variable   | ~x : I64~                      | ~x : I64 = 5~ or ~x := 5~             |
   | Function   | ~F : func(I64) returns I64~    | ~f : func(I64) returns I64 = { ... }~  |
   | Body       | (no separate type decl)        | ~b : Body = { ... }~ or ~b := { ... }~ |
   | Struct     | ~S : struct { ... }~           | (struct decl IS the definition)        |
   | Enum       | ~E : enum { ... }~             | (enum decl IS the definition)          |

** OPEN #55 Lambda functions and first-class bodies
  :DEPENDENCIES: #14 #54
  Based on rstil/doc/first_class.org. Anonymous inline functions (lambdas)
  and first-class Body values (blocks as values).

*** Lambda functions (anonymous inline functions)

   Lambda functions are anonymous FuncDef literals passed directly as
   arguments. Same syntax as named definitions, just without a name:

   #+begin_src til
   Predicate := func(I64) returns Bool
   Reducer := func(I64, I64) returns I64

   // filter with inline lambda
   evens := numbers.filter(func(a: I64) returns Bool { return a.mod(2).eq(0) })
   // [2, 4]

   // reduce with inline lambda
   sum := numbers.reduce(0, func(acc: I64, x: I64) returns I64 { return acc.add(x) })
   // 15

   // FuncSig form lambda (type separated from param names)
   doubled := numbers.map(func(I64) returns I64 = (a) { return a.mul(2) })
   #+end_src

*** First-class bodies (blocks as values)

   A Body is a sequence of statements — the simplest unit of executable
   code. Simpler than a function: no signature, no parameters, no return
   type. A Body is essentially a zero-argument, void closure.

   #+begin_src til
   // Store in a variable
   cleanup := { file.close(); println("done") }

   // Pass as argument
   run_twice := proc(b: Body) {
       b()
       b()
   }
   run_twice({ println("hello") })

   // Store in a struct field
   Task := struct {
       mut on_complete: Body
   }
   t := Task(on_complete = { println("finished") })
   t.on_complete()

   // Store in a collection
   steps := Vec.new("Body")
   steps.push(own { println("step 1") })
   steps.push(own { println("step 2") })
   for step : Body in steps {
       step()
   }
   #+end_src

*** Relationship to functions and closures

   | Construct         | Has params | Has return type | Has captures | Has body |
   |-------------------+------------+-----------------+--------------+----------|
   | Body              | No         | No              | No*          | Yes      |
   | func (no closure) | Yes        | Yes             | No           | Yes      |
   | func (closure)    | Yes        | Yes             | Yes (#54)    | Yes      |

   *Without captures (#54), a Body can only access globals. With captures,
   a Body becomes a zero-argument closure:

   #+begin_src til
   mut total := 0
   data := vec("I64", 1, 2, 3)

   // Body with capture list
   summer := captures mut total, data { for item : I64 in data { total = total.add(item) } }
   summer()
   // total is now 6
   #+end_src

   A Body is to ~proc()~ what a literal is to a named constant — the
   anonymous, inline version. The distinction is that a Body has no
   function overhead (no frame push/pop in the interpreter, no C function
   call in codegen) — it executes in the caller's scope.

*** Use cases

   #+begin_src til
   // Deferred execution (like Go's defer)
   cleanup := { file.close() }
   // ... do work ...
   cleanup()

   // Callbacks and hooks
   Widget := struct {
       mut on_click: Body
       mut on_hover: Body
   }

   // Builder patterns
   with_transaction := proc(db: Database, action: Body) {
       db.begin()
       action()
       db.commit()
   }
   with_transaction(db, { db.insert(record1); db.insert(record2) })

   // Test setup/teardown
   run_test := proc(name: Str, setup: Body, test_body: Body, teardown: Body) {
       println("Running:", name)
       setup()
       test_body()
       teardown()
   }
   #+end_src

*** Scope semantics: inline vs isolated

   Open question: when a Body executes, does it share the caller's scope
   (like an inlined block) or get its own scope (like a function call)?

   - *Option A: Inline* — shared scope (how bare ~{ }~ blocks work today).
     Simple, powerful, zero overhead. But passing a Body as argument is
     weird — which scope does it execute in?
   - *Option B: Isolated* — own scope, captures only (#54). Safe and
     predictable for stored/passed bodies. But needs captures to be useful.
   - *Option C: Both* — bare ~{ }~ blocks are inline (current behavior),
     Body values (stored, passed, returned) are isolated and require
     captures. This is probably the right answer.

*** Comparison

   | Language   | Block-as-value          | Notes                              |
   |------------+-------------------------+------------------------------------|
   | Ruby       | ~{ ... }~ / ~do...end~  | Blocks and Procs are first-class   |
   | Smalltalk  | ~[ ... ]~               | Blocks are objects                 |
   | Swift      | ~{ () -> Void in ... }~ | Closures (always have captures)    |
   | Kotlin     | ~{ ... }~ (lambda)      | Trailing lambda syntax             |
   | Rust       | closure only            | No bare blocks as values           |
   | Go         | ~func() { ... }()~      | Must wrap in function              |
   | TIL        | ~Body = { ... }~        | Simpler than closure               |

   Ruby is the closest precedent. Ruby blocks can be passed to methods
   implicitly (~yield~) or explicitly (~&block~), and ~Proc.new~ wraps a
   block into a first-class value.

** OPEN #54 Explicit capture syntax (closures)
  :DEPENDENCIES: #61
  :DEPENDENCIES_OF: #12 #13
  Add syntax for listing captured variables from outer scopes, both for
  function definitions and anonymous blocks. Based on rstil/doc/first_class.org.
  Split from #17 (global variable access, now fixed).

  Closures = lambdas (#61) + captures. #61 provides anonymous inline
  functions (lambdas); this issue adds the ~captures~ keyword that turns
  a lambda into a closure by binding outer-scope variables.

  til requires *explicit capture lists* — if a function references an outer
  variable, it MUST be listed in the capture list. This makes dependencies
  visible and refactoring mechanical. Inspired by Jai (Jonathan Blow).

*** Capture syntax

   The ~captures~ keyword appears after the parameter list (and before
   ~returns~ if present) in FuncDef form, or after the argument names in
   FuncSig form:

   #+begin_src til
   scale := 10

   // FuncDef form: captures between params and returns
   f := func(a: I64) captures scale returns I64 { return a.mul(scale) }

   // FuncSig form: captures after argument names
   f : func(I64) returns I64 = (a) captures scale { return a.mul(scale) }

   // With a named type alias
   Transformer := func(I64) returns I64
   f : Transformer = (a) captures scale { return a.mul(scale) }
   #+end_src

*** Capture modifiers

   Captures follow the same rules as function parameters — no new rules
   to learn:

   | Capture syntax        | Semantics                     | Same as param mode |
   |-----------------------+-------------------------------+--------------------|
   | ~captures x~          | Const by reference (default)  | ~x: T~             |
   | ~captures mut x~      | Mutable reference             | ~mut x: T~         |
   | ~captures own x~      | Ownership transfer (moved in) | ~own x: T~         |

   #+begin_src til
   mut total := 0
   data := vec("I64", 1, 2, 3)

   // mut capture: closure can modify total
   data.for_each(func(x: I64) captures mut total { total = total.add(x) })

   // own capture: moves data into the closure, caller can't use it after
   consume := proc() captures own data { data.clear() }
   #+end_src

*** Block captures (scope restriction)

   Even without making a function, a capture list on a bare code block
   limits what it can see — an "insurance policy" that the block only
   touches specified state:

   #+begin_src til
   mut total := 0
   mut count := 0
   data := vec("I64", 1, 2, 3)

   // Only total and data are accessible inside this block
   captures total, data {
       for item : I64 in data {
           total = total.add(item)
       }
   }
   // count is guaranteed untouched
   #+end_src

*** Jai-inspired refactoring progression

   Each step is mechanical. The capture list tells you exactly what needs
   to become a parameter when you promote to a pure function:

   #+begin_src til
   // Step 1: inline code
   total = total.add(x)

   // Step 2: block with captures (limit scope, verify dependencies)
   captures total, x {
       total = total.add(x)
   }

   // Step 3: promote to lambda (add params, same captures)
   adder := func(a: I64) captures total returns I64 { return total.add(a) }

   // Step 4: promote to pure function (captures become params, no closure)
   adder := func(a: I64, b: I64) returns I64 { return a.add(b) }
   #+end_src

*** Why explicit captures

   Implicit captures (Python, JavaScript, Rust) feel magical:
   - *Reading*: you see ~scale~ and have to hunt for where it's defined.
     Is it a global? A local three scopes up? A typo for a parameter?
   - *Refactoring*: extracting a block into a function means figuring out
     ALL the implicit dependencies. Miss one and it breaks.
   - *Inlining*: moving a closure back inline means understanding what it
     captures to know where it can safely go.

   Explicit captures solve all three:
   - *Reading*: ~captures scale~ tells you exactly what comes from outside.
   - *Refactoring*: captures become parameters. Mechanical transformation.
   - *Inlining*: captures tell you exactly what the block depends on.

*** Implementation

   - Parser: recognize ~captures~ keyword in func defs (between params
     and returns/body) and bare blocks (before ~{~)
   - AST: store capture list on NODE_FUNC_DEF and NODE_BODY
   - Typer: validate captures exist in outer scope, enforce modifier rules
   - Interpreter: bind captured variables in call_scope from definition scope
   - Codegen: emit closure structs or pass captures as hidden parameters

** OPEN #25 Support for the 4 main operative systems
  :DEPENDENCIES: #12
  til currently only builds and runs on GNU/Linux. Should support the 4
  main operative systems, in priority order:
  1. GNU/Linux/BSD (current, done)
  2. TempleOS
  3. macOS
  4. Windows

  This is not just "til builds natively on each OS." The goal is full
  cross-compilation: build a til binary for any target from any host.
  The matrix is 3×4 + 1×1 = 13 combinations: Linux, macOS, and Windows
  can each target all 4 platforms, but TempleOS as a host only targets
  itself (cross-compiling from TempleOS would mean replacing gcc/clang,
  which is out of scope).

  ctil (the C bootstrap compiler) stays Linux-only. Once til is
  self-hosting (#12), til replaces ctil and handles all platforms.
  Bootstrap path: ctil on Linux compiles self-hosted til → til can
  then cross-compile itself and any .til program to any target.

  The C codegen path helps: emit C + invoke the target's C compiler
  works for most POSIX targets. But TempleOS breaks this pattern
  (no C compiler), requiring a dedicated HolyC codegen backend.

  The builder must select the appropriate codegen backend per target.

  Common areas affected across platforms:
  - Build system (builder that selects backend per target)
  - FFI / dlsym (platform-specific dynamic loading)
  - System primitives (spawn_cmd, readfile, etc. — POSIX vs platform APIs)
  - GUI mode (raylib on POSIX/Windows, native primitives on TempleOS)
  - Audio (miniaudio on POSIX/Windows, native on TempleOS)

*** TempleOS
   :DEPENDENCIES: #12
   The most interesting and challenging target. TempleOS has no C compiler,
   no POSIX, no libc — just HolyC and its own kernel/graphics/audio APIs.

   Requires a new holybuilder backend that emits HolyC instead of C.
   HolyC is C-like enough that the translation should be mostly mechanical
   (similar types, similar control flow), but differences in calling
   conventions, memory management, and string handling will need attention.

   For mode gui: raylib doesn't exist on TempleOS, so we need to recreate
   raylib's interface using TempleOS graphics primitives (GrBlot, GrLine,
   GrRect, etc.) and audio primitives. This is likely easier than most
   people would expect — TempleOS has direct framebuffer access and simple
   audio APIs, and raylib's 2D API surface is not that large. The hard
   part is the impedance mismatch (TempleOS is cooperative multitasking,
   640x480 16-color default, etc.), not the drawing primitives themselves.

   Running the same mode gui program (e.g. music.til) on both GNU/Linux
   via raylib AND TempleOS via native HolyC primitives would be a serious
   language flex — proving til programs are truly platform-independent.

   For mode server: TempleOS docker containers could be extremely small
   (no libc, no userland, minimal kernel), making til a compelling choice
   for lightweight server deployments on TempleOS-based containers.

   Self-hosting (#12) is a hard dependency: TempleOS has no C compiler,
   so til must be able to compile itself to bring its own compiler onto
   the platform. There's no way to cross-compile C for TempleOS from Linux.

   Can test from Linux using QEMU with a TempleOS VM image. Copy til
   files in via a virtual disk, run them, capture output. This is the
   standard way to develop for TempleOS from a modern host.

*** macOS
   Closest to the current GNU/Linux target. All C code (ext.c,
   dispatch.c) is already POSIX-portable — fork, execl, waitpid, dlopen,
   dlsym all work the same on macOS. No cross-compilation needed: ship
   source, user runs `make` with clang natively.

   Makefile changes needed:
   - `-ldl` not needed on macOS (dlopen is in libc)

** OPEN #22 Operator syntax: desugar operators to method calls
  :DEPENDENCIES: #12
  Syntactic sugar so a + b desugars to a.add(b), a < b to a.lt(b), etc.
  Purely a parser transform — no changes needed in typer, interpreter, or
  codegen. The parser rewrites operator expressions into method call AST
  nodes before anything downstream sees them.

  Operator → method mapping:
  - Comparison: < → lt, <= → lte, > → gt, >= → gte, == → eq, != → neq
  - Arithmetic: + → add, - → sub, * → mul, / → div, % → mod

  Precedence (low to high):
  - == != < <= > >=
  - + -
  - * / %

  Parenthesized expressions (a + b) * c override precedence. The parser
  must disambiguate grouping parens from function call parens: ( after an
  identifier/field-access is a call, otherwise it's grouping.

  Parsing approach: Pratt parsing (precedence climbing) or one function
  per precedence level (mutually recursive descent). Both work; Pratt is
  more compact and easier to extend.

  The cmp/initer synergy makes this especially nice for comparisons:
  implement cmp once, the initer auto-generates eq, neq, lt, gt, lte, gte,
  and operator desugaring gives you <, <=, >, >=, ==, != for free. If you
  override eq with a faster implementation, neq (and !=) automatically
  benefits since neq is generated as a.eq(b).not().

*** Phase 1: Comparison operators
   < <= > >= == !=
   All same precedence, binary, no chaining ambiguity.
   Desugars: a < b → a.lt(b), a != b → a.neq(b), etc.

*** Phase 2: Arithmetic operators
   + - * / %
   Introduces real precedence: * / % bind tighter than + -.
   Desugars: a + b * c → a.add(b.mul(c)).
   Requires grouping parentheses support.

   Also: ~++~ / ~--~ as unary sugar for inc/dec (methods added in #34 Phase 2).
   ~i++~ → ~i = i.inc()~, ~i--~ → ~i = i.dec()~.
   ~+=~ / ~-=~ as compound assignment sugar: ~i += 1~ → ~i = i.add(1)~.
