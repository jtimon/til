* OPEN #13 Test runner in til: parallel execution, mode comparison
  :DEPENDENCIES: #5
  Replace tests.sh with a til-based test runner that runs tests in parallel,
  compares interpreter vs codegen output, and reports timing. Inspired by
  rstil/src/tests.til which spawns processes, polls completion, and diffs
  outputs across multiple execution modes.

  Currently tests.sh is a linear bash script that runs each test sequentially
  in both modes. A til-based runner would be faster (parallel), more capable
  (output diffing, timing, better reporting), and serve as a real-world
  exercise for Vec, Str, and the new pointer primitives.

** System primitives needed
   New ext_func/ext_proc in core.til + ext.c + dispatch.c:

   - spawn_cmd(cmd: Str) returns I64 — fork+exec, return child PID
   - check_cmd_status(pid: I64) returns I64 — non-blocking waitpid,
     returns exit code or -1 if still running
   - sleep(ms: I64) — millisecond sleep for poll loops
   - run_cmd(cmd: Str) returns I64 — synchronous popen, captures stdout
     into a Str (via mut param or return)
   - readfile(path: Str) returns Str — read file contents
   - writefile(path: Str, content: Str) — write file contents

   Also useful: clock_ms() for timing, getenv(name: Str) returns Str.

** Test runner design
   A til program (src/tests.til or similar) that:

   1. Collects test file paths (src/test/*.til)
   2. For each test, spawns both interpret and run modes in parallel
   3. Polls for completion, collects exit codes and output
   4. Compares outputs between modes (interpret vs run must match)
   5. Reports pass/fail with diffs on mismatch, timing per test
   6. Configurable parallelism (e.g. 4-8 concurrent processes)

   The old rstil runner used .out files for expected output. Simpler
   approach for ctil: just assert both modes produce identical output
   and exit code 0 (or expected non-zero for error tests).

** Why this blocks self-hosting
   The self-hosted compiler needs to pass the same tests as the C
   implementation. A proper test runner that compares outputs across
   modes is essential for validating correctness during the rewrite.

* OPEN #12 Self-hosting: rewrite ctil in til
  :DEPENDENCIES: #1 #2 #5 #8 #13
  Rewrite the til compiler/interpreter in til itself. This is the ultimate
  test of the language — if til can compile itself, it's expressive enough
  for real systems programming.

  The current C implementation is the reference: lexer, parser, initer,
  typer, interpreter, ccodegen, precomp, scavenger (~6000 lines of C).
  Self-hosting means reimplementing all of these in til, using til's own
  features (structs, enums, ownership, FFI for I/O).

  Depends on most open issues:
  - #1 (consistency): clean backends make porting easier
  - #2 (ref): needed for efficient arg passing without clone overhead
  - #5 (pointers/Vec): FIXED — token lists, AST children, string buffers
  - #8 (variadic): needed for format/println in the self-hosted version

  Does NOT depend on #10 (REPL) or #11 (GUI mode).

* OPEN #11 Mode GUI: raylib integration via FFI
  :DEPENDENCIES: #4
  Add a `mode gui` that auto-includes ext_func/ext_proc declarations for
  raylib, so users can build graphical applications. Similar to how `mode cli`
  calls main(), `mode gui` would call a main loop with raylib setup/teardown.

  Now that #4 (FFI) is done, the mechanism exists: ctil compiles and links
  user .c files, and the interpreter can dlopen shared libraries. For mode gui:
  - Auto-include a til file with raylib ext_func/ext_proc signatures
    (InitWindow, CloseWindow, BeginDrawing, DrawText, etc.)
  - Auto-link raylib (-lraylib) during compilation
  - For the interpreter, dlopen libraylib.so and register all functions
  - Potentially use libffi for the interpreter path to avoid manual
    trampoline entries for each raylib function (raylib has ~500 functions)

  Must work for both `bin/ctil run` and `bin/ctil interpret`.

* OPEN #8 Variadic args for funcs, procs, and macros
  Currently `..Type` variadic parameters only work in ext_func and ext_proc
  (format, println, print). Regular funcs, procs, and macros can't use them.

  What's needed:
  - Parser: allow `..Type` in func/proc/macro parameter lists
  - Typer: validate variadic calls, ensure args match the element type
  - Interpreter: collect variadic args into an array-like structure
  - ccodegen: emit variadic C calls or array-based passing

  This blocks several improvements:
  - Desugaring println/print in the typer (the replacement needs to be variadic)
  - User-defined variadic functions (e.g. custom logging, string builders)
  - Variadic macros for compile-time code generation


* WIP #2 ref keyword: non-owning references for returns, declarations, and struct fields
  Currently all returns are owned by the caller (heap-allocated, caller must
  delete). This breaks when a function wants to return one of its args:

    min := func(a: I64, b: I64) returns I64 {
        if a.lt(b) { return a } else { return b }
    }

  Either a or b would be owned twice and deleted twice (double-free).

** FIXED ref returns
   returns ref I64 — the caller does NOT own the returned value and must
   not delete it. ASAP destruction skips ref return values.

    min := func(a: I64, b: I64) returns ref I64 {
        if a.lt(b) { return a } else { return b }
    }

   Typer validates: return value must be a param or ref variable.

** FIXED ref declarations
   Required when binding the result of a ref-returning function:
   ref my_min := min(34, 35). ASAP destruction skips ref variables.

   Typer validates: RHS must be a ref-returning function call or a
   ref/param variable. Cannot pass ref variable to 'own' parameter.

   ASAP lifetime extension: args to ref-returning calls have their
   lifetimes extended to at least the ref variable's last use, preventing
   use-after-free.

   Interpreter: ref from ident uses scope_set_borrowed (shared cell).
   ccodegen: ref decl emits pointer assignment (no malloc).

** ref struct fields
   A ref field stores a pointer to an existing value (not inline).
   clone does not deep-copy it, delete does not recurse into it —
   just copies/ignores the pointer.

   Open question: default initialization. Inline fields use literal defaults
   (x := 0), but ref fields point to external values that may not exist yet
   at construction time. Options include NULL/nil default, requiring the
   field at construction, or something else. TBD.

** args are already ref by default
   Function args are borrowed (ref) unless explicitly marked own.
   No change needed for args.

* OPEN #1 ccodegen/interpreter consistency: eliminate unnecessary special cases
  The ccodegen and interpreter should be as similar and minimal as possible.
  The typer/initer should do more AST rewriting so backends can be dumb and uniform.

  Architecture has 3 layers:
  - ccore.h/c (168 lines): raw C types (Str, I64, U8, Bool) + raw functions
    with natural C semantics (value params, raw int returns)
  - ext.h/c (131 lines): til_* typedefs + pointer-boxing wrappers for
    ccodegen's calling convention (all args/returns are heap pointers)
  - dispatch.h/c (157 lines): interpreter's ext_function_dispatch, calls
    ccore directly via strcmp chain (47 functions)

  ccodegen (959 lines) links ext.c + ccore.c and emits calls to til_* wrappers.
  The interpreter (725 lines) calls raw ccore functions via dispatch.c.
  core.til uses direct ext_func namespace fields. println/print/format live in
  ext.c as variadic C functions.

  The typer (1650 lines) does all type-aware AST rewriting: clone insertion at
  4 copy sites, ASAP delete insertion, hoisting, delete before reassignment.
  The initer (766 lines) auto-generates clone/delete/enum methods as AST.
  Both backends are largely "dumb" — they execute/emit the pre-rewritten AST.

  Remaining differences fall into 3 categories: actionable, low-priority, and
  acceptable (inherent to having an interpreter vs a C emitter).

** FIXED Enum clone/delete/to_str are real til for ALL enums
   The initer now generates real til-level clone/delete/to_str for all
   enums (both simple and payload). Tagged union structs used in ccodegen
   for all enums. No-payload variants use bare field access syntax
   (Color.Red, Token.Eof — no parens needed).

   Simple enum variants are I64 namespace literals with copy-on-access
   in the interpreter. Payload enum no-payload variants are zero-arg
   ext_func constructors auto-called on field access.

   Clone: payload uses is_/get_/constructor chain, simple uses eq chain.
   Delete: same real til proc for all — free() handles payloads.
   Eliminated ~120 lines of C generation from ccodegen, clone/delete
   dispatch from dispatch.c.

** FIXED Enum method dispatch moved from eval_call to dispatch.c
   Both simple and payload enum methods (constructor, eq, clone, delete,
   is_Variant, get_Variant) were hard-coded inline in eval_call (~65 lines).
   Moved to enum_method_dispatch() in dispatch.c. eval_call now detects
   "this is an enum" and delegates in ~5 lines.

** FIXED enum_has_payloads and variant helpers shared via ast.h
   enum_has_payloads was duplicated in interpreter.c and ccodegen.c.
   Moved to ast.h as static inline, along with enum_variant_tag and
   enum_variant_type. Both backends and dispatch.c use the shared versions.

** FIXED interpreter: dispatch.c strcmp chain replaced with Map
   Was linear scan of 47 string comparisons. Now a Map of DispatchFn
   function pointers, lazy-initialized on first call. Handlers generated
   via H1/H2/HDEL/HCLONE macros. Special cases (free, println, print,
   format) kept as explicit handler functions.

** ccodegen: declaration emission branches on RHS type
   3 branches: struct constructor (malloc + fields), fcall/str (pointer-assign),
   literal/ident (malloc + deref-copy). After hoisting, most RHS are NODE_IDENT
   or NODE_FCALL, so branching is minimal. Accepted.

** Acceptable: struct constructor emission
   Interpreter (eval_call 352-384) allocates StructInstance, iterates fields,
   move semantics for ident args. ccodegen (emit_ctor_fields 203-259) emits
   malloc + field-by-field C assignment with own/inline/nested handling.
   Logic is isomorphic but inherently different — one runs, one emits C.

** Acceptable: ownership tracking
   Interpreter manually tracks ownership via Value types, scope_set_borrowed
   vs scope_set_owned, free_value on reassignment, val_none guards. ccodegen
   delegates to C calling conventions — pointer-assign for compound types,
   deref-assign for scalars, typer-inserted delete calls handle cleanup.
   Fundamentally different execution models.

** Acceptable: namespace field initialization
   Both scan struct/enum defs for non-func namespace fields (interpreter:
   interpreter_init_ns → ns_fields[] array; ccodegen: emit_ns_inits → C
   globals). Could extract shared field iterator but minimal benefit.

** FIXED ccodegen: free() and exit() special cases
   Moved to ext.c as til_free() and til_exit(). Both backends use generic
   call paths — no special-cased emission or dispatch.

** FIXED ccodegen: i64_to_str, u8_to_str, str_eq emitted inline
   Raw implementations in ccore.c. ccodegen links via ext.c wrappers.
   Interpreter calls ccore functions directly via dispatch.c.

** FIXED bool_to_str was a needless builtin
   Replaced with pure til: if b { return "true" } else { return "false" }.
   Removed ext_func, C wrapper from ccodegen, and strcmp case from interpreter.

** FIXED type aliases and PascalCase ext naming
   Added typedefs in ext.h: til_I64, til_U8, til_Bool. PascalCase function
   names (til_I64_add, til_U8_eq) match ccodegen's natural emission.

** FIXED U8 stored as I64 in interpreter
   Interpreter Value union now has a .u8 field (til_U8 *). Clean one-liners.

** FIXED trivial ext_func wrappers in core.til
   Direct ext_func namespace declarations. No wrapper generated in C,
   call sites go straight to ext.c. Eliminated ~50 lines of core.til boilerplate.

** FIXED println/print/format moved to ext.c
   Was type-aware special cases in both backends. Now variadic C functions
   in ext.c. ccodegen emits til_println(n, args...). Interpreter simplified
   to printf loop.

** FIXED ccodegen: struct constructors
   Replaced per-struct til_X_new() constructor functions with malloc +
   field-by-field assignment via emit_ctor_fields(). Handles own fields
   (pointer-assign), inline compound fields (deref + free wrapper), and
   nested constructors (recursive). Fieldless structs just malloc.
   Interpreter uses positional args (typer fills all fields with defaults).

** FIXED ccodegen: deref-copy and reassignment
   Typer inserts Type.delete() before compound-type variable and field
   reassignment. ccodegen uses pointer-assign for compound NODE_ASSIGN,
   deref + free-wrapper for inline NODE_FIELD_ASSIGN. Hoisting skips
   compound-type locals and inline compound constructor/field-assign args.
   Interpreter: free_value before overwriting, ext dispatch nulls own-arg
   cells, evaluates RHS before freeing old value.

** FIXED interpreter: value_deep_copy removed
   Move semantics: NODE_IDENT RHS transfers ownership (copies value, nulls
   source cell). ASAP deletes of moved values are no-ops. Function args
   use scope_set_borrowed (shared cell), matching ccodegen pointer-passing.

** FIXED namespace call path
   Full named-arg resolution in both namespace and regular call paths.
