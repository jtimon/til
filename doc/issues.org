* OPEN #22 Operator syntax: desugar operators to method calls
  :DEPENDENCIES: #12
  Syntactic sugar so a + b desugars to a.add(b), a < b to a.lt(b), etc.
  Purely a parser transform — no changes needed in typer, interpreter, or
  codegen. The parser rewrites operator expressions into method call AST
  nodes before anything downstream sees them.

  Operator → method mapping:
  - Comparison: < → lt, <= → lte, > → gt, >= → gte, == → eq, != → neq
  - Arithmetic: + → add, - → sub, * → mul, / → div, % → mod

  Precedence (low to high):
  - == != < <= > >=
  - + -
  - * / %

  Parenthesized expressions (a + b) * c override precedence. The parser
  must disambiguate grouping parens from function call parens: ( after an
  identifier/field-access is a call, otherwise it's grouping.

  Parsing approach: Pratt parsing (precedence climbing) or one function
  per precedence level (mutually recursive descent). Both work; Pratt is
  more compact and easier to extend.

  The cmp/initer synergy makes this especially nice for comparisons:
  implement cmp once, the initer auto-generates eq, neq, lt, gt, lte, gte,
  and operator desugaring gives you <, <=, >, >=, ==, != for free. If you
  override eq with a faster implementation, neq (and !=) automatically
  benefits since neq is generated as a.eq(b).not().

** Phase 1: Comparison operators
   < <= > >= == !=
   All same precedence, binary, no chaining ambiguity.
   Desugars: a < b → a.lt(b), a != b → a.neq(b), etc.

** Phase 2: Arithmetic operators
   + - * / %
   Introduces real precedence: * / % bind tighter than + -.
   Desugars: a + b * c → a.add(b.mul(c)).
   Requires grouping parentheses support.

* OPEN #17 Global/outer-scope variable access from functions and procs
  :DEPENDENCIES_OF: #12 #13
  Variables declared in script-mode root scope (or any outer scope) are
  invisible at runtime inside function/proc bodies. The typer passes
  (TypeScope chains via parent pointers and finds the variable), but the
  interpreter crashes or errors.

  Reproduction: src/test/globals.til, src/test/globals_cli.til

  func case: "error: undefined variable 'x'" — clean error from NODE_IDENT
  NULL check. proc case: SEGFAULT — NODE_DECL move-semantics path has no
  NULL check on scope_get result.

** Root cause

   expr_uses_var() in typer.c (line 1239) returns 0 for NODE_FUNC_DEF,
   meaning the ASAP pass doesn't see variable references inside nested
   function/proc bodies. When x is declared at top level and referenced
   only inside a proc body, ASAP thinks x is unused and inserts
   I64.delete(x) right after the declaration. When the proc later runs
   and tries to access x, the value has already been freed/nulled.

   func case: I64.clone(x) inside the function body tries to clone x
   whose value is val_none → clean "undefined variable" error (from
   precomp, which also can't find x in its limited scope).

   proc case: I64.clone(x) dereferences v.i64 which is NULL → SEGFAULT.
   The NODE_DECL move-semantics path also lacks a NULL check on
   scope_get result, which would be a secondary crash path.

   Secondary issue: precomp creates its own interpreter scope with only
   func/struct/enum defs (no variable decls), then tries to fold pure
   func calls. If the func accesses a global variable, precomp's
   eval_expr hits the error path and exit(1)s before the interpreter
   even runs.

** DONE Phase 1: Fix global variable access

   1. Added expr_used_in_nested_func() in typer.c: scans for variable
      references inside nested func/proc bodies (excluding shadowed params).
      ASAP pass skips captured variables entirely (no delete inserted),
      so they survive until scope exit — needed because the capturing
      function may run after the enclosing body finishes (e.g. cli mode main).
   2. Added NULL check in interpreter.c NODE_DECL move-semantics path.
   3. Fixed precomp: added func_uses_unknown_globals() to skip folding
      func calls whose body references variables not in the precomp scope.
      Also made track_literal() populate the precomp interpreter scope
      with known constant values (not just the known map).
   4. Fixed ccodegen: top-level variable declarations are emitted as
      file-scope C globals (before function definitions) so they're
      accessible from emitted til functions/procs. This is necessary
      because til functions are emitted as separate C functions, and C
      doesn't support nested functions capturing locals — so top-level
      vars must be file-scope for functions to reference them. The
      values are then assigned inside main() where initialization runs.
      In cli mode, globals are initialized at the start of main()
      before the user's main body.
      Note: ref declarations are skipped from file-scope globals since
      they are pointer aliases (local to main), not owned values that
      other functions need to access.

   Tests: src/test/globals.til (script mode), src/test/globals_cli.til (cli mode).

** Phase 2: Explicit capture syntax

   Add syntax for listing captured variables from outer scopes, both for
   function definitions and anonymous blocks. Based on rstil/doc/first_class.org.

   Function captures (closures):
     scale := 10
     f := func(a: I64) returns I64 [scale] { return a.mul(scale) }

   Capture modifiers:
     [x]      — const reference (default)
     [mut x]  — mutable reference
     [copy x] — snapshot (clone at capture time)
     [own x]  — ownership transfer

   Block captures (restrict access):
     [total, data] {
         // can only see total and data, not other locals
     }

   This enables Jai-inspired mechanical refactoring:
   inline code -> block with captures -> lambda -> pure function

   Implementation:
   - Parser: recognize [ident, ...] before { in func defs and bare blocks
   - AST: store capture list on NODE_FUNC_DEF and NODE_BODY
   - Typer: validate captures exist in outer scope, enforce modifier rules
   - Interpreter: bind captured variables in call_scope from definition scope
   - Codegen: emit closure structs or pass captures as hidden parameters

  Fix: extend collect_refs in scavenger.c to recognize dyn_call*,
  dyn_has_method, array, and vec calls. For array/vec, mark
  "TypeName.size" and "TypeName.clone" and "TypeName.delete" as
  referenced based on the type_name string literal arg.

* OPEN #8 Variadic args for funcs, procs, and macros
  Currently `..Type` variadic parameters only work in ext_func and ext_proc
  (format, println, print). Regular funcs, procs, and macros can't use them.

  Three phases:

** DONE Phase 1: General variadic args for user-defined functions
   Support `..Type` in regular func/proc/macro parameter lists, any type.
   Example: variadic_and := func(args: ..Bool) returns Bool {}

   Internally, variadic args are stored as a til Array. At the call site, an
   Array is built from the passed variadic arguments. Inside the definition,
   the parameter has Array type and the user accesses elements via the Array
   interface:

     print_points := proc(points: ..Point) {
       i := 0
       while i.lt(points.len()) {
         ref p: Point = points.get(i)
         println(p.x, ", ", p.y)
         i = i.add(1)
       }
     }
     print_points(a, b, c)  -- builds Array[Point] from a, b, c

   Ownership: variadic params are implicitly own, non-mut. No need to write
   `own` or `mut` on `..T` params — reject those keywords if present. Callers
   don't write `own` either; ownership transfer is implicit for variadic args.
   Zero and single variadic args are allowed and still use an Array (cap=0/1).

   Parameter ordering: no positional params allowed after the variadic param.
   Only params with defaults (which use named syntax at call site) are allowed
   after it. Without this rule, ambiguous cases would be hard to resolve.

   Implementation: the typer desugars variadic calls into Array.new + Array.set
   statements before the original call, replacing the variadic args with a
   single `_vaN` temp variable. Both interpreter and codegen then see normal
   Array code — no special variadic handling needed in either backend.

   - Parser: rejects own/mut on `..T`, rejects positional params after variadic
   - Typer: `desugar_variadic_calls` transforms call sites before hoisting;
     variadic param typed as Array in function scope; `param_owns[vi] = true`
   - Interpreter: simplified — just binds args positionally (desugaring done)
   - Codegen: emits `til_Array` param type; call sites desugared in AST

** DONE Phase 2: main proc CLI args (including variadic)
   main := proc(args: ..Str) works in both compiled and interpreted modes.
   Needed for #12 (self-hosting) and #13 (test runner in til).

   CLI args (argv[1..]) are validated and parsed at runtime:
   - Arg count checked against param count (or minimum for variadic)
   - Supported param types: Str (identity), I64 (atoll), U8 (range-checked),
     Bool ("true"/"false"). Struct parsing deferred.
   - Variadic ..Str builds Array[Str] from remaining args
   - main := proc() rejects any CLI args

   ctil forwards argv[3..] to both interpret and run commands.
   Parse helpers (til_cli_parse_i64/u8/bool) in ext.c, shared by both paths.

** Phase 3: Variadic args in FFI functions
   Related to #4. Support `..Type` in ext_func/ext_proc parameter lists
   for user-defined companion .c files.

* WIP #2 ref keyword: non-owning references for returns, declarations, and struct fields
  Currently all returns are owned by the caller (heap-allocated, caller must
  delete). This breaks when a function wants to return one of its args:

    min := func(a: I64, b: I64) returns I64 {
        if a.lt(b) { return a } else { return b }
    }

  Either a or b would be owned twice and deleted twice (double-free).

** FIXED ref returns
   returns ref I64 — the caller does NOT own the returned value and must
   not delete it. ASAP destruction skips ref return values.

    min := func(a: I64, b: I64) returns ref I64 {
        if a.lt(b) { return a } else { return b }
    }

   Typer validates: return value must be a param or ref variable.

** FIXED ref declarations
   Required when binding the result of a ref-returning function:
   ref my_min := min(34, 35). ASAP destruction skips ref variables.

   Typer validates: RHS must be a ref-returning function call or a
   ref/param variable. Cannot pass ref variable to 'own' parameter.

   ASAP lifetime extension: args to ref-returning calls have their
   lifetimes extended to at least the ref variable's last use, preventing
   use-after-free.

   Interpreter: ref from ident uses scope_set_borrowed (shared cell).
   ccodegen: ref decl emits pointer assignment (no malloc).

   clone does not deep-copy it, delete does not recurse into it —
   just copies/ignores the pointer.

* OPEN #1 ccodegen/interpreter consistency: eliminate unnecessary special cases
  The ccodegen and interpreter should be as similar and minimal as possible.
  The typer/initer should do more AST rewriting so backends can be dumb and uniform.

  Architecture has 3 layers:
  - ccore.h/c (168 lines): raw C types (Str, I64, U8, Bool) + raw functions
    with natural C semantics (value params, raw int returns)
  - ext.h/c (131 lines): til_* typedefs + pointer-boxing wrappers for
    ccodegen's calling convention (all args/returns are heap pointers)
  - dispatch.h/c (157 lines): interpreter's ext_function_dispatch, calls
    ccore directly via strcmp chain (47 functions)

  ccodegen (959 lines) links ext.c + ccore.c and emits calls to til_* wrappers.
  The interpreter (725 lines) calls raw ccore functions via dispatch.c.
  core.til uses direct ext_func namespace fields. println/print/format live in
  ext.c as variadic C functions.

  The typer (1650 lines) does all type-aware AST rewriting: clone insertion at
  4 copy sites, ASAP delete insertion, hoisting, delete before reassignment.
  The initer (766 lines) auto-generates clone/delete/enum methods as AST.
  Both backends are largely "dumb" — they execute/emit the pre-rewritten AST.

  Remaining differences fall into 3 categories: actionable, low-priority, and
  acceptable (inherent to having an interpreter vs a C emitter).

** Acceptable: struct constructor emission
   Interpreter (eval_call 352-384) allocates StructInstance, iterates fields,
   move semantics for ident args. ccodegen (emit_ctor_fields 203-259) emits
   malloc + field-by-field C assignment with own/inline/nested handling.
   Logic is isomorphic but inherently different — one runs, one emits C.

** Acceptable: ownership tracking
   Interpreter manually tracks ownership via Value types, scope_set_borrowed
   vs scope_set_owned, free_value on reassignment, val_none guards. ccodegen
   delegates to C calling conventions — pointer-assign for compound types,
   deref-assign for scalars, typer-inserted delete calls handle cleanup.
   Fundamentally different execution models.

** Acceptable: namespace field initialization
   Both scan struct/enum defs for non-func namespace fields (interpreter:
   interpreter_init_ns → ns_fields[] array; ccodegen: emit_ns_inits → C
   globals). Could extract shared field iterator but minimal benefit.

** FIXED ccodegen: free() and exit() special cases
   Moved to ext.c as til_free() and til_exit(). Both backends use generic
   call paths — no special-cased emission or dispatch.

** FIXED ccodegen: i64_to_str, u8_to_str, str_eq emitted inline
   Raw implementations in ccore.c. ccodegen links via ext.c wrappers.
   Interpreter calls ccore functions directly via dispatch.c.

** FIXED bool_to_str was a needless builtin
   Replaced with pure til: if b { return "true" } else { return "false" }.
   Removed ext_func, C wrapper from ccodegen, and strcmp case from interpreter.

** FIXED type aliases and PascalCase ext naming
   Added typedefs in ext.h: til_I64, til_U8, til_Bool. PascalCase function
   names (til_I64_add, til_U8_eq) match ccodegen's natural emission.

** FIXED U8 stored as I64 in interpreter
   Interpreter Value union now has a .u8 field (til_U8 *). Clean one-liners.

** FIXED trivial ext_func wrappers in core.til
   Direct ext_func namespace declarations. No wrapper generated in C,
   call sites go straight to ext.c. Eliminated ~50 lines of core.til boilerplate.

** FIXED println/print/format moved to ext.c
   Was type-aware special cases in both backends. Now variadic C functions
   in ext.c. ccodegen emits til_println(n, args...). Interpreter simplified
   to printf loop.

** FIXED ccodegen: struct constructors
   Replaced per-struct til_X_new() constructor functions with malloc +
   field-by-field assignment via emit_ctor_fields(). Handles own fields
   (pointer-assign), inline compound fields (deref + free wrapper), and
   nested constructors (recursive). Fieldless structs just malloc.
   Interpreter uses positional args (typer fills all fields with defaults).

** FIXED ccodegen: deref-copy and reassignment
   Typer inserts Type.delete() before compound-type variable and field
   reassignment. ccodegen uses pointer-assign for compound NODE_ASSIGN,
   deref + free-wrapper for inline NODE_FIELD_ASSIGN. Hoisting skips
   compound-type locals and inline compound constructor/field-assign args.
   Interpreter: free_value before overwriting, ext dispatch nulls own-arg
   cells, evaluates RHS before freeing old value.

** FIXED interpreter: value_deep_copy removed
   Move semantics: NODE_IDENT RHS transfers ownership (copies value, nulls
   source cell). ASAP deletes of moved values are no-ops. Function args
   use scope_set_borrowed (shared cell), matching ccodegen pointer-passing.

** FIXED namespace call path
   Full named-arg resolution in both namespace and regular call paths.
