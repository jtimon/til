** OPEN #149 Add ns_decls container for namespace-section Decls
   Struct and enum definitions carry a ~namespace:~ section whose members
   are parsed as siblings of the instance fields in the same ~body.children~
   Vec, distinguished only by the ~Declaration.is_namespace~ flag.
   ~7 consumer modules read this flag across ~80 sites (builder, typer,
   dispatch, initer, interpreter, scavenger, precomp), each filtering
   body children with ~if dd.is_namespace { ... }~ or its negation.

   This issue adds a dedicated ~ns_decls : Vec<Declaration>~ field next
   to the existing (dead-code) ~fields~ Vec on ~StructDefData~, routes
   namespace Decls there during parsing, and migrates readers off the
   flag. Done in two phases; phase 2 is blocked on prep work.

*** Phase 1 -- structs (this issue)

    1. Add ~StructDefData.ns_decls : Vec<Declaration>~. Parser
       ~parse_struct_def~ (src/self/parser.til:308-333) keeps pushing
       the full Decl Expr into ~body.children~ with ~is_namespace=true~
       (dual-write), AND pushes a cloned ~Declaration~ into
       ~sdd.ns_decls~ with ~default_value~ pointing to the FuncDef
       child (same pattern step 1 of ~#147~ used for instance fields).
       No readers migrated yet -- pure additive commit.

    2. Migrate reader sites one module at a time (one commit per
       module, ~make test~ green at each step). Readers change from
       ~for i in 0..body.child_count() { ... if dd.is_namespace { ... } }~
       to ~for i in 0..sdd.ns_decls.len() { ref dd = sdd.ns_decls.get(i) ... }~,
       then separately from ~if dd.is_namespace.not() { ... process instance field }~
       to the same loop with the flag check removed. Modules in
       approximate dependency order:
       - ~src/self/initer.til~ (~19 reads)
       - ~src/self/typer.til~ (~19 reads)
       - ~src/self/builder.til~ (~26 reads)
       - ~src/self/dispatch.til~ (~8 reads)
       - ~src/self/interpreter.til~ (~7 reads)
       - ~src/self/scavenger.til~ (~7 reads)
       - ~src/self/precomp.til~ (~1 read)

    3. Parser stops pushing namespace Decls into ~body.children~.
       Drop ~Declaration.is_namespace~ reads/writes on the struct
       side (enum parser still sets it -- see phase 2).

*** Phase 2 -- enums (blocked on EnumDef prep)

    ~NodeType.EnumDef~ is a bare variant (src/core/ast.til:175) with
    no payload struct, so there is no place to hang ~ns_decls~ on.
    Phase 2 requires:

    - Introduce ~EnumDefData~ payload struct (parallel to
      ~StructDefData~) and change the variant to ~EnumDef:
      EnumDefData~. Every ~is_variant(x, NodeType.EnumDef)~ and
      constructor site is touched.
    - Then apply the same three-step dual-write / migrate / cut-over
      pattern as phase 1 for enum namespace Decls.

    Kept as a phase within this issue for context; a separate issue
    may be spun off once the ~EnumDefData~ prep is scoped.

*** Notes

    - ~Vec<Declaration>~ chosen over ~Vec<Expr>~: readers that today
      do ~field.child(0)~ to reach the FuncDef switch to reading
      ~dd.default_value~ instead. Declaration already grew a
      ~default_value : Expr~ ref in ~#147~ step 1, which fits.
    - Instance-field handling (~StructDefData.fields~) is out of
      scope. That Vec is still write-only dead code; a separate
      cleanup should either populate it for real or drop it.
    - ~is_namespace~ flag stays on ~Declaration~ throughout phase 1
      until step 3. ~#147~ step 6 ("Remove
      ~Declaration.is_namespace~") becomes trivial once both
      phases land.

** OPEN #147 AST/Context redesign: lessons from rstil
   rstil (the previous Rust implementation, at ~../rstil~) was slower and
   leakier than current til, but several of its architectural choices around
   the AST and Context were cleaner than what til carries today. This issue
   catalogs the ideas worth stealing. It is broader than ~#130~ (which tracks
   field-by-field Expr payload shrinking); this issue is about the root
   pattern: rstil uses ~Context~ as a cross-pass memo, til uses AST fields.

*** Key shape differences

   | Struct      | til (current)                                       | rstil                                                 |
   |-------------+-----------------------------------------------------+-------------------------------------------------------|
   | ~Expr~      | 7 fields: data, til_type, struct_name, children,    | 4 fields: node_type, params, line, col                |
   |             | line, col, path                                     |                                                       |
   | ~Declaration~ | 8 fields: name, explicit_type:Str, is_mut,        | 5 fields: name, value_type:ValueType, is_mut,         |
   |             | is_namespace, own_type, field_offset:I32,           | is_own:Bool, default_value:Option<Expr>               |
   |             | field_size:I32, field_struct_def:*Expr              |                                                       |
   | ~Context~   | 3 fields: mode, path_modes, scope                   | ~18 fields: scope_stack, imported_asts,               |
   |             |                                                     | imports_{init,typer,precomp,eval}_done Sets,          |
   |             |                                                     | anon_{struct,enum,func}_counter + maps,               |
   |             |                                                     | precomp state, temp_enum_payload                      |

*** Root observation: Context vs AST as cache

   rstil's Expr is small because Context carries everything derived or
   cross-pass. til grew Expr and Declaration fields whenever a pass needed
   to remember something -- layout results live on Declaration, type-check
   results live on Expr, call-site resolution lives on FCallData. This is
   why til's Expr is 7 fields to rstil's 4, and why Declaration is 8 to 5.

   til's ~Context~ (src/self/modes.til:19) currently holds only Mode/path
   routing and a TypeScope. Everything else that crosses passes is pushed
   onto AST nodes.

*** Declaration redesign (three biggest wins)

   **** ~explicit_type: Str~ -> ~til_type: TilType~ + ~struct_name: Str~

   Today ~explicit_type~ is read 107 times across 10 files. Every read
   does a scope lookup to resolve "is this a primitive, struct, or enum?".
   rstil resolves type classification once in init phase by using a typed
   enum (~ValueType~: TFunction, TType, TCustom(String), TMulti(String)).

   For til: parser sets ~til_type = TilType.Unknown~ and ~struct_name =
   "MyStruct"~ for user-defined annotations; initer promotes ~til_type~ to
   ~TilType.Struct~ / ~TilType.Enum~ by scope lookup. Later passes read
   the classification directly without re-resolving.

   **** ~is_namespace: Bool~ removal

   Today ~ast.til:337~, ~:351~, ~:367~ all filter by
   ~decl.is_namespace.not()~ when iterating struct/enum bodies. rstil keeps
   namespace members in a separate ~NamespaceDef~ struct on ~StructDef~
   / ~EnumDef~. til can do the same by splitting ~StructDefData.fields~
   into ~fields~ + ~ns_fields~ (and matching for enum body layout).
   Iteration loses the filter; parser routes members by ~namespace:~
   section at parse time.

   **** ~field_offset~ / ~field_size~ / ~field_struct_def~ removal

   These LAYOUT-COMPUTATION fields are set by ~initer.til:1676-1677~ (and
   field_struct_def at ~:1661, :1667~) and read by ~interpreter.til:39,
   698-700, 1416, 1466-1480~. They do not belong on an AST node -- they
   are a per-struct-type fact, not per-declaration syntax. rstil computes
   layout in ~EvalHeap~ on demand; til's natural home is a
   ~Context.struct_layouts: Map<Str, StructLayout>~ keyed by struct name.
   ~StructDefData.total_struct_size~ moves to the same place (this
   subsumes the ~#130~ "total_struct_size duplicates offset/size" item).

   After these three changes, Declaration becomes ~{name, til_type,
   struct_name, is_mut, own_type, default_value}~ -- six fields, pure AST.

*** Context enrichment (rstil patterns)

   Fields rstil's Context has and til could add:

   - ~struct_layouts: Map<Str, StructLayout>~ -- removes three Declaration
     fields plus ~StructDefData.total_struct_size~
   - ~anon_{struct,enum,func}_counter~ + ~anon_func_map~ keyed by
     "path:line:col" -- infrastructure for inline anonymous types (see
     ~#61~, ~#93~)
   - ~imports_{init,typer,precomp,eval}_done~ Sets -- explicit per-pass
     memo of which imported files have been walked (relates to ~#116~,
     ~#66~)
   - ~precomp~ state (forin counter, precomputed_heap_values) -- would
     make precomp.til side effects explicit instead of scattered

** OPEN #146 Unify typer desugar/insert passes into a single walk
   ~infer_body~ in typer.til (lines 2489-2495) runs 7 separate passes in
   sequence over every body, each rebuilding ~body.children~ with a full
   deep clone of kept children:
   - ~desugar_set_literals~
   - ~desugar_map_literals~
   - ~desugar_variadic_calls~
   - ~desugar_kwargs_calls~
   - ~hoist_fcall_args~
   - ~insert_field_deletes~
   - ~insert_free_calls~

   Plus ~insert_exit_deletes~ runs from other call sites.

   Consequences:
   - every body pays 7x (or more) the cost of a single rebuild
   - each pass clones every kept child into its own ~new_ch~ (see ~#144~)
   - the clones cascade through nested bodies recursively
   - most of the remaining ASAN leaks (~#109~) trace back to these
     rebuilds

   Desired direction:
   - merge the desugar/insert passes into one walk per body
   - at each statement, apply all relevant transformations before moving on
   - would also interact with ~#144~: a single-pass rebuild lets
     ~take_children~ / move-based reconstruction work without fighting
     seven separate consumers of the same vector

   Related:
   - ~#144~ clone-and-assign in each pass -- one pass would reduce but
     not eliminate the cost
   - ~#109~ the leaks around ~body.children = new_ch~ are dominated by
     the number of times each body is rebuilt

** OPEN #144 Remove clone-and-assign in typer desugar passes
   The typer's desugar passes (insert_free_calls, desugar_variadic_calls,
   etc.) rebuild body.children via body.children = new_ch. The generated C
   does Vec_delete + Vec_clone + Vec_delete for each pass (10 occurrences).

   Done so far (Option A, partial fix):
   - Added ~Vec.move_from(mut self, own other)~ in ~src/core/vec.til~:
     deletes self's existing elements, reallocs self.data if needed, then
     memcpy's other's buffer contents in. Nulls other.count so the
     ~own~-triggered auto-delete on ~other~ frees only the empty buffer.
   - Replaced all 10 ~body.children = new_ch~ sites in ~src/self/typer.til~
     (also the ~case_body.children~ site) with ~body.children.move_from(own
     new_ch)~. This removes the Vec_clone entirely -- each desugar pass
     now does one memcpy of pointer-sized elements instead of calling
     ~Expr.clone()~ per kept child during the final assignment.

   Remaining (Option C, follow-up):
   - The per-item ~new_ch.push(own stmt.clone())~ in each desugar pass
     still deep-clones every kept statement. Eliminating this requires
     detaching ~body.children~ up front, iterating the detached vec, and
     moving kept items (no clone) into ~new_ch~. More invasive since each
     pass currently uses ~ref~ access to ~body.child(i)~ and mutates
     items in place; the transform would need to run over the detached
     vec instead.

   Original note: could use take_children (now safe to use -- source
   data pointer is nulled after move, fix in f0150ec) to avoid deep
   cloning all children on every desugar pass. A previous attempt used
   ~swap_children~ but that was removed in f0150ec because it zeroed
   lifecycle metadata to avoid UAF, which in turn prevented deep-delete
   at program exit and caused bigger leaks. See #146 for the broader
   plan of unifying the passes so the cost of rebuilding is paid once
   per body instead of 7x.

** OPEN #143 Avoid unnecessary heap round-trips for Expr construction
   ~Expr.new~ itself constructs on the stack, but the generated C for
   callers that store the result does a heap round-trip: the return value
   is heap-allocated, copied to the stack, then freed (~Expr x = *_hp;
   free(_hp)~). Similarly, ~try_eval_call~ in precomp.til allocates
   folded constants on the heap via ~malloc~ + ~memcpy~ just to return
   them. Could be avoided if the codegen supported returning structs by
   value without the heap detour.

   Progress (args side):
   - Commit 4392524: typer's ~hoist_to_temp~ now takes ~is_own~; when
     a hoisted arg flows into an own parameter slot the synthesized
     temp's Declaration is marked ~own_type = Own~, so the builder's
     existing own-local path emits it directly on the heap and skips
     the stack-then-~_oa~-malloc-copy wrap at the call site.
   - Commit 8e8763a: ~insert_free_calls~ in typer.til now runs a
     ~promote_own_transferred_locals~ step after
     ~collect_scope_locals~: user-declared locals with
     ~own_transfer >= 0~ get their Decl's ~own_type~ flipped to Own,
     so they're emitted as heap pointers too.
   - This commit: early-exit deletes emitted by ~insert_exit_deletes~
     now use ~arg_is_own=false, call_free=false~ instead of
     ~true, true~. The ~live~ filter ensures these deletes only fire
     for Shallow (stack) locals that haven't been promoted, so
     ~call_free=false~ is the right semantic (fields cleaned, stack
     container left alone for the frame to unwind). With the own bit
     cleared, emit_as_ptr no longer wraps the arg in the ~_oa~ malloc.

   ~_oa = malloc~ wrap sites in boot/til.c: 2472 (pre-fix) -> 69.

   Still TODO: the ~Expr x = *_hp; free(_hp)~ return round-trip
   mentioned in the issue body (owned struct returns), and the final
   handful of ~_oa wraps for values that aren't directly named user
   locals (complex expressions, variadic arg arrays, etc.).

** OPEN #136 Precomp doesn't fold namespace method calls
  ~Dynamic.size()~ is defined in ~dyn.til~ as ~func() returns USize { return 8 }~
  but is not folded by precomp. ~is_func_call~ in ~precomp.til~ (line 150)

  2) remove the hardcoded sizeof stuff in builder
  Workarounds in builder.til until fixed:
  - ~I64.size()~ instead of ~Dynamic.size()~ for pointer-sized map values
  - ~ref mut prog~ assignment leaks (clones instead of pointer copy)
  - Unnecessary helper procs (~register_lookups_prog~, ~register_funcsig_prog~,
    ~emit_global_inits_prog~) that should be inlined once clone leak is fixed
** OPEN #130 AST / ~Expr~ representation cleanup
  The current AST shape is confusing and duplicates state in ways that make
  parser and ownership bugs harder to reason about.

  Current pressure points:
  - ~ExprData~ is still acting as a giant catch-all payload enum
  - node-kind-specific data is split between ~Expr~, ~ExprData~, and
    sometimes child lists
  - that makes ownership/lifetime of payload pointers much harder to see
  - parser code is forced into fragile payload-rebuild patterns around
    ~ExprData.FuncDef~, ~Decl~, and related nodes

  Intended direction:
  - move toward a clearer node model where node-kind payload data lives in
    the node-kind payload itself
  - keep structural subexpressions in children, instead of duplicating
    state between children and payload structs
  - reduce the amount of duplicated / mirrored metadata spread across
    ~Expr~ and embedded payload structs

  Expected payoff:
  - simpler parser code
  - fewer use-after-free / stale-payload bugs during AST rewrites
  - easier future cleanup of clone/delete semantics around AST nodes

  Done so far:
  - DONE Declaration: ~is_ref~ + ~is_own~ bools -> ~own_type: OwnType~ enum
  - DONE Expr: ~is_own_field~ + ~is_ref_field~ bools -> ~field_own_type: OwnType~
  - DONE TypeBinding: ~is_own~ + ~is_ref~ bools -> ~own_type: OwnType~
  - DONE Parser/typer/builder locals: all ~is_own~/~is_ref~/~is_shallow~ bools
    replaced with ~OwnType~ variables
  - DONE ~FCallData.is_proc~: removed (always false, never read)
  - DONE ~TypeBinding.is_proc: I32~ -> ~func_type: FuncType~ with ~FuncType.None~
  - DONE ~Expr.fn_sig~ moved to ~FCallData.fn_sig~ (single source of truth)
  - DONE Expr: ~is_ns_field~ removed (look up ~Declaration.is_namespace~ instead)
  - DONE Expr: ~field_own_type~ removed (look up ~Declaration.own_type~ instead)
  - DONE Expr: ~save_old_delete~ moved to ~AssignData~ payload
  - DONE Expr: ~is_splat~ moved to ~FCallData.is_splat~
  - DONE Expr: ~is_own_arg~ moved to ~FCallData.own_args~ U64 bitmask
  - DONE Expr: ~is_core~ removed (core and user ASTs kept separate)
  - DONE Expr: ~variadic_index/count~, ~kwargs_index/count~ removed
    (duplicated in ~FCallData~, never used on Expr)
  - DONE Ghost fields ~is_ext~, ~total_struct_size~ removed from Expr

  Remaining opportunities:
  - ~StructDefData.total_struct_size~: duplicates per-field offset/size
    data. See ~#147~ for the broader proposal to move struct layout off
    the AST entirely into a ~Context.struct_layouts~ map.

  See ~#147~ for broader Context/Declaration redesign ideas drawn from
  rstil; this issue stays focused on field-by-field Expr payload
  shrinking.

  Ideas from rstil comparison (rstil's Expr is just 4 fields: node_type,
  params, line, col -- all payload lives in NodeType variants).

  til's Expr is now down to 7 fields: ~data~, ~til_type~, ~struct_name~,
  ~children~, ~line~, ~col~, ~path~. The remaining gap to rstil's 4-field
  shape is type-checker outputs (~til_type~, ~struct_name~) and per-node
  ~path~ duplication.

  Expr-level bloat still present:
  - ~Expr.til_type~ and ~Expr.struct_name~: type-checker outputs, not
    parser outputs. rstil has neither on Expr. Could live in a side table
    or TypeBinding instead of bloating every node.
  - ~Expr.path~: rstil stores ~source_path~ only on FuncDef, not per-Expr.
    All Exprs in one file share the same path -- could be passed as
    context during traversal instead of stored on every node. ~470
    Expr.new call sites would need the path argument removed.

  Payload struct bloat still present:
  - ~Declaration.field_offset~ / ~field_size~ / ~field_struct_def~: struct
    layout computation results. rstil's Declaration has none of them --
    layout is computed during evaluation. Moving these to a separate
    struct-layout table makes Declaration a pure AST node.
  - ~FCallData~: rstil's equivalent (~FCallInfo~) is just ~{does_throw,
    is_bang}~ -- two bools. til's ~variadic_index~ / ~variadic_count~ /
    ~kwargs_index~ / ~kwargs_count~ / ~is_splat~ / ~own_args~ / ~fn_sig~
    are all resolution artifacts. Could shrink FCallData to throw/bang
    flags if variadic resolution moves to a later phase.
  - ~FunctionDef~ sig/def split: rstil cleanly separates ~FuncSig~
    (type-level: arg types, return types, throw types) from ~FuncDef~
    (instance-level: arg names, body). til's ~FunctionDef~ mixes both.
    Splitting could simplify function type checking and make func
    signatures first-class without carrying bodies around.

  rstil ideas already DONE:
  - ~Expr.is_own_arg~ -> ~FCallData.own_args~ U64 bitmask (rstil uses
    an OwnArg NodeType wrapper; til's bitmask avoids tree restructuring)
  - ~Expr.fn_sig~ -> ~FCallData.fn_sig~ (call-site signature lives in
    the call-site payload)
  - ~Expr.is_splat~ -> ~FCallData.is_splat~
  - ~Expr.save_old_delete~ -> ~AssignData.save_old_delete~

  The delete side of this issue is fixed (#126, now in fixed.org):

  - no double-free or aliasing from cloned enums sharing payload buffers

** OPEN #127 Runtime ~Value~/binding ownership cleanup and ~Ptr~ split
  The interpreter runtime currently overloads one mechanism for several
  different ownership cases:
  - scoped values live in ~Cell.val~
  - ~Scope~ bindings own some ~Cell*~ values via ~cell_is_local~
  - one runtime variant, current ~Value.Ptr~, is used for both:
    - borrowed/ref-like storage views
    - owning opaque heap pointers

  That causes two classes of problems:
  - real owner cleanup is missing at some scope teardown / overwrite boundaries
  - dispatcher/runtime code grows manual ~free(...)~ workarounds because the
    runtime value model does not distinguish borrowed pointer views from
    owned dynamic pointers cleanly

  Partial fixes landed (9d8a011 and subsequent):
  - ~scope_free~ now frees Enum payload data for owned cells
  - return values are cloned before ~scope_free~ to prevent dangling
  - Struct data in scope_free is still NOT freed (aliasing risk)
  - ~scope_set_owned~ now calls ~free_value~ on old value before overwriting

  Remaining pressure points:
  - current ~Value.Ptr~ is produced by:
    - ~null~
    - ref field reads
    - own field reads exposing owned storage
    - ~ptr_add(...)~
    - FFI ~Dynamic~ returns
    - container data fields like ~Array.data~ / ~Vec.data~
  - current consumers then guess whether that pointer is borrowed or owned

  Cleanup direction:
  - give runtime pointer-like values distinct meanings
  - stop overloading one variant for both ref-like and owning semantics
  - make the real owner delete runtime values at scope teardown and on
    overwrite
  - then simplify dispatcher helpers like ~val_to_ptr~ so they are only
    borrowed pointer-view helpers instead of ownership escape hatches

  Current candidate shape:
  - split current ~Value.Ptr~ into:
    - ~Value.Ref~ for borrowed/nullable pointer views
    - ~Value.Dynamic~ for owning opaque pointers
  - keep ~val_to_ptr := func(v: Value) returns ref Dynamic~
  - repair runtime binding cleanup before relying on translated dispatcher
    helpers as proof of correctness

  Important constraints discovered during analysis:
  - simple removal of ~Cell~ is not obviously safe yet, because current
    borrowed bindings are used for:
    - ref declarations from identifiers
    - borrowed function argument binding paths
  - own field reads cannot simply be classified globally as borrowed or
    owning without checking the typed/lowered move/clone behavior

  This should be tackled as a focused runtime ownership cleanup, not mixed
  into unrelated dispatcher translations.

** OPEN #91 Containers: store clone/delete as function pointers instead of dyn_call
  Vec/Map/Set currently dispatch clone/delete by type name string via
  ~dyn_call~, which does a linear scan of if/memcmp chains every time.
  Instead, store clone/delete function pointers as fields in the
  container struct. ~Vec.new("Str", Str.size())~ resolves the function
  pointers once at construction, then all subsequent clone/delete calls
  go through the stored pointers directly.

  Proven pattern: ~AlphaBeta~ struct in ~src/lib/alphabeta.til~ uses
  the same function-pointer-on-struct approach — ~h~ and ~nexts~ fields
  called via ~self.h(board)~ and ~self.nexts(board)~. All three game
  examples (checkers, chess, go) use it successfully.

  Design:

  New ext_func to resolve function pointers from type name + method:
  : dyn_get_fn_ptr := ext_func(type_name: Str, method: Str) returns Dynamic {}

  Vec struct gains function pointer fields:
  #+begin_src til
  Vec := struct {
      own mut data : U8 = 0
      mut count : USize = 0
      mut cap : USize = 0
      elem_size : USize = 0
      elem_type := ""
      elem_clone : func(Dynamic) returns Dynamic = default_clone
      elem_delete : proc(Dynamic, Bool) = default_delete
  }
  #+end_src

  Vec.new resolves pointers once at construction:
  #+begin_src til
  new := func(elem_type: Str, elem_size: USize) returns Vec {
      return Vec(
          data = malloc(elem_size),
          cap = 1,
          elem_size = elem_size,
          elem_type = elem_type,
          elem_clone = dyn_get_fn_ptr(elem_type, "clone"),
          elem_delete = dyn_get_fn_ptr(elem_type, "delete"),
      )
  }
  #+end_src

  #+begin_src til
  // Before (dyn_call):
  dyn_call(self.elem_type, "delete", 2, ptr_add(...), false)
  own cloned : Dynamic = dyn_call_ret(self.elem_type, "clone", 1, ptr_add(...))

  // After (function pointer):
  self.elem_delete(ptr_add(...), false)
  own cloned : Dynamic = self.elem_clone(ptr_add(...))
  #+end_src

  val_delete), Set (elem_clone, elem_delete, elem_cmp), Array, Tuple.

  don't work for some case, fix the blocker first, then come back.

  - ~dyn_fn~ renamed from ~dyn_get_fn_ptr~ for brevity
  - Builder generates ~dyn_fn~ dispatch (independent from dyn_call)

  - FuncPtr fields propagated in clone returns

** OPEN #154 Typer error for unused imports
  Follow-up to #86 (fixed). Now that unused function parameters and
  unused local variables are type errors, do the same for unused
  imports.

  #+begin_src til
  import("foo.til")              // error: unused import 'foo'
  import("bar.til") as _bar      // ok: _ prefix suppresses (syntax TBD)

  two reasons:
  - Imports don't introduce a single named binding the way ~x := 42~
    does. An ~import("foo.til")~ call brings every public top-level
    decl from ~foo.til~ into the importing scope; "used" has to mean
    "any symbol the import pulled in was read", not "a binding named
    ~foo~ was read".

    ~_~ prefix convention works for named bindings but there's no
    obvious place to put it on a bare ~import("...")~. Probably needs
    a new form — e.g. ~import("foo.til") as _name~ — before we can
    even talk about flagging unused imports.

  1. Load/import pipeline records, per import, the set of names it
     brought into the importing program's top-level scope.
  2. Typer marks each imported binding as used (same ~used~ flag on
     ~TypeBinding~ used by #86's param/local checks) when any of
     those names is referenced from the importing module.
  3. After type-checking, iterate imports; if no binding from a given
     import was marked used and the import didn't opt out via the
     ~_~ form, emit ~type error: unused import 'path'~.

** OPEN #152 True cross-compilation to macOS (and other host-restricted targets) via precompiled ext.o
  :DEPENDENCIES: #150 (unrelated, just handy to mention)
  :RELATED: rstil Issue #135

  Cross-compiling til programs to macOS from Linux or Windows currently
  is not feasible the way linux-arm64 / linux-riscv64 / windows-x64 are:
  Apple's SDK headers and frameworks aren't redistributable, and
  mingw-style freely available toolchains don't exist for Mach-O. Our
  current cross-compile pipeline assumes each host has a working
  ~{target}-gcc~ that can produce the target's executable format from
  the generated C. macOS breaks that assumption.

  Rstil sketched the same problem in its Issue #135 ("True
  cross-compilation via precompiled ext.o per platform"). The proposed
  shape, adapted for til:

  1. Move every OS-specific call out of the generated C into
     ~src/c/ext.c~ so the emitted ~gen/til/*.c~ ideally only calls into
     a small fixed set of ~til_*~ runtime symbols. The #ifdef _WIN32
     blocks added for windows-x64 are a decent starting audit of which
     functions need this treatment.
  2. Precompile ~ext.c~ once per target on a native runner in CI, ship
     the resulting ~lib/ext-{target}.o~ (e.g. ~ext-macos-x64.o~,
     ~ext-macos-arm64.o~) alongside the sources. For macOS that means a
     ~runs-on: macos-latest~ / ~macos-15-intel~ job whose only output
     is the committed ext object.
  3. Teach ~src/self/builder.til~ to link against the precompiled
     ~ext-{target}.o~ when cross-compiling instead of compiling ext.c
     in-tree. Non-macOS targets (linux-*, windows-x64) keep the current
     compile-ext.c-together flow; macOS uses the precompiled object.
  4. CI adds a ~macos-x64~ / ~macos-arm64~ cross-compile job that
     downloads the committed ext-macos-*.o artifacts and links.

  Rstil wrote up and rejected one iteration of this (branch
  ~claude/fix-issue-135-SMlEh~ in rstil) that removed ~static inline~
  from every ext function and hardcoded 50+ ~extern~ declarations into
  ccodegen. That approach is too brittle; the duplication in both
  ccodegen.rs and ccodegen.til made adding any ext function require
  edits in three places. Whatever til does here needs a lighter coupling
  between ~builder.til~ and ~ext.c~ -- probably a single shared
  ~ext_forward.h~ emitted once and consumed by both the generated C and
  the precompiled ext object.

  Open questions (same as rstil's):
  - How are platform-specific ext.o files versioned / regenerated when
    ext.c changes? Do we fail the main CI if ext-macos-*.o is stale?
  - Ship ~zig cc~ as an alternative? Single toolchain that can target
    everything, at the cost of a ~300MB dependency.
  - Do we keep the current compile-ext.c-together path forever for the
    host target, or migrate fully to the precompiled-per-target model?

** OPEN #115 Remove or simplify ~ext_struct~
  Revisit whether ~ext_struct~ should exist as a separate concept at all.

  Today many uses are really just opaque or fieldless types with methods, for
  example the primitive types split out of ~core.til~. Conceptually, these are
  still just types. The main special behavior ~ext_struct~ is standing in for
  is:

  - "the caller should not assume fields/layout from source syntax"
  - "the type may still define methods in its namespace"
  - "size/clone/delete may need custom behavior"

  That may not justify a dedicated type form.

  Possible direction:

  - use ordinary ~struct {}~ (or fieldless struct declarations) for opaque
    external types too
  - if the type is opaque, do not rely on autogenerated size/clone/delete
  - instead, require explicit methods where the default would be wrong

  In particular, if a type wants to be opaque but still copyable/deletable
  correctly, it should be able to provide:

  - ~size() returns USize~
  - ~clone(...)~
  - ~delete(...)~

  rather than needing a distinct ~ext_struct~ concept.

  Questions to resolve:

  - should opaque external types just be ordinary structs with no fields?
  - should initer only autogenerate ~size/clone/delete~ when the layout is
    fully known from source fields?
  - is there any remaining semantic difference that truly requires
    ~ext_struct~?

  Recent evidence:

  - a dormant ~src/core/i8.til~ plus matching ~ext.c/ext.h~ support was easy
    to add
  - but importing ~I8~ immediately exposed that the current compiler still
    has repeated hardcoded primitive-name lists (~U8~, ~I16~, ~I32~, ~U32~,
    ~U64~, ~I64~, ~F32~, ~Bool~) across builder, initer, typer, and
    ~TilType~
  - those lists decide things like:
    - whether a type is treated as a builtin scalar
    - whether generated headers skip forward declarations for it
    - whether C emission uses the alias-backed primitive representation or
      falls back to ordinary struct handling
  - so the current ~I8~ blocker is not the lack of another ~ext_struct~
    mechanism, but the broader primitive special-casing model

  That pushes this issue in a clearer direction:

  - simplify/remove ~ext_struct~
  - reduce hardcoded primitive special cases to the minimum truly required by
    runtime representation
  - make ordinary type declarations plus explicit methods carry more of the
    behavior today encoded by compiler categories

  This issue is partly about simplifying the language surface and partly about
  reducing one more C-leaning special case in the core model.

** OPEN #58 First-class functions: Type hierarchy and FuncSig/FuncDef
  :DEPENDENCIES: #14 #61
  Based on rstil/doc/first_class.org. Everything in til is a value —
  functions, structs, enums can all be assigned, passed, returned, and
  stored. This issue covers the metatype system that makes it work.

*** Type hierarchy

   ~Type~ is the metatype — the type of all type definitions:

   #+begin_src til
   Type := enum { StructDef, EnumDef, FuncSig, FuncDef }
   #+end_src

   | Metatype    | Defines                      | Instance example                     |
   |-------------+------------------------------+--------------------------------------|
   | ~StructDef~ | Data layout (fields)         | ~p := Point(x=10)~ is a Point        |
   | ~EnumDef~   | Tagged union (variants)      | ~c := Color.Red~ is a Color          |
   | ~FuncSig~   | Calling convention (types)   | ~BinaryOp := func(I64, I64) ret I64~ |
   | ~FuncDef~   | Concrete function (sig+body) | ~add : BinaryOp = (a, b) { ... }~   |

   ~FuncSig~ is a pure type — parameter types, return type, func/proc
   kind. Parameter *names* are NOT part of the signature, only types.

   ~FuncDef~ is a concrete function definition — it has a ~FuncSig~ (its
   type) plus argument names, modifiers, and a body. A FuncDef is to its
   FuncSig what a struct instance is to its StructDef: the signature is
   the type, the definition is the value.

   Function types are *structural*: two signatures with the same parameter
   types, return type, and throws are the same type regardless of name.

*** Explicit type annotations (~: Type =~ syntax)

   Separate the type annotation (~:~) from the value assignment (~=~).
   ~:=~ remains as shorthand for "infer the type":

   #+begin_src til
   // Explicit type + value
   x : I64 = 5
   name : Str = "hello"

   // Type definitions: explicit metatype annotation
   Point : StructDef = struct { mut x: I64 = 0, mut y: I64 = 0 }
   Color : EnumDef = enum { Red, Green, Blue }
   BinaryOp : FuncSig = func(I64, I64) returns I64

   // Inferred (unchanged from today)
   Point := struct { mut x: I64 = 0, mut y: I64 = 0 }   // infers StructDef
   Color := enum { Red, Green, Blue }                      // infers EnumDef
   BinaryOp := func(I64, I64) returns I64                  // infers FuncSig
   #+end_src

*** Type inference rules

   | Declaration                                                | Inferred type |
   |------------------------------------------------------------+---------------|
   | ~x := 5~                                                   | ~I64~         |
   | ~name := "hello"~                                          | ~Str~         |
   | ~Point := struct { ... }~                                  | ~StructDef~   |
   | ~Color := enum { ... }~                                    | ~EnumDef~     |
   | ~BinaryOp := func(I64, I64) returns I64~                   | ~FuncSig~     |
   | ~add := func(a: I64, b: I64) returns I64 { return ... }~   | ~FuncDef~     |

   The distinction mirrors struct/enum:
   - ~Point := struct { ... }~ defines a *type* (infers StructDef)
   - ~p := Point(x=10)~ creates an *instance* (infers Point)
   - ~BinaryOp := func(I64, I64) returns I64~ defines a *type* (infers FuncSig)
   - ~add := func(a: I64, b: I64) returns I64 { ... }~ creates an *instance* (infers FuncDef)

   A bodyless func/proc is a FuncSig (type definition). A bodied func/proc
   is a FuncDef (value). The presence of a body (and parameter names)
   distinguishes them.

*** Collections of types

   #+begin_src til
   Vec.new("BinaryOp")   // Vec of function pointers with the same signature
   Vec.new("FuncSig")    // Vec of function signatures (different signatures)
   Vec.new("FuncDef")    // Vec of function definitions (different definitions)
   Vec.new("StructDef")  // Vec of struct definitions
   Vec.new("Type")       // Vec of any type definition
   #+end_src

*** Type parameters

   Two-phase plan:

   *Phase now*: ~func(T: Type)~ — no new syntax needed. ~Type~ is already
   a special parameter type. Macros handle the template use case by
   executing arbitrary code at compile time (like Jai's comptime, Zig's
   comptime):

   #+begin_src til
   // Type as a regular parameter
   swap := func(T: Type, mut a: T, mut b: T) { ... }
   swap(I64, x, y)

   Pair := macro(A: Type, B: Type) returns Type {
       return struct { mut first: A, mut second: B }
   }
   IntStr := Pair(I64, Str)
   #+end_src

   *Phase later*: ~func<T>()~ as sugar that desugars to macro calls.
   Use ~<>~ for familiarity (Rust, C++, Java):

   #+begin_src til
   // Sugar (later):
   swap<T> := func(mut a: T, mut b: T) { ... }
   swap(x, y)       // T inferred
   swap<I64>(x, y)  // T explicit

   // Desugars to:

** OPEN #55 Lambda functions and first-class bodies
  :DEPENDENCIES: #14 #54
  Based on rstil/doc/first_class.org. Anonymous inline functions (lambdas)
  and first-class Body values (blocks as values).

*** Lambda functions (anonymous inline functions)

   Lambda functions are anonymous FuncDef literals passed directly as
   arguments. Same syntax as named definitions, just without a name:

   #+begin_src til
   Predicate := func(I64) returns Bool
   Reducer := func(I64, I64) returns I64

   // filter with inline lambda
   evens := numbers.filter(func(a: I64) returns Bool { return a.mod(2).eq(0) })
   // [2, 4]

   // reduce with inline lambda
   sum := numbers.reduce(0, func(acc: I64, x: I64) returns I64 { return acc.add(x) })
   // 15

   // FuncSig form lambda (type separated from param names)
   doubled := numbers.map(func(I64) returns I64 = (a) { return a.mul(2) })
   #+end_src

*** First-class bodies (blocks as values)

   A Body is a sequence of statements — the simplest unit of executable
   code. Simpler than a function: no signature, no parameters, no return
   type. A Body is essentially a zero-argument, void closure.

   #+begin_src til
   // Store in a variable
   cleanup := { file.close(); println("done") }

   // Pass as argument
   run_twice := proc(b: Body) {
       b()
       b()
   }
   run_twice({ println("hello") })

   // Store in a struct field
   Task := struct {
       mut on_complete: Body
   }
   t := Task(on_complete = { println("finished") })
   t.on_complete()

   // Store in a collection
   steps := Vec.new("Body")
   steps.push(own { println("step 1") })
   steps.push(own { println("step 2") })
   for step : Body in steps {
       step()
   }
   #+end_src

*** Relationship to functions and closures

   | Construct         | Has params | Has return type | Has captures | Has body |
   |-------------------+------------+-----------------+--------------+----------|
   | Body              | No         | No              | No*          | Yes      |
   | func (no closure) | Yes        | Yes             | No           | Yes      |
   | func (closure)    | Yes        | Yes             | Yes (#54)    | Yes      |

   *Without captures (#54), a Body can only access globals. With captures,
   a Body becomes a zero-argument closure:

   #+begin_src til
   mut total := 0
   data := vec("I64", 1, 2, 3)

   // Body with capture list
   summer := captures mut total, data { for item : I64 in data { total = total.add(item) } }
   summer()
   // total is now 6
   #+end_src

   A Body is to ~proc()~ what a literal is to a named constant — the
   anonymous, inline version. The distinction is that a Body has no
   function overhead (no frame push/pop in the interpreter, no C function
   call in codegen) — it executes in the caller's scope.

*** Use cases

   #+begin_src til
   // Deferred execution (like Go's defer)
   cleanup := { file.close() }
   // ... do work ...
   cleanup()

   // Callbacks and hooks
   Widget := struct {
       mut on_click: Body
       mut on_hover: Body
   }

   // Builder patterns
   with_transaction := proc(db: Database, action: Body) {
       db.begin()
       action()
       db.commit()
   }
   with_transaction(db, { db.insert(record1); db.insert(record2) })

   // Test setup/teardown
   run_test := proc(name: Str, setup: Body, test_body: Body, teardown: Body) {
       println("Running:", name)
       setup()
       test_body()
       teardown()
   }
   #+end_src

*** Scope semantics: inline vs isolated

   Open question: when a Body executes, does it share the caller's scope
   (like an inlined block) or get its own scope (like a function call)?

   - *Option A: Inline* — shared scope (how bare ~{ }~ blocks work today).
     Simple, powerful, zero overhead. But passing a Body as argument is
     weird — which scope does it execute in?
   - *Option B: Isolated* — own scope, captures only (#54). Safe and
     predictable for stored/passed bodies. But needs captures to be useful.
   - *Option C: Both* — bare ~{ }~ blocks are inline (current behavior),
     Body values (stored, passed, returned) are isolated and require
     captures. This is probably the right answer.

*** Comparison

   | Language   | Block-as-value          | Notes                              |
   |------------+-------------------------+------------------------------------|
   | Ruby       | ~{ ... }~ / ~do...end~  | Blocks and Procs are first-class   |
   | Smalltalk  | ~[ ... ]~               | Blocks are objects                 |
   | Swift      | ~{ () -> Void in ... }~ | Closures (always have captures)    |
   | Kotlin     | ~{ ... }~ (lambda)      | Trailing lambda syntax             |
   | Rust       | closure only            | No bare blocks as values           |
   | Go         | ~func() { ... }()~      | Must wrap in function              |
   | TIL        | ~Body = { ... }~        | Simpler than closure               |

   Ruby is the closest precedent. Ruby blocks can be passed to methods
   implicitly (~yield~) or explicitly (~&block~), and ~Proc.new~ wraps a
   block into a first-class value.

** OPEN #54 Explicit capture syntax (closures)
  :DEPENDENCIES: #61
  :DEPENDENCIES_OF: #12 #13
  Add syntax for listing captured variables from outer scopes, both for
  function definitions and anonymous blocks. Based on rstil/doc/first_class.org.
  Split from #17 (global variable access, now fixed).

  Closures = lambdas (#61) + captures. #61 provides anonymous inline
  functions (lambdas); this issue adds the ~captures~ keyword that turns
  a lambda into a closure by binding outer-scope variables.

  til requires *explicit capture lists* — if a function references an outer
  variable, it MUST be listed in the capture list. This makes dependencies
  visible and refactoring mechanical. Inspired by Jai (Jonathan Blow).

*** Capture syntax

   The ~captures~ keyword appears after the parameter list (and before
   ~returns~ if present) in FuncDef form, or after the argument names in
   FuncSig form:

   #+begin_src til
   scale := 10

   // FuncDef form: captures between params and returns
   f := func(a: I64) captures scale returns I64 { return a.mul(scale) }

   // FuncSig form: captures after argument names
   f : func(I64) returns I64 = (a) captures scale { return a.mul(scale) }

   // With a named type alias
   Transformer := func(I64) returns I64
   f : Transformer = (a) captures scale { return a.mul(scale) }
   #+end_src

*** Capture modifiers

   Captures follow the same rules as function parameters — no new rules
   to learn:

   | Capture syntax        | Semantics                     | Same as param mode |
   |-----------------------+-------------------------------+--------------------|
   | ~captures x~          | Const by reference (default)  | ~x: T~             |
   | ~captures mut x~      | Mutable reference             | ~mut x: T~         |
   | ~captures own x~      | Ownership transfer (moved in) | ~own x: T~         |

   #+begin_src til
   mut total := 0
   data := vec("I64", 1, 2, 3)

   // mut capture: closure can modify total
   data.for_each(func(x: I64) captures mut total { total = total.add(x) })

   // own capture: moves data into the closure, caller can't use it after
   consume := proc() captures own data { data.clear() }
   #+end_src

*** Block captures (scope restriction)

   Even without making a function, a capture list on a bare code block
   limits what it can see — an "insurance policy" that the block only
   touches specified state:

   #+begin_src til
   mut total := 0
   mut count := 0
   data := vec("I64", 1, 2, 3)

   // Only total and data are accessible inside this block
   captures total, data {
       for item : I64 in data {
           total = total.add(item)
       }
   }
   // count is guaranteed untouched
   #+end_src

*** Jai-inspired refactoring progression

   Each step is mechanical. The capture list tells you exactly what needs
   to become a parameter when you promote to a pure function:

   #+begin_src til
   // Step 1: inline code
   total = total.add(x)

   // Step 2: block with captures (limit scope, verify dependencies)
   captures total, x {
       total = total.add(x)
   }

   // Step 3: promote to lambda (add params, same captures)
   adder := func(a: I64) captures total returns I64 { return total.add(a) }

   // Step 4: promote to pure function (captures become params, no closure)
   adder := func(a: I64, b: I64) returns I64 { return a.add(b) }
   #+end_src

*** Why explicit captures

   Implicit captures (Python, JavaScript, Rust) feel magical:
   - *Reading*: you see ~scale~ and have to hunt for where it's defined.
     Is it a global? A local three scopes up? A typo for a parameter?
   - *Refactoring*: extracting a block into a function means figuring out
     ALL the implicit dependencies. Miss one and it breaks.
   - *Inlining*: moving a closure back inline means understanding what it
     captures to know where it can safely go.

   Explicit captures solve all three:
   - *Reading*: ~captures scale~ tells you exactly what comes from outside.
   - *Refactoring*: captures become parameters. Mechanical transformation.
   - *Inlining*: captures tell you exactly what the block depends on.

*** Implementation

   - Parser: recognize ~captures~ keyword in func defs (between params
     and returns/body) and bare blocks (before ~{~)
   - AST: store capture list on NODE_FUNC_DEF and NODE_BODY
   - Typer: validate captures exist in outer scope, enforce modifier rules
   - Interpreter: bind captured variables in call_scope from definition scope
   - Codegen: emit closure structs or pass captures as hidden parameters

** OPEN #25 Support for the 4 main operative systems
  :DEPENDENCIES: #12
  til currently only builds and runs on GNU/Linux. Should support the 4
  main operative systems, in priority order:
  1. GNU/Linux/BSD (current, done)
  2. TempleOS
  3. macOS
  4. Windows

  This is not just "til builds natively on each OS." The goal is full
  cross-compilation: build a til binary for any target from any host.
  The matrix is 3×4 + 1×1 = 13 combinations: Linux, macOS, and Windows
  can each target all 4 platforms, but TempleOS as a host only targets
  itself (cross-compiling from TempleOS would mean replacing gcc/clang,
  which is out of scope).

  ctil (the C bootstrap compiler) stays Linux-only. Once til is
  self-hosting (#12), til replaces ctil and handles all platforms.
  Bootstrap path: ctil on Linux compiles self-hosted til → til can
  then cross-compile itself and any .til program to any target.

  The C codegen path helps: emit C + invoke the target's C compiler
  works for most POSIX targets. But TempleOS breaks this pattern
  (no C compiler), requiring a dedicated HolyC codegen backend.

  The builder must select the appropriate codegen backend per target.

  Common areas affected across platforms:
  - Build system (builder that selects backend per target)
  - FFI / dlsym (platform-specific dynamic loading)
  - System primitives (spawn_cmd, readfile, etc. — POSIX vs platform APIs)
  - GUI mode (raylib on POSIX/Windows, native primitives on TempleOS)
  - Audio (miniaudio on POSIX/Windows, native on TempleOS)

*** TempleOS
   :DEPENDENCIES: #12
   The most interesting and challenging target. TempleOS has no C compiler,
   no POSIX, no libc — just HolyC and its own kernel/graphics/audio APIs.

   Requires a new holybuilder backend that emits HolyC instead of C.
   HolyC is C-like enough that the translation should be mostly mechanical
   (similar types, similar control flow), but differences in calling
   conventions, memory management, and string handling will need attention.

   For mode gui: raylib doesn't exist on TempleOS, so we need to recreate
   raylib's interface using TempleOS graphics primitives (GrBlot, GrLine,
   GrRect, etc.) and audio primitives. This is likely easier than most
   people would expect — TempleOS has direct framebuffer access and simple
   audio APIs, and raylib's 2D API surface is not that large. The hard
   part is the impedance mismatch (TempleOS is cooperative multitasking,
   640x480 16-color default, etc.), not the drawing primitives themselves.

   Running the same mode gui program (e.g. music.til) on both GNU/Linux
   via raylib AND TempleOS via native HolyC primitives would be a serious
   language flex — proving til programs are truly platform-independent.

   For mode server: TempleOS docker containers could be extremely small
   (no libc, no userland, minimal kernel), making til a compelling choice
   for lightweight server deployments on TempleOS-based containers.

   Self-hosting (#12) is a hard dependency: TempleOS has no C compiler,
   so til must be able to compile itself to bring its own compiler onto
   the platform. There's no way to cross-compile C for TempleOS from Linux.

   Can test from Linux using QEMU with a TempleOS VM image. Copy til
   files in via a virtual disk, run them, capture output. This is the
   standard way to develop for TempleOS from a modern host.

*** macOS
   Closest to the current GNU/Linux target. All C code (ext.c,
   dispatch.c) is already POSIX-portable — fork, execl, waitpid, dlopen,
   dlsym all work the same on macOS. No cross-compilation needed: ship
   source, user runs `make` with clang natively.

   Makefile changes needed:
   - `-ldl` not needed on macOS (dlopen is in libc)
   - `-lffi` needs `brew install libffi` + Homebrew `-I`/`-L` paths
   - `-rdynamic` works with clang
   - Platform detection to conditionally apply the above

   Runtime differences:
   - `.dylib` instead of `.so` for dynamic libraries (mode gui, FFI)
   - Homebrew paths for raylib headers and libs

   No `#ifdef __APPLE__` needed in the C source itself — only in the
   Makefile and possibly in the codegen output (linker flags when
   compiling .til programs that use mode gui).

*** Windows
   Furthest from POSIX, but possibly dying — don't wait for self-hosting.
   Options: MSVC, MinGW, or cross-compile with zig cc. Dynamic loading
   via LoadLibrary/GetProcAddress instead of dlopen/dlsym. Process
   spawning via CreateProcess instead of fork/exec. raylib supports
   Windows natively, so mode gui should work once the build system
   handles .dll and Windows linking.

   Can test from Linux using Wine as a first step before touching a
   real Windows machine.
