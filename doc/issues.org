** OPEN #50 Shallow types: stack allocation for structs/enums without own fields
  Currently only primitive ext_structs (I64, U8, Bool, etc.) get the stack
  allocation optimization — declared as stack variables instead of heap-allocated
  pointers. This can be extended to any struct or enum that has no ~own~ fields
  or payloads, recursively.

  A type is "shallow" (in the copy-safety sense) if:
  - It has no ~own~ fields (structs) or ~own~ payloads (enums)
  - All its inline (non-ref, non-own) fields/payloads are themselves shallow,
    recursively
  - ~ref~ fields are fine — they're borrowed pointers, shallow-copying them
    is safe since we don't manage their lifetime

  The check is mutually recursive: structs can have enum fields, enums can
  have struct or enum payloads, so both must be checked together.

  Shallow types can be:
  - Stack-allocated instead of heap-allocated (the optimization)
  - Shallow-copied safely (memcpy, no deep copy needed)

  Important: shallow types still need ~delete~ called (unlike primitives whose
  delete is a no-op). Custom delete methods may do non-memory work (close file
  descriptors, connections, etc.). The guarantee to the user is that delete is
  called ASAP after the last use of an instance. What changes is: no ~free~ at
  the end (since it wasn't malloc'd), and no deep-copy clone (just memcpy).

  This is a different notion from the ~shallow~ keyword on args/returns, which
  is about passing by value vs pointer based on the size of the type relative
  to the platform pointer size. The "shallow type" concept here is about copy
  semantics — whether a type can be safely shallow-copied without ownership
  issues. In the future, the ~shallow~ keyword could be computed automatically
  from this property plus size considerations.


** OPEN #46 Optimize Str with with_capacity in concat, join, etc.
  str.til methods like concat, join, and others build strings by repeated
  push_byte or append without pre-allocating. This causes excessive
  reallocations. Use Str.with_capacity to pre-compute the final size and
  allocate once upfront. Should measurably improve performance since Str
  operations are on the hot path for the compiler.

*** Architecture: core.til as ctil's runtime library

   core.til (which imports str.til, vec.til, map.til, set.til, array.til)
   is built as a library via ~ctil build core.til~, producing core.c/core.h.
   ctil itself links against this generated core.c instead of having its
   own handwritten str.c, vec.c, map.c. The .til files become the single
   source of truth.

   core.til uses ~link_c("ext.c")~ — the builder emits ~#include "ext.c"~
   in the generated core.c, making it self-contained. ccore.c is NOT
   link_c'd because it overlaps with ext.c (both define I64_add, Bool_eq,
   etc.). ext.c is the canonical source for scalar ops and system prims.
   ccore.c's unique parts (Token/Expr/TypeScope/Mode clone/delete for
   til_bridge FFI) stay in a small compiler-only file.

   Bootstrap: generated core.c/core.h are copied to src/bootstrap/ and checked
   into the repo. People who clone the repo get pre-generated files.
   ~make libs~ rebuilds from core.til when needed.

   Vec/Map/Set conflict: the generated types have different struct layouts
   than the compiler's internal versions (e.g. generated Vec has I64 fields
   + Str elem_type, compiler Vec has U32 fields + void* data). Resolution:
   rename the compiler's internal types to CVec/CMap/CSet.

   Str signature change: generated Str functions return I64*/Bool* (heap
   pointers) instead of scalar I64/Bool. All compiler call sites (~60+)
   need adapting (*Str_eq(...) style dereferences). Compiler-only helpers
   (Str_val, Str_new, Str_new_len, Str_eq_c) stay in a compiler-only file.

*** Order
   1. Builder: link_c emits #include in generated .c (not separate compile)
   2. Rename compiler Vec/Map/Set → CVec/CMap/CSet (avoid conflicts)
   3. Build core.til with link_c("ext.c"), copy to src/bootstrap/
   4. Wire ctil to use src/bootstrap/core.c, adapt Str callers, delete str.c
   5. Replace src/c/lexer.c with til-compiled liblexer
   6. Replace src/c/parser.c + ast.c with til-compiled libparser
   7. Continue upward: initer, typer, interpreter, builder
   8. ctil.c becomes a thin CLI linking only til libs
   9. ctil.c itself replaced by til.til binary → C code fully eliminated

  At each step, run benchmarks to verify no significant performance
  regressions from the til versions vs the C originals. The test suite
  (tests.til) runs four configurations: C interpreted, C compiled, til
  interpreted, til compiled — all must produce identical results.

** OPEN #43 Remove --allow-multiple-definition linker workaround
  compile_c in builder.c passes -Wl,--allow-multiple-definition to the
  linker. This was added because til.til links C modules (str.c, vec.c,
  map.c, ccore.c) that define functions also generated by the builder
  from core.til (Str_clone, Vec_push, Set_delete, etc.).

  The proper fix: make the builder skip emitting core struct method bodies
  when link_c files already provide them, OR make core.til struct methods
  ext_func (provided by the C files, not generated). This requires
  coordination between the builder and the link_c mechanism.

  The workaround is harmless for normal programs (no duplicate symbols
  = flag is a no-op) but masks potential real conflicts.

** OPEN #25 Support for the 4 main operative systems
  :DEPENDENCIES: #12
  til currently only builds and runs on GNU/Linux. Should support the 4
  main operative systems, in priority order:
  1. GNU/Linux/BSD (current, done)
  2. TempleOS
  3. macOS
  4. Windows

  This is not just "til builds natively on each OS." The goal is full
  cross-compilation: build a til binary for any target from any host.
  The matrix is 3×4 + 1×1 = 13 combinations: Linux, macOS, and Windows
  can each target all 4 platforms, but TempleOS as a host only targets
  itself (cross-compiling from TempleOS would mean replacing gcc/clang,
  which is out of scope).

  ctil (the C bootstrap compiler) stays Linux-only. Once til is
  self-hosting (#12), til replaces ctil and handles all platforms.
  Bootstrap path: ctil on Linux compiles self-hosted til → til can
  then cross-compile itself and any .til program to any target.

  The C codegen path helps: emit C + invoke the target's C compiler
  works for most POSIX targets. But TempleOS breaks this pattern
  (no C compiler), requiring a dedicated HolyC codegen backend.

  The builder must select the appropriate codegen backend per target.

  Common areas affected across platforms:
  - Build system (builder that selects backend per target)
  - FFI / dlsym (platform-specific dynamic loading)
  - System primitives (spawn_cmd, readfile, etc. — POSIX vs platform APIs)
  - GUI mode (raylib on POSIX/Windows, native primitives on TempleOS)
  - Audio (miniaudio on POSIX/Windows, native on TempleOS)

*** TempleOS
   :DEPENDENCIES: #12
   The most interesting and challenging target. TempleOS has no C compiler,
   no POSIX, no libc — just HolyC and its own kernel/graphics/audio APIs.

   Requires a new holybuilder backend that emits HolyC instead of C.
   HolyC is C-like enough that the translation should be mostly mechanical
   (similar types, similar control flow), but differences in calling
   conventions, memory management, and string handling will need attention.

   For mode gui: raylib doesn't exist on TempleOS, so we need to recreate
   raylib's interface using TempleOS graphics primitives (GrBlot, GrLine,
   GrRect, etc.) and audio primitives. This is likely easier than most
   people would expect — TempleOS has direct framebuffer access and simple
   audio APIs, and raylib's 2D API surface is not that large. The hard
   part is the impedance mismatch (TempleOS is cooperative multitasking,
   640x480 16-color default, etc.), not the drawing primitives themselves.

   Running the same mode gui program (e.g. music.til) on both GNU/Linux
   via raylib AND TempleOS via native HolyC primitives would be a serious
   language flex — proving til programs are truly platform-independent.

   For mode server: TempleOS docker containers could be extremely small
   (no libc, no userland, minimal kernel), making til a compelling choice
   for lightweight server deployments on TempleOS-based containers.

   Self-hosting (#12) is a hard dependency: TempleOS has no C compiler,
   so til must be able to compile itself to bring its own compiler onto
   the platform. There's no way to cross-compile C for TempleOS from Linux.

   Can test from Linux using QEMU with a TempleOS VM image. Copy til
   files in via a virtual disk, run them, capture output. This is the
   standard way to develop for TempleOS from a modern host.

*** macOS
   Closest to the current GNU/Linux target. All C code (ccore.c, ext.c,
   dispatch.c) is already POSIX-portable — fork, execl, waitpid, dlopen,
   dlsym all work the same on macOS. No cross-compilation needed: ship
   source, user runs `make` with clang natively.

   Makefile changes needed:
   - `-ldl` not needed on macOS (dlopen is in libc)
   - `-lffi` needs `brew install libffi` + Homebrew `-I`/`-L` paths
   - `-rdynamic` works with clang
   - Platform detection to conditionally apply the above

   Runtime differences:
   - `.dylib` instead of `.so` for dynamic libraries (mode gui, FFI)
   - Homebrew paths for raylib headers and libs

   No `#ifdef __APPLE__` needed in the C source itself — only in the
   Makefile and possibly in the codegen output (linker flags when
   compiling .til programs that use mode gui).

*** Windows
   Furthest from POSIX, but possibly dying — don't wait for self-hosting.
   Options: MSVC, MinGW, or cross-compile with zig cc. Dynamic loading
   via LoadLibrary/GetProcAddress instead of dlopen/dlsym. Process
   spawning via CreateProcess instead of fork/exec. raylib supports
   Windows natively, so mode gui should work once the build system
   handles .dll and Windows linking.

   Can test from Linux using Wine as a first step before touching a
   real Windows machine.

** OPEN #22 Operator syntax: desugar operators to method calls
  :DEPENDENCIES: #12
  Syntactic sugar so a + b desugars to a.add(b), a < b to a.lt(b), etc.
  Purely a parser transform — no changes needed in typer, interpreter, or
  codegen. The parser rewrites operator expressions into method call AST
  nodes before anything downstream sees them.

  Operator → method mapping:
  - Comparison: < → lt, <= → lte, > → gt, >= → gte, == → eq, != → neq
  - Arithmetic: + → add, - → sub, * → mul, / → div, % → mod

  Precedence (low to high):
  - == != < <= > >=
  - + -
  - * / %

  Parenthesized expressions (a + b) * c override precedence. The parser
  must disambiguate grouping parens from function call parens: ( after an
  identifier/field-access is a call, otherwise it's grouping.

  Parsing approach: Pratt parsing (precedence climbing) or one function
  per precedence level (mutually recursive descent). Both work; Pratt is
  more compact and easier to extend.

  The cmp/initer synergy makes this especially nice for comparisons:
  implement cmp once, the initer auto-generates eq, neq, lt, gt, lte, gte,
  and operator desugaring gives you <, <=, >, >=, ==, != for free. If you
  override eq with a faster implementation, neq (and !=) automatically
  benefits since neq is generated as a.eq(b).not().

*** Phase 1: Comparison operators
   < <= > >= == !=
   All same precedence, binary, no chaining ambiguity.
   Desugars: a < b → a.lt(b), a != b → a.neq(b), etc.

*** Phase 2: Arithmetic operators
   + - * / %
   Introduces real precedence: * / % bind tighter than + -.
   Desugars: a + b * c → a.add(b.mul(c)).
   Requires grouping parentheses support.

   Also: ~++~ / ~--~ as unary sugar for inc/dec (methods added in #34 Phase 2).

** WIP #17 Global variable access, explicit capture syntax
  :DEPENDENCIES_OF: #12 #13
  Variables declared in script-mode root scope (or any outer scope) are
  invisible at runtime inside function/proc bodies. The typer passes
  (TypeScope chains via parent pointers and finds the variable), but the
  interpreter crashes or errors.

  Reproduction: src/test/globals.til, src/test/globals_cli.til

  func case: "error: undefined variable 'x'" — clean error from NODE_IDENT
  NULL check. proc case: SEGFAULT — NODE_DECL move-semantics path has no
  NULL check on scope_get result.

*** Root cause

   expr_uses_var() in typer.c (line 1239) returns 0 for NODE_FUNC_DEF,
   meaning the ASAP pass doesn't see variable references inside nested
   function/proc bodies. When x is declared at top level and referenced
   only inside a proc body, ASAP thinks x is unused and inserts
   I64.delete(x) right after the declaration. When the proc later runs
   and tries to access x, the value has already been freed/nulled.

   func case: I64.clone(x) inside the function body tries to clone x
   whose value is val_none → clean "undefined variable" error (from
   precomp, which also can't find x in its limited scope).

   proc case: I64.clone(x) dereferences v.i64 which is NULL → SEGFAULT.
   The NODE_DECL move-semantics path also lacks a NULL check on
   scope_get result, which would be a secondary crash path.

   Secondary issue: precomp creates its own interpreter scope with only
   func/struct/enum defs (no variable decls), then tries to fold pure
   func calls. If the func accesses a global variable, precomp's
   eval_expr hits the error path and exit(1)s before the interpreter
   even runs.

*** DONE Phase 1: Fix global variable access

   Program-scope globals live for the program's lifetime — they should
   never be ASAP-deleted. The scavenger handles removing unused globals.

   1. Skip ASAP deletion for program-scope variables: added skip_delete
      flag to LocalInfo in typer.c. Variables with a declaration at program
      scope (is_program_scope && decl_idx >= 0) get skip_delete=1. The
      ownership-transfer check still runs for them (use-after-own errors),
      but no delete calls are inserted. Variables captured by nested
      functions also get skip_delete=1 (expr_used_in_nested_func check).
   2. Added NULL check in interpreter.c NODE_DECL move-semantics path.
   3. Fixed precomp: added func_uses_unknown_globals() to skip folding
      func calls whose body references variables not in the precomp scope.
      Also made track_literal() populate the precomp interpreter scope
      with known constant values (not just the known map).
   4. Fixed builder: top-level variable declarations are emitted as
      file-scope C globals (before function definitions) so they're
      accessible from emitted til functions/procs. This is necessary
      because til functions are emitted as separate C functions, and C
      doesn't support nested functions capturing locals — so top-level
      vars must be file-scope for functions to reference them. The
      values are then assigned inside main() where initialization runs.
      In cli mode, globals are initialized at the start of main()
      before the user's main body.
      Note: ref declarations are skipped from file-scope globals since
      they are pointer aliases (local to main), not owned values that
      other functions need to access.

   Tests: src/test/globals.til (script mode), src/test/globals_cli.til (cli mode).

*** Phase 2: Explicit capture syntax

   Add syntax for listing captured variables from outer scopes, both for
   function definitions and anonymous blocks. Based on rstil/doc/first_class.org.

   Function captures (closures):
     scale := 10
     f := func(a: I64) returns I64 [scale] { return a.mul(scale) }

   Capture modifiers:
     [x]      — const reference (default)
     [mut x]  — mutable reference
     [copy x] — snapshot (clone at capture time)
     [own x]  — ownership transfer

   Block captures (restrict access):
     [total, data] {
         // can only see total and data, not other locals
     }

   This enables Jai-inspired mechanical refactoring:
   inline code -> block with captures -> lambda -> pure function

   Implementation:
   - Parser: recognize [ident, ...] before { in func defs and bare blocks
   - AST: store capture list on NODE_FUNC_DEF and NODE_BODY
   - Typer: validate captures exist in outer scope, enforce modifier rules
   - Interpreter: bind captured variables in call_scope from definition scope
   - Codegen: emit closure structs or pass captures as hidden parameters
