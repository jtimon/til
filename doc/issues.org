* OPEN #35 Switch statement
  Multi-phase implementation. Desugars in the typer to if/else chains — no new
  codegen or interpreter support needed.

  Syntax: ~switch expr { case val: body  case: body }~. Implicit break at end
  of every case (no fall-through). Default case is ~case:~ (not ~default:~).
  Tokens TOK_SWITCH, TOK_CASE already reserved in lexer.

  rstil had full switch with enum pattern matching (see ../rstil/src/test/enums.til).
  This reimplements the same feature set in phases.

** DONE Phase 1: Simple value matching
   Switch on I64, U8, Str, Bool — cases compare with ~.eq()~.
   Also works for enums since ~.eq()~ is auto-generated.

   #+begin_src til
   switch x {
   case 1: println("one")
   case 2: println("two")
   case: println("other")
   }
   #+end_src

   Desugars to:
   #+begin_src til
   {
       _sw := x
       if _sw.eq(1) { println("one") }
       else if _sw.eq(2) { println("two") }
       else { println("other") }
   }
   #+end_src

   Parser: NODE_SWITCH with children[0] = expr, children[1..] = case nodes.
   Each case node stores the match expression (or none for default) and body.
   Typer: desugar to if/else chain in anonymous scope.

** Phase 2: Range cases
   Leverages the existing Range struct from #34.

   #+begin_src til
   switch c {
   case '0'..'9': println("digit")
   case 'a'..'z': println("lower")
   case: println("other")
   }
   #+end_src

   A range case ~case A..B:~ checks ~_sw.gte(A).and(_sw.lt(B))~ (half-open,
   consistent with Range semantics). The ~..~ is already parsed as Range.new()
   by the parser — the typer recognizes Range.new() calls in case expressions
   and emits the range check instead of ~.eq()~.

** Phase 3: Enum variant matching
   Match on enum variants without payload extraction.

   #+begin_src til
   switch color {
   case Color.Red: println("red")
   case Color.Blue: println("blue")
   case: println("other")
   }
   #+end_src

   Desugars ~case EnumType.Variant:~ to ~if _sw.is_Variant()~, using the
   auto-generated ~is_Variant()~ method.

** Phase 4: Enum payload extraction
   Pattern matching with payload binding.

   #+begin_src til
   switch opt {
   case Option.Some(value):
       println(value.to_str())
   case Option.None:
       println("nothing")
   }
   #+end_src

   Desugars ~case EnumType.Variant(name):~ to:
   #+begin_src til
   if _sw.is_Variant() {
       name := _sw.get_Variant()
       body
   }
   #+end_src

   Uses auto-generated ~get_Variant()~ method. The bound variable ~name~
   is scoped to the case body.

** Phase 5: Nested patterns
   Full pattern matching as in rstil — nested enum patterns, struct
   field matching, wildcard ~_~. Lower priority; phases 1-4 cover
   most practical use cases.


* OPEN #32 Keyword arguments (kwargs) capture and forwarding
  Allow functions to capture named arguments as a map and forward them to
  other functions. Similar to Python's ~**kwargs~.

  til already has named args at call sites (~Point(x = 5, y = 10)~), but
  there's no way to capture arbitrary named args or forward them.

  Possible syntax:
  #+begin_src til
  // Capture: **kwargs param collects unmatched named args as Map
  wrapper := proc(prefix: Str, **opts: Map) {
      create(prefix, **opts)  // forward as named args
  }
  #+end_src

  Language support for reference:
  | Language   | Kwargs capture     | Kwargs forward     |
  |------------+--------------------+--------------------|
  | Python     | ~**kwargs~         | ~f(**kwargs)~      |
  | Ruby       | ~**kwargs~         | ~f(**kwargs)~      |
  | Go         | None               | None               |
  | JavaScript | None (object conv) | None (~{...opts}~) |
  | Kotlin     | None               | None               |
  | Swift      | None               | None               |

  Only Python and Ruby have true kwargs with forwarding. Most other languages
  use map/object passing as a convention instead.

  Low priority — named args already cover most use cases. Kwargs would mainly
  help with generic wrappers and delegation patterns.

* OPEN #31 Variadic argument forwarding (splat)
  When a variadic function receives args as an Array, there's no way to forward
  them to another variadic function. Need a splat/spread syntax at call sites.

  Example use case:
  #+begin_src til
  TODO := proc(parts: ..Str) {
      panic("TODO", format(..parts))
  }
  #+end_src

  Proposed syntax: ~..array_expr~ at call site (mirrors ~..Type~ in declaration).

  Other languages for reference:
  - Python: ~def f(*args): g(*args)~
  - Go: ~func f(args ...T) { g(args...) }~
  - JS: ~function f(...args) { g(...args) }~
  - Ruby: ~def f(*args); g(*args); end~
  - Kotlin: ~fun f(vararg args: T) { g(*args) }~

  Would allow TODO, custom logging, and other wrapper functions to forward
  variadic args without manually iterating.

* OPEN #28 Aliases: implicit ref for comptime-immutable declarations
  When `a := b` and b is provably immutable and comptime, `a` should be
  an alias (implicit ref — pointer to same storage) instead of a clone.
  No new syntax needed: the compiler detects when aliasing is safe.

  A value is alias-eligible when it is:
  - A literal (42, "hello", true)
  - Another alias or constant
  - The result of a pure func or macro call where all args are comptime
  - NOT a proc call result
  - NOT a mut variable
  - NOT a ref to a mutable binding

  The existing is_comptime helper from precomp.c can help identify
  compile-time-evaluable expressions. The key insight: if the value
  can never change, there's no reason to copy it — just point to it.

  Benefits:
  - Zero-cost naming (naming a constant doesn't allocate)
  - Enables the compiler to share storage for repeated constants
  - No user-visible behavior change (immutable either way)

  Ownership: only the original value is deleted, aliases are skipped
  by ASAP (same as ref declarations today). The typer/ASAP already
  handles ref lifetime correctly — this extends it to auto-detected cases.


* OPEN #25 Support for the 4 main operative systems
  :DEPENDENCIES: #12
  til currently only builds and runs on GNU/Linux. Should support the 4
  main operative systems, in priority order:
  1. GNU/Linux/BSD (current, done)
  2. TempleOS
  3. macOS
  4. Windows

  This is not just "til builds natively on each OS." The goal is full
  cross-compilation: build a til binary for any target from any host.
  The matrix is 3×4 + 1×1 = 13 combinations: Linux, macOS, and Windows
  can each target all 4 platforms, but TempleOS as a host only targets
  itself (cross-compiling from TempleOS would mean replacing gcc/clang,
  which is out of scope).

  ctil (the C bootstrap compiler) stays Linux-only. Once til is
  self-hosting (#12), til replaces ctil and handles all platforms.
  Bootstrap path: ctil on Linux compiles self-hosted til → til can
  then cross-compile itself and any .til program to any target.

  The C codegen path helps: emit C + invoke the target's C compiler
  works for most POSIX targets. But TempleOS breaks this pattern
  (no C compiler), requiring a dedicated HolyC codegen backend.

  The builder must select the appropriate codegen backend per target.

  Common areas affected across platforms:
  - Build system (builder that selects backend per target)
  - FFI / dlsym (platform-specific dynamic loading)
  - System primitives (spawn_cmd, readfile, etc. — POSIX vs platform APIs)
  - GUI mode (raylib on POSIX/Windows, native primitives on TempleOS)
  - Audio (miniaudio on POSIX/Windows, native on TempleOS)

** TempleOS
   :DEPENDENCIES: #12
   The most interesting and challenging target. TempleOS has no C compiler,
   no POSIX, no libc — just HolyC and its own kernel/graphics/audio APIs.

   Requires a new holybuilder backend that emits HolyC instead of C.
   HolyC is C-like enough that the translation should be mostly mechanical
   (similar types, similar control flow), but differences in calling
   conventions, memory management, and string handling will need attention.

   For mode gui: raylib doesn't exist on TempleOS, so we need to recreate
   raylib's interface using TempleOS graphics primitives (GrBlot, GrLine,
   GrRect, etc.) and audio primitives. This is likely easier than most
   people would expect — TempleOS has direct framebuffer access and simple
   audio APIs, and raylib's 2D API surface is not that large. The hard
   part is the impedance mismatch (TempleOS is cooperative multitasking,
   640x480 16-color default, etc.), not the drawing primitives themselves.

   Running the same mode gui program (e.g. music.til) on both GNU/Linux
   via raylib AND TempleOS via native HolyC primitives would be a serious
   language flex — proving til programs are truly platform-independent.

   For mode server: TempleOS docker containers could be extremely small
   (no libc, no userland, minimal kernel), making til a compelling choice
   for lightweight server deployments on TempleOS-based containers.

   Self-hosting (#12) is a hard dependency: TempleOS has no C compiler,
   so til must be able to compile itself to bring its own compiler onto
   the platform. There's no way to cross-compile C for TempleOS from Linux.

   Can test from Linux using QEMU with a TempleOS VM image. Copy til
   files in via a virtual disk, run them, capture output. This is the
   standard way to develop for TempleOS from a modern host.

** macOS
   Closest to the current GNU/Linux target. All C code (ccore.c, ext.c,
   dispatch.c) is already POSIX-portable — fork, execl, waitpid, dlopen,
   dlsym all work the same on macOS. No cross-compilation needed: ship
   source, user runs `make` with clang natively.

   Makefile changes needed:
   - `-ldl` not needed on macOS (dlopen is in libc)
   - `-lffi` needs `brew install libffi` + Homebrew `-I`/`-L` paths
   - `-rdynamic` works with clang
   - Platform detection to conditionally apply the above

   Runtime differences:
   - `.dylib` instead of `.so` for dynamic libraries (mode gui, FFI)
   - Homebrew paths for raylib headers and libs

   No `#ifdef __APPLE__` needed in the C source itself — only in the
   Makefile and possibly in the codegen output (linker flags when
   compiling .til programs that use mode gui).

** Windows
   Furthest from POSIX, but possibly dying — don't wait for self-hosting.
   Options: MSVC, MinGW, or cross-compile with zig cc. Dynamic loading
   via LoadLibrary/GetProcAddress instead of dlopen/dlsym. Process
   spawning via CreateProcess instead of fork/exec. raylib supports
   Windows natively, so mode gui should work once the build system
   handles .dll and Windows linking.

   Can test from Linux using Wine as a first step before touching a
   real Windows machine.

* OPEN #23 shallow keyword: direct C calling convention for ext_func/ext_proc
  til's FFI currently uses a pointer calling convention: all args are heap
  pointers (til_I64 *, til_Str *, etc.) and returns are heap-allocated.
  External C libraries (raylib, SDL, etc.) use natural C types — int, float,
  const char *, structs by value. This mismatch forces users to write wrapper
  .c files that dereference pointers and convert types.

  A `shallow` keyword on ext_func/ext_proc would tell the FFI to use C's
  native calling convention instead of til's pointer convention. C doesn't
  distinguish ref from own — it's all just pointers or values — so the user
  maps til types to C semantics, and the FFI handles the conversion:

    init_window := shallow ext_proc(w: I64, h: I64, title: Str) {}
    // FFI passes: (int64_t)w, (int64_t)h, null-terminated char* from title
    // Instead of: til_I64*, til_I64*, til_Str*

  Type mapping for shallow:
  - I64 → int64_t by value (dereference the pointer)
  - U8 → uint8_t by value
  - Bool → bool by value
  - Str → const char * (null-terminated copy of data)
  - Struct → pointer to flat buffer (same as current, already C-compatible)

  This eliminates the need for wrapper .c files for most external libraries.
  The companion .c file (#4 TODO) would only be needed for complex cases
  (custom memory management, opaque types, callbacks).

  Implementation:
  - Parser: recognize `shallow` before ext_func/ext_proc
  - AST: flag on FUNC_DEF (e.g. is_shallow)
  - dispatch.c: use ffi_type_sint64/uint8/etc. instead of ffi_type_pointer,
    dereference values before passing, handle Str null-termination
  - builder.c: emit direct C types instead of til_* wrappers
  - dlsym: look up function name directly (no til_ prefix)

* OPEN #22 Operator syntax: desugar operators to method calls
  :DEPENDENCIES: #12
  Syntactic sugar so a + b desugars to a.add(b), a < b to a.lt(b), etc.
  Purely a parser transform — no changes needed in typer, interpreter, or
  codegen. The parser rewrites operator expressions into method call AST
  nodes before anything downstream sees them.

  Operator → method mapping:
  - Comparison: < → lt, <= → lte, > → gt, >= → gte, == → eq, != → neq
  - Arithmetic: + → add, - → sub, * → mul, / → div, % → mod

  Precedence (low to high):
  - == != < <= > >=
  - + -
  - * / %

  Parenthesized expressions (a + b) * c override precedence. The parser
  must disambiguate grouping parens from function call parens: ( after an
  identifier/field-access is a call, otherwise it's grouping.

  Parsing approach: Pratt parsing (precedence climbing) or one function
  per precedence level (mutually recursive descent). Both work; Pratt is
  more compact and easier to extend.

  The cmp/initer synergy makes this especially nice for comparisons:
  implement cmp once, the initer auto-generates eq, neq, lt, gt, lte, gte,
  and operator desugaring gives you <, <=, >, >=, ==, != for free. If you
  override eq with a faster implementation, neq (and !=) automatically
  benefits since neq is generated as a.eq(b).not().

** Phase 1: Comparison operators
   < <= > >= == !=
   All same precedence, binary, no chaining ambiguity.
   Desugars: a < b → a.lt(b), a != b → a.neq(b), etc.

** Phase 2: Arithmetic operators
   + - * / %
   Introduces real precedence: * / % bind tighter than + -.
   Desugars: a + b * c → a.add(b.mul(c)).
   Requires grouping parentheses support.

   Also: ~++~ / ~--~ as unary sugar for inc/dec (methods added in #34 Phase 2).
   ~i++~ → ~i = i.inc()~, ~i--~ → ~i = i.dec()~.
   ~+=~ / ~-=~ as compound assignment sugar: ~i += 1~ → ~i = i.add(1)~.

* OPEN #18 Tuple: fixed-size heterogeneous collections
  Holds a fixed number of elements of different types.
  Immutable (elements set at construction, no push/pop).

  Syntax:
    point := (10, 20)              // inferred as (I64, I64)
    record := (42, "alice", true)  // inferred as (I64, Str, Bool)
    x := record.0                  // I64, value 42
    name := record.1               // Str, value "alice"
    t : (Bool, I64, Str) = (false, 1, "two")  // explicit type annotation

  Literal-only construction (no Tuple() constructor). Minimum 2 elements.
  Access by numeric index (.0, .1, .N) with compile-time bounds checking.
  Type annotation uses parenthesized type list: (Bool, I64, Str).

** Option A: Auto-generated structs (desugared in typer)
   Each tuple shape generates a hidden struct. The typer rewrites tuple
   literals into struct constructors before interpreter/codegen runs.

   (I64, Str, Bool) generates:
     Tuple_I64_Str_Bool := struct { _0 := 0; _1 := ""; _2 := false }

   Pros:
   - Reuses all existing struct machinery (codegen, interpreter, initer,
     clone/delete, field access) — very little new code
   - Zero runtime overhead (typed C struct fields, no offset calcs)
   - .N becomes field access on _N — works through existing NODE_FIELD_ACCESS
   - One representation for both interpreter and codegen

   Cons:
   - Many hidden struct defs (one per unique tuple shape)
   - Name mangling gets ugly with nesting: Tuple_Tuple_I64_I64_Str
   - Each shape is a distinct type — can't write generic "any tuple" functions

   Changes needed:
   - AST: add NODE_TUPLE_LIT
   - Parser: (expr, expr, ...) syntax; TOK_NUMBER after DOT for .N access;
     (Type, Type, ...) in type position for annotations
   - Typer: synthesize struct defs, rewrite NODE_TUPLE_LIT to NODE_FCALL
   - Interpreter/codegen/initer: no changes (struct machinery handles it)

** Option B: Single Tuple struct with parallel Vecs (like old rstil List)
   One Tuple type for all shapes. Raw data buffer + parallel Vecs of
   type_names and type_sizes as runtime metadata. Compiler validates
   .N access at compile time using its own type tracking.

   Tuple := struct {
       mut data : Ptr = Ptr()
       mut _len : I64 = 0
       mut type_names : Vec = Vec()  // Vec<Str> per element
       mut type_sizes : Vec = Vec()  // Vec<I64> per element
   }

   Pros:
   - One type for all tuples — simpler type system
   - Functions can accept any Tuple regardless of shape
   - Familiar pattern from old rstil List

   Cons:
   - Runtime overhead (offset calculations, Vec allocations for metadata)
   - Codegen needs casts: *(int64_t*)(tuple->data + offset)
   - Clone/delete must iterate Vecs at runtime with dyn_call per element
   - Doesn't reuse struct machinery — lots of new code needed
   - .N return type known at compile time but runtime extraction is dynamic

** Option C: Dynamic list (Lisp-style)
   Fully dynamic — each element stored as Dynamic, no compile-time type
   tracking. Like Python lists or Lisp cons cells.

   list := [42, "alice", true]   // all elements are Dynamic
   x := list.0                   // returns Dynamic, needs cast to use

   Pros:
   - Simplest implementation — basically a Vec of Dynamic values
   - Most flexible — can hold anything, mix types freely
   - No struct generation, no type metadata Vecs

   Cons:
   - No compile-time type safety — .N returns Dynamic
   - User must cast/assert types at point of use
   - Runtime type errors instead of compile-time
   - Boxing overhead for primitives
   - Requires allowing Dynamic fields/locals (currently forbidden)

* WIP #17 Global variable access, explicit capture syntax
  :DEPENDENCIES_OF: #12 #13
  Variables declared in script-mode root scope (or any outer scope) are
  invisible at runtime inside function/proc bodies. The typer passes
  (TypeScope chains via parent pointers and finds the variable), but the
  interpreter crashes or errors.

  Reproduction: src/test/globals.til, src/test/globals_cli.til

  func case: "error: undefined variable 'x'" — clean error from NODE_IDENT
  NULL check. proc case: SEGFAULT — NODE_DECL move-semantics path has no
  NULL check on scope_get result.

** Root cause

   expr_uses_var() in typer.c (line 1239) returns 0 for NODE_FUNC_DEF,
   meaning the ASAP pass doesn't see variable references inside nested
   function/proc bodies. When x is declared at top level and referenced
   only inside a proc body, ASAP thinks x is unused and inserts
   I64.delete(x) right after the declaration. When the proc later runs
   and tries to access x, the value has already been freed/nulled.

   func case: I64.clone(x) inside the function body tries to clone x
   whose value is val_none → clean "undefined variable" error (from
   precomp, which also can't find x in its limited scope).

   proc case: I64.clone(x) dereferences v.i64 which is NULL → SEGFAULT.
   The NODE_DECL move-semantics path also lacks a NULL check on
   scope_get result, which would be a secondary crash path.

   Secondary issue: precomp creates its own interpreter scope with only
   func/struct/enum defs (no variable decls), then tries to fold pure
   func calls. If the func accesses a global variable, precomp's
   eval_expr hits the error path and exit(1)s before the interpreter
   even runs.

** DONE Phase 1: Fix global variable access

   Program-scope globals live for the program's lifetime — they should
   never be ASAP-deleted. The scavenger handles removing unused globals.

   1. Skip ASAP deletion for program-scope variables: added skip_delete
      flag to LocalInfo in typer.c. Variables with a declaration at program
      scope (is_program_scope && decl_idx >= 0) get skip_delete=1. The
      ownership-transfer check still runs for them (use-after-own errors),
      but no delete calls are inserted. Variables captured by nested
      functions also get skip_delete=1 (expr_used_in_nested_func check).
   2. Added NULL check in interpreter.c NODE_DECL move-semantics path.
   3. Fixed precomp: added func_uses_unknown_globals() to skip folding
      func calls whose body references variables not in the precomp scope.
      Also made track_literal() populate the precomp interpreter scope
      with known constant values (not just the known map).
   4. Fixed builder: top-level variable declarations are emitted as
      file-scope C globals (before function definitions) so they're
      accessible from emitted til functions/procs. This is necessary
      because til functions are emitted as separate C functions, and C
      doesn't support nested functions capturing locals — so top-level
      vars must be file-scope for functions to reference them. The
      values are then assigned inside main() where initialization runs.
      In cli mode, globals are initialized at the start of main()
      before the user's main body.
      Note: ref declarations are skipped from file-scope globals since
      they are pointer aliases (local to main), not owned values that
      other functions need to access.

   Tests: src/test/globals.til (script mode), src/test/globals_cli.til (cli mode).

** Phase 2: Explicit capture syntax

   Add syntax for listing captured variables from outer scopes, both for
   function definitions and anonymous blocks. Based on rstil/doc/first_class.org.

   Function captures (closures):
     scale := 10
     f := func(a: I64) returns I64 [scale] { return a.mul(scale) }

   Capture modifiers:
     [x]      — const reference (default)
     [mut x]  — mutable reference
     [copy x] — snapshot (clone at capture time)
     [own x]  — ownership transfer

   Block captures (restrict access):
     [total, data] {
         // can only see total and data, not other locals
     }

   This enables Jai-inspired mechanical refactoring:
   inline code -> block with captures -> lambda -> pure function

   Implementation:
   - Parser: recognize [ident, ...] before { in func defs and bare blocks
   - AST: store capture list on NODE_FUNC_DEF and NODE_BODY
   - Typer: validate captures exist in outer scope, enforce modifier rules
   - Interpreter: bind captured variables in call_scope from definition scope
   - Codegen: emit closure structs or pass captures as hidden parameters

* WIP #2 ref keyword: non-owning references for returns, declarations, and struct fields
  Currently all returns are owned by the caller (heap-allocated, caller must
  delete). This breaks when a function wants to return one of its args:

    min := func(a: I64, b: I64) returns I64 {
        if a.lt(b) { return a } else { return b }
    }

  Either a or b would be owned twice and deleted twice (double-free).

** FIXED ref returns
   returns ref I64 — the caller does NOT own the returned value and must
   not delete it. ASAP destruction skips ref return values.

    min := func(a: I64, b: I64) returns ref I64 {
        if a.lt(b) { return a } else { return b }
    }

   Typer validates: return value must be a param or ref variable.

** FIXED ref declarations
   Required when binding the result of a ref-returning function:
   ref my_min := min(34, 35). ASAP destruction skips ref variables.

   Typer validates: RHS must be a ref-returning function call or a
   ref/param variable. Cannot pass ref variable to 'own' parameter.

   ASAP lifetime extension: args to ref-returning calls have their
   lifetimes extended to at least the ref variable's last use, preventing
   use-after-free.

   Interpreter: ref from ident uses scope_set_borrowed (shared cell).
   builder: ref decl emits pointer assignment (no malloc).

** ref struct fields
   A ref field stores a pointer to an existing value (not inline).
   clone does not deep-copy it, delete does not recurse into it —
   just copies/ignores the pointer.

   Open question: default initialization. Inline fields use literal defaults
   (x := 0), but ref fields point to external values that may not exist yet
   at construction time. Options include NULL/nil default, requiring the
   field at construction, or something else. TBD.

** args are already ref by default
   Function args are borrowed (ref) unless explicitly marked own.
   No change needed for args.
