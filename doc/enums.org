# Enum semantics and follow-up cleanup notes.

* Proper enums (#113)

Current intended semantic split:

- ~x.is(E.V)~
  Tag-only check. This should answer "is this enum value of variant ~V~?"
  without comparing payload contents.

- ~x.eq(y)~
  Full enum equality. For no-payload enums this is equivalent to tag
  equality. For payload enums it must compare both:
  - variant tag
  - payload value

Examples:

#+begin_src til
Direction := enum { Up, Down }
Token := enum { Num: I64, Eof }

Direction.Up.is(Direction.Up)        # true
Direction.Up.eq(Direction.Up)        # true

Token.Num(42).is(Token.Num)          # true
Token.Num(42).eq(Token.Num(42))      # true
Token.Num(42).eq(Token.Num(99))      # false
Token.Num(42).is(Token.Eof)          # false
#+end_src

Why both are needed:

- ~is~ is the branchy / UFCS-friendly check:
  ~if tok.is(Token.Num) { ... }~
- ~eq~ is ordinary value equality:
  ~if tok.eq(Token.Num(42)) { ... }~

* Switches

Desired rule for enum ~switch~:

- enum switches should be exhaustive by default
- if all variants are not covered, typer should reject them
- unless the user provides an explicit default case

That gives:

- safer enum control flow
- cleaner intent than relying on accidental fallthrough of missing cases

Examples:

#+begin_src til
switch tok {
case Token.Num(n): ...
case Token.Eof: ...
}
#+end_src

This should be accepted only if ~Token~ has exactly those variants.

#+begin_src til
switch tok {
case Token.Num(n): ...
case:
    ...
}
#+end_src

This should be accepted as explicitly non-exhaustive because it has a
default case.

* Payload mutability (#211)

Enum variant payloads are mutable by default, like struct instance
fields. In a ~switch~ case binding, the payload variable inherits the
matched value's mutability:

  - matched value is ~mut~ -> payload binding is ~ref mut~, so the case
    body can write through it (and the change is visible after the
    switch).
  - matched value is immutable -> payload binding is ~ref~, writes are
    rejected by the typer.

#+begin_src til
mut tok := Token.Num(0)
switch tok {
case Token.Num(n): n = 42       // OK: tok is mut, payload binding is ref mut
case Token.Eof:
}
// tok is now Token.Num(42)
#+end_src

To opt out, mark the variant ~const~: ~const Variant: Type~ declares an
immutable payload, and writes to its case binding are rejected even on
a mut enum. This is the parallel to ~const field~ on structs.

#+begin_src til
Tok := enum { const Num: I64, Eof }
mut t := Tok.Num(7)
switch t {
case Tok.Num(p): p = 99     // type error: cannot assign to immutable 'p'
case Tok.Eof:
}
#+end_src

Plain mut/const on the variant tag itself (no payload, e.g. ~mut Eof~)
is not meaningful and has no effect.

Known limitation: the compiled (C) backend propagates writes through
the binding into the enum's payload storage. The AST interpreter binds
payloads by copy, so writes do not propagate under ~til interpret~ --
the typer accepts the assignment in both modes but only the compiled
path observes the in-place update.
