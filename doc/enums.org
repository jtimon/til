# Enum semantics and follow-up cleanup notes.

* Proper enums (#113)

Current intended semantic split:

- ~x.is(E.V)~
  Tag-only check. This should answer "is this enum value of variant ~V~?"
  without comparing payload contents.

- ~x.eq(y)~
  Full enum equality. For no-payload enums this is equivalent to tag
  equality. For payload enums it must compare both:
  - variant tag
  - payload value

Examples:

#+begin_src til
Direction := enum { Up, Down }
Token := enum { Num: I64, Eof }

Direction.Up.is(Direction.Up)        # true
Direction.Up.eq(Direction.Up)        # true

Token.Num(42).is(Token.Num)          # true
Token.Num(42).eq(Token.Num(42))      # true
Token.Num(42).eq(Token.Num(99))      # false
Token.Num(42).is(Token.Eof)          # false
#+end_src

Why both are needed:

- ~is~ is the branchy / UFCS-friendly check:
  ~if tok.is(Token.Num) { ... }~
- ~eq~ is ordinary value equality:
  ~if tok.eq(Token.Num(42)) { ... }~

* Switches

Desired rule for enum ~switch~:

- enum switches should be exhaustive by default
- if all variants are not covered, typer should reject them
- unless the user provides an explicit default case

That gives:

- safer enum control flow
- cleaner intent than relying on accidental fallthrough of missing cases

Examples:

#+begin_src til
switch tok {
case Token.Num(n): ...
case Token.Eof: ...
}
#+end_src

This should be accepted only if ~Token~ has exactly those variants.

#+begin_src til
switch tok {
case Token.Num(n): ...
case:
    ...
}
#+end_src

This should be accepted as explicitly non-exhaustive because it has a
default case.

* Payload mutability (#211)

Enum variant payloads are mutable by default, like struct instance
fields. In a ~switch~ case binding, the payload variable inherits the
matched value's mutability:

  - matched value is ~mut~ -> payload binding is ~ref mut~, so the case
    body can write through it (and the change is visible after the
    switch).
  - matched value is immutable -> payload binding is ~ref~, writes are
    rejected by the typer.

#+begin_src til
mut tok := Token.Num(0)
switch tok {
case Token.Num(n): n = 42       // OK: tok is mut, payload binding is ref mut
case Token.Eof:
}
// tok is now Token.Num(42)
#+end_src

To opt out, mark the variant ~const~: ~const Variant: Type~ declares an
immutable payload, and writes to its case binding are rejected even on
a mut enum. This is the parallel to ~const field~ on structs.

#+begin_src til
Tok := enum { const Num: I64, Eof }
mut t := Tok.Num(7)
switch t {
case Tok.Num(p): p = 99     // type error: cannot assign to immutable 'p'
case Tok.Eof:
}
#+end_src

Plain mut/const on the variant tag itself (no payload, e.g. ~mut Eof~)
is not meaningful and has no effect.

Writes propagate in both backends: the compiled (C) backend lowers the
case binding to a real ~T *p~ pointer into the enum's payload, so
~p = 99~ becomes ~*p = 99~. The AST interpreter detects the typer-emitted
~ref T = get_payload(<ident>)~ shape, registers a payload-alias entry
on the scope, and mirrors each assignment to the binding into the
source enum's ~ei.data + ENUM_PAYLOAD_OFFSET~ via the typed
~write_i64~ / ~write_u8~ / ... primitives (see ~src/c/ext.c~).
Primitives (I64, U8, I8, I32, U32, U64, F32, Bool) and ~Str~ payloads
propagate in both backends. The compile backend lowers ~p = "lit"~ on
a ~ref mut Str~ payload binding to ~*p = (Str){.c_str=..., ...}~ (a
write-through compound literal, not a heap rebind, see #221). The AST
interpreter mirrors the assignment into ~ei.data + ENUM_PAYLOAD_OFFSET~
via either a typed primitive write (~write_i64~ / ~write_u8~ / ... in
~src/c/ext.c~) or, for ~Str~ / user struct payloads, by ~memcpy~'ing the
struct bytes after first calling ~Str_delete~-equivalent cleanup so
owned sources don't leak. Struct payloads (~Variant: SomeStruct~) work
too: the typer accepts a constructor RHS on the payload-alias ref-mut
target (the rejection that flagged "ref target cannot be assigned an
owned value" is gated by ~is_payload_alias~), the builder lowers it to
~{ Type_delete(p, &(Bool){0}); Type *_new = (Type *)Type(...); *p =
*_new; free(_new); }~, and the interp writeback memcpys the struct
bytes via the existing ~Value.Struct~ path. Nested-enum payloads
(~Variant: SomeOtherEnum~) also propagate: the interp writeback's
~Value.Enum~ branch deep-copies the new inner enum into a fresh heap
~EnumInstance~ and stores its pointer at the outer payload slot
(mirroring ~write_field~'s Ref-typed enum shape; the compile path
already worked via the same ~is_payload_alias~ compound branch). The
remaining piece is freeing the OLD inner ~EnumInstance~ at the slot
before overwriting -- a bounded per-write leak under interp; flagged
as a follow-up.

* match: the expression form (issue #244)

~switch~ is statement-only: each arm runs statements and the whole
~switch~ yields no value. ~match~ is its expression counterpart -- each
arm is a single value-expression and the whole ~match~ yields a value,
so it can appear anywhere an expression can (initializers, call
arguments, ~return~, an assignment RHS, another match's arm):

#+begin_src til
name := match c {
case Color.Red:   "red"
case Color.Green: "green"
case Color.Blue:  "blue"
}
#+end_src

~match~ reuses the whole ~switch~ machinery: the same ~case <pattern>:~
arms, payload bindings (~case Token.Num(v):~), guards
(~case <pattern> if <expr>:~), and a bare ~case:~ default. Rules that
differ from ~switch~:

- *All arms must yield the same type* -- that common type is the type of
  the match. A mismatch is reported as "match arms have incompatible
  types: 'X' vs 'Y'".
- *Exhaustive on every path* -- a match must produce a value no matter
  what. An enum subject must cover every variant (same check ~switch~
  already enforces); any other subject must supply a ~case:~ default.

Lowering (typer): a ~match~ is rewritten to a result temp plus a
synthetic ~switch~ whose every arm assigns the arm's value to the temp
(~mut _m := <default>; switch subj { case P: _m = v ... }~). The temp
declaration and switch are spliced in as statements just before the
enclosing statement, and the match node is replaced by an ~_m~
reference. The synthetic switch then goes through the ordinary switch
desugar, so nested matches, guards, bindings and enum-exhaustiveness all
fall out of the existing code (see ~infer_match_expr~ /
~lower_one_match~ in ~src/self/typer.til~).

Cross-suggesting errors steer each keyword to its role:
- ~x := switch ...~ (switch in value position) -> "switch is a statement
  and yields no value; use 'match' to select a value".
- a bare ~match ...~ statement -> "match yields a value that is unused
  here; use 'switch' for arms that run statements".
