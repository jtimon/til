* Qualifier-to-C Reference

Complete mapping of til qualifiers to generated C, by context.

** Mutability rule (~mut~ vs ~const~)

   til picks the most common mutability per context as the default,
   and the opposite keyword is rejected as redundant. The split is
   symmetric:

   | Context             | Default | Opt-out keyword | Redundant keyword (rejected) |
   |---------------------+---------+-----------------+------------------------------|
   | Local               | const   | ~mut~           | ~const~                      |
   | Parameter           | const   | ~mut~           | ~const~                      |
   | Namespace decl      | const   | ~mut~           | ~const~                      |
   | Struct field        | mut     | ~const~         | ~mut~                        |
   | Enum variant payload| mut     | ~const~         | ~mut~                        |

   In every row, only the non-default keyword carries information,
   and the typer/parser rejects the redundant one with a "redundant
   '<kw>' on <context>: <context>s are <default> by default; drop it"
   message. Diagnostics are tested by
   ~test/qualifier_const_local_error.til~,
   ~test/qualifier_const_param_error.til~,
   ~test/qualifier_const_namespace_error.til~,
   ~test/qualifier_mut_field_error.til~, and
   ~test/qualifier_mut_payload_error.til~.

   Issue #164 adds the third leg: a ~mut~ local, ~mut~ parameter,
   or ~priv mut~ namespace declaration that the body (or, for ns
   decls, the declaring file) never uses to write is also flagged.
   The detector counts these as writes: a direct assignment
   (~name = ...~), a field-chain assignment (~name.f = ...~), a UFCS
   method call on the name (~name.method(args)~), passing the name
   (or one of its fields) to a callee parameter declared ~mut~/~own~
   or typed ~Dynamic~ (a raw pointer the callee may write through,
   e.g. ~move~/~memcpy~/~write_ptr~), the name as the source of a
   ~case V(p):~ ref-mut payload binding, and the name as the source
   of a ~ref mut~ declaration (a launder use that requires the
   source to be ~mut~). Namespace declarations additionally record
   a write when a ~Type.member = ...~ FieldAssign type-checks.

   Note that ~mut~ only governs the binding and its direct fields
   (~p = ...~, ~p.f = ...~). Mutating state reachable *through* a
   non-mut binding -- a child element via a ~for x in p.children~
   loop variable, or the buffer behind a pointer field -- does not
   require ~mut~ on the binding, so such params are correctly
   flagged as redundant. The diagnostic is intentionally conservative
   on the write side: it would rather miss a redundant ~mut~ (e.g. a
   binding only passed to a read-only ~Dynamic~ param like ~is_null~)
   than wrongly reject a ~mut~ that is actually needed.

   Namespace decls are flagged only when both ~priv~ and ~mut~ are
   present and no write is seen in the declaring file. ~priv~ blocks
   downstream consumers from writing, so the single-file view is
   complete; a public ~mut~ ns decl (e.g. ~HashMap.hasher~) is left
   alone because a downstream consumer may legitimately write it
   even when the library itself does not.

** Arguments (function/proc parameters)

| til                       | C                | semantics                            |
|---------------------------+------------------+--------------------------------------|
| ~arg: Type~               | ~Type *arg~      | borrowed ref (read-only)             |
| ~mut arg: Type~           | ~Type *arg~      | mutable ref (callee can modify)      |
| ~own arg: Type~           | ~Type *arg~      | ownership transfer (callee frees)    |
| ~shallow arg: Type~       | ~Type arg~       | pass by value (copy on stack)        |

** Return types

| til                       | C return type    | semantics                            |
|---------------------------+------------------+--------------------------------------|
| ~returns Type~            | ~Type *~         | heap-allocated, caller owns          |
| ~returns shallow Type~    | ~Type~           | value return (on stack)              |
| ~returns ref Type~        | ~Type *~         | borrowed pointer, caller must NOT free |

** Locals

| til                       | C (scalar)       | C (compound)     | semantics                            |
|---------------------------+------------------+------------------+--------------------------------------|
| ~x := expr~               | ~Type x = ...~   | ~Type *x = ...~  | owned immutable local (default)      |
| ~mut x := expr~           | ~Type x = ...~   | ~Type *x = ...~  | owned mutable local                  |
| ~ref x : Type = expr~     | ~Type *x = ...~  | ~Type *x = ...~  | borrowed immutable local             |
| ~ref mut x : Type = expr~ | ~Type *x = ...~  | ~Type *x = ...~  | borrowed mutable local               |

Scalar types: I64, U8, I8, I16, I32, U32, U64, F32, Bool.
Compound types: Str, user-defined structs, enums.

** Struct fields

Unlike locals and parameters, struct instance fields are MUTABLE by
default (#211): a struct is usually a bag of state updated over its
lifetime, so the common case needs no annotation. ~const~ opts a field
out (immutable after construction). An explicit ~mut~ on a field is
redundant and rejected by the typer.

| til                          | C struct member  | semantics                            |
|------------------------------+------------------+--------------------------------------|
| ~field := default~           | ~Type field~     | inline value, mutable (default)      |
| ~const field := default~     | ~Type field~     | immutable inline value               |
| ~own field := default~       | ~Type *field~    | owned heap pointer (mutable)         |
| ~own const field := default~ | ~Type *field~    | immutable owned heap pointer         |
| ~ref field : Type = NULL~    | ~Type *field~    | borrowed pointer (caller owns data)  |
| ~ref const field : Type = NULL~ | ~Type *field~ | immutable borrowed pointer           |

** Key rules

- ~type_name_to_c("I64")~ = ~I64 *~ (pointer form, for normal args/returns/locals)
- ~type_name_to_c_value("I64")~ = ~I64~ (value form, for ~shallow~ args/returns)
- In generated C, ~DEREF(x)~ dereferences a pointer: ~(*(x))~
- Args are borrowed by default, so explicit ~ref~ parameters are redundant
- Returns are owned by default, so explicit ~returns own Type~ is redundant
- Locals are owned by default, so explicit ~own~ on locals is redundant:
  ~own x := ...~, ~own mut x := ...~, ~own x : Type = ...~, and
  ~own mut x : Type = ...~ are all rejected
- Struct fields are mutable by default, so explicit ~mut~ on a field is
  redundant and rejected; use ~const~ to make a field immutable
  (namespace decls inside a struct keep the const-by-default rule and
  still opt in with ~mut~)
- Locals, parameters, and namespace declarations are immutable by
  default; ~const~ is redundant in those positions and the parser
  rejects it ("redundant 'const' on local/parameter/namespace
  declaration ... drop it"). ~const~ is meaningful only on struct
  fields and on enum variant payloads
- Enum variant payloads are mutable by default (same rule as struct
  fields); the payload binding in a ~switch~ case inherits the matched
  value's mutability (~ref mut~ when the enum value is ~mut~, ~ref~
  otherwise). ~const Variant: Type~ opts a payload out (writes to the
  case binding are then rejected even on a mut enum); a redundant
  ~mut~ on a variant declaration is rejected with the symmetric
  "redundant 'mut' on variant payload ... drop it" message. See
  doc/enums.org "Payload mutability"
- ~Dynamic~ is the raw owned-pointer exception:
  - use ~buf : Dynamic = malloc(...)~ for owned allocations
  - use ~ref elem : T = buf~ or ~ref elem : T = ptr_add(buf, i)~ for typed access
  - a local owned ~Dynamic~ must either transfer ownership to an ~own~ arg/field
    or be explicitly freed with ~free(own buf)~
- ~shallow~ is for args/returns only, not locals or fields
- Issue #223 (auto-shallow): primitives (I8/I16/I32/I64, U8/U16/U32/U64,
  F32/F64, Bool, ...) are treated as ~shallow~ in func/proc args and return
  values even without the keyword -- a plain ~x: I64~ param and a plain
  ~returns I64~ are passed/returned by value. The qualifier is inferred for
  ordinary func/proc and funcptr signatures; it does NOT apply to
  ~ext_func~/~ext_proc~ (their qualifiers map to a fixed C signature), to
  ~mut~/~own~ primitive params, or to params of a ~returns ref T~ function
  (those stay borrowed pointers so the ref can escape)
- Issue #223 step 2: because the qualifier is now inferred, an explicit
  ~shallow~ on a primitive where auto-shallow applies (ordinary func/proc,
  not ~returns ref~) is redundant, and the typer now REJECTS it: "redundant
  'shallow' on primitive ...; drop it". ~shallow~ remains required on
  ~ext_func~/~ext_proc~ (it describes the fixed C signature) and on a
  ~returns ref T~ function's primitive params. (The funcptr-signature
  variant of the rule is sketched but not yet active -- ~infer_func_sig_expr~
  is a ~func~ with no ~ctx~ to raise the diagnostic from.)
- Issue #223 step 3: a STRUCT is auto-shallow (treated as ~shallow~ in
  args/returns without the keyword) when it is safe to copy byte-for-byte:
  ~MyStruct.size().lte(UPtr.size())~ (<= 8 bytes), no ~own~ fields, no
  ~ref~ fields, and every field's type is itself auto-shallow (a primitive
  or another auto-shallow struct). Otherwise it stays pointer-passed. The
  transitive field check closes the hole where a size-8 struct embeds a
  deep struct inline.
- Issue #223 enum follow-up: a PAYLOADLESS enum (a bare tag, every variant
  without a payload) is auto-shallow when its size fits a machine word
  (~size <= UPtr.size()~). A bare tag is a plain integer, always
  value-copy-safe. Payload-BEARING enums are not auto-shallow yet: the
  generic ~get_payload~ helper hardcodes the payload offset at
  ~sizeof(void*)~ (8), which is wrong for a <= 4-byte-aligned payload (union
  at offset 4) -- a pre-existing limitation tracked as a separate follow-up.
- Issue #223 step 4: on a STRUCT (or enum), an explicit ~shallow~ is a
  force/override, NOT an error. Unlike a primitive (step 2, rejected as
  redundant), a struct ~shallow~ is always honored: it forces by-value
  passing and bypasses the step-3 eligibility check, so even a struct too
  large or otherwise ineligible for auto-shallow is passed by value when the
  user writes ~shallow~ (memory safety of value-copying a struct with ~own~
  fields is then the user's responsibility, as in C). Needs no special case:
  the step-1/3 normalization only promotes the Ref/Own default to Shallow, so
  an already-~Shallow~ qualifier is left untouched; and the redundant-shallow
  rejection is gated on ~is_primitive~, so it never fires for a struct/enum.
- ~shallow~ params are already values — no ~DEREF~ needed, but need ~&(Type){val}~ to pass to pointer-expecting functions
- ~returns shallow~ functions return ~Type~ not ~Type *~ — callers must not dereference the result
- Scalar locals are stack values: ~I64 x = value~, not ~I64 *x = malloc(...)~
- Scalar locals use ~&(Type){x}~ when passed to pointer-expecting functions
- Delete calls for scalar locals are suppressed (no heap allocation to free)
- Ordinary non-~ref~, non-~Dynamic~ locals are stack-backed values and are passed
  as ~&x~ to pointer-expecting functions
- ~Dynamic~ locals stay on the raw-pointer path

** Internal representation

All qualifier contexts (args, returns, locals, fields) use the same
~OwnType~ enum: ~Own~, ~Ref~, ~Shallow~.

| Context | Default   | Meaning of Shallow       | Meaning of Own          | Meaning of Ref        |
|---------+-----------+--------------------------+-------------------------+-----------------------|
| Args    | ~Ref~     | pass by value (stack)    | ownership transfer      | (redundant, = default)|
| Returns | ~Own~     | value return (stack)     | (redundant, = default)  | borrowed pointer      |
| Locals  | ~Shallow~ | owned stack value        | (forbidden, = default)  | borrowed pointer      |
| Fields  | ~Shallow~ | inline value in struct   | owned heap pointer      | borrowed pointer      |

For ~Param~ and ~FunctionDef~, the enum is used directly (~own_type~,
~return_own_type~).  For ~Declaration~ (locals and fields), the same
~own_type~ field replaces the former ~is_own~ / ~is_ref~ booleans.
~Shallow~ is the natural default for declarations: inline for fields,
stack-owned for locals.


** Visibility (priv keyword)

   Issue #179. til is public by default; the ~priv~ keyword opts
   individual declarations out of visibility. This is the inverse
   of Rust, where items are private by default and ~pub~ opts them
   in to exposure. The default keeps libraries that expose almost
   everything noise-free; the unusual case (genuinely internal
   helpers) carries the marker.

*** Where ~priv~ may appear

    | Context                                  | Effect                                         |
    |------------------------------------------+------------------------------------------------|
    | Top-level ~func~ / ~proc~ declaration    | hidden from importers (cross-file only)        |
    | Top-level ~struct~ declaration           | type name + constructor hidden from importers  |
    | Top-level ~enum~ declaration             | type name + variants hidden from importers     |
    | Top-level value (~mut~ / immutable)      | name hidden from importers                     |
    | Field inside a ~struct~ body             | hidden outside the struct's namespace methods  |
    | Function inside a ~namespace:~ block     | hidden outside the struct's namespace methods  |

*** Cross-file vs in-struct rules

    Top-level ~priv~ items follow a *cross-file* rule: they are
    visible everywhere within the declaring file (top-level helpers,
    other top-level functions, even outside the struct that uses
    them) and invisible from any importer.

    Struct members (fields and namespace methods) marked ~priv~
    follow a stricter *in-struct* rule: they are visible only from
    inside the struct's own namespace methods. Same-file code that
    is not part of the struct cannot touch them either.

*** Examples

    #+begin_src til
    // Library file: lib_helper.til
    mode lib

    // Public function -- visible to importers.
    add_one := func(x: I64) returns I64 {
        return x.add(internal_inc(1))
    }

    // Private to this file -- importers cannot reference it.
    priv internal_inc := func(x: I64) returns I64 {
        return x.add(1)
    }

    // Private struct -- importers cannot construct it or name it.
    priv Secret := struct {
        mut value := 0
    namespace:
        show := func(self: Secret) returns I64 { return self.value }
    }

    // Public struct with a private field and a private method.
    Box := struct {
        mut tag := 0
        priv mut hidden := 0
    namespace:
        // Public method may freely read the priv field of the
        // same struct.
        hidden_plus_one := func(self: Box) returns I64 {
            return self.hidden.add(1)
        }
        // Private namespace method -- only callable from another
        // Box method.
        priv bump := func(mut self: Box) {
            self.hidden = self.hidden.add(1)
        }
        // Public wrapper that calls the priv method internally.
        bump_and_show := func(mut self: Box) returns I64 {
            Box.bump(self)
            return self.hidden
        }
    }

    // Private top-level globals -- importers cannot read or write.
    priv MAGIC : I64 = 42
    priv mut counter := 0
    #+end_src

*** Not in scope

    - No module/crate-level visibility levels (no ~pub(crate)~,
      no ~pub(super)~). til's import graph is flat: a thing is
      either visible to importers or it isn't.
    - ~priv~ has no effect on builtins, ~ext_func~, ~ext_proc~,
      or ~ext_struct~.
    - The keyword is purely additive. Any existing declaration
      stays public unless ~priv~ is added explicitly.

*** Diagnostics

    - Cross-file violation: ~error: 'X' is private to '<decl-file>'~
    - In-struct violation:  ~error: 'Type.member' is private to its struct~
    - Unused priv item (issue #187): a ~priv~ declaration that no
      code in its declaring file references is dead code -- no
      importer can possibly reach it, since the priv keyword
      blocks that. The typer flags it and suggests removing the
      keyword:

      | Kind                  | Diagnostic                                                       |
      |-----------------------+------------------------------------------------------------------|
      | top-level func        | ~error: unused priv func 'X'; remove 'priv' to expose it~        |
      | top-level struct      | ~error: unused priv struct 'X'; remove 'priv' to expose it~      |
      | top-level enum        | ~error: unused priv enum 'X'; remove 'priv' to expose it~        |
      | top-level value       | ~error: unused priv value 'X'; remove 'priv' to expose it~       |
      | struct field          | ~error: unused priv field 'Foo.x'; remove 'priv' to expose it~   |
      | namespace member      | ~error: unused priv ns member 'Foo.f'; remove 'priv' to expose it~ |

      The escape hatch from #86 (~_~-prefixed names skip the
      check) still applies, but the recommended fix is "remove
      priv" rather than "rename to _": hiding-by-default doesn't
      fit a type or function name. If you genuinely want to keep
      a dead helper as scaffolding, prefer ~_~-prefix; if you
      want it exposed to importers, drop the ~priv~.

      Auto-generated bodies (clone / delete / size / enum
      eq / to_str / variant constructors) do *not* count as
      uses for this check. Without that gate, every priv field
      and ns member would always look "used" because the
      auto-gen clone reads every field by name. ~FunctionDef~
      carries an ~auto_generated~ flag set by initer's helpers;
      the typer tracks ~auto_gen_depth~ around such bodies and
      skips "mark used" sets while inside one. *Hand-written*
      ~clone~/~delete~/~size~ stay regular user code
      (~auto_generated == false~) and DO mark their reads, so
      ~priv~ fields used only by a user-supplied clone are not
      misreported as unused (regression test in
      ~test/priv.til:test_user_clone_marks_priv_used~).

*** Parser-synthesised declarations

    The parser emits a small number of compiler-internal Decls
    that the user never sees by name:

    - ~_lambda_<N>~: the lifted carrier for an anonymous
      ~func() {...}~ literal, created by ~maybe_lift_lambda~ so
      the existing fn-ptr machinery can pick it up.
    - ~_Fn_*~: the synthetic FuncSig type for an inline
      ~Fn(T1, T2) returns T~ annotation, created by
      ~parse_fn_signature~.

    Both are marked ~priv~ at construction. They're file-local
    by construction (only referenced by the synthetic Ident
    written immediately after); the ~priv~ marker means the
    unused-priv check would catch any future drift if a phase
    started leaking these names cross-file. The ~_~ prefix
    keeps the unused-decl check itself quiet for the synthetic
    names.
