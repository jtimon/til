* Pipeline

This is the current high-level pipeline implemented in
[[file:../src/til.til][src/til.til]].
It is intentionally short and code-oriented.

* Orchestration and shared state

Two modules sit underneath every phase below:

- Load:
  - `prepare_program(...)` and friends
  - Implemented in til: [[file:../src/self/loader.til][src/self/loader.til]]
  - Drives the whole pipeline: discovers and reads source files,
    resolves recursive `import(...)`, picks the mode, then runs
    tokenize -> parse -> init -> type -> asap -> constfolder -> scavenge
    in order before handing the merged AST to a back end. The CLI
    (`src/til.til`) and the REPL both go through here.

- Context:
  - `Context`, `TypeScope`, `TypeBinding`, ...
  - Implemented in til: [[file:../src/self/context.til][src/self/context.til]]
  - The shared state threaded through the phases: imported units,
    the active mode, the type scope and its name/type bindings, and
    the FFI map. Every phase reads and mutates the same `Context`.

1. Tokenize

- `tokenize(source, path)`
- Implemented in til: [[file:../src/self/lexer.til][src/self/lexer.til]]
- Turns source text into a stream of tokens.

2. Parse

- `parse(tokens, path, mode_out)`
- Implemented in til: [[file:../src/self/parser.til][src/self/parser.til]]
- The CLI loads:
  - `src/core/core.til`
  - user source
  - recursive `import(...)` files
  - optional mode auto-import file from `src/modes/*.til`
- Result: one merged AST.

3. Init declarations

- `init_declarations(ast, scope)`
- Implemented in til: [[file:../src/self/initer.til][src/self/initer.til]]
- Responsibilities:
  - pre-register types, funcs, globals, aliases
  - build initial `TypeScope`
  - auto-generate helpers such as enum/struct namespace methods
  - prepare enough metadata for typing

4. Type check

- `type_check(ast, scope, mode)`
- Implemented in til: [[file:../src/self/typer.til][src/self/typer.til]]
- Responsibilities:
  - resolve names and types
  - infer expression types
  - lower/check some constructs during typing
  - lower throw/catch to plain til (mut locals + if-guards) so
    downstream stages don't need new node kinds; see
    [[file:./throws.org][doc/throws.org]] for the surface and recipes
  - reject invalid programs

5. Asap destruction

- `insert_free_calls(body, scope, owns_scope)` (per body)
- Implemented in til: [[file:../src/self/garbager.til][src/self/garbager.til]]
- Responsibilities:
  - per-binding live-range analysis (~collect_scope_locals~,
    ~extend_ref_local_lifetimes~)
  - diagnose use-after-ownership-transfer
  - promote Decl ownership to Own where needed
  - insert ~Type.delete()~ / ~free()~ at the last use of every
    local, before reassignments, and on every early-exit edge
    (return / break / continue)
- After asap insertion, every owned local has an explicit cleanup
  call on every path that leaves its scope.
- The asap code now lives in its own ~mode lib~ module
  (src/self/garbager.til) and the program-body asap runs as its own
  phase, ~garbager_destroy(ctx)~, at the tail of
  ~init_and_type_program~ (see [[file:../issues/fixed/204.org][#204]]).
  The remaining per-body (function/if/while/block/catch) garbager
  calls are still invoked inline from the typer's scope-management
  sites because per-body scopes are stack-locals freed when
  ~infer_body~ returns; lifting them into the single
  ~garbager_destroy~ walk needs the per-body scope cache on ~Context~
  and is tracked under [[file:../issues/open/158.org][#158]].

6. Constfolder

- `constfolder(ast)`
- Implemented in til: [[file:../src/self/constfolder.til][src/self/constfolder.til]]
- Responsibilities:
  - fold macros
  - fold comptime-known pure calls recursively when possible
  - track known literal values
- After constfolder, macros should be gone from the AST.

7. Scavenge

- `scavenge(ast, mode, run_tests)`
- Implemented in til: [[file:../src/self/scavenger.til][src/self/scavenger.til]]
- Usually removes dead/unreachable declarations after earlier passes.
- This is skipped in some library translate/build paths.

8. Back end

There are two main back ends.

- Interpret:
  - `interpret(ast, ...)`
  - [[file:../src/self/interpreter.til][src/self/interpreter.til]]
  - For calls into native C libraries the interpreter routes through
    its own FFI dispatcher, in the same
    [[file:../src/self/interpreter.til][src/self/interpreter.til]]
    (libffi-backed `ext_func`/`ext_proc` calls).

- Build:
  - `build(ast, mode, run_tests, path, c_path)`
  - [[file:../src/self/builder.til][src/self/builder.til]]
  - emits C, headers, and optional til bindings
  - then `compile_c(...)` / `compile_lib(...)` compiles the emitted C
  - target/platform selection (the `Target` and `Lang` enums) lives in
    [[file:../src/self/targets.til][src/self/targets.til]]

9. Runtime/ext layer

- A small set of primitives is still implemented in handwritten C and linked
  into the self-hosted pipeline:
  - [[file:../src/c/ext.c][src/c/ext.c]]
- Til sees these through `ext_func` / `ext_proc` declarations imported through
  `modes.til`, and handwritten C sees til declarations through generated
  `boot/til_forward.h`.

* Bindings generator (separate tool)

- `generate_bindings(in_path, out_path)`
- Implemented in til: [[file:../src/self/binder.til][src/self/binder.til]]
- Not part of the translate/interpret pipeline above. Driven from
  `bin/til bindings <header> -o <out>`: reads one C header and emits a
  `mode lib` til file mirroring its typedefs, enums, function
  declarations and numeric `#define`s, so til can `import` the vendor
  binding files under `vendor/bindings/`.

* Notes

- Loader orchestrates the phases; Context carries the shared state.
- Lexer turns source into tokens.
- Parser builds the AST.
- Initer registers and synthesizes declarations.
- Typer resolves and validates.
- Garbager inserts the automatic destructor calls.
- Constfolder folds comptime work.
- Scavenger removes unused leftovers.
- Builder/interpreter are the final execution back ends; the interpreter
  also handles its own FFI calls and targets defines build targets.
- Binder is a standalone C-header -> til bindings generator, outside
  the main pipeline.

That is the current practical phase order to keep in mind when debugging:

- file loading / import resolution / mode selection: `src/self/loader.til`
- shared `Context` / `TypeScope` state: `src/self/context.til`
- tokenize problem: `src/self/lexer.til`
- parse problem: `src/self/parser.til`
- missing declaration / auto-generated helper: `src/self/initer.til`
- name/type resolution or enum switch behavior: `src/self/typer.til`
- missing/extra `delete`/`free` call, use-after-move: `src/self/garbager.til`
- comptime folding issue: `src/self/constfolder.til`
- generated C / linkage issue: `src/self/builder.til`
- cross-compile target/lang issue: `src/self/targets.til`
- interpreter FFI call into native C: `src/self/interpreter.til`
- runtime behavior mismatch: `src/self/interpreter.til` or `src/c/ext.c`
- generated vendor bindings wrong: `src/self/binder.til`
