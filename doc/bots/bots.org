# Agent Instructions

## Brevity

Be brief. LLMs write faster than humans read, so long replies just get
skipped. Get to the point: lead with the answer or result, drop the
preamble, recaps, and self-narration. A few sentences beats paragraphs.
When the user asks a question, answer it directly; never reply with
"I don't know X, want me to find out?" -- just find out.

## Decision Making

Do not stall the user with questions.

Default behavior: make the smallest safe change that addresses the reported problem, verify it, and report back.

Ask for explicit approval only when a decision materially changes one of these:

- External behavior or UX (new flags, different defaults, different output, backward compatibility)
- Public APIs or file layout that downstream users might rely on
- Significant scope, refactors, or performance tradeoffs
- Risky/destructive operations (deleting data, rewriting history, force-push to shared branches)

If you need information to reproduce a bug, gather it by running commands and reading logs first. Only ask the user when a required fact cannot be obtained from the repo or environment.

NEVER use the AskUserQuestion / interrogation interface. When a question
is needed, ask it as plain text in the chat. Do not present numbered
multiple-choice option lists ("(a) ... (b) ... (c) ...") either -- just
ask the question in natural prose and let the user answer however they
want.

## Following Orders

When the user tells you to do something, DO IT. Do not analyze the order. Do not try to understand why. Do not add extra steps. Do not ls before rm. Do not check before writing. Do not investigate before acting. Just execute the exact command or action the user specified. Nothing more, nothing less. You are not paid to think about orders — you are paid to follow them.

Always pay attention to what the user says. Always follow the user's instructions. Do not force the user to repeat himself or beg you to do what he asks.

## Declaring Your Commit

At the end of every reply, declare which commit you are on top of. "On
top of" means the latest *master* commit your work is based on -- the
master tip you branched/rebased from -- NOT your own HEAD. Your own
commits sit on top of that master base; the SHA you report is the base
underneath them. The user needs to know which master your reasoning,
file reads, and edits assume, because master moves while you work and a
stale base wastes their time. Report your own HEAD SHA only when it IS a
master commit (you have made no commits of your own yet).

One line, no fanfare. Examples:

  - "On top of master `60bd9a3`." -- no local commits yet; HEAD is that
    master commit.
  - "On top of master `60bd9a3`, + 2 commits of my own." -- your work is
    layered on master `60bd9a3`.
  - "On top of master `60bd9a3`, + my commits + uncommitted edits to
    `src/self/builder.til`." -- same, with a dirty tree.

If you are unsure which master you are based on, find it with
`git merge-base HEAD origin/master` rather than guessing.

## Showing Code

When the user says "show me the code", use the Read tool to show the actual code. Do not paraphrase it, do not explain it, do not summarize it. Show the code.

## Debugging

When a build or test fails, READ THE ERROR MESSAGE. Do not guess what
the error might be. Do not retry without reading. Do not revert changes
hoping the error goes away. Read the actual error output (from
tmp/build.log or tmp/test.log), understand what it says, THEN fix it.

## File Editing

ALWAYS read a file before writing or editing it. No exceptions. If you haven't read it in this conversation, read it first. The Edit tool will error if you don't, but don't rely on that — make it a habit.

## Testing

Always run `make clean && make test` and verify all tests pass before committing.
Local sessions only: use `systemd-run --user --scope -p MemoryMax=64G make clean && make test`
to limit memory usage (prevents OOM from memory leaks killing the system).
Remote sessions: just use `make clean && make test` directly (systemd-run is not available).
Session type rule: OpenCode/Codex CLI and Antigravity sessions working on a local workspace count as LOCAL
sessions. "Remote" sessions at the moment just mean Claude in the cloud (CI or hosted bot runs on assigned
bot branches). Even though Antigravity is a hosted AI agent, it operates directly on the user's local
worktree/system and is therefore a LOCAL session.
See doc/self.org for the build workflow.

## Worktrees (local sessions)

When a task spans multiple files or multiple commits, create a dedicated git
worktree so the task stays isolated and the main tree is free for other work.

Directory: `~/sync/data_common/code/jt/til_worktrees/<descriptive-name>`

Setup::

  mkdir -p ~/sync/data_common/code/jt/til_worktrees
  git worktree add ~/sync/data_common/code/jt/til_worktrees/<name> [HEAD|master]

For a new branch, add `-b <branch-name>`.

All `make` targets work from the worktree. The main tree stays clean.

Merging to master (local sessions)::

  # Start this sequence from a committed worktree branch. If your fix is
  # still uncommitted, finish the edit/test/commit cycle first.
  #
  # 1. Cherry-pick the single worktree commit onto master.
  git checkout master
  git cherry-pick <worktree-branch-name>

  # Note: The human runs the merge script directly. Local sessions
  # just need to cherry-pick to master when told.
  # Do NOT run tests, make clean, or any extra commands after cherry-picking
  # to master, unless the user explicitly tells you to.

  # 2. Only now remove the worktree.
  git worktree remove <path>
  git worktree prune

Teardown (when no merge to master is needed)::

  git worktree remove <path>
  git worktree prune

The til_worktrees directory lives outside the repo, so worktrees never appear
as untracked files in `git status` of the main tree.

## ASAN (remote sessions)

Remote sessions may hit `AddressSanitizer failed to allocate shadow memory`
due to `ulimit -d` restrictions. Run `ulimit -d unlimited` before ASAN:

    ulimit -d unlimited && bin/til_asan translate test/strings.til > tmp/asan.log 2>&1
    grep "SUMMARY" tmp/asan.log

Remote sessions must NOT update the ASAN baseline recorded in
`doc/bots/asan.org`. The baseline is updated automatically by the merge script
run by the human.
section above). A remote session may report its own ASAN numbers for context,
but should not edit the baseline table or treat small deltas as a blocker --
the merger decides whether to accept them.

- NEVER edit boot/ files. Fix .til or src/c/ sources instead.
- Use `tmp/` dir for ad-hoc test files.

## Issue Tracking

- Issues live in `issues/open/`, one file per issue (e.g. `issues/open/189.org`).
- Each file starts with `# priority: high` or `# priority: low`, then `** OPEN #NNN Title`.
- Process is in `issues/process.org`; the counter is in `issues/NEXT_ISSUE.txt`.
- New issues go at the top of their priority, numbered from `issues/NEXT_ISSUE.txt`.
- When work starts, change OPEN -> WIP in the title line.
- Fixed issues move to `issues/fixed/`, cancelled to `issues/cancelled/`.
- `make test` regenerates `issues/summary.org`. Increment `issues/NEXT_ISSUE.txt` manually.

## Build Output

- `make` and `make test` are expensive (full recompile + test suite).
  NEVER run them multiple times to grep/head/tail the output separately.
  Run once, redirect to `tmp/build.log`, then read that file as needed.
  Example: `make > tmp/build.log 2>&1` then use Read/Grep on `tmp/build.log`.
  ALWAYS use `> file 2>&1` redirection, NEVER use `| tee` (causes permission re-prompts).

## Background vs Foreground

NEVER run commands in the background. No exceptions. This is the single
most important operational rule on this machine: backgrounded builds
pile up, run concurrently, thrash the CPU, and have to be hand-killed.
Every time you background a build you force the human to stop what they
are doing and `kill` your orphans. DO NOT DO IT.

ONE TASK AT A TIME. Start a command, WAIT for it to finish in the
foreground, read its output, THEN start the next one. Never have two
builds or tests in flight at once. Never start a new build while another
is still running. If you catch yourself about to launch a second thing
"while the first one runs", STOP -- that is the exact mistake that
spawns the orphan pile.

Specifically, NEVER:

- pass `run_in_background: true` or set `WaitMsBeforeAsync` < `3600000` in the run_command tool. Not for builds, not
  for tests, not for "I'll check on it later", not ever. There is no
  situation in this repo where backgrounding is the right call.
- end a shell command with `&` to backgroundize it.
- `disown`, `nohup`, `setsid`, or `(cmd &)` -- same effect.
- spawn polling loops (`while pgrep ...; do sleep N; done`) to "wait".
- use `schedule`, `timer`, or `manage_task` tools to background a command and poll it. Always run commands synchronously in the foreground.

If a build/test is slow, just WAIT for it -- the Bash tool blocks until
it finishes, however long that takes. Pass a large `timeout` (the tool
accepts up to 3600000 ms / 60 minutes) and let the single foreground
command run to completion. Do NOT background it so you can "do other
things meanwhile" -- the next step depends on the result anyway, so
there is nothing useful to do in parallel.

If you think you need to background something to stay responsive: you
don't. Run it in the foreground, wait, then continue. A slow synchronous
command is always better than an orphaned background task the human has
to clean up.

Polling is also forbidden. A `while pgrep ...; do sleep N; done` loop
is a background task in disguise: it stalls the turn, hides errors in
the loop's stdout, and the harness will background it anyway when the
inner command exits 0. If you started a command and need to wait, you
already waited -- the Bash tool already blocked. Read the captured log
file (`tmp/build.log`).

## Recovery

- The last commit ALWAYS works. til_boot is built from it via git.
- `make clean && make test` always works.
- `git checkout HEAD boot/` restores boot/ from HEAD if corrupted.
- After context compaction, don't assume the repo is broken — the last commit is good.

## ASCII Only

Use plain ASCII everywhere -- in .til source files, comments, string
literals, commit messages, and documentation. No Unicode arrows,
bullets, em-dashes, or fancy quotes. til's lexer cannot parse Unicode.
Use -> not ->, -- not --, * not bullet, ' and " not curly quotes.
Preserve existing ~REM~ comments unless the change makes them false or
obsolete. They are memory aids for future sessions; do not delete them
just because the surrounding code was edited.

## Naming

Do not introduce identifiers that start with a leading underscore
(~_x~, ~_foo_bar~, ~_t_Bool_1234~, etc).  til is not C, and the
"reserved for the implementation" convention does not apply here.
A leading underscore is reserved for codegen-emitted temporaries
(boot/til.c writes ~_t_<Type>_<id>~ and similar) and for variables
that are deliberately unused (~for _ in 0..n~).  Any other ~_name~
is noise and obscures intent.

If you find yourself reaching for a leading underscore to dodge a
name collision, rename the binding to something descriptive instead.
If you are translating from existing code that uses leading
underscores (eg the ~_nv~/~_b~ style), drop the underscore on the
way through.

## Commits

- Always include ALL modified files in commits.
- When boot files are regenerated, include ALL modified generated boot artifacts, not just boot/*.c.
  This includes boot/*.c, boot/*.h, boot/*_forward.h, and any other changed files under boot/.
- Local agents: include doc/totals.csv and img/totals.svg in commits
  (this includes OpenCode/Codex CLI and Antigravity sessions working on a local checkout).
- Remote agents (specifically Claude in the cloud, GitHub Actions, etc.): do NOT include doc/totals.csv
  or img/totals.svg. After `make test` regenerates them, DISCARD the changes
  before committing -- either form is fine:
      git checkout HEAD -- doc/totals.csv img/totals.svg
      git restore        doc/totals.csv img/totals.svg
  This is an explicit exception to the "no destructive git commands" rule
  below: totals files are auto-generated and safe to discard in remote
  sessions. Both files must be discarded together (never just one).
  Leaving them dirty is not acceptable -- the working tree must be clean
  after the commit, otherwise the stop hook will block the session.
- Use `make test` before every commit (skip when only documentation changed)
- Prefix documentation-only commits with "Doc: " (e.g. "Doc: Update issue #105 progress")
- Do NOT start commit message subjects with `#` (e.g. "#167 Foo"). Lead with
  a word and reference the issue number inline ("Issue #167: foo" or
  "Foo (#167)"). Lines beginning with `#` are comments in git's commit
  message buffer and get stripped by some tools.
- Never run destructive git commands (revert, restore, stash, reset --hard, checkout -- files)
- Local sessions do NOT fetch and do NOT push. Humans handle all syncing with remotes. If rebasing in a local session, rebase on the local `master` branch directly.
- Do NOT remind Jorge to push. Local pushes are handled manually and don't
  need bot follow-up nudges.
- Remote sessions may push only to their own assigned feature branches
  (for example `claude/...` or other explicitly assigned bot branches).
  Remote sessions must NOT push to master.
- Force-pushing to your own assigned feature branch (e.g. `claude/...`) is
  pre-authorized for remote sessions -- do NOT ask for confirmation. Use
  `git push --force-with-lease -u origin <branch>`. This applies after a
  rebase or after the previous tip was squash-merged into master and the
  remote rejects the fast-forward. Force-pushing to master or to a branch
  not assigned to you still requires explicit authorization.
- NEVER look for excuses to reduce scope, skip changes, or defer work. If the user says to change something, CHANGE IT. Don't analyze whether it's "needed" or "used" -- just do it.
- When told "in the whole codebase", that means every file, every path, every occurrence. No exceptions.
- When the user asks you a question, stop whatever you're doing and answer his questions. They're not rhetorical questions and even if they are, or socratic method, you answer them the best you can and completely honestly.

## Remote master ref can mirror your own branch (remote sessions)

In the hosted sandbox the remote `master` ref is NOT a reliable view of
the human's authoritative master. After you push (or force-push) your
assigned feature branch, `git fetch origin master` / `origin/master` can
come back pointing at your OWN just-pushed commit -- so `git merge-base
HEAD origin/master` equals your HEAD and the branch looks "already
merged" when it is not. (This is the misunderstanding behind a session
that rebased, pushed, then wrongly reported "already merged, nothing to
do" instead of surfacing the branch.)

Never conclude your work is merged, and never skip pushing, on that
basis. A remote session's job ends at: commit, push the feature branch,
and report the branch name so the human can merge it manually. Always
push and always surface the branch name even when origin/master appears
to already contain your commit. For "Declaring Your Commit", take the
true master base from the SHA the human gives you rather than trusting
origin/master.

## Rebasing (remote sessions)

When rebasing a feature branch on top of origin/master, do NOT abort on
conflicts in automatically generated files (boot/*). Those files are
regenerated from source, so it's safe to accept whatever master has for
them. Remote sessions don't commit doc/totals.csv or img/totals.svg, so
those files don't enter rebase conflicts.

The base procedure is:

1. `git fetch origin master`
2. `git rebase origin/master`
3. If the rebase completes cleanly: done. Remote session: push to your own
   assigned feature branch. Local session: stop there; a human handles the push.
4. If conflicts ONLY in generated files (boot/*):
   a. `git checkout --ours boot/til.c boot/til_forward.h` (and any other
      conflicted boot/ files), then `git add` them and
      `git rebase --continue`.
   b. After rebase finishes, soft-reset and regenerate so the boot/
      artifacts match the rebased source: `git reset --soft HEAD~1`,
      then run the build/test target that matches the commit's
      prefix:
         - Plain commits: `make clean && make test`.
         - `Two-pass: `-prefixed commits: `make clean && make test_two_pass`.
           These commits depend on the new compiler's self-applied
           output -- pass 1 alone would commit a `boot/til.c` that
            the local merger's `make test_two_pass` (run via the merge script)
            would regenerate
           differently. See `doc/self.org` "Two-pass build" for
           the bootstrap reasoning.
      If the branch mixes prefixes, choose `make test_two_pass` (it
      is a superset of `make test`). Then re-stage and commit.
      Discard totals with `git restore doc/totals.csv img/totals.svg`
      before committing.
   c. Remote session: push (force-with-lease since history was rewritten)
      to your own assigned feature branch. Local session: stop there; a
      human handles the push.
5. If source files (src/**, .github/**, Makefile, CLAUDE.md, etc.) have
   conflicts: resolve them manually. Do NOT `--ours` source files.
   After resolution, regenerating boot/ is recommended (`make clean &&
   make test`, or `make clean && make test_two_pass` if any pending
   commit is `Two-pass: `-prefixed) so the boot artifacts match the
   resolved source.

Soft-reset + regenerate is only required when (4) the rebase produced
boot/* conflicts; a clean rebase needs no soft-reset on remote sessions.

Exception: when the user says the rebased branch is *next to merge*
(no other remote branches will land between this push and the local
merger picking it up), the soft-reset + regenerate step at (4b) /
(5) can be skipped. The local merger's `make test` / `make test_two_pass`
(run via the merge script) regenerates boot/* against the rebased source anyway,
so the work would just be thrown away. Discard doc/totals.csv + img/totals.svg
if they are modified, then commit and force-push. Only the human can grant this
exception -- never assume it; redo the regen by default.

The next-to-merge exception is REVOKED the moment the rebase produces
a conflict in any file OUTSIDE the regen set (boot/*). It only covers
rebases where every conflict is confined to generated boot/* files and
all source files (src/**, .github/**, Makefile, etc.) auto-merge
cleanly. If a source file conflicts -- even one resolved trivially --
the exception no longer applies: resolve the source conflict and then
do the full soft-reset + regenerate (4b / 5) so boot/* matches the
hand-resolved source. The reasoning: a source-level conflict means the
rebased source is not a clean superset of what either side generated
boot/* from, so skipping regen could ship a boot/* that no longer
corresponds to the committed source.


NEVER `git rebase --abort` on conflicts in generated files alone. Only
abort if a source-file conflict is genuinely unresolvable.
