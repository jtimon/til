* FIXED #72 link_c files can't see til-defined types and functions

  enum tag enums, and function declarations. link_c files get it via
  ~-include~ flag when compiled to .o separately.

  C typedef — forward.h provides them from the til declarations.

* FIXED #17 Global variable access
  :DEPENDENCIES_OF: #12 #13
  Variables declared in script-mode root scope (or any outer scope) were
  invisible at runtime inside function/proc bodies. The typer passed
  (TypeScope chains via parent pointers and finds the variable), but the
  interpreter crashed or errored.

  Reproduction: src/test/globals.til, src/test/globals_cli.til

  func case: "error: undefined variable 'x'" — clean error from NODE_IDENT
  NULL check. proc case: SEGFAULT — NODE_DECL move-semantics path had no
  NULL check on scope_get result.

** Root cause

   expr_uses_var() in typer.c returned 0 for NODE_FUNC_DEF, meaning the
   ASAP pass didn't see variable references inside nested function/proc
   bodies. When x was declared at top level and referenced only inside a
   proc body, ASAP thought x was unused and inserted I64.delete(x) right
   after the declaration. When the proc later ran and tried to access x,
   the value had already been freed/nulled.

   func case: I64.clone(x) inside the function body tried to clone x
   whose value was val_none → clean "undefined variable" error (from
   precomp, which also couldn't find x in its limited scope).

   proc case: I64.clone(x) dereferenced v.i64 which was NULL → SEGFAULT.
   The NODE_DECL move-semantics path also lacked a NULL check on
   scope_get result, which was a secondary crash path.

   Secondary issue: precomp created its own interpreter scope with only
   func/struct/enum defs (no variable decls), then tried to fold pure
   func calls. If the func accessed a global variable, precomp's
   eval_expr hit the error path and exit(1)d before the interpreter
   even ran.

** Fix

   Program-scope globals live for the program's lifetime — they should
   never be ASAP-deleted. The scavenger handles removing unused globals.

   1. Skip ASAP deletion for program-scope variables: added skip_delete
      flag to LocalInfo in typer.c. Variables with a declaration at program
      scope (is_program_scope && decl_idx >= 0) get skip_delete=1. The
      ownership-transfer check still runs for them (use-after-own errors),
      but no delete calls are inserted. Variables captured by nested
      functions also get skip_delete=1 (expr_used_in_nested_func check).
   2. Added NULL check in interpreter.c NODE_DECL move-semantics path.
   3. Fixed precomp: added func_uses_unknown_globals() to skip folding
      func calls whose body references variables not in the precomp scope.
      Also made track_literal() populate the precomp interpreter scope
      with known constant values (not just the known map).
   4. Fixed builder: top-level variable declarations are emitted as
      file-scope C globals (before function definitions) so they're
      accessible from emitted til functions/procs. This is necessary
      because til functions are emitted as separate C functions, and C
      doesn't support nested functions capturing locals — so top-level
      vars must be file-scope for functions to reference them. The
      values are then assigned inside main() where initialization runs.
      In cli mode, globals are initialized at the start of main()
      before the user's main body.
      Note: ref declarations are skipped from file-scope globals since
      they are pointer aliases (local to main), not owned values that
      other functions need to access.

   Tests: src/test/globals.til (script mode), src/test/globals_cli.til (cli mode).

   Explicit capture syntax split out as #54.

* FIXED #28 Aliases: implicit ref for comptime-immutable declarations
  When ~a := b~ and b is provably immutable, ~a~ becomes an auto-alias
  (implicit ref — pointer to same storage) instead of a clone. The
  compiler detects when aliasing is safe with no new syntax needed.

  Auto-alias conditions (all must hold):
  - Destination is not ~mut~ (immutable target)
  - RHS is a ~NODE_IDENT~ (variable reference)
  - Source is not ~mut~ (immutable source)
  - Source is not ~own~ (not an ownership transfer)
  - Source is not an explicit ~ref~ (unless it's itself an auto-alias)
  - Source is not a parameter (params are borrowed, aliasing them
    and returning would create dangling refs)

  Implementation:
  - typer.c: auto-alias detection sets ~is_ref=true~ and new ~is_alias=true~
    flag on both NODE_DECL and TypeBinding. Alias chaining supported:
    ~x := 42; y := x; z := y~ — all three share the same storage.
  - typer.c: returning an auto-alias auto-inserts clone (prevents
    use-after-free since alias shares storage with a local).
  - initer.h: added ~is_alias~ field to TypeBinding to distinguish
    auto-aliases from explicit ~ref~ declarations.
  - Interpreter and builder already handle ~is_ref~ correctly — no
    changes needed. Aliases use ~scope_set_borrowed~ (interpreter)
    and pointer assignment (builder), same as explicit refs.
  - ASAP lifetime extension already handles refs transitively via
    fixed-point propagation — works for alias chains too.

  Tests: src/test/aliases.til (17 tests covering I64/Str/Bool/struct
  aliases, alias chaining, deep chains, returned aliases, mut exclusion,
  struct field cloning, param cloning).

* FIXED #53 Payload enum in struct field: codegen bug
  Compiled mode produced wrong values when a payload enum was stored as
  an inline struct field. Interpreter worked correctly.

  Root cause: varargs desugaring in typer.c (desugar_variadic_calls)
  only cloned NODE_IDENT arguments before passing to Array_set, not
  NODE_FIELD_ACCESS. When println(c.name) was desugared to
  Array_set(_va, 0, c.name), the builder emitted &c->name as the val
  argument. Array_set does free(val) after memcpy — since c.name is at
  offset 0, free(&c->name) freed the entire Container. Subsequent
  mallocs reused that memory, corrupting c.kind.

  Fix: added NODE_FIELD_ACCESS to the clone check in
  desugar_variadic_calls (typer.c:1357). Now field accesses are cloned
  before passing to Array_set, preventing the interior pointer free.

  Tests: src/test/enums.til test_payload_enum_field,
  test_payload_enum_field_b, test_payload_enum_field_default,
  test_payload_enum_field_mut.

* FIXED #40 Func returning borrowed param: use-after-free in both interpret and run
  When a func returns a borrowed parameter (~return a~ where ~a~ is not
  ~own~), the returned pointer IS the same pointer as the caller's argument.
  The typer's ASAP system hoists literal args to temps and deletes them
  after the call, freeing the return value's memory.

  Root cause: the typer hoists ~max_val(5, 3)~ into ~_t1 := 5; _t2 := 3;
  r := max_val(_t1, _t2); I64.delete(_t1); I64.delete(_t2)~. Since
  ~max_val~ returns borrowed ~a~ directly, ~r~ and ~_t1~ are the same
  pointer — deleting ~_t1~ frees ~r~'s data (use-after-free in interpret,
  double-free crash in run).

  Affects both interpreter and builder because both evaluate the typer's
  ASAP-inserted delete calls. Not mode-specific — appeared in cli mode
  because that's where the pattern was first used.

  Fix: auto-insert clone in the typer when a non-ref-returning function
  returns a borrowed parameter (~return <ident>~ where ident is a non-own
  param). Added ~returns_ref~ parameter to ~infer_body~ so ref-returning
  functions (which intentionally return the same pointer) are not cloned.

* FIXED #41 Typer: ASAP delete before reassignment causes use-after-free
