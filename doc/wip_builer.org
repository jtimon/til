** ROOT CAUSE: Makefile doesn't use bootstrap/parser.c

This is the SAME issue that lexer had before it was migrated.

The Makefile (line 5) filters out ~src/c/lexer.c~ and adds ~src/bootstrap/lexer.c~
(generated from lexer.til). But it does NOT do this for parser:

#+begin_src makefile
SRCS := $(wildcard src/*.c) $(filter-out src/c/ext.c src/c/lexer.c, $(wildcard src/c/*.c)) src/bootstrap/lexer.c
HDRS := $(filter-out src/c/lexer.h, $(wildcard src/c/*.h)) src/bootstrap/lexer.h
#+end_src

~src/c/parser.c~ is still linked. ~src/bootstrap/parser.c~ (generated from
parser.til by ~ctil_core~, lines 40-42) exists but is never used.

This causes a *symbol collision* via ~--allow-multiple-definition~:

- ~src/c/parser.c~ has ~dyn_call_clone~ that knows about: TokenType, Token,
  NodeType, FuncType, TilType, DeclData, FuncDefData, ExprData, Expr, Parser
- ~gen/c/til.c~ has ~dyn_call_clone~ that also knows about: Mode, TypeScope,
  CollectionInfo, DynCallInfo (types from builder.til / typer)
- The linker picks parser.c's version (linked first), so at runtime ~dyn_call_clone~
  fails for Mode etc with: ~dyn_call: unknown type for clone~

The same collision applies to ~dyn_call_delete~, ~dyn_call_cmp~, ~dyn_call_eq~,
~dyn_call_to_str~ — any dispatch function.

This is the SAME bug pattern as the I64_to_str/U8_to_str issue in ccore.c
(already fixed in this branch).

** Fix needed

Update Makefile to use ~src/bootstrap/parser.c~ instead of ~src/c/parser.c~:

#+begin_src makefile
SRCS := $(wildcard src/*.c) $(filter-out src/c/ext.c src/c/lexer.c src/c/parser.c, $(wildcard src/c/*.c)) src/bootstrap/lexer.c src/bootstrap/parser.c
HDRS := $(filter-out src/c/lexer.h src/c/parser.h, $(wildcard src/c/*.h)) src/bootstrap/lexer.h src/bootstrap/parser.h
#+end_src

This should resolve the dyn_call collision because ~src/bootstrap/parser.c~
(generated from parser.til) won't have conflicting dyn_call definitions — those
will only exist in gen/c/til.c which is linked for bin/c/til.

Actually, BOTH parser.c files have dyn_call_clone (generated code always includes it).
The real fix may require either:
1. Linking gen/c/til.c BEFORE bootstrap files so its dyn_call wins
2. Or renaming/namespacing the dyn_call functions per translation unit
3. Or ensuring the bootstrap parser.c's dyn_call includes all necessary types

This needs investigation when resuming.

** Architecture decisions

*** fprintf → concat
Every ~FILE *f~ parameter becomes ~mut out: Str~.
Each ~fprintf(f, ...)~ becomes ~out = out.concat(...)~.
At the end, ~writefile(path, out)~.
The swap/move codegen fix (commit 7a8c240) makes this pattern safe for mut params.

*** Static variables → module-level mut vars
The 12 file-scope statics in builder.c become module-level ~mut~ variables in builder.til.

*** AST access via bridge functions
Opaque Expr handles through ~30 bridge functions in ~src/c/til_bridge.c~.

*** Name conflict
~mode~ is a til keyword → ~til_mode~ in both builder.c and builder.til.

** Key lessons learned

*** format() does NOT do template substitution
~format("hello {} world", name)~ produces ~"hello {} world" + name~, NOT ~"hello name world"~.
format() just concatenates all its arguments. Every ~format("...{}...", arg)~ must become
~format("...", arg, "...")~ or ~"...".concat(arg).concat("...")~.

*** --allow-multiple-definition is dangerous
When two .c files define the same function with different signatures or different bodies,
the linker silently picks the first one. This caused:
- I64_to_str: ccore.c had ~I64_to_str(I64 v)~ (value), generated code had
  ~I64_to_str(I64 *val)~ (pointer) → pointer address interpreted as integer
- dyn_call_clone: parser.c version doesn't know Mode → runtime crash

*** Str corruption from ext_c_path
The garbage-after-ext.c error is NOT a format() bug (those were fixed). It's the
dyn_call collision causing memory corruption during translation, which corrupts
heap-allocated Str objects used later in compile_c.
