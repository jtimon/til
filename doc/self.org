* Self-hosting build

  See Makefile header comments and ~make help~ for the full reference.

** Key concepts

  - ~bin/til_boot~: built entirely from the last commit via git.
    Uses HEAD's boot/ AND HEAD's src/c/. Completely isolated from
    any uncommitted changes. Always safe. Never fails.
  - ~bin/til~: built by til_boot from current sources. The resulting
    binary IS self-hosted (its compilation logic comes from the
    current .til sources), but the C it just emitted into boot/ was
    generated using HEAD's logic.
  - ~boot/~: generated C, committed for til_boot. Regenerated every
    build so the next commit's til_boot has current code.

** Single-pass build

  The Makefile builds til in one pass:

  1. ~til_boot~ (from HEAD) translates current .til sources to C,
     regenerates boot/, builds ~bin/til~

  In most cases this is enough: the C in ~boot/~ is the output of
  HEAD's compiler applied to the current source, and that compiles
  into a working ~bin/til~. ~make clean && make test~ on every
  commit confirms that the next commit's ~til_boot~ will build.

** When a preparation commit is needed

  Because boot/ is generated by HEAD's logic (not the current
  source's logic), changes that alter how the compiler EMITS code
  cannot take effect in the same commit that uses them. The most
  common case is auto-generated methods -- e.g. a new rule that
  synthesises ~clone~ or ~delete~ on structs/enums that lack them.
  HEAD's compiler doesn't know about the new rule, so it won't
  emit those methods into boot/ for the commit that introduces it.

  When that happens, split the work into two commits:

  1. Preparation commit: introduce the new auto-gen logic (or other
     emit-side change). After this commit lands, ~til_boot~ on the
     NEXT build will have the new logic baked in.
  2. Use commit: rely on the new behaviour. til_boot now emits the
     expected output, so boot/ and bin/til are consistent.

  This is uncommon -- most changes (new syntax desugaring that
  reuses existing emit paths, type-checker changes, runtime fixes,
  src/c/ tweaks, library code) take effect in a single commit.
  Only changes that the BOOTSTRAP compiler must already understand
  in order to emit them need the two-commit dance.

  If in doubt, run ~make clean && make test~. If it passes, the
  commit stands on its own.

** Two-pass build (single commit alternative)

  When you would otherwise need a preparation commit, running the
  build twice locally is an alternative: pass 1 compiles current
  sources with HEAD's til_boot (old emit), pass 2 recompiles using
  the freshly-built bin/til (new emit). The boot/til.c committed at
  the end is the result of the new compiler self-applied, exactly
  the state that two separate commits would have converged to.

  Use ~make two_pass~ for the build, ~make test_two_pass~ for the
  full test suite. Commits depending on this MUST be prefixed
  "Two-pass: " so the local merger reproduces the same regen
  procedure (the merging procedure in
  ~doc/bots/merging_from_remote.org~ runs ~make test~ by default and
  needs to switch to ~make test_two_pass~ for these).

  Choosing between the two approaches:

  - Preparation + use commits: each commit verifies independently
    with ~make test~. Bisect-friendly. Preferred when the change
    is large or the emit-side logic might want a second look on
    its own.
  - Two-pass single commit: more compact history, the change ships
    atomically. Useful for small self-applied tweaks where the
    intermediate "logic without users" state is uninteresting.

** Staged migration across master merges

  Some changes are NOT a single emit-side step but a change to the
  source-level contract that the loader/parser enforces. Example:
  switching ~import("loader.til")~ to ~import("loader")~ requires
  the loader to start appending ~.til~ internally AND the call
  sites to drop the suffix. HEAD's til_boot has the old loader
  baked in; if a single commit ships both the new loader source
  and the new bare call sites, HEAD's til_boot rejects the new
  call sites at parse/resolve time and the build never reaches
  pass 2 -- ~two_pass~ does not help here because pass 1 itself
  fails.

  These migrations need a sequence of commits, each one merged to
  master before the next is authored:

  1. Transitional commit: extend the loader/parser to accept BOTH
     the old and new forms (e.g. loader strips trailing ~.til~ if
     present, then appends ~.til~). Call sites stay in the old
     form so HEAD's til_boot still parses them. ~make clean &&
     make test~ passes. Merge to master.
  2. Migration commit: now that master's til_boot understands the
     new form, update call sites to the new form and (optionally)
     reject the old form with a clear error. ~make clean && make
     test~ passes against the master from step 1. Merge to master.

  The remote agent can author commit 2 only after commit 1 has
  landed on origin/master, because til_boot is built from the last
  commit on the agent's base. A reviewer rebasing a feature branch
  that contains BOTH commits at once cannot use the soft-reset +
  regen recipe -- the intermediate til_boot they need is the one
  baked into commit 1's boot/til.c, which only exists once commit
  1 has been merged.

** Staged interface migration with a temporary alias

  Some bootstrap-safe source interface changes need THREE commits, not
  two. This happens when existing source call sites must keep working
  for HEAD's til_boot, but you still want the final public name to
  change.

  Example pattern: changing ~Set.new("Str")~ into ~Set.new(Str)~.
  If commit 1 changed ~Set.new~ directly, old bootstrapped code that
  still emits string-arg calls could break before the source tree is
  fully migrated.

  Use this sequence instead:

  1. Add a temporary alias with the NEW interface, under a NEW name
     (for example ~Set.new_newi(T: Type)~), while keeping the old
     ~Set.new(elem_type: Str)~ interface intact. Rewrite all source
     call sites and desugars to use the temporary alias. ~make clean &&
     make test~ passes. Merge to master.
  2. Flip the original public name to the NEW interface (for example
     ~Set.new(T: Type)~), and rewrite all source call sites back to
     the original name. Keep the temporary alias as a forwarding
     wrapper so bootstrapped/generated callers from commit 1 still
     work. ~make clean && make test~ passes. Merge to master.
  3. Remove the temporary alias once nothing in source or generated
     output still calls it. ~make clean && make test~ passes. Merge to
     master.

  This is the right pattern when the final interface name must stay the
  same, but the argument contract changes in a way that old boot output
  cannot consume immediately.

** Breaking changes in standalone scripts (~examples/~, ~tools/~)

  ~make test~ exercises the compiler, the test suite, and every example
  the suite knows how to build. But standalone scripts that are NOT
  hooked into the suite (the merge helper at ~examples/merge_commit.til~,
  ad-hoc scripts under ~examples/~ that nobody runs as part of CI) won't
  be caught when a commit makes a previously non-throwing function newly
  throw -- e.g. ~Vec.get throws OutOfBounds~. Bare ~vec.get(i)~ in such a
  script compiles fine before the change lands, fails after with
  "missing argument for parameter '_err_kind'".

  The breaking-change commit's ~make test~ passes regardless: every
  caller the suite covers gets a ~!~ or a ~?~ in the same commit. The
  standalone script isn't in the suite, so it falls outside that proof.

  *** The merge helper has a special constraint ***

  The local merger runs the helper using ~master's bin/til~, against the
  candidate commit's tree. So at merge time the helper file taken from
  the candidate commit is being compiled by the OLD compiler:

  - Bundle the script adaptation INTO the breaking commit -> at merge
    time, master's ~bin/til~ sees the candidate's adapted script (with
    new ~!~ markers) and rejects it: "function does not throw, remove
    the '!'". The merge can't even start.
  - Keep the candidate's script bare (matching master) -> at merge
    time, master's ~bin/til~ sees the candidate's bare script and
    compiles it fine. The merger proceeds, lands the commit. AFTER the
    commit lands on master, ~bin/til~ regenerates with the new throws
    behaviour and the bare ~.get(i)~ in the merger source now fails to
    build. The human has to hand-fix that follow-up before the next
    merge.

  So the merge helper specifically must stay at master's form in the
  breaking commit. The script fix is a follow-up commit on master that
  the human applies by hand (the merger that would normally apply it
  is itself broken in the window between the two).

  *** Other standalone scripts ***

  For scripts that aren't part of the merge workflow, either approach
  works:

  - Same-commit bundle: simple, no broken window for users of the
    script.
  - Follow-up: keeps the breaking diff narrow. Acceptable if no tool
    chain depends on the script being compilable from master.

  Patterns when you do adapt:

  - Bare ~.get(i)~ becomes ~v[i]~ for panic-on-OOB (the parser
    desugars subscript into ~.get(i)!~), or stay with ~.get(i)?~ when
    you want to catch instead of panic.
  - Nested ~v[i]~ inside another call (e.g.
    ~trim_spaces(cells[i]).to_i64()~) hits the Dynamic-hoist limit on
    the bang temp and needs an explicit ref binding first:
      ref c : Str = cells[i]
      trim_spaces(c).to_i64()

** Commands

  - ~make test~ builds + runs all tests (uses xvfb-run). Commit includes boot/.
  - ~make two_pass~ builds, then rebuilds with the freshly-built ~bin/til~
    so ~boot/til.c~ reflects the new compiler self-applied. Use for
    "Two-pass: "-prefixed commits.
  - ~make test_two_pass~ ~two_pass~ + run the full test suite.
  - ~make test_asan~ builds + runs tests with ASAN leak detection (~--til-bin bin/til_asan~).
  - ~make test_nogui~ skips GUI build tests (no xvfb needed).
  - ~make bin/til_asan~ ASAN build for memory debugging.
  - ~make bin/til_debug~ debug build for gdb.
  - ~make clean && make test~ always works.
  - ~git checkout HEAD boot/~ for recovery.
