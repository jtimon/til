* Self-hosting build

  See Makefile header comments and ~make help~ for the full reference.

** Key concepts

  - ~bin/til_boot~: built entirely from the last commit via git.
    Uses HEAD's boot/ AND HEAD's src/c/. Completely isolated from
    any uncommitted changes. Always safe. Never fails.
  - ~bin/til~: built by til_boot from ALL current sources (.til AND
    src/c/). Both .til and C changes take effect immediately.
  - ~boot/~: generated C, committed for til_boot. Regenerated every
    build by ~til_boot translate~ (before ~til_boot build~) so
    src/c/ files see current signatures via boot/ headers.

** Bootstrap does NOT constrain commit structure

  til_boot is frozen at HEAD. It translates current .til to
  regenerate boot/, then compiles current .til AND current src/c/
  into bin/til. All in one ~make test~ run.

  This means the bootstrap NEVER forces you to split changes across
  multiple commits. You can change .til files, src/c/ files, or both
  at the same time. Commit structure is purely a matter of what makes
  sense for the work -- the self-hosting build imposes no constraints.

  If you think you need multiple commits "for bootstrap reasons",
  you are wrong. Re-read this section.

  ONE EXCEPTION: auto-generated method signatures (e.g. GEN_CMP_FUNC
  in initer.c). These are generated by C code inside til_boot, not
  from .til sources. Changing their signatures in initer.c requires
  two commits: (1) change initer.c, (2) update C callers. This is
  because til_boot translate uses HEAD's initer.c to generate boot/.

** Commands

  - ~make test~ does everything (uses xvfb-run). Commit includes boot/.
  - ~make test_nogui~ skips GUI build tests (no xvfb needed).
  - ~make bin/til_asan~ ASAN build for memory debugging.
  - ~make bin/til_debug~ debug build for gdb.
  - ~make clean && make test~ always works.
  - ~make revert_boot~ for recovery.
