* Self-hosting strategy

til self-hosting proceeds bottom-up (#39): replace C modules one at a time
with til-compiled equivalents. The top-down path (#12) is complete — ctil.c
has been deleted and til.til is the sole compiler entry point.

** Current state

*** Top-down (#12): COMPLETE

til.til (src/til.til) replaced ctil.c as the compiler entry point.
It links C modules via link_c() and calls them through FFI bridge
functions (til_bridge.c). As #39 replaces C modules, til.til's
link_c() list shrinks and can switch to importing .til modules
directly (#77).

** Order of operations

*** Phase 1: Simple modules (current)
lexer, parser, ast, initer, scavenger, modes — these are relatively
independent and mechanical. Each swap follows the same pattern:
1. Write/verify the .til equivalent
2. Generate bootstrap .c+h
3. Update Makefile, adapt C call sites
4. Delete old .c+h

*** Phase 2: Builder
Medium complexity. Generates C code from the AST. No dependencies on
interpreter. Can be converted once the simpler modules are done.

*** Phase 3: Typer
The semantic core — type inference, checking, resolution. This is where
most of the compiler's intelligence lives. Convert last among the
"main path" modules because:
- Highest risk of subtle behavioral changes
- Every other module will be battle-tested in til by then
- Any bugs in earlier conversions surface here (typer exercises everything)

*** Phase 4: Precomp + Interpreter (deferred)
These two are interdependent: precomp evaluates compile-time expressions
by calling the interpreter. Converting them is deferred because:
- The interpreter is the slowest execution path (the builder/compile
  path is the primary target for performance)
- Interdependency makes the conversion harder — both must move together
- Once everything else is in til, these can be replaced last
- For practical purposes, even a fully self-hosted til can keep the C
  interpreter as a fallback without shame

** Bootstrap build workflow

*** Build flow

   ~make~ (or ~make test~ which depends on it):
   1. First cc: compile src/c/*.c + bootstrap/ → bin/til_bootstrap
   2. ~til_core~: bin/til_bootstrap translates .til sources → regenerates bootstrap/
   3. Second cc: recompile with new bootstrap → bin/til_bootstrap (new code)
   4. ~til_core~: regenerate again with the new binary
   5. Third cc: recompile → bin/til_bootstrap (final)

   Pass 3 (steps 4-5) is needed because the pass-1 binary may have old
   compiler behavior (e.g. different scavenger/builder logic). The pass-2
   binary has the new code, so its til_core output is correct. All three
   cc passes compile the SAME source files.

*** Making changes to .til files only
   Just run ~make clean && make test~. The three-pass build handles it:
   - First cc uses committed bootstrap (old .til output) → works
   - First til_core regenerates bootstrap (using old binary)
   - Second cc uses new bootstrap → binary has new code
   - Second til_core regenerates bootstrap (using new binary → correct)
   - Third cc uses correct bootstrap → final binary
   - Commit everything: .til files + regenerated bootstrap/

*** Changing C files that call bootstrap functions
   If you rename or change a function in .til that C code calls
   (e.g. ~split~ → ~Str_split~):
   1. Change the .til files first (the source of truth)
   2. ~make clean && make test~ — regenerates bootstrap with new names
   3. Now change the C files to use the new names
   4. ~make clean && make test~ again — both passes match
   5. Commit everything together: .til + C + bootstrap

*** Recovery targets
   - ~make revert_bootstrap~ — restore bootstrap/ from last commit
   - ~make rescue~ — build compiler from previous commit's bootstrap

*** Rules
   - NEVER edit bootstrap/ files manually. They are generated.
   - Always commit .til changes + regenerated bootstrap together.
   - The committed bootstrap must be compatible with committed C source.
   - ~make clean && make test~ from a fresh clone must always work.

** Design notes

- The typer is dangerous territory. It contains the actual semantics of
  the language. Converting it too early risks introducing subtle bugs
  that are hard to trace. Better to have every other module proven first.

- precomp's dependency on interpreter means they form a unit. You can't
  convert precomp without converting (or at least bridging) interpreter.
  Since interpreter is the least performance-critical path (everyone
  uses ~bin/til_bootstrap run~ not ~interpret~ for real work), there's
  no urgency.

- As each C module gets replaced, til.til benefits too — it can import
  the .til version instead of linking the .c version, gradually becoming
  a pure til program.
