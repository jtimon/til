* Self-hosting bootstrap workflow

This document covers the two-pass build, how to make breaking changes,
and recovery. For self-hosting progress tracking, see #39 in doc/issues.org.

** Build flow

   ~make~ (or ~make test~ which depends on it):
   1. First cc: compile src/c/*.c + bootstrap/ → bin/til_bootstrap
   2. ~til_core~: bin/til_bootstrap translates .til sources → regenerates bootstrap/
   3. Second cc: recompile with new bootstrap → bin/til_bootstrap (final)

   ~make clean~ only deletes bin/* — bootstrap/ is never touched.
   The committed bootstrap is always available for pass 1.

   Two passes handle ALL changes, including compiler behavior changes
   in .til, because ~til_core~ regenerates both ~modes.c~ and ~til.c~:
   - ~modes.til~ is ~mode lib~ and imports the whole chain
     (ast.til, lexer.til, core.til) → scavenging is skipped →
     ~modes.c~ always has all function bodies
   - ~--allow-multiple-definition~ lets the linker find any symbols
     missing from ~til.c~ in ~modes.c~

** Bootstrap facade: modes.til

   ~bootstrap/modes.c~ is the single generated file in SRCS.
   modes.til sits at the top of the import chain:

   ~modes.til → ast.til → lexer.til → core.til~

   As modules are ported, modes.til's import changes to point to
   the newest module. E.g. when parser is ported:

   ~modes.til → parser.til → ast.til → lexer.til → core.til~

   SRCS always stays ~bootstrap/modes.c~ — no Makefile changes.
   One ~_til_lib_init~ constructor, one header, no collisions.

** Making changes to .til files only

   Just run ~make test~. The two-pass build handles it:
   - First cc uses committed bootstrap → works
   - til_core regenerates bootstrap with current .til
   - Second cc uses new bootstrap → final binary
   - Commit everything: .til files + regenerated bootstrap/

   ~make clean~ is not needed — ~make~ detects changed .til deps
   and rebuilds automatically.

** Changing C files that call bootstrap functions

   If you rename or change a function in .til that C code calls
   (e.g. ~split~ → ~Str_split~):
   1. Change the .til files first (the source of truth)
   2. ~make test~ — regenerates bootstrap with new names
   3. Now change the C files to use the new names
   4. ~make test~ again — both passes match
   5. Commit everything together: .til + C + bootstrap

** Changing .til and C files together

   When .til and C code must change together (struct layout changes,
   builder codegen changes, or any case where the old bootstrap is
   incompatible with the new C code):

   Correct workflow:
   1. Change .til files AND C files together
   2. ~make til_core~ — regenerates bootstrap using the existing
      bin/til_bootstrap from the previous build
   3. ~make test~ — both passes now use the same regenerated bootstrap

   Do NOT run ~make clean~ in this workflow. ~make clean~ destroys
   the binary needed by ~make til_core~. ~make test~ alone is
   sufficient — ~make~ detects changed deps and rebuilds.

   If you accidentally ran ~make clean~ and lost the binary, use
   ~make rescue~ to recover (see Recovery targets).

** Removing bridge functions or changing signatures

   When a function is removed from src/c/ that the old bootstrap
   still calls, the first-pass link will fail with undefined references.
   Do NOT fix this by editing bootstrap/ — the fix is always in the
   .til or src/c/ sources.

   Two-step approach:
   1. Change .til source only (e.g. rename ~til_foo~ to ~foo~). Build.
      First pass uses old bootstrap (calls ~til_foo~, bridge provides it).
      ~til_core~ regenerates bootstrap with ~foo~. Second pass uses new
      bootstrap. Commit.
   2. Remove bridge function (now dead code). Build. Both passes use
      ~foo~ directly. Commit.

   If the function doesn't exist in the underlying C module either
   (truly new or truly removed), the .til source must be restructured
   so the generated code doesn't need it.

** Recovery targets

   - ~make revert_boot~ — restore bootstrap/ from last commit
   - ~make rescue~ — build compiler from last commit's sources

   ~make rescue~ is needed when you accidentally ran ~make clean~
   before ~make til_core~ and lost the binary. It builds a compiler
   from the last commit's sources (both bootstrap/ and src/c/).

   Recovery workflow:
   1. ~make rescue~ → builds ~tmp/rescue/til~ from last commit's sources
   2. ~make til_core TIL=tmp/rescue/til~ → regenerates bootstrap
   3. ~make test~

** When to use ~make clean~

   ~make clean~ is rarely needed. Prefer ~make test~ for normal
   development — it's faster and preserves the binary for ~til_core~.

   Use ~make clean~ only when:
   - Switching branches (stale object files)
   - Wanting a fully fresh build to verify from scratch
   - After ~make test~ passes and before committing, to verify
     that ~make test~ works from a clean state (optional)

   NEVER run ~make clean~ before ~make til_core~. It destroys the
   binary that ~til_core~ needs.

** Rules

   - NEVER edit bootstrap/ files. If the build fails with errors
     in bootstrap/, fix the .til sources or src/c/ instead.
   - Always commit .til changes + regenerated bootstrap together.
   - The committed bootstrap must be compatible with committed C source.
   - ~make test~ from a fresh clone must always work.
   - Prefer ~make test~ over ~make clean && make test~ during development.

** Limitations while C modules remain (#39)

   - til.til links C modules via link_c() and calls them through FFI
     bridge functions (til_bridge.c). As #39 replaces C modules, the
     link_c() list shrinks.
   - #77 tracks switching til.til from link_c to import for modules
     already converted (ast.til done, parser.til next). Blocked on
     builder emitting imported function bodies in per-module .c files.
   - precomp depends on interpreter — they form a unit and will be
     converted together (or last). The C interpreter can remain as
     a fallback without shame.
