* Long-term reflections

Where the truthful, long-term answer to a design call differs from
what we ship today, log it here.  Short-term workarounds left in the
tree get a paragraph so the debt is visible to future readers.  This
file is for "what we would build if we cared only about getting it
right for the long run", not for sprint-sized fixes.

** Builder stack-local Str delete special-case

Tracked as [[../issues/open/209.org][#209]].

src/self/builder.til ~1413-1437 force-emits
~Str_delete(&name, &(Bool){0})~ for stack-local Str deletes,
overriding whatever bool the typer put in the AST.  It exists because
the typer commits to the cleanup bool at type-check time, but the
builder later chooses to stack-allocate the binding (shallow-return
funcs like ~str_view~, LiteralStr RHS, etc.).  ~call_free=true~ on a
stack pointer is UB, so the builder rewrites the call at emission.

The post-precomp fold case (FCall RHS folded to LiteralStr) is now
handled in the scavenger (~fix_stale_litstr_in_body~): the AST gets
re-aligned to ~Shallow~ + ~bool=false~ so DCE can act on it and the
emission stays correct without help.  The builder special-case still
covers the shallow-return-FCall case (e.g. ~str_view~), because no
post-typer pass yet rewrites those.

Long-term answer: make the AST the single source of truth for memory
semantics.  Move the stack-vs-heap allocation decision out of the
builder and into the typer (or a dedicated post-typer pass), record
it on the Decl, and derive the cleanup bool from that record.  The
builder then becomes a pure translator -- no special-cases, no hidden
invariants, no cleanup deltas for future passes to chase.

The cost is one design pass plus an audit across existing codegen
optimizations.  The payoff compounds: every future optimization that
stack-allocates something would otherwise grow another special-case
in the builder; doing it right once reduces surface area permanently
and makes the AST trustworthy to linters, alternative backends, and
future contributors who shouldn't have to reverse-engineer the
typer/builder disagreement.

** Interpreter read_field returns dangling StructInstance.struct_name

~src/self/interpreter.til:read_field~ for an inline-struct field does:

  mut ftype := dd.explicit_type
  ...
  return make_struct_value(ftype, sdef3, ptr, true)

Then ~make_struct_value~ stores ~ftype~ into the returned
~StructInstance.struct_name~ field, which is declared
~ref mut struct_name : Str = NULL~ -- a borrow.  But ~ftype~ is a
local Str, cloned from ~dd.explicit_type~ via the typer's auto-clone
for Ident/FieldAccess Struct-typed RHSes.  When ~read_field~ returns,
~ftype~ goes out of scope and its destructor frees its ~c_str~ buffer
-- the returned ~StructInstance.struct_name~ now points at freed
memory.

Today's auto-gen ~Line_clone~ never trips this because it wraps each
field in ~Point.clone(self.start)~, producing an owning ~Value.Struct~
that the caller copies into the new ~Line~'s start slot immediately,
before ~ftype~ is freed.  Try to skip the ~.clone()~ wrap (recursive
POD struct clone-skip in ~initer.til:is_pod_type_for_clone~) and the
borrowed ~Value.Struct~ flows through ~write_field~'s deep-clone path
(~val.clone()~ -> ~StructInstance.clone~ -> ~Str_eq~ on the dangling
~struct_name~), which is the use-after-free ASAN catches in
~test/structs.til:test_clone~ once recursive POD is enabled.

Compiled path is fine -- C-level struct copy doesn't care about the
~struct_name~ pointer's validity at that moment.

Long-term answer: ~StructInstance~ should OWN its ~struct_name~ (or
the interpreter should pull the name from a longer-lived source like
the ~StructDef~ binding's name, not from per-field declarations).
Either fix removes the lifetime trap; the second is cheaper because
struct names are interned in the scope's bindings already.  Closing
this unblocks recursive POD struct clone-skip (~Line_clone~ becomes a
single C compound-literal expression), which feeds back into the
builder's POD-ctor optimisation chain.

** Per-literal malloc in dyn_type_to_str returns

Tracked as [[../issues/open/210.org][#210]].

~emit_dyn_type_to_str_body~ in src/self/builder.til emits ~17 malloc'd
Str wrappers (one per Type variant) for the literal-name returns, even
though the literal data is static.  The wrapper can't be made static
without either reworking the ~swap~-based ~mut Str~ lowering (so
~Str.delete~ can skip free for ~cap == CAP_LIT~) or splitting the
function's heterogeneous return (static literals vs owned ~Str_clone~
type names).  See #210 for the full analysis; shares the ~mut Str~/
~swap~ machinery with #209.
