** Known issues

*** Shallow Value passthrough in widen_numeric
C code does bitwise copy (*r = *v) for unchanged Values. til's Value.clone()
deep-clones, which changes ownership semantics for struct/enum Values.
Options:
1. Add a shallow_copy/bitwise_copy primitive to til
2. Restructure callers so widen_numeric only receives numeric Values
3. Keep widen_numeric in C permanently

*** Manual struct allocation (scope_new, scope_set_*)
til cannot malloc(sizeof(Scope)) and ref-cast the result, or build
Binding structs and memcpy them into Map slots. Need:
1. Add a way to construct structs into malloc'd buffers
2. Add Map helpers that accept struct constructors directly
3. Keep scope functions in C permanently

*** StructInstance.data byte offset access
str_view, make_str_value, make_str_value_own, read_field, write_field all
manipulate StructInstance.data with hardcoded sizeof offsets. These are
platform-dependent (sizeof(U8*) = 8 on 64-bit). Options:
1. Add ext_func helpers: str_data_ptr(inst), str_count(inst), etc.
2. Use Str struct constructor directly (needs the cached_str_def global)
3. Keep these functions in C and translate higher-level functions that call them

*** Function pointer dispatch (dispatch_init, ext_dispatch_builtin)
dispatch_init stores C function pointers via REG macro. til cannot take
the address of a function. ext_dispatch_builtin retrieves and calls
function pointers from a Map. Options:
1. Generate a giant if/else chain in til (like dyn_call dispatch in builder)
2. Keep dispatch_init and ext_dispatch_builtin in C permanently
3. Add function-pointer-as-Dynamic support to til
