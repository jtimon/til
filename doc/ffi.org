* FFI Bindings

How til talks to C, and the conventions that keep the boundary safe.

** Why bindings need a human reviewer

The til-side qualifiers that govern call semantics --
~ext_proc~ vs ~ext_func~, ~own~ vs ~ref~ vs ~shallow~ on args,
and ~own~ vs ~ref~ vs ~shallow~ on returns -- carry information
that the C header simply does not encode.  C has no syntax for:

- *purity* / side effects.  ~int GetScreenWidth(void)~ and
  ~int CRC32(const char *)~ look identical to a header parser,
  but the first reads a process-global and the second is a pure
  function of its input.  til's precomp pass folds pure calls at
  compile time; calling ~GetScreenWidth()~ at compile time
  silently returns 0 (no window) and bakes that 0 into the
  generated binary.  Folded vs not-folded is a til-side
  decision: ~ext_func~ means "pure, fold freely",
  ~ext_proc~ means "side effects, never fold".
- *pointer ownership on parameters*.  Does ~Foo *foo~ as an arg
  mean "callee borrows this pointer" or "callee takes ownership
  and will free it"?  Same C signature, opposite til-side
  contract.
- *pointer ownership on returns*.  Does the ~char *~ returned by
  ~GetClipboardText()~ point at memory the caller must free, or
  at raylib-internal scratch that mustn't be freed by anyone but
  raylib?

So the auto-generator (~src/self/binder.til~, run via
~make update_c_libs~) produces a starting point: structs spelled
correctly, primitive args mapped through, but every function
defaults to ~ext_func~ with ~shallow~ everywhere it could
guess.  A human then walks the output and tags the cases the
generator could not see.

** The audit checklist

When regenerating ~vendor/bindings/*.til~, scan each ~ext_func~
in the output and ask:

1. *Does the C function read or modify global state?*  Window
   metrics, input queues, time, allocator state, file system,
   GPU, audio device, PRNG seed -- anything that depends on or
   touches the world outside the call's arguments.  If yes:
   change ~ext_func~ to ~ext_proc~.  Return type stays
   (~ext_proc~ may return a value -- the proc/func distinction
   is purity, not arity).

2. *Does the C function take a pointer arg that it stores or
   frees?*  By default the binder emits ~shallow~ for primitives
   passed by value.  For a single-level pointer arg it follows the
   C ~const~ qualifier: a ~const T *p~ (read-only pointee) maps to
   the borrowed default ~p : T~, while a non-const ~T *p~ maps to
   ~mut p : T~ (a mutable borrow -- C args are mutable unless
   ~const~, til args are immutable unless ~mut~, so the generator
   bridges the two).  If the C side takes ownership of the buffer,
   change the arg to ~own~ -- til will then transfer the heap
   allocation across the call and not try to free it on this side.

3. *Does the C function return a pointer the caller does not
   own?*  raylib's ~GetMonitorName~, ~GetClipboardText~,
   ~GetWorkingDirectory~ return internal static buffers; the
   matching binding return type should be ~returns ref I8~ (not
   ~returns I8~, which would mean "caller owns and must free").
   The binder emits ~ref~ for raw-pointer returns by default,
   which is the safe choice -- audit the few cases where the C
   call actually hands ownership over.

If any of (1)-(3) flips, also walk the matching wrapper in
~src/modes/gui.til~ (or wherever the user-facing surface lives)
and propagate the same qualifier change.  A ~func~ wrapper
around an ~ext_proc~ has to itself become a ~proc~, or precomp
will still try to fold the wrapper.  A ~ref~-returning ext call
threaded through a ~returns Str~ wrapper would leak the
caller's expectation that the wrapper owns the buffer.

** Quick reference

Mapping from "what the C function does" to the right binding
shape.  See [[file:qualifiers.org][doc/qualifiers.org]] for the
qualifier semantics themselves; this table is just the FFI
layer.

| C behaviour                                | til binding form                                            |
|--------------------------------------------+-------------------------------------------------------------|
| Pure function of args, returns a value     | ~ext_func(args) returns shallow T {}~                       |
| Reads/mutates global state, returns void   | ~ext_proc(args) {}~                                         |
| Reads/mutates global state, returns value  | ~ext_proc(args) returns shallow T {}~                       |
| Takes ownership of a heap pointer arg      | ~ext_proc(own buf : T)~ or ~ext_func(own buf : T) ...~      |
| Borrows a ~const T *~ pointer arg (read-only)| ~ext_*(buf : T)~ (default; ~ref~ keyword is redundant)    |
| Mutates a non-const ~T *~ pointer arg       | ~ext_*(mut buf : T)~ (the generator's default for non-const)|
| Returns an internal pointer (do not free)  | ~ext_*(...) returns ref T {}~                               |
| Returns a heap pointer the caller owns     | ~ext_*(...) returns T {}~ (default; ~own~ keyword redundant) |

For passing scalars and small structs by value (the normal C
ABI shape), use ~shallow~ on the arg and ~returns shallow T~ on
the return -- those map to plain ~T arg~ / ~T return~ in the
generated C.

** ext_struct vs struct: the field-layout contract

~ext_struct~ is not just "a struct whose methods live in C".  Its
reason to exist is field layout:

- ~struct~ reserves til's right to reorder fields (for packing,
  alignment, or any future layout optimization).  Source order is
  not a promise; two structs with the same fields in a different
  order may compile to the same layout.
- ~ext_struct~ guarantees declaration order == memory order.  The
  fields are laid out exactly as written, because the type has to
  match a C struct's ABI for FFI.  This holds even though the
  reordering optimization for ~struct~ is not implemented yet: it is
  a contract about what each form is *allowed* to do, not a snapshot
  of today's codegen.

Consequence: any type that crosses the C boundary by value (or whose
address is handed to C and read back field-by-field) must be an
~ext_struct~, never a plain ~struct~.  You cannot express "match this
exact C layout" with a form that is permitted to reorder.

~ext_struct~ takes an optional C struct tag for the cases where the C
library names the aggregate twice (the ~struct foo~ tag vs the
~typedef~ name):

- ~ext_struct {}~          -- anonymous (no C struct tag)
- ~ext_struct("foo") {}~   -- emits ~struct foo~

There is no ~ext_struct("")~ form: an empty tag is the same as no
tag, so the parser rejects it and points you at bare ~ext_struct {}~.

** noop_proc: forcing precomp to skip a wrapper

Some user-level wrappers depend on runtime state but cannot
inherit a ~proc~ tag from their callees -- e.g. the wrapper
delegates to an ~ext_func~ that genuinely is pure on its
arguments, but the wrapper itself adds context (reads a global,
queries the current frame, etc) that precomp must not evaluate
at compile time.  Drop a call to ~noop_proc~ anywhere in the
function body:

#+begin_src til
import("core/sys")  // for noop_proc

frame_relative_x := func(world_x: F32) returns F32 {
    noop_proc()  // mark this func impure; never fold
    return world_x.sub(camera_offset.x)  // camera_offset is a global
}
#+end_src

~noop_proc~ is a one-liner ~void noop_proc(void) {}~ in
~src/c/ext.c~, declared as ~ext_proc() {}~ in
~src/core/sys.til~ and listed in
~is_side_effecting_name~ inside ~src/self/precomp.til~.  The
call has zero runtime cost (the C body is empty) and exists
purely as a flag for the constant folder.

This is the escape hatch for "the binding is correct but the
*wrapper* still depends on state."  When you find yourself
reaching for it, double-check that the underlying ~ext_func~
really is pure -- often the right fix is to flip *it* to
~ext_proc~ instead.

** Regenerating the bindings

~make update_c_libs~ runs ~bin/til bindings~ against
~vendor/raylib/src/raylib.h~, ~vendor/raylib/src/rcamera.h~,
~vendor/tinyfiledialogs/tinyfiledialogs.h~,
~vendor/nng/include/nng/nng.h~, and libffi's
~ffi.h~, and writes the output to
~vendor/bindings/{raylib,tinyfd,nng,libffi}.til~.

*This destroys any hand-edits.*  Re-do the audit (see above)
before committing the regenerated files.

nng has four hand-edits the generator cannot produce, all of
which must be re-applied after regeneration:

0. *Every binding is ext_proc.*  The binder defaults to ext_func,
   but nng has no pure functions -- each call mutates or reads live
   socket/pipe/dialer/listener/ctx/aio/msg state, does I/O,
   allocates, or returns an internal pointer.  Folding any of them
   in precomp (opening sockets at compile time, or baking a
   compiler-process pointer into the output) is wrong, so flip the
   whole file: ~sed -i 's/:= ext_func(/:= ext_proc(/g'~.

1. *~nng_logger~ alias.*  The binder leaves the ~nng_logger~
   function-pointer typedef undefined; add ~nng_logger := Dynamic~
   before the logging functions.
2. *Non-shallow logger param.*  ~nng_log_set_logger~ takes a
   ~nng_logger~ (now ~Dynamic~), so its parameter must NOT be
   ~shallow~ (shallow requires a scalar/struct type).
3. *Protocol constructors.*  ~nng_pair0_open~, ~nng_pub0_open~,
   ~nng_sub0_open~, etc. live in ~nng/protocol/*/*.h~, not in
   ~nng.h~, and reference ~NNG_DECL~/~nng_socket~ without including
   ~nng.h~ -- so they do not preprocess standalone.  Re-append the
   hand-written "Protocol constructors" block (each is
   ~ext_proc(mut sock : nng_socket) returns shallow I32 {}~, taking
   the socket out-parameter by address -- ~mut~ because the call
   writes the socket through that non-const pointer).

Useful workflow:

1. Save the current ~vendor/bindings/raylib.til~ somewhere
   outside the repo as a reference for the previous audit.
2. Run ~make update_c_libs~.
3. Diff against the saved file; transfer every ~ext_proc~
   tag, every ~own~/~ref~ override, and every comment that
   explained a non-obvious choice.  New C-side additions
   (new raylib functions) get fresh audits.
4. Run ~make clean && make test~.  If a wrapper in
   ~src/modes/gui.til~ used a renamed-or-deleted C symbol,
   the build will tell you.
5. Verify the games (~bin/shootil~, ~bin/chess~, ~bin/go~) still
   render their HUDs and respond to mouse / keyboard.  The
   regressions that motivated this whole layer
   (issue #208 / commit ~3897ed9~) are interaction-shaped, so
   the test suite alone won't catch them.

Routine builds (~make~, ~make test~, ~make two_pass~) never
touch ~vendor/bindings/~ -- they consume the committed files.

** Ownership audit (raylib), 2026-05

A pass over every pointer-returning and free-style function in
~vendor/bindings/raylib.til~.  Conclusion: the binder's defaults are
correct for raylib's *manual* memory model, where the user pairs
each ~LoadX~ with an explicit ~UnloadX~ / ~MemFree~ and til stays
out of the lifecycle.  Specifically:

- *Pointer returns are all ~returns ref T~.*  ~GetClipboardText~,
  ~GetMonitorName~, ~Get*Name~, the whole ~Text*~ family, ...
  return raylib-internal/static buffers the caller must NOT free.
  The ~Load*~ family (~LoadFileData~, ~LoadImageColors~,
  ~LoadFontData~, ~LoadCodepoints~, ~LoadWaveSamples~,
  ~LoadMaterials~, ~LoadModelAnimations~, ~CompressData~, the
  ~Compute*~ hashes, ...) return heap the caller owns but frees via
  the matching ~UnloadX~ / ~MemFree~, NOT via ~free()~.  Either way
  til must not auto-free, so ~ref~ (til does not own) is correct.
  Promoting these to owned (~returns T~) would make til emit a
  generic ~free()~ at scope end -- a double-free against the user's
  ~UnloadX~, and wrong anyway for compound unloads
  (~UnloadModelAnimations~ frees nested arrays; ~free()~ wouldn't).

- *~Unload*~ / ~MemFree~ args are borrowed (or ~shallow~ for
  by-value structs).*  This is the correct mirror of the ~ref~
  returns: the value isn't til-owned, so there's no ownership to
  transfer with ~own~.  By-value resource structs (~Texture~,
  ~Image~, ~Sound~, ~Font~, ~Model~, ~Wave~, ~Music~, ...) pass
  ~shallow~; raylib frees their internal GPU/heap state.  No til
  pointer ownership is involved.

- *No arg needs ~own~.*  raylib copies or borrows every pointer arg
  (e.g. ~Load*FromMemory~ copies ~fileData~; ~UpdateTexture~ copies
  ~pixels~); none stores or frees a caller buffer in a way that maps
  to til's ownership-transfer.

*** Watch-list: ambiguous ~returns Dynamic~

A handful of functions return raw ~void *~ / ~void **~ as
~Dynamic~, which is til's *owned* raw-pointer type.  None has a til
caller today (no ~src/modes/gui.til~ wrapper), so there is no live
bug, but a wrapper author MUST pick the right shape per function
rather than letting til treat the result as owned:

- ~MemAlloc~, ~MemRealloc~: genuinely owned (caller frees via
  ~MemFree~).  ~Dynamic~ is right.
- ~GetWindowHandle~: returns the platform/GLFW window handle --
  *borrowed*, must never be freed.  A wrapper should treat it as a
  borrowed pointer, not an owned ~Dynamic~.
- ~TextSplit~: returns ~char **~ into raylib's internal static
  buffer -- *borrowed*, do not free.
- ~LoadTextLines~: allocates; freed via ~UnloadTextLines~ (compound,
  not a plain ~free()~), so a wrapper must route cleanup through
  ~UnloadTextLines~, not til's owned-Dynamic auto-free.

tinyfd (~tinyfd_*Dialog~ -> ~returns ref I8~ into a static buffer)
and libffi (~ffi_closure_alloc~ -> ~Dynamic~, freed via
~ffi_closure_free~) follow the same rules and were spot-checked
clean.
